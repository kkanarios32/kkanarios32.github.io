1. Why Custom Kernels?

  • March 23, 2026
Example. In attention kernel, we perform \(QK^T\) in VMEM then copy back to HBM, then copy back to VMEM and do the softmax. Instead we want to fuse this into one kernel.
  • Logit & Max fusion:
    • 34.45% flop utilization
      • Use MXU to get over 350TFLOP/s.
    • 18.74% HBM bandwidth utilization.
    • 3.55% On-chip read utilization
  • Softmax denom:
    • 0.32% flop utilization.
    • 0.0% HBM bandwidth utilization.
      • Keeps 1024x1024 logit matrix resident in VMEM.
    • 22.61% On-chip read utilization
      • Uses VPU to compute softmax which consumes data from VMEM at 6.6 TB/s.
  • Attention output:
    • 35.82% flop utilization.
    • 19.49% HBM bandwidth utilization.
    • 7.39% On-chip read utilization.

HBM Bottleneck: The default XLA attention does not fuse the softmax with the logit computation, requiring the entire logit matrix to be written back to HBM.

Vector Unit Limitations: MXU can be left idle while VPU is performing copmlex exponential and sum operations.

VMEM Residency and Loop Control: Pallas allows you to control data movement, so that we can explicitly enforce online softmax rather than writing back to HBM.

Temporal Control via LLO: Pallas also allows control over the temporal flow, preventing the compiler from reordering your manual optimizations.

2. Pallas Fundamentals

  • March 23, 2026
 Ignore the HLO pass.

Writing a kernel:

  • Inside the kernel, data has already been moved from HBM to VMEM.
  • If data exceeds VMEM, we need to tile with BlockSpec.
  • Also need to pipeline i.e. while doing computation on a given block, we need to be moving the next block from HBM to VMEM.
  • By default, pipelining is done for you, but there are other APIs to control this(?)
    • For-Loop scheduler.
    • Asynchronous overlap.
    • Buffer reuse. Uses same VMEM buffer across iterations.

Two-level Tiling:

  • Macro-tiling: this is done by Pallas / user. You define the larger software blocks i.e. 1024x1024 to hide latency DMA transfers. Pallas manages pipeling.
  • Micro-tiling: this is done by compiler / hardware. Translates larger block into hardware native VREG 8x128.

VPU Pipelining:

  • VLOAD: fetching inputs for chunk (8x128)(N + 1)th from VMEM and move to VRegs
  • VALU: Perform elementwise operations for current (Nth) chunk
  • VSTORE: write completed result for previous (N - 1)th chunk from VRegs back to VMEM.

3. Flash Attention

  • March 23, 2026

Online softmax:

  • Each block does a local softmax calculation.
  • Tracks a shared maximum and adjusts results.

Can Now fuse score calculation, softmax, and value aggregation into single pass.

No longer need to store NxN attention matrix in HBM.

Can break \(Q, K, V\) into small blocks that fit in VMEM.

Kernel available here.

  • Dynamic scaling \((\alpha )\): Computes \(e^{m_{prev} - m_{next}}\) to track change in maximum logit across blocks for numerical stability.
  • Re-scale accumulation: Multiplies by \(\alpha \) to the previously accumulated sums before adding current block output to accurately rescale all past results.
  • Additive updates: partial results (current softmax and outputs) are added to running totals via fast multiply and addition ops.
  • Delayed normalization: only divide by normalization once at the very end of the loop because division is slow.

Block-size tuning: to maximize compute-to-load ratio need biggest block sizes that fit into VMEM.

4. Splash Attention

  • March 23, 2026

With certain masks (i.e. causal) can skip alot of computation for parts of the attention matrix that are being masked out.

  • Finer blocks -> can skip more but then have poorer utilization.

Pre-processing Mask Info:

  • Sparsity pruning: Divide mask into blocks and discard all inactive blocks (all zeros).
  • Data compression: see how many unique mask patterns are needed to describe the remaining active blocks.
  • Execution map and pre-fetching: (active_rows, mask_next) generated to map the computation order and tell the MXU which mask pattern to prefetch for zero-latency application.

Metadata structures:

  • Block filtering (block_mask): categorizes QK blocks as zero, non-zero, partial.
  • Coordinate mapping (active_ros/cols): provides the hardware with (Q, KV) grid indices, ensuring efficient loading of active blocks from HBM.
  • Mask pipelining (mask_next): tells the hardware which mask pattern to prefetch.
  • SMEM-resident metadata: computed once per mask and stored in SMEM. Enable high-speed access to sparse execution map without need to re-transfer from HBM.

Pallas Call

  • Grid definition: defined by (num_q_heads, grid_size). Grid size is set dynamically using the sparese meta data fwd_mask_info.num_active_blocks[0].
  • SMEM Inputs: num_scalar_prefetch instructs hardware to prefetch critical scalar/metadata inputs into SMEM before thread begins computing.
  • VMEM Inputs: (Q, K, V) that must be moved from HBM to VMEM for kernel to execute.
