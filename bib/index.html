      This covers the math behind the online softmax algorithm required by Flash Attention (v1).

      Safe softmax

      • May 20, 2026
      • Kellen Kanarios

      The first thing we must consider is numerical precision. Namely, the original softmax operation is given by

      \[\mathrm {softmax}(\mathbf {x}) = \frac {\exp (\mathbf {x})}{\sum _{i=1}^n \exp (\mathbf {x}_i)}\]

      However, when operating with low precision data types (i.e. float16) the maximum value is \(65536\). Therefore, with \(\exp \) we can easily get numerically overflow.

      To get around this, the online softmax algorithm adheres to the safe softmax. For safe softmax, we simply multiply the numerator and denominator by \(\exp (-m)\), where \(m = \max \{x_{1}, \ldots , x_n\}\). This gives

      \[\mathrm {softmax}(\mathbf {x}) = \frac {\exp (\mathbf {x} - m)}{\sum _{i=1}^n \exp (\mathbf {x}_i - m)}\]

      Since \(x_i - m \leq 0\), this ensures that each \(\exp (x_i - m)\) is bounded in a safe range.

      Problem. Note that computing the \(\max \) over \(\mathbf {x}\) means that we must have all of the logits in a row of \(QK^T\) materialized in SMEM, or first loop over all of them to compute the max before computing the normalization constant.

      Fusing the max and normalization

      • May 20, 2026
      • Kellen Kanarios

      The online softmax algorithm fuses the max reduction into the normalization pass, so a single sweep over the row suffices to compute the normalization with the max. The key difficulty is that the denominator depends on \(n\) for each \(i\) i.e.

      \[d = \sum _{i=1}^{n} \exp (x_i - m_n)\]

      Therefore, we must find a recursive relationship such that \(d_{i+1} = f(d_i, m_i, x_{i+1})\) for some function \(f\). This would then allow us to locally compute \(d_{i+1}\) with only the local information of current \(x_{i+1}\) and the two running statistics \(d_i\) and \(m_i\). This is exactly the kind of algorithm that is conducive to memory tiling.

      Question. What is \(f(d_i, m_i, x_{i+1})\)?
      Answer. Consider \(d_{i+1} = d_i \cdot \exp (m_{i} - m_{i + 1}) + \exp (x_{i + 1} - m_{i + 1})\).
      Proof. This can be seen by directly rewriting \(d_{i + 1}\). \[ \begin {align*} d_{i + 1} &= \sum _{j = 1}^{i+1} \exp (x_j - m_{i + 1}) \\ &= \left (\sum _{j = 1}^{i} \exp (x_j - m_{i+1})\right ) + \exp (x_{i+1} - m_{i+1}) \\ &= \left (\sum _{j = 1}^{i} \exp (x_j - m_{i})\right )\exp (m_{i} - m_{i + 1}) + \exp (x_{i+1} - m_{i+1}) \\ &= d_{i} \exp (m_{i} - m_{i + 1}) + \exp (x_{i+1} - m_{i+1}) \end {align*} \]

      The running denominator \(d\) is rescaled by \(e^{m - m_{\text {new}}}\) each time a new maximum is found, keeping it consistent with the safe-softmax shift.

      Remark. Unfortunately, we cannot get around the second pass to compute the normalized output for \(\mathbf {p}\). However, this is not the case in Flash Attention (v1).

      Flash Attention (Fwd) [OGCS0]

      • May 27, 2026
      • Kellen Kanarios

      Unlike in the online softmax, we are not interested in the outputted probability vector \(\mathbf {p}\). Instead we are interested in the entire output matrix \(O\) given by

      \[O = \mathrm {softmax}(QK^T) V\]

      For more details see self-attention.

      Notation. In correspondence with the notation in the online softmax, we define the following: \[x_{i,j} = \mathbf {q}_i \cdot \mathbf {k}_j,\] where \(\mathbf {q}_i\) is the \(i\)th row of \(Q\) and \(\mathbf {k}_j\) is the \(j\)th row of \(K\). Similarly, \(\mathbf {o}_i\) is the \(i\)th row of the output matrix \(O\) given by \[\mathbf {o}_i = \sum _{j = 1}^{n} \frac {\exp (x_{i,j} - m_n)}{d_n} \cdot \mathbf {v}_j \]

      Now following the template provided by the online softmax, we must find a recurrence for \(\mathbf {o}_i\). Namely, we will consider the sequence \(\mathbf {o}_{i,k}'\) given by \[\mathbf {o}_{i,k}' = \sum _{j = 1}^{k} \frac {\exp (x_{i,j} - m_k)}{d_k} \cdot \mathbf {v}_j, \] such that \(\mathbf {o}_{i,n}' = \mathbf {o}_n\)

      Claim. We have the following recurrence for \(\mathbf {o}_{i,k}'\): \[ \mathbf {o}_{i,k}' = \mathbf {o}_{i, k-1} \cdot \frac {d_{k - 1} \cdot \exp (m_{k-1} - m_{k})}{d_k} + \frac {\exp (x_{i,j} - m_k)}{d_k} \cdot \mathbf {v}_k \]
      Proof. We will proceed by directly rewriting the recurrence for \(\mathbf {o}_{i,k}'\) as \[ \begin {align*} \mathbf {o}_{i,k}' &= \sum _{j = 1}^{k} \frac {\exp (x_{i,j} - m_k)}{d_k} \cdot \mathbf {v}_j \\ &= \left (\sum _{j = 1}^{k - 1} \frac {\exp (x_{i,j} - m_{k-1})}{d_{k-1}} \cdot \mathbf {v}_j\right ) \frac {d_{k - 1} \cdot \exp (m_{k-1} - m_{k})}{d_k} + \frac {\exp (x_{i,k} - m_k)}{d_k} \cdot \mathbf {v}_k \\ &= \mathbf {o}_{i, k-1} \cdot \frac {d_{k - 1} \cdot \exp (m_{k-1} - m_{k})}{d_k} + \frac {\exp (x_{i,k} - m_k)}{d_k} \cdot \mathbf {v}_k .\end {align*} \]

      Flash Attention (Bwd) [OGCRZ]

      • May 27, 2026
      • Kellen Kanarios

      To understand the innovation of the backward pass in flash attention, we must first find all of the gradients we must compute.

      Notation. Following the notation in the original paper, we have the following matrices \[\mathbf {S} = \mathbf {Q} \mathbf {K}^T, \quad \mathbf {P} = \mathrm {softmax}(\mathbf {S}), \quad \mathbf {O} = \mathbf {P} \mathbf {V}\] Now suppose their is some scalar loss function \(\phi \). Then we define \[\frac {\partial \phi }{\partial \mathbf {O}}, \frac {\partial \phi }{\partial \mathbf {K}}, \frac {\partial \phi }{\partial \mathbf {V}} = d\mathbf {O}, d\mathbf {K}, d\mathbf {V}\]

      We will compute each gradient individually.

      1. \(d\mathbf {P}\): this is just a straightforward application of the chain rule. In particular, \(\mathbf {O} = \mathbf {P} \mathbf {V}\) so that \[ d\mathbf {P} = \frac {\partial \phi (\mathbf {O})}{\partial \mathbf {P}} = \frac {\partial \phi (\mathbf {O})}{\partial \mathbf {O}} \frac {\partial \mathbf {O}}{\partial \mathbf {P}} = d\mathbf {O} \mathbf {V}^T \]

        To sanity check, let us look at the shapes. \(d\mathbf {O}\) is the partial with respect to the outputs \(O\), so we need an entry for each output and it is \(N \times d\). \(V^T\) is \(d \times N\), so that the result \(d\mathbf {P}\) is \(N \times N\) as desired.

        Furthermore, we can see this by writing \(\mathbf {O}_{ik} = \sum _{j} \mathbf {P}_{ij} \mathbf {V}_{jk}\). Then

        \[ \begin {align*} d\mathbf {P}_{ij} &= \sum _{i} \frac {\partial \phi }{\partial \mathbf {O}_{ik}} \cdot \frac {\partial \mathbf {O}_{ik}}{\partial \mathbf {P}_{ij}} \\ &= \sum _{i} d\mathbf {O}_{ik} \mathbf {V}_{jk} .\end {align*} \]
      2. \(d\mathbf {V}\): this is just the same as \(d\mathbf {P}\). In particular, \(\mathbf {O} = \mathbf {P} \mathbf {V}\) so that \[ d\mathbf {V} = \frac {\partial \phi (\mathbf {O})}{\partial \mathbf {V}} = \frac {\partial \phi (\mathbf {O})}{\partial \mathbf {O}} \frac {\partial \mathbf {O}}{\partial \mathbf {V}} = \mathbf {P}^T d\mathbf {O} \]

        Furthermore, we can see this by writing \(\mathbf {O}_{ik} = \sum _{j} \mathbf {P}_{ij} \mathbf {V}_{jk}\). Then

        \[ \begin {align*} d\mathbf {V}_{jk} &= \sum _{i} \frac {\partial \phi }{\partial \mathbf {O}_{ik}} \cdot \frac {\partial \mathbf {O}_{ik}}{\partial \mathbf {V}_{jk}} \\ &= \sum _{i} d\mathbf {O}_{ik} \mathbf {P}_{ij} \\ &= \sum _{i} \frac {\exp \left (\mathbf {Q}_{i} \cdot \mathbf {K}_{j}\right )}{\mathbf {L}_{i}} d\mathbf {O}_{ik} .\end {align*} \]
      3. \(d\mathbf {S}\): now we apply chain rule with \(d\mathbf {P}\). However, we must take the gradient through the softmax. Exiting matrix notation hell, if we consider the scalar case. Let \(\mathbf {s}\) and \(\mathbf {p}\) just be rows. Then using the scalar softmax derivation \[ \begin {align*} d\mathbf {s}_j &= \sum _{i} d\mathbf {p}_i \cdot \frac {\partial \mathbf {p}_i}{\partial \mathbf {s}_j} \\ &= \sum _{i} d\mathbf {p}_i \cdot (\delta _{ij}\mathbf {p}_i - \mathbf {p}_{ij}) \\ &= d \mathbf {p}_{j} \mathbf {p}_{j} - \mathbf {p}_{j}\sum _{i} d\mathbf {p}_{i} \mathbf {p}_{i} .\end {align*} \] Now if we rewrite this as the row vector, we get \[ \begin {align*} d\mathbf {S}_{k} &= \mathbf {P}_{k} \odot (d\mathbf {P}_{k} - \mathbf {P}_k \cdot d\mathbf {P}_k) .\end {align*} \] To get back to the full matrix, define \(\mathbf {D}_i = \mathbf {P}_i \cdot d\mathbf {P}_i\) i.e. a vector whose \(i\)th entry is \(\mathbf {P}_i \cdot d\mathbf {P}_i\). Then we have that \[ \begin {align*} d\mathbf {S} &= \mathbf {P} \odot (d\mathbf {P} - \mathbf {D}) .\end {align*} \]
      4. \(d\mathbf {Q}\): To get \(d\mathbf {Q}\), we just apply the chain rule to \(d\mathbf {S}\). Since \(\mathbf {S} = \mathbf {Q} \mathbf {K}^T\), \[ \begin {align*} d\mathbf {Q}_{i} &= \sum _{j} d\mathbf {S}_{ij} \mathbf {K}_{j} \\ &= \sum _{j} \mathbf {P}_{ij} \cdot (d\mathbf {P}_{ij} - \mathbf {D}_i) \mathbf {K}_{j} .\end {align*} \]
      5. \(d\mathbf {K}\): Same as \(d\mathbf {Q}\) i.e. \[ \begin {align*} d\mathbf {K}_{j} &= \sum _{j} d\mathbf {S}_{ij} \mathbf {Q}_{i} \\ &= \sum _{j} \mathbf {P}_{ij} \cdot (d\mathbf {P}_{ij} - \mathbf {D}_i) \mathbf {Q}_{i} .\end {align*} \]
      Problem. \(\mathbf {P}\) and \(d\mathbf {P}\) are \(N \times N\) matrices. Materializing this is equivalent to materialing the attention matrix in the forward pass.
      💡 Insight. recomputing \(\mathbf {P}_{ij}\) and \(d\mathbf {P}_{ij}\).

      First, we can easily recompute \(\mathbf {P}_{ij}\) with \(\mathbf {Q}\) and \(\mathbf {K}\) (which are both \(N \times d\) and must be resident in memory) via

      \[\mathbf {P}_{ij} = \frac {\exp \left (\mathbf {Q}_{i} \cdot \mathbf {K}_{j}\right )}{\mathbf {L}_{i}}\]

      We can apply the same trick to \(d\mathbf {P}_{ij}\) with \(d\mathbf {O}\) and \(\mathbf {V}\) (both \(N \times d\)). Namely,

      \[d\mathbf {P}_{ij} = \sum _{k} d\mathbf {O}_{ik} \mathbf {V}_{jk} = d\mathbf {O}_{i}^T \mathbf {V}_j\]

      Never need an \(N \times N\) matrix.

      Now we can rewrite \(d\mathbf {Q}\) and \(d\mathbf {K}\) as \[ d\mathbf {Q}_{i} = \sum _{j} \frac {\exp \left (\mathbf {Q}_{i} \cdot \mathbf {K}_{j}\right )}{\mathbf {L}_{i}} \cdot (d\mathbf {O}_{i}^T \mathbf {V}_j - \mathbf {D}_i) \mathbf {K}_{j} \] and \[ d\mathbf {K}_{j} = \sum _{j} \frac {\exp \left (\mathbf {Q}_{i} \cdot \mathbf {K}_{j}\right )}{\mathbf {L}_{i}} \cdot (d\mathbf {O}_{i}^T \mathbf {V}_j - \mathbf {D}_i) \mathbf {Q}_{i} \]

      Problem. \(\mathbf {D}\) requires computing the global dot product \(\mathbf {P}_i \cdot d\mathbf {P}_i\).
      💡 Insight.

      To get around this, we precompute \(\mathbf {D}\) using the \(\mathbf {O}\) values from the forward pass and the current \(d\mathbf {O}\) values. In particular,

      \[ \begin {align*} \mathbf {D}_i &= \mathbf {P}_{i} \cdot d\mathbf {P}_{i} \\ &= \sum _{j} \frac {\exp \left (\mathbf {Q}_{i} \cdot \mathbf {K}_{j}\right )}{\mathbf {L}_{i}} d\mathbf {P}_{ij} \\ &= \sum _{j} \frac {\exp \left (\mathbf {Q}_{i} \cdot \mathbf {K}_{j}\right )}{\mathbf {L}_{i}} d\mathbf {O}_{i}^{T} \mathbf {V}_{j} \\ &= \sum _{j} d\mathbf {O}_{i}^{T} \left (\frac {\exp \left (\mathbf {Q}_{i} \cdot \mathbf {K}_{j}\right )}{\mathbf {L}_{i}} \cdot \mathbf {V}_{j}\right ) \\ &= d\mathbf {O}_{i}^{T} \sum _{j} \frac {\exp \left (\mathbf {Q}_{i} \cdot \mathbf {K}_{j}\right )}{\mathbf {L}_{i}} \cdot \mathbf {V}_{j} \\ &= d\mathbf {O}_{i}^{T} \mathbf {O}_{i} .\end {align*} \] Thus, we can compute \(\mathbf {D}_i\) directly from the vectors \(d\mathbf {O}_i\), \(\mathbf {O}_i\).

      Summary

      • May 27, 2026
      • Kellen Kanarios
      • In the forward pass,
        1. Compute and store \(\mathbf {O}\) values in memory.
        2. Compute and store \(\mathbf {L}\) values in memory
      • Between the forward and backward pass,
        1. Compute \(d\mathbf {O}\). (This is done automatically in JAX)
        2. Use \(\mathbf {O}\) and \(d\mathbf {O}\) to compute \(\mathbf {D}\). \(O(n)\) memory.
      • In the backward pass,
        1. Compute \(d\mathbf {V}_j\) via \(\mathbf {Q}, \mathbf {K}, d\mathbf {O}\). \(O(d)\) memory.
        2. Compute \(d\mathbf {Q}_i\) via \(\mathbf {Q}_i, \mathbf {K}_j, \mathbf {L}_i, \mathbf {D}_i\). \(O(d)\) memory.
        3. Compute \(d\mathbf {K}_j\) via \(\mathbf {Q}_i, \mathbf {K}_j, \mathbf {L}_i, \mathbf {D}_i\). \(O(d)\) memory.

      The first thing we need to do is understand all the things that can go awry.

      First in the valid-invalid protocol example we were given they only have two virtual channels. However, we need 3 channels to handle, request messages, forwarded messages, and response messages.

      Let us briefly understand why this is the case. Detecting a deadlock is equivalent to detecting a cycle in a dependency graph, so it is sufficient to show a cyclic dependency.

      • If we issue a GetS this goes to the directory, giving us an edge N1 -> D.
      • Then the directory issues a Fwd-GetS to the ownder, giving us the chain N1 -> D -> N2
      • The owner then needs to send data to the requester, giving us the chain N1 -> D -> N2 -> N1.
      • Namely, the data must go through the same channel as all of the requests depending on it.

      To avoid this, we partition into 3 channels:

      • Request: GetS, GetM, PutM
      • Forwarded: Fwd-GetS, Fwd-GetM, PutAck, Inv
      • Response: InvAck, Data

       For this note, we will consider a maximization linear program. Namely, an optimization problem of the following form \[ \begin {align*} \max _{\mathbf {x}} \quad &\mathbf {c}^{\top } \mathbf {x} \\ \text {s.t.} \quad &A\mathbf {x} \leq b .\end {align*} \] In the spirit of lagrange multipliers, we can also get a dual formulation. To do so, we add a multiplier \(y_i\) for each constraint i.e. \[y_i \mathbf {a}_i^T \mathbf {x} \leq y_i b_i\] we then consider the summed constraint (which ends up being tight enough), \[\mathbf {y}^T A \mathbf {x} \leq \mathbf {y}^T\mathbf {b}\] To relate this back to our primal problem, we enforce the additional constraint that \[\mathbf {y}^T A = \mathbf {c}^T \iff A \mathbf {y} = \mathbf {c} \]. The reason for this is that for a feasible \(\mathbf {y}\), we have \[\mathbf {y}^T A \mathbf {x} = \mathbf {c}^T \mathbf {x} \leq \mathbf {y}^T \mathbf {b}, \] so that \(\mathbf {y}^T \mathbf {b}\) is an upper bound for our primal objective \(\mathbf {c}^T \mathbf {x} \). Intuitively, to recover \(\mathbf {c}^T \mathbf {x}^* \), we can just minimize over \(\mathbf {y}\), resulting in the following dual program \[ \begin {align*} \min _{\mathbf {y}} \quad &\mathbf {b}^T\mathbf {y} \\ \text {s.t.} \quad &A \mathbf {y} = \mathbf {c} \\ & \mathbf {y} \geq 0 .\end {align*} \]

      Rapidly exploring random trees (RRT) is a very simple, elegant and widely used algorithm for motion planning. The algorithm proceeds as follows:

      Algorithm. For \(k = 1\) to \(K\) do
      1. \(x_{rand} \gets \mathsf {RANDOM\_STATE}()\)
      2. \(x_{near} \gets \mathsf {NEAREST\_NEIGHBOR}(x_{rand}, \mathcal {T})\)
      3. \(u \gets \mathsf {SELECT\_INPUT}(x_{near})\)
      4. \(x_{new} \gets \mathsf {NEW\_STATE}(x_{near}, u, \Delta t)\)
      5. \(\mathcal {T}.add\_vertex(x_{new})\)
      6. \(\mathcal {T}.add\_edge((x_{near}, u, x_{new}))\)
      return \(\mathcal {T}\)
      Remark. Given some dynamics model \(\dot {x} = f(x, u)\), \(\mathsf {NEW\_STATE}\) is some approximate integration, such as Euler integration i.e. \[x_{new} = x_{near} + f(x_{near}, u) \Delta t\] Preferable to use higher-order method like Runge-Kutta.

      Eligibility Traces [DJPO]

      • July 24, 2025
      • Kellen Kanarios

      As I grow more interested in abandoning batched updates and the replay buffer, the importance of elibility traces grows with me. Due to this, I thought it finally time to re-prove / understand what makes these traces so eligible.

      The reference followed is Sutton and Barto (Ch 12.).

      The eligibility trace vector is given by \[ \begin {align*} \mathbf {z}_{-1} &= 0 \\ \mathbf {z}_t &= \gamma \lambda \mathbf {z}_{t - 1} + \nabla \hat {v}(S_t; \mathbf {w}_t) .\end {align*} \] Recall the temporal difference error is given by \[\begin {align*} \mathbf {\delta }_t = r_{t + 1} + \gamma \hat {v}(s_{t + 1}; \mathbf {w}_t) - \hat {v}(s_t; \mathbf {w}_t) .\end {align*}\] For TD(\(\lambda \)), the incremental update is given by \[\begin {align*} \mathbf {w}_{t + 1} = \mathbf {w}_t + \alpha \mathbf {\delta }_{t} \mathbf {z}_t .\end {align*}\]

      We can rewrite the \(\lambda \)-return recursively in terms of reward and itself as follows: \[\begin {align*} G_t^{\lambda } &= (1 - \lambda ) \sum _{n=1}^{\infty } \lambda ^{n-1}G_{t:t+n} \\ G_t^{\lambda } &= (1 - \lambda ) \sum _{n=1}^{\infty } \lambda ^{n-1}(R_{t + 1} + \gamma G_{t+1:t+n}) \\ G_t^{\lambda } &= R_{t + 1} + \gamma (1 - \lambda ) \left [\hat {v}(S_{t + 1}; \mathbf {w}_t) + \sum _{n=2}^{\infty } \lambda ^{n-1} G_{t+1:t+n}\right ] \\ G_t^{\lambda } &= R_{t + 1} + (1 - \lambda )\gamma \hat {v}(S_{t + 1}; \mathbf {w}_t) + \gamma \lambda G_{t + 1}^{\lambda } .\end {align*}\]

      Now we will show that the prediction error of TD(\(\lambda \)) can be re-written as the sum of TD-errors. The prediction error is written as \[PE = G_t^{\lambda } - \hat {v}(S_t, \mathbf {w}_t)\] Rewriting the \(\lambda \)-return as above \[ \begin {align*} PE &= R_{t + 1} + \gamma \hat {v}(S_{t + 1} ; \mathbf {w}_t) + \gamma \lambda G_{t + 1}^{\lambda } - \gamma \lambda \hat {v}(S_{t + 1}, \mathbf {w}_t) - \hat {v}(S_t; \mathbf {w}_t) &= (R_{t + 1} + \gamma \hat {v}(S_{t + 1} ; \mathbf {w}_t) - \hat {v}(S_t, \mathbf {w}_t)) + \gamma \lambda G_{t + 1}^{\lambda } - \gamma \lambda \hat {v}(S_{t + 1}; \mathbf {w}_t) &= \mathbf {\delta }_t + \gamma \lambda G_{t + 1}^{\lambda } - \gamma \lambda \hat {v}(S_{t + 1}; \mathbf {w}_t) .\end {align*} \]

      We now are left with \(\gamma \lambda (G_{t + 1}^{\lambda } - \hat {v}(S_{t + 1}, \mathbf {w}_t))\), where we can proceed in the same manner recursively, such that \[PE = \sum _{t=1}^{\infty } (\gamma \lambda )^{t - 1} \delta _t\] It is easy to see the same thing goes through for the finite time slice \(\lambda \)-return i.e. \[G_{t:t + k}^{\lambda } - \hat {v}(S_t, \mathbf {w}_{t - 1}) = \sum _{i = t}^{t + k} (\gamma \lambda )^{i - t} \delta _i\]

      Going back to the eligibility trace update, if we instead do not apply each elibgibility trace update but apply the sum of all the updates for the same fixed \(\mathbf {w}_t\), then we get that the total update is \[ \begin {align*} \mathbf {w}_{t+1} &= \sum _{t=1}^{T} \alpha \delta _t \mathbf {z}_t \\ &= \sum _{t=1}^{T} \alpha \delta _t \sum _{i=0}^{t} (\lambda \gamma )^{i} \nabla \hat {v}(S_i, \mathbf {w}_t) \\ &= \sum _{t=1}^{T} \alpha (r_t + \gamma \hat {v}(S_{t+1}, \mathbf {w}_t) - \hat {v}(S_t, \mathbf {w}_t)) \underbrace {\sum _{i=0}^{t} (\lambda \gamma )^{i} \nabla \hat {v}(S_i, \mathbf {w}_t)}_{X_t} .\end {align*} \] Then \[\begin {align*} &= \sum _{t=1}^{T} \alpha r_t X_t + \gamma \hat {v}(S_{t+1}, \mathbf {w}_t) X_t - \hat {v}(S_t, \mathbf {w}_t) X_t .\end {align*}\]

      This shows that the backward-view and forward-view are "equivalent". The forward-view in practice makes an update at each step, so it is typically better.

      • \(b\) batch dimension,

      The total FLOPS for attention are roughly \(bnd^2\). This comes from
      1. \(3nd^2\) FLOPS to compute \(Q = XW_Q, K = XW_K, V = XW_V\).
      2. Need to do this \(b\) times for each input in batch.
      and the total memory accesses are roughly \(\underbrace {bnd}_{X} + \underbrace {bhn^2}_{\text {softmax}} + \underbrace {d^2}_{\text {projection}}\). This gives high arithmetic intensity \[O\left (\left (\frac {h}{d} + \frac {1}{bn}\right )^{-1}\right )\]

      This is the follow up to the previous note on the ELBO. Mainly, here we aim to show how we can use the ELBO to find \(\theta ^*\) i.e. \[ \begin {align*} \theta ^* &= \arg \max \mathbb {E}_{x \sim \mathcal {D}} \log p(x; \theta ) \\ &= \arg \max \mathbb {E}_{x \sim \mathcal {D}} \log \int p(x, z; \theta ) \mathrm {d}z .\end {align*} \] Notably, ELBO gives us the lower bound \[\log p(x; \theta ) \geq \mathbb {E}_{z \sim q}\left [\log \frac {p(x, Z; \theta )}{q(z)}\right ] \eqqcolon \mathsf {ELBO}(x, q, \theta )\] for a fixed theta and \(q\).       • July 1, 2025

      The above motivates the expectation step of the EM algorithm. Namely how can we compute the lower bound expectation for the best possible \(q\).

      Recall (or review) from the discussion on ELBO that \(q(z) = p(z | x; \theta )\) maximizes the ELBO for a given \(x\) and \(\theta \). Therefore, we actually want to learn \(q_{\phi }(z, x, \theta ) \approx p(z | x; \theta )\). To do so, we can simply maximize the ELBO (duh). Formally, \[\phi _{t+1} = \argmax _{\phi } \mathbb {E}_{x \sim \mathcal {D}, z \sim q}\left [\log \frac {p(x, Z; \theta _t)}{q_{\phi }(Z, x, \theta _t)}\right ]\]

      Question. Now this is all cool theoretically but how TF do we actually maximize this thing?
      Answer. It is highly dependent on the parametrization of \(p\) and \(q\).
      Example.

      We define the problem as

      1. \(p(z) \sim N(0, 1)\)
      2. \(p(x | z; \theta ) \sim N(\mu _{\theta }(z), \sigma _{\theta }(z))\)
      3. \(q_{\phi }(z | x, \theta ) \sim N(\mu _{\phi }(x), \sigma _\phi (x))\)
      where \(\mu _{\{\phi , \theta \}}, \sigma _{\{\phi , \theta \}}\) are parametrized neural networks. Then we can rewrite the ELBO as (ignoring constants) \[ \begin {align*} \mathsf {ELBO}(x, q, \theta ) &= \mathbb {E}_{x \sim \mathcal {D}, z \sim q(\cdot \mid x, \theta )} \left [\log p(Z) - \log \frac {p(x \mid Z; \theta )}{q_{\phi }(Z, x, \theta )}\right ] \\ &= \mathbb {E}_{x \sim \mathcal {D}, z \sim q(\cdot \mid x, \theta )} \left [-\frac {Z^2}{2} - \frac {\sigma _{\phi }(x)^2}{\sigma _{\theta }(z)^2}\big ((Z - \mu _{\theta }(Z))^2 - (Z - \mu _{\phi }(x))^2\big )\right ] .\end {align*} \]

      Problem. We need to sample realization \(z \sim q_{\phi }(\cdot |x, \theta )\). So isn't our realization \(z\) a function of \(\phi \). How would we optimize this?
      Answer. The reparametrization trick!!

      Reparametrization Trick [0089]

      • July 1, 2025

      For \(q_{\phi }(\cdot | x) \sim N(\mu _{\phi }(x), \sigma _{\phi }(x))\). The reparametrization trick allows us to differentate through a random sample by sampling from \(q_{\phi }(\cdot | x)\) via \[z(\phi ) = \mu _{\phi }(x) + \sigma _{\phi }(x) \cdot \epsilon ,\] where \(\epsilon \sim N(0, 1)\). Then, for a given sample \(\epsilon \), we can easily compute \[\nabla _{\phi } z(\phi ) = \nabla _{\phi } \mu _{\phi }(x) + \nabla _{\phi }\sigma _{\phi }(x) \cdot \epsilon \]

      Now for a given sample \(x_i \sim \mathcal {D}\), we can get a sample \(z_i(\phi ) \sim q(\cdot \mid x_i, \theta )\) that is differentiable with respect to \(\phi \). Using this, we can get the stochastic gradient estimate \[\nabla _q \widetilde {\mathsf {ELBO}}(x, q, \theta ) = \nabla _{\phi } \left [\frac {z_i(\phi )^2}{2} - \frac {\sigma _{\phi }(x)^2}{\sigma _{\theta }(z)^2}\big ((z_i(\phi ) - \mu _{\theta }(z_i(\phi )))^2 - (z_i(\phi ) - \mu _{\phi }(x))^2\big )\right ]\]

      Algorithm. input \(\theta _t\)
      1. Sample \(x_i\) from dataset \(\mathcal {D}\)
      2. Sample \(z_i(\phi ) \sim q_{\phi }(\cdot \mid x_i, \theta _t)\) via trick.
      3. Compute \(\nabla _q \widetilde {\mathsf {ELBO}}\)
      4. Update via SGD

      Evidence Lower Bound [0082]

      • June 30, 2025

      Setup. We are given a joint distribution \(p(X, Z; \theta )\), where \(X\) and \(Z\) are random variables with their joint parametrized by \(\theta \). We are also given a dataset \(\mathcal {D}\) containing realizations of only \(X\) i.e. \(Z\) is a latent random variable.

      Goal. We want to learn \(p(x; \theta )\).

      However, \[p(x; \theta ) = \int p(x, z; \theta ) \mathrm {d}z = \int p(x | z; \theta )p(z) \mathrm {d}z,\] which is computationally intractable for most continuous \(Z\).

      Question. Can we update \(\theta \) for a given realization of \(x\) and \(z\)?
      Answer. Yes!!!

      Motivation. Suppose we had some \(q(z)\) that we could sample realizations of \(Z\) from. Then we want to find some reasonable \(\ell _{\theta }\), such that \[\theta ^* \approx \arg \max _{\theta } \mathbb {E}_{x, \sim p(x; \theta ^*), z \sim q(z)}[\ell _{\theta }(X, Z)]\]

      Remark. \(q(z)\) can potentially (and will) depend on the sample \(x_i\). This is OK because we are given \(x_i\)'s, which we can then just pass to \(q\) to get the corresponding \(z_i\).
      Why is this the case? Because then (for well-behaved \(\ell _{\theta }\)) \[\theta _{t + 1} = \theta _{t} - \alpha \nabla _{\theta _{t}}\ell _{\theta _{t}}(x_i, z_i) \to \theta ^*\] i.e. SGD is a reasonable thing to do, where the gradient is for a single sample of \(x_i\) and \(z_i\) which we have readily available from our dataset \(\mathcal {D}\) and our distribution \(q\) that we can sample from. This idea comes up a lot in optimization, such as the \(\log \) trick in the original policy gradient.

      Similar to the \(\log \) trick, the Evidence Lower Bound (ELBO) comes from a simple multiply and divide trick. Namely, \[\begin {align*} \log p(x; \theta ) &= \log \int p(x, z; \theta ) \mathrm {d}z \\ &= \log \int p(x, z; \theta ) \frac {q(z)}{q(z)} \mathrm {d}z \\ &= \log \mathbb {E}_{z \sim q(z)} \frac {p(x, Z; \theta )}{q(z)} \\ &\geq \mathbb {E}_{z \sim q(z)} \log \frac {p(x, Z; \theta )}{q(z)} .\end {align*}\] The last inequality follows directly from Jensen's Inequality. However, note that this is only a lower bound and therefore not exactly what we want. For example, if we were to pick \(q \sim N(0, 1)\) and then optimize the ELBO for \(\theta \) we would not necessarily find a good \(\theta \) for the likelihood \(\log p(x; \theta )\). Luckily, we can actually get the difference \(\log p(x; \theta ) - \)ELBO in closed form! Namely, \[\begin {align*} \log p(x; \theta ) - \mathrm {ELBO} &= \log p(x; \theta ) - \mathbb {E}_{z \sim q(z)} \log \frac {p(x, Z; \theta )}{q(z)} \\ &= \int \left (\log p(x; \theta ) - \log \frac {p(x, z; \theta )}{q(z)}\right )q(z) \\ &= \int \left (\log \frac {p(x; \theta )q(z)}{p(x, z; \theta )}\right )q(z) \\ &= \int \left (\log \frac {p(x; \theta )q(z)}{p(z | x; \theta )p(x; \theta )}\right )q(z) \\ &= \int \left (\log \frac {q(z)}{p(z | x; \theta )}\right )q(z) \\ &= \mathrm {KL}\big (q(z) \mid \mid p(z | x; \theta )\big ) .\end {align*}\] Thus, to tighten the lower bound we can optimize \(q_{\phi }(z)\) to try to match \(p(z | x; \theta )\). As a sanity check, lets make sure these match when \(q(z) = p(z \mid x; \theta )\). \[\begin {align*} \mathbb {E}_{z \sim q(z)} \log \frac {p(x, Z; \theta )}{q(z)} &= \int \log \frac {p(x, z; \theta )}{q(z)}\mathrm {d}z \\ &= \int \log \frac {p(x; \theta )p(z \mid x; \theta )}{p(z \mid x; \theta )}\mathrm {d}z \\ &= \int \log p(x; \theta )\mathrm {d}z \\ &= \log (x; \theta ) .\end {align*}\] You may be wondering how we use this algorithmically to find \(\theta ^*\)? This will be the main idea of expectation maximization. Feel free to check it out!

      Stack vs. Heap [0025]

      • December 28, 2024
      • Kellen Kanarios

      I often hear this question come up amongst performance-driven developers, and for the longest time I could never get a straight answer as to why the stack was so much better?       Why stack is faster.

      • December 28, 2024
      • Kellen Kanarios
       This all comes down to what happens during each memory allocation. Prior to actually running the program, the operating system allocates some amount of memory for both the stack and the heap (I think around 2-4MB?). When you allocate memory on the stack i.e. int x = 1;
      • Stack just increments stack pointer.
      • Writes value to new address pointed to by stack pointer.
      When you allocate memory on the heap i.e. int *x = malloc(sizeof(int)), we must instead choose how best to fulfill the request, which is a whole research direction in itself.
      • One such implementation is the buddy algorithm
      However, malloc itself is not a system call and therefore is not terribly slow. A real issue occurs when we run out of pre-allocated heap memory. This requires invoking a system call like sbrk to increment the size of the allocated memory for the heap.
      • If no available pages, OS must evict and write to disk...
      • Formally, this is known as "heap growth".
      Generally, if you know how much memory you need it is good practice to just allocated all up front to minimize any potential system calls. With this, the performance difference is not too drastic :)

      The first thing we need to do is understand all the things that can go awry.

      First in the valid-invalid protocol example we were given they only have two virtual channels. However, we need 3 channels to handle, request messages, forwarded messages, and response messages.

      Let us briefly understand why this is the case. Detecting a deadlock is equivalent to detecting a cycle in a dependency graph, so it is sufficient to show a cyclic dependency.

      • If we issue a GetS this goes to the directory, giving us an edge N1 -> D.
      • Then the directory issues a Fwd-GetS to the ownder, giving us the chain N1 -> D -> N2
      • The owner then needs to send data to the requester, giving us the chain N1 -> D -> N2 -> N1.
      • Namely, the data must go through the same channel as all of the requests depending on it.

      To avoid this, we partition into 3 channels:

      • Request: GetS, GetM, PutM
      • Forwarded: Fwd-GetS, Fwd-GetM, PutAck, Inv
      • Response: InvAck, Data

      On-chip interconnection networks [667T]

      • April 8, 2026
      • Kellen Kanarios

      The network connecting cores, L2/L3 banks, and memory controllers on a Multicore chip. The same abstractions that govern off-chip networks (topology, routing, flow control) apply, but the constraints are different: wires are abundant but die area is scarce, latency is dominated by hops rather than propagation, and the network is part of the Cache coherence critical path.

      On-chip network metrics [OMZU]

      • May 12, 2026
      • Kellen Kanarios

      The measurements used to compare network topologies and traffic patterns.

      Traffic-independent

      • May 12, 2026
      • Kellen Kanarios
      • Degree: number of links at each node. A proxy for router complexity — each port costs a buffer, crossbar input, state machine.
      • Bisection bandwidth: bandwidth across a cut that splits the network into two equal-sized halves. The minimum over all such cuts. A proxy for peak achievable throughput, especially for uniformly random traffic.
      • Diameter: maximum hop distance between any two nodes. The worst-case hop count.
      Remark. Bisection bandwidth is less useful for on-chip networks than off-chip because global on-chip wiring is plentiful.
      Remark. Ring bisection is 2 (two cuts to bisect); torus bisection is 4.

      Traffic-dependent

      • May 12, 2026
      • Kellen Kanarios
      • Hop count: hops between two specific nodes. The average over a traffic distribution is a useful proxy for latency.
      • Maximum channel load: the most-loaded link's load relative to per-node injection bandwidth, for a given traffic pattern. The proxy for sustained throughput.
      • Path diversity: number of distinct shortest paths between two nodes. Higher path diversity allows adaptive routing to spread load.

      Performance regimes

      • May 12, 2026
      • Kellen Kanarios
      • Zero-load latency: average hop count with no contention.
      • Saturation throughput: the injection rate at which latency curves upward exponentially.
      Remark. Hop count is not the same as latency. Per-hop pipeline depth and link propagation matter — more short hops can beat fewer slow ones. Router pipeline depth isn't always known in advance.

      Direct vs indirect networks [95AC]

      • May 12, 2026
      • Kellen Kanarios

      Two organizational families for on-chip networks.

      • Direct — every terminal node (core, memory controller) is associated with a router. Most on-chip designs are direct because area is precious and there's no point spending a router with no compute attached. The "\(k\)-ary \(n\)-cube" family (Ring topology, Mesh topology, Torus topology) is direct.
      • Indirect — routers may exist that connect only to other routers, with no terminal attached. Butterfly network and Clos network are indirect; a router in their middle stages just shuffles bits between input and output stages.
      Remark. Indirect networks can shorten the diameter for the same node count, but they pay for extra silicon area for routers that don't serve traffic locally. Often the right call off-chip; usually the wrong call on-chip.

      Topologies

      • April 8, 2026
      • Kellen Kanarios

      Crossbar [YV5J]

      • May 12, 2026
      • Kellen Kanarios

      The simplest indirect network: an \(n \times m\) grid of switch points connecting \(n\) inputs to \(m\) outputs. Any input can be connected to any free output simultaneously — non-blocking.

      • One hop per transit (minimum possible).
      • Serializes if multiple senders target the same receiver.
      • Switch area grows as \(O(n \cdot m)\) — the dominant cost as systems grow.
      Remark. Crossbar router area scales as the square of the port count, which is why high-radix routers are dramatically more expensive than low-radix ones.

      Crossbars are common as the inner switch of larger structured networks (Clos network, Fat tree) — large flat crossbars are impractical at scale, but small crossbars are cheap.

      Ring topology [UEA0]

      • May 12, 2026
      • Kellen Kanarios

      A direct network where each node connects to two neighbors, forming a closed loop. Bidirectional rings allow either direction of travel; unidirectional rings allow only one. The simplest topology to implement and reason about.

      In the \(k\)-ary \(n\)-cube family, a ring is a \(k\)-ary 1-cube.

      • Degree 2 (every node).
      • Diameter \(\lfloor k/2 \rfloor \) for a bidirectional \(k\)-node ring.
      • Bisection bandwidth 2 — cut the ring at exactly two links.
      • Hardware overhead very low: routers have two ports.
      Problem. Worst-case hop count is \(O(k)\). Doesn't scale beyond a small number of nodes.
      Remark. Despite the poor scaling, rings are common in commercial CPUs (Intel's L3 ring bus on Sandy/Ivy Bridge) because they're cheap and 8–16 cores is enough.

      Mesh topology [9SA2]

      • May 12, 2026
      • Kellen Kanarios

      A 2D grid of nodes with neighbors connected on the four compass directions, no wrap-around. The dominant on-chip topology in commercial designs (Xeon Phi, recent server x86 chips) because it lays out naturally on a 2D die.

      • Interior nodes degree 4; edge nodes degree 3; corners degree 2.
      • Average hop count for an even-\(k\) mesh: \[H_{avg} = \frac {nk}{3}\]
      • Maximum channel load for uniform random traffic: \(k/4\) — twice that of a Torus topology.
      Problem. Non-uniform node degree. Edge nodes carry less traffic than they could, while interior nodes bottleneck on traffic flowing through them.
      Remark. Mesh dominates Torus topology in practice despite the topology metrics being worse, because torus wrap-around wires are physically long and force expensive folding to equalize link lengths. Manufacturing wins over theoretical performance.

      Torus topology [DDKT]

      • May 12, 2026
      • Kellen Kanarios

      A Mesh topology with wrap-around links connecting the ends of each dimension. In the \(k\)-ary \(n\)-cube classification, a \(k\)-ary \(n\)-cube. Uniform node degree (every node has degree \(2n\)), uniform load distribution, half the hop count of a mesh.

      • Every node has degree \(2n\).
      • Average hop count for even \(k\): \[H_{avg} = \frac {nk}{4}\]
      • Maximum channel load for uniform random traffic: \(k/8\) — half that of a Mesh topology.
      • Good path diversity: many shortest paths between any two nodes.
      Problem. Wrap-around links are physically long. To equalize wire delays, the torus must be folded, doubling effective wire lengths and link area. The wrap-around also doubles the link count versus a mesh of the same dimensions.
      Remark. Despite the theoretical advantage, mesh wins on real chips. Tori show up in HPC interconnects (Cray) where the wires can run on dedicated cables.

      Variants:

      • \(k\)-ary \(n\)-cube: \(k\) nodes along each of \(n\) dimensions, all ends wrapped.
      • Asymmetric: \((i, j, k)\)-ary — different node count per dimension.
      • Mesh: same as torus but without wrap-around.
      • Ring: \(k\)-ary 1-cube.

      • May 12, 2026
      • Kellen Kanarios

      A multistage indirect network. A \(k\)-ary \(n\)-fly has \(k^n\) terminal nodes connected through \(n\) stages of \(k \times k\) crossbar switches. At each stage, the routing decision selects which of \(k\) sub-groups to enter; after \(n\) stages, every terminal can reach every other.

      • Hop count \(n - 1\).
      • Under uniform random traffic, maximum channel load is 1 (perfectly balanced).
      • Channel load drops further for non-uniform traffic that respects the natural partitioning.
      Problem. Zero path diversity. There is exactly one path between any source and destination. A single hot link wedges traffic.
      Solution. Add shuffle stages that permute paths after each routing stage, so the same source-destination pair can be reached by multiple routes.

      Clos network [YKAN]

      • May 12, 2026
      • Kellen Kanarios

      A three-stage indirect network parameterized by \((m, n, r)\): \(r\) input switches and \(r\) output switches each of size \(n \times m\), with \(m\) middle-stage switches each \(r \times r\). Total terminal nodes \(N = nr\). Non-blocking when \(m \geq 2n - 1\).

      • Number of total switching elements scales as \(N^{1.5}\) versus \(N^2\) for a flat Crossbar while preserving non-blocking.
      • Path diversity is \(m\) — multiple distinct paths between any source and destination.
      Proof. There are \(rmn\) input-switch elements, \(mr^2\) middle-switch elements, and \(rmn\) output-switch elements: \[S = 2rmn + mr^2.\] With \(r = N/n\): \[S = 2Nm + m(N/n)^2.\] Choosing \(n = \sqrt {N}\) and \(m \geq 2n - 1 = O(\sqrt {N})\): \[S = 2mN + 2m\sqrt {N} = O(N^{1.5}).\]
      Remark. A folded Clos network — input and output stages combined, making \(n\) input ports and \(m\) output ports bidirectional — is structurally the same as a Fat tree.

      Fat tree [DFC7]

      • May 12, 2026
      • Kellen Kanarios

      A tree topology with bandwidth that doubles at every level toward the root. Each interior switch has logical degree 4, but as you climb the tree, link width doubles to absorb the combined traffic of its subtree — preserving bisection bandwidth.

      Structurally, a fat tree is a folded Clos network: the input and output stages of a Clos are merged into a single bidirectional stage at each level of the tree, and the middle stage becomes the higher levels of the tree.

      • Each switch has logical degree 4 (2 down, 2 up — with link doubling).
      • Bisection bandwidth scales linearly with the number of leaves.
      • Path diversity grows toward the root.
      Remark. Fat trees are the dominant topology for off-chip HPC interconnects (InfiniBand fat-tree clusters). Less common on-chip because the doubling links are hard to lay out on a 2D die.

      Network routing strategies [VTS4]

      • May 12, 2026
      • Kellen Kanarios

      Given a topology and a packet's source and destination, which path through the network does it take? Three classes of answer:

      • Greedy (shortest-path) — always pick the shortest available route.
        • (+) Minimizes latency under light load.
        • (-) Overloads central links; worst-case throughput poor.
      • Uniform / oblivious random — pick a direction at random.
        • (+) Spreads load; avoids systematic hotspots.
        • (-) Wastes bandwidth on non-minimal paths; latency unpredictable.
      • Adaptive — at each hop, pick the outgoing link with the lowest local congestion.
        • (+) Dynamically balances load; best worst-case throughput.
        • (-) Routers need congestion state; risk of livelock without care.
      Example. Tornado traffic on an 8-node bidirectional ring: node \(i\) sends to \((i+3) \bmod 8\).
      • Greedy: all packets travel counter-clockwise. Bidirectional bandwidth wasted; \(\frac {1}{3}\) utilization.
      • Uniform: randomly clockwise/counter-clockwise. Counter-clockwise gets \(\frac {2}{3}\) utilization, clockwise \(\frac {2}{5}\); worst-case \(\frac {2}{5}\).
      • Adaptive: sends \(\frac {5}{8}\) counter-clockwise to match traffic exactly. Best.
      Remark. Adaptive routing tends to need Network backpressure signaling to know where congestion is, and care to avoid livelock when many adaptive decisions chase each other.
      Remark. Oblivious strategies like Dimension-order routing and Valiant's routing algorithm give deterministic guarantees (deadlock-free, bounded path length) at the cost of giving up the load-balancing of true adaptive routing.

      Dimension-order routing [83P2]

      • May 12, 2026
      • Kellen Kanarios

      An oblivious Network routing strategies for \(k\)-ary \(n\)-cube networks (Mesh topology, Torus topology): always exhaust the offset in one dimension completely before changing to the next. E.g. in 2D mesh routing, route X-then-Y — never turn back to X after starting Y.

      The dimension order prevents routing deadlock: cycles in the channel dependency graph are impossible because no packet ever requests a channel in a "lower" dimension once it has started using a higher one. The dependency graph is therefore acyclic, and the standard cycle-equals-deadlock argument applies.

      • (+) Provably deadlock-free without virtual channels.
      • (-) Zero path diversity. Hot spots can't be routed around.
      • (-) Unaware of current network state — oblivious.

      The standard remedy for the path-diversity problem is Valiant's routing algorithm, which keeps the deadlock-freedom of dimension-order routing while spreading load with a randomized intermediate hop.

      Valiant's routing algorithm [XLDA]

      • May 12, 2026
      • Kellen Kanarios

      An oblivious Network routing strategies that achieves provably optimal worst-case throughput by introducing randomness in path selection:

      1. For a packet going from \(s\) to \(d\), randomly pick an intermediate node \(w\).
      2. Route \(s \to w\) using Dimension-order routing.
      3. Eject the packet at \(w\), then re-inject it.
      4. Route \(w \to d\) using Dimension-order routing.

      To preserve deadlock-freedom, the intermediate node must remove the packet from the network before re-injecting — otherwise the channel dependencies of the two halves could interlock.

      Remark. Restricting \(w\) to lie in the sub-quadrant defined by \(s\) and \(d\) keeps paths near-minimal while still spreading load — the practical compromise between guaranteed throughput and added latency.
      Remark. Valiant's algorithm trades average-case latency (paths are longer) for worst-case throughput (any traffic pattern looks roughly uniform once randomized). The right call for adversarial workloads; overkill for uniform traffic.

      Routing deadlock [DWV1]

      • May 12, 2026
      • Kellen Kanarios

      Routing deadlock occurs when a packet cannot make progress because the buffer at its next link is full, and the packet holding that buffer is itself blocked waiting on a buffer this packet holds — a cycle in the channel dependency graph.

      Two structural responses:

      1. Avoid deadlock by construction. Dimension-order routing enforces an ordering on dimension transitions so the dependency graph is acyclic by design.
      2. Detect and recover from deadlock. Allow all turns. Detect cycles via timeouts. Drain stuck packets onto a backup network (a "slow channel" — physical, or more commonly an escape virtual channel) that uses dimension-order routing.

      The relationship to other related concepts:

      • Backpressure (Network backpressure) is the signaling mechanism that lets a router downstream tell upstream it is congested, before the upstream actually fills the link.
      • Adaptive routing avoids deadlock locally; backpressure lets adaptive routing react to remote congestion.

      Network backpressure [EZDD]

      • May 12, 2026
      • Kellen Kanarios

      A signaling mechanism that lets a downstream router tell upstream "I am congested — slow down" before its buffers actually overflow. With backpressure information, adaptive routing can route around remote congestion, not just local.

      The mechanism propagates: a congested link signals back to its sender, which then knows its next link will fill, which signals further back. Useful information flows in the opposite direction of the data.

      Problem. Acting only on local state can cause livelock — packets bounce between hotspots, never converging.

      Backpressure breaks the livelock by giving local routers global-ish information without an actual global view.

      Remark. Backpressure is mechanism; adaptive routing is policy. The two together are what real high-radix on-chip networks use.

      Interface with the cache hierarchy

      • April 8, 2026
      • Kellen Kanarios

      The on-chip network connects each core (with its L1) and the L2 cache banks (which may be shared or private). Each L2 bank has a cache controller (manages Cache coherence) and, at memory boundaries, a memory controller (talks to DRAM).

      Two cache-organization choices interact with the network design:

      • Private L2 caches: each tile owns its L2 slice; cross-tile traffic happens only on private-L2 miss. (+) Lower hit latency; less interconnect traffic. (-) Replicated data; more off-chip traffic on misses.
      • Shared L2 cache: a single logical L2 split into banks, with each address mapped to a home bank. (+) Better utilization, no replication. (-) Every L1 miss goes on the network.

      The interface to the network sits at the cache controller:

      • Miss status handling register (MSHR): on a cache miss, records what request the cache is waiting on. Matched against the reply when it arrives.
      • Transaction status handling register (TSHR): tracks outstanding memory transactions to off-chip memory from the directory cache.
      Remark. Memory controllers are commonly placed at the chip perimeter, near the I/O pads. A memory controller can share a router with a core/cache tile (saves area; needs arbitration) or have its own (more area; no contention).
      Remark. Hardware-managed shared memory is one design point; another is explicit message passing, where software (not hardware) is responsible for moving data between processes.

      Implementation considerations

      • April 8, 2026
      • Kellen Kanarios

      Links. Thicker wires with larger spacing → lower resistance, higher speed, better energy efficiency. The wire budget determines how many ports you can afford per router.

      Routers. Each port costs a buffer, a crossbar input, state logic, and a link to its neighbor. Crossbar area scales as the square of port count, so high-radix routers are dramatically expensive. Rings have very simple routers but poor performance.

      Floorplan. Tori need to be folded to equalize wire lengths, doubling effective wire length; meshes don't. Tori also have twice as many links as meshes, which may force narrower links and lower per-link bandwidth.

      Folding can also equalize path lengths inside a network at the cost of increasing average latency.

      Concentration allocates more cores per router. (-) Decreases throughput per node. (+) Programs with high locality do little remote communication, so concentration is a good fit there.

      ILP vs TLP [RJEB]

      • May 13, 2026
      • Kellen Kanarios

      The two strategies a microarchitecture can use to keep arithmetic units busy in the face of memory latency and dependencies. Instruction-level parallelism mines parallelism from within a single instruction stream; Thread-level parallelism mines it across many. Every modern design is a position on this axis, and the position determines the rest of the microarchitecture.

      The choice is not really a choice — it is forced by the workload. ILP works only when the single thread has independent work to find; once dependency chains and branch mispredictions saturate the issue window, no amount of silicon spent on reorder buffers, wakeup logic, or speculation buys more throughput. TLP works only when many independent threads exist; throwing more warps at a single sequential dependency chain accomplishes nothing.

      The downstream consequences split everything else:

      • Control overhead. ILP machines spend most of their area on out-of-order execution machinery — Reorder buffer, Reservation stations, register renaming, CAM-based wakeup-select. TLP machines spend almost none; the schedule is "pick any ready thread."
      • Caches. ILP needs large caches because a single stalled thread blocks the whole pipeline. TLP can afford small caches because the latency is hidden by switching threads.
      • Register file. ILP keeps one (renamed) register set hot. TLP keeps every resident thread's register set hot simultaneously — GPU register files are an order of magnitude larger than CPU register files for this reason.
      • Readiness tracking. ILP pays O(N²) for CAM-based wakeup-select across N in-flight instructions; TLP pays O(1) per instruction via a per-register scoreboard.

      The slogan: CPUs spend a lot of silicon to find parallelism in one thread; GPUs spend almost none because they have many threads. Everything downstream — pipeline depth, cache hierarchy, scheduling policy, energy per operation — falls out of which side of this axis the chip sits on.

      Theorem. Mutual information is nonnegative [TB4P]

      • July 11, 2026
      • Kellen Kanarios

      For any pair of random variables, \(I(X;Y)\geq 0\), with equality iff \(X\) and \(Y\) are independent.

      This is Gibb's inequality in disguise: mutual information is the KL divergence between the joint distribution and the product of marginals, \[I(X;Y)\ =\ D_{\mathrm {KL}}\left (P_{XY}\,\|\,P_X P_Y\right )\ \geq \ 0,\] and Gibbs' inequality forces every KL divergence to be nonnegative, vanishing exactly when the two arguments coincide — i.e. when \(P_{XY}=P_XP_Y\). It is the root fact from which Conditioning reduces entropy follows.

      Theorem. Conditioning reduces entropy [H7J5]

      • July 11, 2026
      • Kellen Kanarios

      Observing another variable can only lower entropy on average: \[H(X\mid Y)\ \leq \ H(X),\] with equality iff \(X\) and \(Y\) are independent.

      Immediate from Mutual information is nonnegative and the identity \(I(X;Y)=H(X)-H(X\mid Y)\): since \(I(X;Y)\geq 0\), the conditional entropy cannot exceed the unconditional one. The caveat is that this holds on average over \(Y\) — a particular outcome \(Y=y\) can raise the entropy of \(X\); it is only the expectation \(H(X\mid Y)=\mathbb {E}_Y[H(X\mid Y=y)]\) that is guaranteed to drop.

      Theorem. Entropy is subadditive [C6M8]

      • July 11, 2026
      • Kellen Kanarios

      The joint entropy of a collection never exceeds the sum of the individual entropies: \[H(X_1,\dots ,X_n)\ \leq \ \sum _{i=1}^{n} H(X_i),\] with equality iff the \(X_i\) are mutually independent.

      Expand the joint entropy by the chain rule, \(H(X_1,\dots ,X_n)=\sum _i H(X_i\mid X_1,\dots ,X_{i-1})\), and apply Conditioning reduces entropy term by term: each \(H(X_i\mid X_{<i})\leq H(X_i)\). Conditioning on the past can only shrink each term, so dropping the conditioning gives an upper bound.

      Theorem. The uniform distribution maximizes entropy [6FHG]

      • July 11, 2026
      • Kellen Kanarios

      Over a finite alphabet \(\mathcal {X}\), entropy is bounded by the log of the alphabet size, \[H(X)\ \leq \ \log |\mathcal {X}|,\] with equality iff \(X\) is uniform. No distribution on \(\mathcal {X}\) is more uncertain than the uniform one.

      Again a face of Gibb's inequality: measuring \(P\) against the uniform law \(U(x)=1/|\mathcal {X}|\), \[D_{\mathrm {KL}}(P\,\|\,U)\ =\ \log |\mathcal {X}| - H(X)\ \geq \ 0,\] so \(H(X)\leq \log |\mathcal {X}|\), and Gibbs' equality condition \(P=U\) pins down the maximizer. This is the discrete base case of the general maximum-entropy principle: absent constraints, the flattest distribution wins.

      Log-Sum Inequality [004F]

      • February 1, 2025
      • Kellen Kanarios

      If \(a_{i},b_{i}>0\) for \(i\in \{1,\ldots ,n\}\); \(A=\sum _{i=1}^{n}a_{i},\quad B=\sum _{i=1}^{n}b_{i}\) Then, we have: \(\sum _{i=1}^{n}a_{i}\log a_{i}/b_{i}\geq A\log A/B\)

      Proof. Define \(p_i = \frac {a_i}{A}\), \(q_i = \frac {b_i}{B}\), then \[ \begin {align*} D_{\mathrm {KL}}(p || q) &= \sum \frac {a_i}{A} \log {\frac {B a_i}{A b_i}} \\ &= \sum \frac {a_i}{A} \log {\frac {a_i}{b_i}} - \sum \frac {a_i}{A} \log {\frac {A}{B}} \\ &= \sum \frac {a_i}{A} \log {\frac {a_i}{b_i}} - \log {\frac {A}{B}} \\ \end {align*} \] By Gibb's inequality, \[ \begin {align*} \sum \frac {a_i}{A} \log {\frac {a_i}{b_i}} - \log {\frac {A}{B}} \geq 0 &\implies \sum \frac {a_i}{A} \log {\frac {a_i}{b_i}} \geq \log {\frac {A}{B}} \\ &\iff A \sum a_i \log {\frac {a_i}{b_i}} \geq A \log {\frac {A}{B}} \end {align*} \] as desired.

      Consequences of Log-Sum Inequality. The log-sum inequality upgrades nonnegativity to convexity.

      Definition. Markov chain as conditional independence [UWYD]

      • July 11, 2026
      • Kellen Kanarios

      The information-theoretic reading of a Markov chain: three variables form a chain \(U\to V\to W\) when \(W\) depends on \(U\) only through \(V\), \[P(W=w\mid U=u, V=v)\ =\ P(W=w\mid V=v).\] Equivalently, \(U\) and \(W\) are conditionally independent given \(V\), written \((U\perp W)\mid V\): \[P(U=u, W=w\mid V=v)\ =\ P(U=u\mid V=v)\,P(W=w\mid V=v).\]

      Two consequences make this the right object for information theory. The relation is reversible — \(U\to V\to W \iff W\to V\to U\) — because conditional independence is symmetric in \(U\) and \(W\). And any estimator forms a chain \(X\to Y\to \hat X\): an estimate \(\hat X\) is computed from the observation \(Y\) alone, never touching \(X\) directly. That structure is exactly what the data processing inequality needs — you cannot manufacture information about \(X\) by processing \(Y\).

      Theorem. Data Processing Inequality [004Z]

Therefore, \[ \begin {align*} I(U; V) &= I(U; W) + I(U; V | W) \\ &\geq I(U; W) \end {align*} \]

      Data race [P0NQ]

      • May 12, 2026
      • Kellen Kanarios

      Two accesses to the same memory location, from different threads, where at least one is a write and there is no happens-before relationship between them. The happens-before relation is what synchronization primitives like Locks and atomics induce — when one access happens before another, they are not racing.

      Remark. Competing for the same lock is not a data race. Whichever thread acquires first happens before the next; the access inside each critical section is sequentially ordered.

      Data-race freedom (DRF) is the implicit contract behind every relaxed memory model: if your program has no data races, Relaxed consistency hardware behaves as if it were Sequential consistency. Conversely, in the presence of a data race, the result is undefined by the language and the model.

      Problem. Writing data-race-free programs is hard. Treating every shared access as a synchronization point destroys performance.

      SC-preserving compilers approach: forbid the compiler from reordering shared accesses, leaving the hardware free to do TSO-style reordering. The optimization most disrupted is common-subexpression elimination across shared accesses — solved by inserting a "monitor" check on the coherence bit between expressions.

      Hardware approach. Treat all pages as local/safe by default and extend page protection bits to the thread level: a non-owning thread accessing the page triggers a fault, exposing the data race at runtime.

      Hardware interrupt path [K325]

      • May 15, 2026
      • Kellen Kanarios

      What the CPU does, entirely in hardware, the moment an interrupt arrives — whether a device IRQ, a fault like a Page fault, or a software int N. The point of the hardware path is to land in the kernel at a clean instruction boundary with enough saved state that user execution can resume later as if nothing happened.

      Four things happen, in order, with no software involvement:

      1. Reach a clean instruction boundary. The CPU cannot pause halfway through, say, a memory write — architectural state would be inconsistent. So the current instruction either finishes normally (typical for asynchronous interrupts like a timer) or is rolled back as if it never started (faults, where the instruction couldn't complete anyway and needs to be retried). A page-fault handler maps in the missing page and UHNL returns to re-execute the faulting instruction; a timer handler returns to the next instruction.
      2. Save the minimum resume state to the kernel stack. The CPU pushes RIP (where to resume), RSP (the user stack pointer it's about to abandon), the privilege level (CS — were we in ring 3 or ring 0?), and RFLAGS. The save target is the kernel stack, whose address the CPU reads from the Task state segment that the OS set up at boot. This is a security boundary: the kernel never trusts the user stack, since user code could have corrupted it or pointed RSP at something malicious. Note what is not saved: general-purpose registers. The handler itself pushes RAX, RBX, ... if it uses them, which keeps the hardware path fast.
      3. Jump to the handler via the Interrupt descriptor table. The CPU takes the interrupt number \(N\), indexes entry \(N\) of the IDT (found via the IDTR register), loads the handler address into RIP, and loads the new code segment, which switches the CPU from ring 3 to ring 0. After this step, the CPU is running kernel code on the kernel stack at the right privilege level — all set up automatically.
      4. Flush the Reorder buffer of work past the interrupt point. In an out-of-order execution CPU the ROB may already contain instructions from after the interrupt — they've been issued speculatively because the front-end ran ahead. Those entries are discarded so the handler sees clean architectural state, as if execution had stopped exactly at the interrupt point. When UHNL returns, those instructions re-execute normally. This is the same squash machinery used for branch mispredictions in Speculative execution — and, famously, the reason Spectre-class side channels exist: speculative work leaves observable traces in caches even after the ROB is flushed.

      Once the hardware path finishes, the OS's interrupt handler is running. Its first job is typically to save the remaining registers before doing real work — the rest is software.

      Remark. Steps 2 and 3 are why an OS needs to set up the IDT and TSS before enabling interrupts. Until both structures point somewhere valid, the first interrupt would either jump to garbage or push to a user-controlled stack — both fatal.

      Dennard scaling [S0TN]

      • May 12, 2026
      • Kellen Kanarios

      The observation that as transistor dimensions shrink by a factor \(\alpha \), you can also drop voltage by \(\alpha \), keeping power per transistor constant:

      \[P = C \left (\frac {V}{\alpha }\right )^2 f\]

      In the Dennard era, every process shrink got more transistors at the same power budget — the foundation of decades of frequency scaling.

      Post-Dennard. Below ~65nm, voltage can no longer be scaled because the supply-to-threshold ratio \(V_{dd}/V_{th}\) must stay above ~3 for transistors to switch reliably. The consequences:

      • Power per transistor scales only as \(1/\alpha \), not \(1/\alpha ^2\).
      • Power density therefore scales as \(\alpha \) — denser chips dissipate more per unit area.

      This is the same constraint that drove the Power wall in commercial CPUs: single-core frequency scaling stopped working, and the industry pivoted to multicore and energy-efficient architectures.

      Cache hierarchy [KBYI]

      • May 12, 2026
      • Kellen Kanarios

      Multiple tiers of progressively-larger, progressively-slower storage between the CPU's registers and main memory. Each tier holds the most-recently-used contents of the next slower tier, exploiting Spatial locality and Temporal locality in the workload.

      Approximate tiers on a modern x86 chip:

      • Registers — ns, dozens of bytes per core.
      • L1 cache — 2–8 ns, 8–128 KB per core. Often split into instruction and data L1s.
        • Built from static RAM (4–6 transistors per cell), which is fast but area-expensive vs. DRAM's single-transistor cell.
      • L2 cache — 5–12 ns, 0.5–8 MB. Closes the latency gap between L1 and DRAM.
      • L3 cache — slower, shared across cores.
      • Main memory — 10–60 ns, GBs.
      Example. A byte's journey. L1 checked first; on miss, L2; on miss, main memory; on miss in main memory, Page fault to disk.
      Remark. L1 is split into instruction and data caches because the locality profiles of code and data are very different — re-running the same code over different data has high temporal locality for code but low for data.

      Butterfly network [5WX9]

      • May 12, 2026
      • Kellen Kanarios

      A multistage indirect network. A \(k\)-ary \(n\)-fly has \(k^n\) terminal nodes connected through \(n\) stages of \(k \times k\) crossbar switches. At each stage, the routing decision selects which of \(k\) sub-groups to enter; after \(n\) stages, every terminal can reach every other.

      • Hop count \(n - 1\).
      • Under uniform random traffic, maximum channel load is 1 (perfectly balanced).
      • Channel load drops further for non-uniform traffic that respects the natural partitioning.
      Problem. Zero path diversity. There is exactly one path between any source and destination. A single hot link wedges traffic.
      Solution. Add shuffle stages that permute paths after each routing stage, so the same source-destination pair can be reached by multiple routes.

      Definition. Thread-level parallelism [G4ML]

      • May 13, 2026
      • Kellen Kanarios

      Parallelism extracted by running multiple independent threads concurrently and switching between them whenever any one of them stalls. Where Instruction-level parallelism mines a single instruction stream for hidden concurrency, TLP gives up on that ambition and instead assumes that enough independent work exists in the program (or workload) that some thread is always ready to make progress.

      TLP makes the per-thread microarchitecture cheap. There is no need for a deep issue window, aggressive speculation, or CAM-based wakeup-select: if the current thread can't proceed, the scheduler picks another. The cost shifts from per-instruction machinery to context-holding — registers, program counters, and stack pointers for every resident thread must live somewhere fast.

      Two dominant implementation styles:

      • Coarse-grained TLP. Conventional multicore CPUs and SMT: a handful of threads share a core, switched on long-latency events or interleaved cycle-by-cycle.
      • Massive TLP. GPUs keep dozens of warps resident per GPU architecture SM and switch on any stall. The GPU register scoreboard is the readiness check that makes this cheap.

      TLP fails when the workload genuinely has only one thread of useful work, or when the threads contend on shared state (locks, Cache coherence traffic). It wins overwhelmingly on data-parallel kernels where the threads are independent by construction.

      Theorem. Second-order optimality conditions [U3L4]

      • July 11, 2026
      • Kellen Kanarios

      Let \(f:\mathbb {R}^n\to \mathbb {R}\) be twice continuously differentiable in an open neighborhood of \(\mathbf {x_\star }\).

      Necessary. If \(\mathbf {x_\star }\) is a local minimizer, then \[(i)\ \nabla f(\mathbf {x_\star })=\mathbf {0};\qquad (ii)\ \nabla ^2 f(\mathbf {x_\star })\succeq \mathbf {0}.\] That is, it is a stationary point whose Hessian is positive semidefinite.

      Sufficient. If instead \(\nabla f(\mathbf {x_\star })=\mathbf {0}\) and \(\nabla ^2 f(\mathbf {x_\star })\succ \mathbf {0}\) (Hessian strictly positive definite), then \(\mathbf {x_\star }\) is a strict local minimizer. The gap between \(\succeq \) and \(\succ \) is exactly the ambiguity the first-order condition alone cannot resolve.
