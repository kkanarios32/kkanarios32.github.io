A two-loop scheduler makes warp readiness a constant-time scoreboard check [7SGF]

  • July 11, 2026
  • Kellen Kanarios

Deciding which warp to issue each cycle by scanning every warp's operands against the scoreboard would be prohibitive — 64 warps against 4 operands each is 256 comparisons. A two-loop scheduler makes the readiness check constant-time instead. The first loop selects a warp with room in its instruction buffer, looks up its PC, and fetches; a small per-warp bit-vector of outstanding register dependencies rides alongside the fetched instruction and clears as writes retire. The second loop issues any buffered instruction whose dependency vector is clean.

Because readiness is a single per-warp bit-vector test rather than a global scan, the scheduler stays cheap even with dozens of resident warps — the design tradeoff that lets a GPU keep far more work in flight than a CPU's CAM-based wakeup-select ever could. It is the scheduling-side counterpart to why the scoreboard replaces a CPU's dependency-tracking hardware.

    Warp scheduling hides memory latency by switching among resident warps [1PWL]

    • July 11, 2026
    • Kellen Kanarios

    A GPU keeps many warps resident on each Streaming Multiprocessor (SM) and switches between them on every stall. When a warp issues a memory op that misses cache it is dead for hundreds of cycles — but with dozens of resident warps the scheduler always has a runnable one, so aggregate throughput stays high even though single-warp latency is awful. This is the Thread-level parallelism design point cashed out on the GPU: spend silicon on resident warp state rather than on latency-reduction machinery. The switch is free because each SM's register file is large enough to hold every resident warp's registers at once, so a context switch is just changing which warp the scheduler picks — not saving state to memory the way a CPU context switch must.

    Two things bound how well this works. Occupancy — how many warps are resident — sets the ceiling on latency hiding: with too few warps the scheduler runs out of runnable candidates. And the scheduler itself must stay cheap enough to choose a warp every cycle, which it does with a two-loop scoreboard check, marshalling each issued instruction's register reads through an operand collector.

      Definition. GPU register scoreboard [XPAA]

      • May 13, 2026
      • Kellen Kanarios

      The readiness-tracking structure that lets a GPU issue instructions out of program order without any of the CAM machinery a CPU needs. A scoreboard is a small table — typically one busy bit per architectural register — that records whether some in-flight instruction is still computing a value that something else might want to read.

      The protocol is trivial. When an instruction issues, it sets the busy bit on its destination register. When it writes back, it clears the bit. Before any instruction can read its sources, the hardware checks the corresponding bits: if any source is busy, the instruction (or its entire warp) waits. The check is a single table lookup per source, so the per-instruction cost is \(O(1)\) regardless of how many other instructions are in flight.

      This is enough because GPUs lean on Thread-level parallelism rather than Instruction-level parallelism. When a warp's next instruction reads a busy register, the GPU architecture scheduler does not stall — it picks a different ready warp from the dozens of resident warps and runs that one instead. The scoreboard's job is only to answer "is this warp ready? yes/no," which is far cheaper than the broadcast/compare a CPU Issue queue wakeup-select does to find dependency satisfaction in a single thread's instruction window.

      The size difference falls out of this. CPU issue queues hold tens of entries; GPU instruction windows can hold many warps' worth of in-flight work, because every additional warp costs a row in the scoreboard and a register-file allocation, not a quadratic CAM expansion. See ILP vs TLP for why the design fork sits where it does.

      Remark. The scoreboard's simplicity is what lets GPUs spend their transistor budget on ALUs and the Register file instead of control logic. It is one of the most visible places where the Thread-level parallelism design choice cashes out as silicon savings.

      Definition. Issue queue wakeup-select [UJPD]

      • May 13, 2026
      • Kellen Kanarios

      The mechanism by which a modern out-of-order CPU notices that a previously-stalled instruction now has all its operands ready. The issue queue holds the N in-flight instructions whose operands are not yet available; each entry watches a small set of physical register tags (the operands it is waiting for). When a producer finishes, it broadcasts "tag T is ready" to every entry, and each entry compares that tag against its own waiting tags through a content-addressable memory (CAM). Entries whose last operand arrives become eligible to issue this cycle; a separate select stage picks which of the now-ready entries actually fires.

      The cost is the reason CPU issue queues are small. Each entry runs a CAM lookup against each broadcast tag, so an N-entry queue with N producers per cycle scales as \(O(N^2)\) in comparators, wires, and power. The select stage compounds this — priority encoders across N entries are themselves a quadratic-area structure on the critical timing path. The cumulative wakeup-select latency is what bounds clock frequency once N grows past a few tens.

      This is the structural reason CPU out-of-order windows stay in the tens-to-low-hundreds of entries even as transistor budgets grow. Wider doesn't help: timing closure on the CAM degrades faster than the additional ILP repays. GPUs sidestep the whole problem by leaning on Thread-level parallelism — a cheap per-register scoreboard plus a warp scheduler covers stalls without any broadcast/compare structure.

      Remark. Wakeup-select sits at the boundary between Reservation station (which holds the instruction) and register renaming (which named the tags being broadcast). All three structures together are what people mean by "the out-of-order back-end."

      Warp [X9VZ]

      • May 12, 2026
      • Kellen Kanarios

      A warp is the unit of scheduling on an NVIDIA GPU: a group of ~32 threads that issue one instruction per cycle in lockstep. Each functional unit in the SM has as many lanes as a warp has threads, so a single fetched instruction fans out across all 32 lanes at once. The warp — not the thread — is what the hardware fetches, schedules, and tracks; the per-thread SIMT programming model is an illusion built on top of it.

      Because the warp is the atom of execution, the two facts that dominate GPU performance are both properties of the warp: it hides memory latency by switching among resident warps instead of stalling, and it serializes both sides of a branch when its threads diverge.
