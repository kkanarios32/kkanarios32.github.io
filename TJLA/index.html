    Processing in Memory for ML [2BGU]

    • April 6, 2026
    • Kellen Kanarios

    Notes on guest lecture by Yufeng Gu titled Architecture and System Co-Design for Scalable Large Language Model Inference

    TFLOPs increasing much faster than DRAM bandwidth -> Hardware scaling is hitting the memory wall.

    Data center power consumption also increasing \(\approx \) 10% a year.

    GPUs are under low utilization for LLM inference

    • Prefix stage: parallel
    • Decoding stage: sequential -> bottleneck!

    As batch size increases, the KV-cache dominates memory usage.

    • KV cache size grows with context length.
    • Tradeoff between context length and batch size.
    Remark. Unlike fully-connected cannot batch self-attention because it depends on the individual prompts.

    Recall. operational intensity is \[\frac {\text {compute throughput}}{\text {memory bandwidth}}\]

    Inference much lower operational intensity than GEMM!

    • Inference only doing matrix \(\times \) row
    • If embedding dim is \(D\), then this is only \(D\) ops / byte.

    Processing in memory (PIM) enables faster memory bandwidth.

    Problem. Has lower memory capacity (fewer GB) due to added processing units (PU).
    Solution. Use Compute eXpress Link (CXL) to do memory expansion to get high memory bandwidth + expend memory capacity.
    Remark. Put high use operations i.e. vector multiplication, element-wise multiplication in PIM module (99% usage). Put more complicated operations on PNM accelerator. TODO: ISAs for each specified on slides (not available yet).
    Remark. CENT supports inter and intra-device communication via connecting communication controller to shared buffer and CXL port.

    Communcation primitives. multicast and gather

    • Multicast: send to all CXL devices.
    • Gather: receive data sequentially from all CXL devices.
    • Multicsat-gather -> tensor parallel.

    Peer-to-peer: can also support pipeline parallelism.

    Remark. For fast communication, use tensor parallelism i.e. if connected via NVLink. If slower communication use pipeline parallelism. Can mix.
    Remark. Longer context -> better performance for PIM (TODO: why?). HBM + PIM does not scale. Cannot arbitrarily add PIM.

    POTW: Fast Transformer Decoding: One Write-Head is All You Need [W4YP]

    • July 14, 2025
    • Kellen Kanarios

    Notes on the paper Fast Transformer Decoding: One Write-Head is All You Need.

    For,

    • \(b\) batch dimension,
    • \(n\) sequence length,
    • \(d\) hidden dimension,
    • \(h\) number of heads.
    The total FLOPS for attention are roughly \(bnd^2\). This comes from
    1. \(3nd^2\) FLOPS to compute \(Q = XW_Q, K = XW_K, V = XW_V\).
    2. Need to do this \(b\) times for each input in batch.
    and the total memory accesses are roughly \(\underbrace {bnd}_{X} + \underbrace {bhn^2}_{\text {softmax}} + \underbrace {d^2}_{\text {projection}}\). This gives high arithmetic intensity \[O\left (\left (\frac {h}{d} + \frac {1}{bn}\right )^{-1}\right )\]

    However, if we do incremental inference then we must multiply our total number of memory accesses by \(n\) i.e. \(bn^2d + nd^2\). This gives arithmetic intensity \[O\left (\left (\frac {n}{d} + \frac {1}{b}\right )^{-1}\right ),\] which requires large batch and short sequence length.

    Remark. I believe we just ignore the softmax memory contribution in the inference case because it does not scale with \(n\) anymore as we are computing the logits just for the next token. Therefore, it becomes a \(\frac {d}{h}\) term, which we can safely ignore?

    The key is that the \(\frac {n}{d}\) term comes from the \(bn^2d\) term, where we are loading \(h\) \((b \times n \times d / h)\) \(K\) and \(V\) matrices \(n\) times. Thus, we can improve the \(\frac {n}{d}\) term by a factor of \(h\) by simply not using a different \(K\) and \(V\) matrix for each head. This is the entire idea behind MQA.

    Remark. Similar to the KV-cache, we still can use \(h\) \(Q\) matrices because we do not need to load \(n\) of them into memory because only the last one matters for inference.

    Group Query Attention [XUVV]

    • July 14, 2025
    • Kellen Kanarios

    Group query attention is the same idea as MQA but instead of using one \(K\) and \(V\) for every head, they use a group of them. Where obviously the number of groups needs to be less than the number of heads.

    Also inspired by this idea of reducing the dependence of \(K\) and \(V\) on the sequence length is sliding window attention.

    Sliding Window Attention [WM3M]

    • July 14, 2025
    • Kellen Kanarios

    The idea behind sliding window attention is to ensure the number of weight matrices needed for the keys \(K\) and values \(V\) does not scale with the sequence length. Intuitively, this means allowing each word to "attend" to only some fixed number of previous words rather than the whole sequence
