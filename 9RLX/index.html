Programmed I/O [9RLX]

  • July 13, 2026

Simplified I/O Device Protocol: requires three "registers". One is the status register that tells the OS whether the device is ready or currently performing an operation. The second is the command register, which tells the device what we want to do. Lastly, is the data register that stores the actual data we want moved. The procedure operates as follows [arpaci2018operating, 36.3]

  1. OS polls status register in while loop until status == READY
  2. OS writes data register with data to be moved
  3. OS writes command register with command to be performed
  4. Device begins command and sets status = BUSY
  5. OS polls status register in while loop until status == READY

The above protocol requires that the CPU polls the device. This is a huge waste if the device is very slow because the CPU is unable to do any work while waiting. One solution is to use interrupts. In particular, the CPU follows the above procedure but after writing the command it resumes doing other work. Upon completion, the device issues an interrupt to the CPU to tell it that it has finished the command and that the CPU can resume working on that line of instructions.

Question. How is copying memory to device actually done in programmed I/O? Is the device address space mapped into the CPU address space and the CPU has 1. load from CPU address space to register, 2. write to device address from register?
Answer. There are two ways. The first is there are special I/O instructions (typically protected) that allow the CPU to directly write to the device. In x86, these are in, out. The second way is through memory-mapped I/O (MMIO). This is where the device is actually allocated a region of virtual address space and writing to / reading this region is routed to the device rather than the on-chip memory hierarchy [arpaci2018operating, 36.6].

        Hardware interrupt path [K325]

        • May 15, 2026
        • Kellen Kanarios

        What the CPU does, entirely in hardware, the moment an interrupt arrives — whether a device IRQ, a fault like a Page fault, or a software int N. The point of the hardware path is to land in the kernel at a clean instruction boundary with enough saved state that user execution can resume later as if nothing happened.

        Four things happen, in order, with no software involvement:

        1. Reach a clean instruction boundary. The CPU cannot pause halfway through, say, a memory write — architectural state would be inconsistent. So the current instruction either finishes normally (typical for asynchronous interrupts like a timer) or is rolled back as if it never started (faults, where the instruction couldn't complete anyway and needs to be retried). A page-fault handler maps in the missing page and IRET returns to re-execute the faulting instruction; a timer handler returns to the next instruction.
        2. Save the minimum resume state to the kernel stack. The CPU pushes RIP (where to resume), RSP (the user stack pointer it's about to abandon), the privilege level (CS — were we in ring 3 or ring 0?), and RFLAGS. The save target is the kernel stack, whose address the CPU reads from the Task state segment that the OS set up at boot. This is a security boundary: the kernel never trusts the user stack, since user code could have corrupted it or pointed RSP at something malicious. Note what is not saved: general-purpose registers. The handler itself pushes RAX, RBX, ... if it uses them, which keeps the hardware path fast.
        3. Jump to the handler via the Interrupt descriptor table. The CPU takes the interrupt number \(N\), indexes entry \(N\) of the IDT (found via the IDTR register), loads the handler address into RIP, and loads the new code segment, which switches the CPU from ring 3 to ring 0. After this step, the CPU is running kernel code on the kernel stack at the right privilege level — all set up automatically.
        4. Flush the Reorder buffer of work past the interrupt point. In an out-of-order execution CPU the ROB may already contain instructions from after the interrupt — they've been issued speculatively because the front-end ran ahead. Those entries are discarded so the handler sees clean architectural state, as if execution had stopped exactly at the interrupt point. When IRET returns, those instructions re-execute normally. This is the same squash machinery used for branch mispredictions in Speculative execution — and, famously, the reason Spectre-class side channels exist: speculative work leaves observable traces in caches even after the ROB is flushed.

        Once the hardware path finishes, the OS's interrupt handler is running. Its first job is typically to save the remaining registers before doing real work — the rest is software.

        Remark. Steps 2 and 3 are why an OS needs to set up the IDT and TSS before enabling interrupts. Until both structures point somewhere valid, the first interrupt would either jump to garbage or push to a user-controlled stack — both fatal.
