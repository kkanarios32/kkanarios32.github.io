Definition. Release consistency [ZWHV]

  • May 24, 2026
  • Kellen Kanarios

A Relaxed consistency discipline that recovers data-race freedom (see Data race) under Relaxed memory order not by issuing fewer fences, but by issuing weaker ones. The full-fence cost of Sequential consistency is replaced by a pair of half-fences placed exactly at the boundaries of critical sections — acquire on entry, release on exit.

The intuition is that correctness around a critical section only requires that work cannot leak out of it; work is free to sink into it. Half-fences encode exactly this asymmetry, which is why they are the right primitive for lock-based and lock-free synchronization alike.

Definition 1. Full fence [S80U]

  • May 24, 2026
  • Kellen Kanarios

A Memory fence across which no memory operation may move in either direction. Prior accesses must become globally visible before any subsequent access may issue. x86's mfence is the canonical example — the hammer that drains the Store buffer and serializes all prior memory operations against all subsequent ones.

Full fences are the default ordering primitive under Sequential consistency — every memory operation behaves as if a full fence sat on either side. They are also the only fence one needs to reason about correctness, which is why they are the building block for the half-fences of Release consistency.

Definition 2. Half fence [S5D0]

  • May 24, 2026
  • Kellen Kanarios

A Memory fence that constrains motion in only one direction, splitting the symmetric guarantee of a Full fence into two cheaper primitives:

  • Acquire — operations after the fence may not move above it. Used at the entry to a critical section: subsequent reads and writes cannot be hoisted before the lock is held.
  • Release — operations before the fence may not move below it. Used at the exit of a critical section: prior writes cannot be deferred past the unlock.

The asymmetry is exactly the freedom one wants around a critical section: expanding the section is fine — work may sink into it (past acquire) or rise into it (before release) — but work may not leak out of it. This is the basis of Release consistency and the reason C++ std::atomic exposes memory_order_acquire / memory_order_release as its core primitives.

C++ std::atomic compiles to half-fences in the right places, which is why it is the standard primitive for portable lock-free code.

Context

    Relaxed consistency [BCFG]

    • May 12, 2026
    • Kellen Kanarios

    A memory model weaker than Sequential consistency that allows the hardware (and compiler) to reorder memory operations for performance, with explicit Memory fence instructions where the programmer wants to forbid reordering.

    A fence is implemented with a counter — incremented on issue of each memory op, decremented on completion. The fence stalls dispatch until the counter reaches zero, which forces the Store buffer to drain before later accesses may proceed.

    Real machines pick a relaxation along a spectrum: Total store order permits only store-load reordering and is what x86 ships; Relaxed memory order permits anything and pushes the ordering burden onto the programmer. Either way, the next move is to make the fences themselves cheaper, which is what Release consistency does.

    Definition. Total store order [VH8Y]

    • May 24, 2026
    • Kellen Kanarios

    A Relaxed consistency relaxation in which loads may be hoisted above prior stores, but no other reorderings are permitted. The motivation: a store can sit in the Store buffer waiting for cache, while subsequent loads — which do not touch the buffered address — proceed without stalling.

    Same-core reads of a still-buffered store get the right value through Store-to-load forwarding; cross-core reads see the old value until the buffer drains. This asymmetry is the entire point of TSO's relaxation: the store buffer hides latency from the issuing thread without forcing other threads to wait. x86 implements TSO.

    Remark. Under TSO, Dekker's algorithm is still broken without Memory fence — the classic example where TSO is too weak for naive software synchronization.
    Remark. The IBM 370 variant is TSO without Store-to-load forwarding — it preserves global store atomicity at the cost of same-core stall-on-self-read.

    Definition. Relaxed memory order [WT44]

    • May 24, 2026
    • Kellen Kanarios

    A Relaxed consistency relaxation in which any pair of memory operations may be reordered. RMO is the most permissive end of the consistency spectrum; the programmer recovers ordering only through explicit Memory fence instructions.

    Each stronger reordering still has to be justified per case:

    • Load → store — no performance gain to reorder; leave alone.
    • Store → store — keep ordering with a slightly larger Store buffer.
    • Load → load — speculate the second load while waiting for the first; the reorder is invisible if the speculation is verified.

    Because every fence is expensive under RMO, the natural next move is to weaken the fences themselves into half-fences, which is what Release consistency does.

      Definition. Memory fence [R03A]

      • May 13, 2026
      • Kellen Kanarios

      An instruction that forces an ordering constraint between memory operations on either side of it. Fences exist because Relaxed consistency hardware reorders accesses aggressively for performance, and the programmer occasionally needs a point in the instruction stream past which prior accesses must be globally visible before subsequent ones may issue. A fence is the explicit knob for "wait until the world catches up here."

      The x86 family carries three:

      • sfence — store fence. All prior stores must drain from the Store buffer before any later store may issue. Used to publish a buffer whose writes must be visible to consumers in order.
      • lfence — load fence. All prior loads must complete before any later load issues. Less commonly needed under TSO, but useful as a speculation barrier (Spectre mitigation cases) and on architectures with load-load reordering.
      • mfence — full fence. Drains the entire store buffer and serializes all prior memory operations against all subsequent ones. The hammer.

      What a fence actually does in hardware is force the store buffer to drain (or wait for it to drain) rather than just waiting for Reorder buffer commit. This is the distinction worth keeping straight: ROB commit makes a store architecturally real but invisible to other cores; the fence is what makes it visible. The cost is whatever latency the cache hierarchy adds — fences are expensive precisely because they undo the latency-hiding the Store buffer exists to provide.

      Under Sequential consistency every memory operation behaves as if a full fence sat on either side of it, which is why SC is fast to reason about and slow to implement. Real-world consistency models pick relaxations and require fences only at the points the programmer marks as ordering-sensitive — acquire and release Half fences (see Release consistency) are the common compromise.

      Remark. A fence does not flush caches or invalidate anything — coherence traffic does that. A fence only constrains the issue order observed by the rest of the system.

        Data race [P0NQ]

        • May 12, 2026
        • Kellen Kanarios

        Two accesses to the same memory location, from different threads, where at least one is a write and there is no happens-before relationship between them. The happens-before relation is what synchronization primitives like Locks and atomics induce — when one access happens before another, they are not racing.

        Remark. Competing for the same lock is not a data race. Whichever thread acquires first happens before the next; the access inside each critical section is sequentially ordered.

        Data-race freedom (DRF) is the implicit contract behind every relaxed memory model: if your program has no data races, Relaxed consistency hardware behaves as if it were Sequential consistency. Conversely, in the presence of a data race, the result is undefined by the language and the model.

        Problem. Writing data-race-free programs is hard. Treating every shared access as a synchronization point destroys performance.

        SC-preserving compilers approach: forbid the compiler from reordering shared accesses, leaving the hardware free to do TSO-style reordering. The optimization most disrupted is common-subexpression elimination across shared accesses — solved by inserting a "monitor" check on the coherence bit between expressions.

        Hardware approach. Treat all pages as local/safe by default and extend page protection bits to the thread level: a non-owning thread accessing the page triggers a fault, exposing the data race at runtime.

        Sequential consistency [OW4D]

        • May 12, 2026
        • Kellen Kanarios

        The strongest practical memory model. The result of any execution is the same as if the operations of all threads were executed in some single sequential order, and the operations of each thread appear in that order in the same sequence as the program specifies.

        Equivalent operational definition: imagine a single switch that, at each step, selects one thread to run its next operation. Any valid execution is the result of some sequence of such selections. There is no reordering across threads, no reordering within a thread, and no buffering that becomes visible to other threads.

        Sequential consistency is what most programmers reason about by default — and what real hardware does not provide, because it forbids almost every optimization that makes a CPU fast. Relaxed consistency is what we get in exchange.

        Remark. SC does not provide guarantees between threads beyond ordering. Two threads racing on the same variable are still racing — SC just promises a consistent global timeline.
        Remark. Enforcing SC purely in hardware requires loads to wait for the Store buffer to drain and stores to retire strictly in order. Both kill performance. SC-preserving compilers try to push the constraint to compile time by refusing to reorder shared accesses.
