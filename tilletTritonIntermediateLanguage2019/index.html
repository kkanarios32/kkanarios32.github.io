    MLIR [IR7B]

    • https://www.lei.chat/posts/mlir-vector-dialect-and-patterns/

    Dialects. This is effectively a syntax. For your given DSL, you must write each of the operations in this MLIR dialect in order to inherit the benefits of the MLIR ecosystem.

    The whole structured CodeGen paradigm in MLIR prefers co-designing op semantics and transformations and let ops structurally encode and guarantee certain properties to simplify analyses and transformations.
    — Lei Zhang

    The general paradigm. It seems that when you are writing a DSL backed by MLIR that you

    1. Write an MLIR dialect for your DSL
    2. You can write custom compiler passes for your MLIR dialect.
      • This is what is done in Triton.
      • Triton has two custom MLIR dialects: ttir and ttgir.
        • The ttir dialect is the backend agnostic IR.
        • The ttgir dialect is the Triton GPU IR.
    3. Write a lowering pass from your MLIR dialect to the MLIR dialect of an MLIR with codegen backing i.e.
      • MLIR has an LLVM dialect
      • Alternatively, XLA ingests StableHLO which is an MLIR dialect. In theory, this means that if your DSL is similar to that of a JAX / LinAlg dialect, then you can get a lot of the optimizations Google has come up with for free.
        • However, XLA works by translating StableHLO to HLO, which is then optimized by XLA. HLO is NOT an MLIR dialect and therefore in order to use XLA your backend needs to have an appropriate conversion from HLO to an executable on their backend.
        • This is what is done for trainium. Namely, the neuronx-cc compiler performs the translation from optimized HLO to optimized executable code on their ASIC.
      • Triton and the GPU backend for XLA are the LLVM + NVVM MLIR, which is the MLIR dialect for LLVM with added GPU instructions.
