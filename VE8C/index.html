Limited Direct Execution [VE8C]

  • July 19, 2025
  • Kellen Kanarios

The OS lets user code run directly on the CPU for speed, but with hardware-enforced limits that prevent a buggy or malicious program from compromising the rest of the system. Two questions motivate the mechanism:

Question.
  1. How do we make sure a program doesn't do what we don't want it to do?
  2. How do we enable time-sharing?

The first question is answered by hardware User mode vs kernel mode enforced via the Trap table, with a controlled Syscall mechanism as the only path from one to the other. The second is answered by periodic timer interrupts that trap to the OS, which performs a Context switch and runs the scheduler.

The whole stack is bootstrapped during the Boot sequence, when the kernel installs the trap table and starts the timer before handing control to user space.

1. User mode vs kernel mode [QUQC]

  • May 12, 2026

Two privilege levels supported by the CPU. User mode restricts the running process to a limited set of instructions and a limited view of memory — usually just its own address space. Kernel mode runs the OS with full access to all instructions and all of physical memory.

Switching between modes is mediated by hardware via a Trap table populated at boot. The OS sets up the table; the CPU enforces it. A process cannot promote itself to kernel mode arbitrarily — it must invoke a Syscall mechanism, which traps to a fixed entry point in the OS.

Remark. Kernel mode is dangerous because the address space is effectively the entire machine. The mode bit + the trap table together are what makes it safe to time-share a CPU between mutually distrusting programs.

2. Trap table [0NVY]

  • May 12, 2026

A hardware-readable table installed by the OS at boot that maps each kind of exceptional control-flow event (syscall, interrupt, page fault, illegal instruction, ...) to the address of an OS handler. When the CPU takes a trap, it consults this table to decide where to jump, switches into kernel mode, and runs the handler.

The table is what makes it safe to expose privileged operations to user mode: user code never gets to pick the destination of a mode-switch, only the trap number. The OS controls every entry point, the CPU enforces the indirection.

Remark. Installing the trap table is itself a privileged operation. It happens once, early in boot, before any user process exists.

3. Syscall mechanism [A5S6]

  • May 12, 2026

A syscall is a user-mode-initiated request for a privileged service from the kernel. Because user code cannot jump into kernel code directly, the mechanism is built around a single trap instruction plus a pre-agreed register convention identifying which service to perform.

The end-to-end sequence:

  1. User code places the syscall number in a pre-defined register and invokes the trap instruction.
  2. Hardware saves enough state to resume the process and switches to kernel mode.
  3. Hardware jumps to the OS-installed handler via the Trap table.
  4. The handler reads the syscall number from the agreed register and dispatches to the corresponding OS routine.
  5. The OS validates the request, performs the work, and arranges return values per the calling convention.
  6. The OS restores user state and returns the process to user mode.
Remark. Because the register convention is fixed by ABI, the user-mode side of a syscall is typically hand-written in assembly within libc.
Remark. Interrupts work via a similar trap-handler mechanism, but are initiated by hardware (timer, device) rather than user code.
Remark. The kernel must validate any pointers or arguments the user supplies — user code cannot be trusted not to ask the kernel to read kernel memory on its behalf.

4. Context switch [OKCI]

  • May 12, 2026

The operation of saving the full execution state of one process (or thread) and restoring another's so the CPU can resume it as if it had never been interrupted. On a trap into the kernel, the hardware stashes the user-visible state (PC, condition codes, stack pointer) into a known location, and the OS saves the remaining general-purpose registers as part of the switching code.

Two flavors of state are involved:

  1. User registers: the current program's context (general-purpose registers, stack vars).
  2. Kernel registers: meta-information used to resume the trap correctly (PC at trap, syscall number, etc.).

Context switches are not free — they incur the cost of cache and TLB pollution in addition to the direct cost of saving/restoring state. This is the cost amortized by larger scheduling quanta and a recurring concern for any Concurrency system.

5. Boot sequence [XSFO]

  • May 12, 2026

The fixed series of steps a machine performs from power-on until a usable operating system is running. Each stage runs from a known location and hands off control to the next.

  1. Power signal resets the CPU to a known state.
  2. The CPU fetches from a hard-wired address — the BIOS (or UEFI firmware). The BIOS runs basic hardware checks (POST).
  3. The BIOS loads and jumps to the bootloader (e.g. GRUB) from a known location on disk.
  4. The bootloader places the kernel image at the correct address and jumps to its entry point.
  5. The kernel performs early setup: installs the Trap table (a special privileged instruction), starts the timer interrupt, sets up memory management, and finally enters user space.
Remark. The hardware is hard-wired only to find the BIOS. Everything past that is software the user can replace, which is why dual-booting and custom bootloaders work.

    Processes [SE66]

    The process abstraction, the syscalls for working with processes on Unix, the hardware-software contract that makes time-sharing safe, and the scheduling policies that decide who runs when.

    The abstraction

    • August 16, 2025

    Process [FCHM]

    • May 12, 2026

    A running program, plus all the state the OS needs to pause it and resume it later. A process consists of:

    1. Memory: an Address space holding the program's code, stack, and heap.
    2. Registers: general-purpose registers carrying current computation, plus special-purpose registers:
      • Program counter (PC): the next instruction to run.
      • Stack pointer and frame pointer: where the process sits on its stack.

    A process moves between three states:

    • running: currently on a CPU.
    • ready: runnable, waiting for the Scheduling to give it a CPU.
    • blocked: waiting on I/O or another event, not runnable.
    Remark. Threads (see Concurrency Intro) share an address space within a process but each have their own PC, register set, and stack.

    Process API [JJ45]

    • May 12, 2026

    The minimum operations any OS must expose for working with Processes:

    1. Create: load a program from disk, set up its address space (stack, heap, code), and jump to main. The OS also initializes argc/argv.
    2. Destroy: kill a misbehaving or unwanted process.
    3. Wait: block until another process completes — useful for ordering dependent work.
    4. Miscellaneous control: e.g. suspend / resume.
    5. Status: report on running processes.
    Remark. The stack is allocated up front. The heap starts small and grows via malloc, which calls into the OS (brk/sbrk) when it runs out of headroom.

    On Unix, the concrete realization of this API uses fork(), wait(), and exec() — covered separately.

    Process control block [JE53]

    • May 12, 2026

    The kernel-side data structure holding all per-process state. There is one PCB per Process, threaded together into the process list. The PCB is what the OS reads and writes during a Context switch: the saved register set lives here, as does the process state, the open files, the parent pointer, and so on.

    An example from xv6:

    // the registers xv6 will save and restore
// to stop and subsequently restart a process
struct context {
  int eip;
  int esp;
  int ebx;
  int ecx;
  int edx;
  int esi;
  int edi;
  int ebp;
};
// the different states a process can be in
enum proc_state { UNUSED, EMBRYO, SLEEPING, RUNNABLE, RUNNING, ZOMBIE };
// the information xv6 tracks about each process
// including its register context and state
struct proc {
  char *mem;                 // Start of process memory
  uint sz;                   // Size of process memory
  char *kstack;              // Bottom of kernel stack for this process
  enum proc_state state;     // Process state
  int pid;                   // Process ID
  struct proc *parent;       // Parent process
  void *chan;                // If non-zero, sleeping on chan
  int killed;                // If non-zero, have been killed
  struct file *ofile[NOFILE]; // Open files
  struct inode *cwd;         // Current directory
  struct context context;    // Switch here to run process
  struct trapframe *tf;      // Trap frame for the current interrupt
};
    Remark. Threads need an analogous thread control block; see Concurrency Intro.

    Unix process syscalls

    • August 16, 2025

    fork() [O1S9]

    • May 12, 2026

    The Unix system call that duplicates the calling Process. After fork() returns, two processes exist where there was one. The child gets its own copy of:

    1. Address space.
    2. General-purpose registers.
    3. Program counter — the child returns from fork() at the same instruction the parent does.

    The two processes are distinguished by fork()'s return value:

    int main()
  int rc = fork();
  if (rc < 0)
    printf("error with fork");
  else if (rc == 0)
    printf("in child");
  else
    printf("in parent");        // rc holds child PID
    Remark. Which process is scheduled first after fork() is non-deterministic. Use wait() if you need ordering.
    Remark. The copy is logical, not physical: real implementations use Copy-on-write so the duplicated address space only allocates fresh pages when one side writes to them.

    wait() [J6VZ]

    • May 12, 2026

    Blocks the calling Process until one of its children terminates. The OS marks the child's PCB ZOMBIE on exit, keeping enough state (exit status, resource usage) for the parent to retrieve via wait(); only when the parent collects it is the child fully reaped.

    wait() returns the PID of the child that died. Variant waitpid(pid, ...) blocks on a specific child — useful when a parent has multiple children and only one is interesting.

    Used to enforce ordering between parent and child code paths after fork(), or to know when an exec()'d program finishes.

    Remark. Calling wait() with no children to wait on returns -1.

    exec() [YHJR]

    • May 12, 2026

    Replaces the calling Process's program with a new executable. exec() does not return on success — the original code, data, stack, and heap are discarded and the new program runs in their place. The PID is unchanged.

    What exec() does internally:

    1. Loads the new executable's code and static data, overwriting the current.
    2. Re-initializes stack and heap.
    3. Jumps to the new program's entry point with arguments passed as argv.

    The canonical Unix pattern is fork() then exec() — the shell uses this to launch a command without becoming that command:

    int main()
  while (true) {
    command = getcommand();
    int rc = fork();
    if (rc < 0)
      printf("error with fork");
    else if (rc == 0)
      exec(command);            // child becomes the command
    else
      int rc = wait(NULL);      // parent waits for it
  }

    Redirection is implemented by opening a file in the child between fork() and exec(), attached to STDOUT_FILENO:

    if (rc == 0)
  close(STDOUT_FILENO);
  open("output.txt");
  exec(command);
    Remark. The many variants — execl, execle, execlp, execv, execvp, execvP — differ in how they take arguments (list vs. vector), whether they use the PATH environment variable, and whether they take an explicit environment.

    Unix signals [9V3H]

    • May 12, 2026

    An asynchronous notification delivered to a Process. A process sends a signal with kill(pid, signum) (the name is misleading — most signals don't kill); the recipient can install a handler with signal(signum, handler) to "catch" it, otherwise the default action runs (often termination).

    Signals are the OS's mechanism for events the recipient didn't ask to be told about: SIGINT from Ctrl-C, SIGSEGV from a Page fault on an invalid page, SIGCHLD when a child exits, SIGTERM from an administrative kill.

    Who can signal whom depends on the user owning the process — a user can signal their own processes; the superuser can signal any process.

    Scheduling [B8GC]

    • July 26, 2025

    The scheduler decides which runnable Process to put on the CPU at each moment. There is no single right answer because the metrics conflict:

    1. Turnaround time: time from arrival to completion, \(T_{\text {completion}} - T_{\text {arrival}}\). Minimized by running short jobs first.
    2. Response time: time from arrival to first scheduling. Minimized by frequent rotation.
    3. Fairness: every job makes progress; nobody starves.

    FIFO scheduling [JA6Z]

    • May 12, 2026

    First-in, first-out. Jobs are run to completion in the order they arrive. Trivially fair under any reasonable notion of fairness, but suffers from the convoy effect: a single long-running job at the head delays every shorter job behind it, inflating average turnaround time.

    Problem. If the first arriving job is much longer than the rest, all subsequent jobs wait at least its full duration before getting any CPU.

    SJF scheduling [VJ4F]

    • May 12, 2026

    Shortest job first. Run the job with the smallest total work first, run-to-completion. If all jobs arrive at the same time, SJF provably minimizes average turnaround time.

    Problem. Real workloads don't all arrive at \(t = 0\). A long job that arrives slightly before a short one still gets to monopolize the CPU until it finishes — SJF degenerates to FIFO scheduling under sequential arrivals.

    Removing the run-to-completion constraint and allowing preemption gives STCF scheduling.

    STCF scheduling [7R07]

    • May 12, 2026

    Shortest time-to-completion first. A preemptive refinement of SJF scheduling: whenever the scheduler is invoked, it picks the job with the least remaining work. A short job arriving mid-execution can preempt a long one.

    STCF is provably optimal for minimizing average turnaround time over all preemptive schedulers.

    Problem. Long jobs starve in response time. The lowest-priority work might not get a CPU slice for a very long time, which is unacceptable for interactive workloads.

    Round-robin scheduling [NEL8]

    • May 12, 2026

    Cycle through all runnable jobs, giving each a fixed time slice — the stride, time slice, or quantum length — before preempting and moving to the next. The best you can do at minimizing response time: every job sees the CPU within at most \(N\) quanta where \(N\) is the runnable-job count.

    The quantum length is a tradeoff:

    • Short quantum: better response time, more frequent context switches, worse cache locality.
    • Long quantum: amortizes context-switch cost, worse response time.

    Worst-case response time for \(N\) jobs at quantum \(S\) is \(S(N-1)\).

    Remark. Round-robin minimizes response time but is poor for turnaround time relative to STCF scheduling. A scheduler that wants both has to combine ideas — see Multi-level feedback queue.

    Multi-level feedback queue [4WG5]

    • May 12, 2026

    A practical scheduler that approximates STCF scheduling's turnaround properties and Round-robin scheduling's response properties without requiring foreknowledge of job lengths. Maintain multiple priority queues; within a queue, schedule round-robin. Never service a lower-priority queue while a higher-priority queue has a runnable job.

    The priority of a job is not assigned by the user — it is learned from behavior:

    • New jobs enter at the top priority.
    • Each priority level has an allotment — the total CPU time a job may consume before being demoted.
    • Once a job exhausts its allotment at a level, it drops to the next level.

    Short jobs that finish quickly never leave the top queue (good turnaround). Interactive jobs that yield often stay at the top (good response time). CPU-bound jobs sink to the bottom and get scheduled when nothing better is runnable.

    Problem. Gaming the allotment: a job can issue I/O just before its allotment expires to avoid demotion, since I/O time doesn't count.
    Solution. Charge the job's allotment for total time at the priority level, including time spent blocked.
    Problem. Starvation of low-priority queues if higher queues never empty.
    Solution. Periodically reset all jobs to the top priority. Cheap and effective.

    Lottery scheduling [AM8U]

    • May 12, 2026

    A proportional-share scheduler. Each runnable job holds some number of tickets, drawn from a fixed total. On each scheduling decision, the scheduler picks a random ticket and runs the job that owns it. Over time, a job receives CPU time proportional to its share of the tickets.

    The implementation is trivial: one random number per decision, plus a sum table. No global state about historical CPU usage is needed.

    Mechanisms layered on top:

    • Ticket currency — give each user a fixed number of tickets to split among their processes however they like.
    • Ticket transfer — a client process can hand tickets to a server to boost its priority while servicing the request.
    • Ticket inflation — a process can grow its share temporarily, with bounds set by trust.
    Remark. The unsolved problem is how to assign tickets in the first place. The mechanism is simple; the policy is not.

    A deterministic version of the same idea is Stride scheduling, which avoids the variance of random sampling at the cost of needing global state.

    Stride scheduling [HCJ7]

    • May 12, 2026

    A deterministic proportional-share scheduler. Each job has a stride inversely proportional to its ticket count (the more tickets, the smaller the stride) and a running pass counter that accumulates its stride every time it's scheduled.

    The decision is:

    1. Pick the job with the lowest pass.
    2. Run it for one quantum.
    3. Increment its pass by its stride.

    A job with twice as many tickets has half the stride, so its pass grows half as fast, so it gets scheduled twice as often. Same long-run behavior as Lottery scheduling, but with zero variance.

    Problem. New jobs need an initial pass value. Setting it to 0 lets them dominate; setting it to the global minimum lets them merge in cleanly but requires tracking that minimum.
    Remark. Each context switch is not free. Choosing a longer time slice amortizes that cost at the price of response time.
    Remark. I/O-bound jobs are commonly handled by splitting a job into its non-I/O segments and scheduling them individually — turning one long-but-mostly-blocked job into many short, runnable ones.

      Remark. Round-robin minimizes response time but is poor for turnaround time relative to STCF scheduling. A scheduler that wants both has to combine ideas — see Multi-level feedback queue.

      Multi-level feedback queue [4WG5]

      • May 12, 2026

      A practical scheduler that approximates STCF scheduling's turnaround properties and Round-robin scheduling's response properties without requiring foreknowledge of job lengths. Maintain multiple priority queues; within a queue, schedule round-robin. Never service a lower-priority queue while a higher-priority queue has a runnable job.

      The priority of a job is not assigned by the user — it is learned from behavior:

      • New jobs enter at the top priority.
      • Each priority level has an allotment — the total CPU time a job may consume before being demoted.
      • Once a job exhausts its allotment at a level, it drops to the next level.

      Short jobs that finish quickly never leave the top queue (good turnaround). Interactive jobs that yield often stay at the top (good response time). CPU-bound jobs sink to the bottom and get scheduled when nothing better is runnable.

      Problem. Gaming the allotment: a job can issue I/O just before its allotment expires to avoid demotion, since I/O time doesn't count.
      Solution. Charge the job's allotment for total time at the priority level, including time spent blocked.
      Problem. Starvation of low-priority queues if higher queues never empty.
      Solution. Periodically reset all jobs to the top priority. Cheap and effective.

      Lottery scheduling [AM8U]

      • May 12, 2026

      A proportional-share scheduler. Each runnable job holds some number of tickets, drawn from a fixed total. On each scheduling decision, the scheduler picks a random ticket and runs the job that owns it. Over time, a job receives CPU time proportional to its share of the tickets.

      The implementation is trivial: one random number per decision, plus a sum table. No global state about historical CPU usage is needed.

      Mechanisms layered on top:

      • Ticket currency — give each user a fixed number of tickets to split among their processes however they like.
      • Ticket transfer — a client process can hand tickets to a server to boost its priority while servicing the request.
      • Ticket inflation — a process can grow its share temporarily, with bounds set by trust.
      Remark. The unsolved problem is how to assign tickets in the first place. The mechanism is simple; the policy is not.

      A deterministic version of the same idea is Stride scheduling, which avoids the variance of random sampling at the cost of needing global state.

      Stride scheduling [HCJ7]

      • May 12, 2026

      A deterministic proportional-share scheduler. Each job has a stride inversely proportional to its ticket count (the more tickets, the smaller the stride) and a running pass counter that accumulates its stride every time it's scheduled.

      The decision is:

      1. Pick the job with the lowest pass.
      2. Run it for one quantum.
      3. Increment its pass by its stride.

      A job with twice as many tickets has half the stride, so its pass grows half as fast, so it gets scheduled twice as often. Same long-run behavior as Lottery scheduling, but with zero variance.

      Problem. New jobs need an initial pass value. Setting it to 0 lets them dominate; setting it to the global minimum lets them merge in cleanly but requires tracking that minimum.
      Remark. Each context switch is not free. Choosing a longer time slice amortizes that cost at the price of response time.
      Remark. I/O-bound jobs are commonly handled by splitting a job into its non-I/O segments and scheduling them individually — turning one long-but-mostly-blocked job into many short, runnable ones.
