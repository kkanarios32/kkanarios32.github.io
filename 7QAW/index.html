An operand collector serializes register reads to dodge register-file bank conflicts [7QAW]

  • July 11, 2026
  • Kellen Kanarios

A warp's 32 threads each read 2–4 registers per instruction, but the register file is banked and cannot serve two reads to the same bank in one cycle. An operand collector absorbs this: it gathers a warp's register reads across multiple cycles and routes the collected values to the lanes once all are available, turning what would be a stall into a few cycles of latency the scheduler hides anyway.

These register-file bank conflicts are invisible to the programmer — unlike shared-memory bank conflicts, which the programmer controls directly — but they shrink as register pressure drops. Fewer registers per thread means fewer simultaneous reads competing for banks, one more reason low register pressure raises throughput.

    Warp scheduling hides memory latency by switching among resident warps [1PWL]

    • July 11, 2026
    • Kellen Kanarios

    A GPU keeps many warps resident on each Streaming Multiprocessor (SM) and switches between them on every stall. When a warp issues a memory op that misses cache it is dead for hundreds of cycles — but with dozens of resident warps the scheduler always has a runnable one, so aggregate throughput stays high even though single-warp latency is awful. This is the Thread-level parallelism design point cashed out on the GPU: spend silicon on resident warp state rather than on latency-reduction machinery. The switch is free because each SM's register file is large enough to hold every resident warp's registers at once, so a context switch is just changing which warp the scheduler picks — not saving state to memory the way a CPU context switch must.

    Two things bound how well this works. Occupancy — how many warps are resident — sets the ceiling on latency hiding: with too few warps the scheduler runs out of runnable candidates. And the scheduler itself must stay cheap enough to choose a warp every cycle, which it does with a two-loop scoreboard check, marshalling each issued instruction's register reads through an operand collector.

      Bank conflicts serialize shared-memory accesses to the same bank [OGCRU]

      • May 25, 2026
      • Kellen Kanarios

      Shared memory is divided into banks (typically 32). Each bank can service one access per cycle. If two threads in a warp access different addresses in the same bank, the accesses serialize — a bank conflict.

      Three behaviors when warp threads access shared memory:

      • Different banks: full parallelism. No conflict.
      • Same bank, same word: hardware broadcasts the word to all threads. No conflict.
      • Same bank, different words: serialize.
      Question. How to avoid bank conflicts?
      Answer. Change the access pattern so threads hit different banks (often by transposing or by adding padding to break the stride that causes the conflict).

      Warp [X9VZ]

      • May 12, 2026

      A warp is the unit of scheduling on an NVIDIA GPU: a group of ~32 threads that issue one instruction per cycle in lockstep. Each functional unit in the SM has as many lanes as a warp has threads, so a single fetched instruction fans out across all 32 lanes at once. The warp — not the thread — is what the hardware fetches, schedules, and tracks; the per-thread SIMT programming model is an illusion built on top of it.

      Because the warp is the atom of execution, the two facts that dominate GPU performance are both properties of the warp: it hides memory latency by switching among resident warps instead of stalling, and it serializes both sides of a branch when its threads diverge.

      Register file [ETWN]

      • May 12, 2026

      The CPU's nearest tier of storage — the first NUMA level. Registers sit physically next to the ALUs and are addressed by name in the instruction encoding rather than by memory address. Their count is limited by the chip's surface area, which is why main memory exists at all.

      The register file is the structure that holds the architecturally-visible registers. To support a superscalar machine, the file needs enough read and write ports to feed every ALU simultaneously: for an \(n\)-ALU chip with two operands per op, that's \(2n\) read ports and \(n\) write ports.

      Problem. Die area for a register file scales quadratically in the number of ports.
      Solution. Split it. Different execution units (integer, floating-point, vector) get their own smaller register files. Fewer ports per file, faster access, less area.

      Beyond the architecturally-named registers, modern chips keep many extra rename registers used by register renaming to break false data dependencies.

      Remark. Instructions distinguish "immediate types" from "register types" so the decoder knows whether a field encodes a register name or a literal value.
