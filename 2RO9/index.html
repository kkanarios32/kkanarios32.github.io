    Register renaming breaks false dependencies [5CT9]

    • May 12, 2026
    • with contributions from Kellen Kanarios

    Register renaming is the data hazard mitigation that makes out-of-order execution viable: the hardware dynamically remaps each architectural-register write to a fresh physical register from a larger pool, so two instructions that reuse the same architectural name for independent values never collide. The false data dependency — same architectural register, unrelated values — simply disappears, because each write lands in a different physical register.

    Example. Consider: 
    add r1, r2, r3      // r2 := r1 + r3
add r4, r5, r2      // r2 := r4 + r5   (reuses r2 for an independent value)
    The second add only depends on r2 as a destination, not as a source. Without renaming, it would have to wait for the first instruction's writeback to avoid clobbering. With renaming, the second write goes to a fresh physical register, and the two instructions can proceed in parallel.

    The programming model exposes only the architecturally-named registers; the Register file hides a larger pool of rename registers underneath. This is one of the central tricks that makes out-of-order execution viable.

      Rebuilding My (Neo)Vim Config From Scratch [005F]

      • March 4, 2025
      • Kellen Kanarios

      I used LazyVim for a long time, but I kept hitting a wall: understanding how LazyVim did something was consistently harder than if I had just written the thing myself. So I gave myself one day to rebuild from scratch, with the modest goal of first-class support for \(\TeX \), Python, Forester, and C/C++. That one-day sprint is long past, and the config has kept evolving since — this note describes where it landed, not the frantic first pass. The through-line is that almost everything now leans on Neovim's own primitives instead of a framework, which is exactly the transparency I was missing.

      Remark. The whole config lives in my dotfiles. It is a thin init.lua that loads four modules — config.lazy, config.options, config.keymaps, and core.lsp — with the actual plugin specs under lua/plugins/ and per-server LSP tables under lsp/.

      Sane defaults

      • March 4, 2025
      • Kellen Kanarios

      To my surprise, a lot of the behavior I had taken for granted turned out to be options LazyVim set internally. I was greeted on first launch by 8-space indents and a clipboard that would not survive a trip between terminal instances. So the first job was archaeology: I dug through LazyVim for every option I actually liked and copied them into config/options.lua. System clipboard, sensible folding via Treesitter (foldexpr = v:lua.vim.treesitter.foldexpr()), a global statusline, conceallevel for markup — the unglamorous foundation everything else sits on.

      Plugin management

      • March 4, 2025
      • Kellen Kanarios

      The plugin manager is lazy.nvim, the de facto standard, and it genuinely just works. The bootstrap snippet from their docs clones it on first run, and after that every file under lua/plugins/ that returns a spec table is imported automatically. No central manifest to keep in sync — a new file is a new plugin.

      Native LSP, no framework

      • March 4, 2025
      • Kellen Kanarios

      This is where the config diverged most sharply from where it started. The first pass used mason.nvim, mason-lspconfig, and nvim-lspconfig — the standard stack. All three are now gone. Neovim 0.11 ships a native LSP configuration API: vim.lsp.config defines a server and vim.lsp.enable turns it on, and any table returned from lsp/<name>.lua on the runtime path is picked up as that server's config. So core.lsp is just a list —

      vim.lsp.enable({ "forester", "texlab", "clangd", "basedpyright", "lua_ls", "ruff" })

      — and each server gets a plain Lua file in lsp/. No package manager, no bridge plugin translating between two config schemas. I install the servers themselves through my system package manager and let Neovim wire them up. It is dramatically less machinery for the same result, and when something breaks there is exactly one place to look.

      The Python setup is the clearest upgrade. I dropped pyright for basedpyright and split responsibilities: basedpyright does type checking and inlay hints, while ruff runs as its own language server handling import organization and diagnostics. In the LazyVim days I fought constantly with import resolution, hand-writing a function to locate each project's virtualenv and pin pythonPath to its interpreter. basedpyright resolves environments well enough on its own that the hack is gone. clangd, texlab, and lua_ls round out the set, each configured in its own file — texlab in particular carries a fair amount of custom command wiring for building, forward search into Zathura, and environment manipulation.

      A shared LspAttach autocommand in core.lsp binds the buffer-local LSP keymaps once, for every server: go-to-definition and friends route through fzf-lua pickers, and document highlighting, inlay-hint toggling, and native completion get enabled per-buffer based on what each client actually supports.

      Forester finally has a real language server

      • March 4, 2025
      • Kellen Kanarios

      The original post described an ugly hack to get Forester completion. The forester.nvim plugin exposed a completion source, but only for nvim-cmp; blink.cmp (which I use — see below) needs a blink.compat shim, and that shim expected a plugin to be a source rather than to bury one inside a larger feature set. Getting reference completion working meant reading blink.compat internals and registering the source by hand.

      None of that survives, because Forester now ships an actual language server. My lsp/forester.lua just launches forester lsp and hands it the forest root. The one wrinkle worth recording: forester locates the forest by its process working directory and ignores the LSP rootUri, and Neovim does not spawn the server in root_dir by default. So the config resolves the nearest forest.toml and starts the RPC process with an explicit cwd — otherwise the server exits with status 1 whenever Neovim's cwd has no forest.toml (e.g. when I open a tree from $HOME). Reference completion — indispensable for writing this very blog — now flows through the standard LSP path with no shim in sight.

      Completion

      • March 4, 2025
      • Kellen Kanarios

      Completion is blink.cmp, the successor that finally displaced nvim-cmp. Its sources are lsp, path, snippets, and omni, with LuaSnip as the snippet engine. Because I lean on native LSP, the on_attach handler calls vim.lsp.completion.enable with autotrigger, so the server's completions feed the picker directly.

      Layered on top is supermaven-nvim for AI ghost-text suggestions — accept-suggestion, accept-word, and clear bound to <C-l>, <C-j>, and <C-]>. It fills the role Copilot-style tools play elsewhere without getting in the way of the structured LSP and snippet completions.

      Snippets

      • March 4, 2025
      • Kellen Kanarios

      Going all the way back to the Gilles Castel blog post, I have always wanted snippets that auto-expand. Under LazyVim I had settled for friendly-snippets on native Neovim snippets; rebuilding from scratch, I refused to compromise. The engine is LuaSnip, which blink.cmp supports natively, and I have to plug the luasnip-latex-snippets repo that ports the original UltiSnips math snippets over. My own snippets live in a small local plugin, custom-luasnip-snippets.nvim, vendored under lua/local/.

      The trick I had wanted for years and can finally do: load LaTeX math snippets while inside math environments in Forester. The snippet plugin runs with use_treesitter = true, so it inspects the Forester Treesitter grammar, finds the node types for math environments, and only expands the math snippets when the cursor is actually inside one. Writing \(\TeX \) inside a .tree file feels exactly like writing it in a .tex file.

      Formatting and linting

      • March 4, 2025
      • Kellen Kanarios

      Formatting runs through conform.nvim on save. Python is handled by ruff_format followed by pyink, Lua by stylua — this replaced the old black + isort pairing, folding most of it into the same ruff that already runs as a language server. Formatting falls back to the LSP formatter when no dedicated tool is configured for a filetype.

      Fuzzy finding and forest integration

      • March 4, 2025
      • Kellen Kanarios

      The fuzzy finder is fzf-lua, which replaced Telescope. File and grep pickers are scoped to a detected project root — a small utils.root module (lifted from LazyVim) walks LSP workspace folders, .git, and cwd to find it — with a dedicated tex override that restricts live-grep to \(\TeX \) source files.

      The piece I am most fond of is forester-new, a local plugin that wires the editor into the forest's own note pipeline. It shells out to scripts/new so every tree gets the standard frontmatter and routing, then commands like :ForesterNew, :ForesterStub, :ForesterCitek, and :ForesterTransient handle the follow-through: creating a tree and inserting a \transclude, stubbing a concept and dropping a [[id]] link at the cursor, or — in visual mode — lifting a selection into a new tree and replacing it in place with the transclude. Stub-and-link without ever leaving the sentence, which is exactly the friction the forest workflow is supposed to remove.

      Anki

      • March 4, 2025
      • Kellen Kanarios

      I originally built a whole in-editor Anki workflow on top of AnkiConnect and a vendored anki.nvim. That has since moved out of the editor and into the forest itself: cards are just \taxon{Card} trees, and scripts/anki-sync mirrors them into Anki — creating the note on first sync and updating in place thereafter. Writing a card is now the same as writing any other tree, snippets and all, and the sync is one command away. This is an example of a card.
