Definition. GPU register scoreboard [XPAA]

  • May 13, 2026
  • Kellen Kanarios

The readiness-tracking structure that lets a GPU issue instructions out of program order without any of the CAM machinery a CPU needs. A scoreboard is a small table — typically one busy bit per architectural register — that records whether some in-flight instruction is still computing a value that something else might want to read.

The protocol is trivial. When an instruction issues, it sets the busy bit on its destination register. When it writes back, it clears the bit. Before any instruction can read its sources, the hardware checks the corresponding bits: if any source is busy, the instruction (or its entire warp) waits. The check is a single table lookup per source, so the per-instruction cost is \(O(1)\) regardless of how many other instructions are in flight.

This is enough because GPUs lean on Thread-level parallelism rather than Instruction-level parallelism. When a warp's next instruction reads a busy register, the GPU architecture scheduler does not stall — it picks a different ready warp from the dozens of resident warps and runs that one instead. The scoreboard's job is only to answer "is this warp ready? yes/no," which is far cheaper than the broadcast/compare a CPU Issue queue wakeup-select does to find dependency satisfaction in a single thread's instruction window.

The size difference falls out of this. CPU issue queues hold tens of entries; GPU instruction windows can hold many warps' worth of in-flight work, because every additional warp costs a row in the scoreboard and a register-file allocation, not a quadratic CAM expansion. See ILP vs TLP for why the design fork sits where it does.

Remark. The scoreboard's simplicity is what lets GPUs spend their transistor budget on ALUs and the Register file instead of control logic. It is one of the most visible places where the Thread-level parallelism design choice cashes out as silicon savings.

Context

    GPU architecture [82U4]

    • May 13, 2026
    • Kellen Kanarios

    A GPU sacrifices single-thread latency for aggregate throughput. The whole machine is built around the Thread-level parallelism design point — keep many threads resident, switch on every stall, spend almost no silicon on per-instruction control. The opposite end of the axis from a CPU:

    CPU GPU
    Caches Large Small (boost memory throughput)
    Control Sophisticated (branch prediction, OoO) Simple (no branch prediction, no forwarding)
    ALUs Powerful Energy-efficient, heavily pipelined
    Optimizes for Single-thread latency Aggregate throughput

    The GPU equivalent of a core is a streaming multiprocessor (SM). Each SM executes the same instruction across many threads in lockstep — the SIMT programming model — and uses a per-register GPU register scoreboard rather than CPU-style wakeup-select to decide which warp can issue next.

    Definition. SIMT [7F19]

    • May 13, 2026
    • Kellen Kanarios

    Single instruction, multiple threads. A GPU streaming multiprocessor issues one instruction per cycle to a group of threads — a warp — but each thread within the group keeps its own program counter and register state. The hardware enforces lockstep execution within the warp while preserving the illusion of independent threads to the programmer.

    SIMT is the GPU's answer to a question the Vector ISA / SIMD world struggled with: how do you get the throughput of wide vector execution without forcing the programmer to manually vectorize? Two mechanisms make this work.

    Warp divergence serializes both sides of a branch [OGCSM]

    • July 6, 2026
    • Kellen Kanarios

    When threads in a warp take different branches of a conditional, the warp does not split into smaller warps. Instead, the active mask selects which lanes execute each branch; lanes on the inactive side run dummy instructions until the warp reconverges at the post-conditional join point. The programmer writes ordinary scalar code with if/else, and the hardware silently masks the inactive lanes.

    This is the chief ergonomic win over SIMD: the compiler does not have to prove that all lanes will take the same path. The cost is that divergent code pays for both sides — a 32-wide warp where 16 threads take the if-branch and 16 take the else-branch executes both branches sequentially with half the lanes idle each time.

    The hardware sequences the two paths with a reconvergence stack — or, on recent GPUs, a barrier-based equivalent that avoids its deadlocks. The standard software mitigation is JKNL: group threads with the same control-flow trajectory into one warp so the branch never diverges in the first place.

    Grouping flexibility

    • May 13, 2026
    • Kellen Kanarios

    Threads are assigned to warps based on warp-ID, but the hardware and compiler are free to regroup. JKNL re-bundles threads with similar control-flow trajectories into the same warp so that future branches stay non-divergent. This is a class of optimization that pure SIMD cannot express — a SIMD ISA fixes the lane assignment at compile time.

    The hardware cost of SIMT relative to SIMD is the per-thread PC and register state. The benefit is that a single binary runs efficiently across a wide range of dynamic control flow, which is why the model has dominated GPU programming since CUDA. The scoreboard mechanism described in GPU register scoreboard operates per-warp rather than per-thread precisely because SIMT treats the warp as the unit of issue.

    Remark. SIMT is sometimes described as "SIMD with a per-lane PC," which captures the implementation but understates the programming-model impact: vectorization becomes a runtime, not a compile-time, decision.

    Warp scheduling hides memory latency by switching among resident warps [1PWL]

    • July 11, 2026
    • Kellen Kanarios

    A GPU keeps many warps resident on each Streaming Multiprocessor (SM) and switches between them on every stall. When a warp issues a memory op that misses cache it is dead for hundreds of cycles — but with dozens of resident warps the scheduler always has a runnable one, so aggregate throughput stays high even though single-warp latency is awful. This is the Thread-level parallelism design point cashed out on the GPU: spend silicon on resident warp state rather than on latency-reduction machinery. The switch is free because each SM's register file is large enough to hold every resident warp's registers at once, so a context switch is just changing which warp the scheduler picks — not saving state to memory the way a CPU context switch must.

    Two things bound how well this works. Occupancy — how many warps are resident — sets the ceiling on latency hiding: with too few warps the scheduler runs out of runnable candidates. And the scheduler itself must stay cheap enough to choose a warp every cycle, which it does with a two-loop scoreboard check, marshalling each issued instruction's register reads through an operand collector.

    Warp [X9VZ]

    • May 12, 2026
    • Kellen Kanarios

    A warp is the unit of scheduling on an NVIDIA GPU: a group of ~32 threads that issue one instruction per cycle in lockstep. Each functional unit in the SM has as many lanes as a warp has threads, so a single fetched instruction fans out across all 32 lanes at once. The warp — not the thread — is what the hardware fetches, schedules, and tracks; the per-thread SIMT programming model is an illusion built on top of it.

    Because the warp is the atom of execution, the two facts that dominate GPU performance are both properties of the warp: it hides memory latency by switching among resident warps instead of stalling, and it serializes both sides of a branch when its threads diverge.

    CUDA Hierarchy [OGCS4]

    • June 1, 2026
    • Kellen Kanarios

    Grid: Top-level define how many blocks are needed to cover the computation across SMs.

    Block: Ensures all warps in a block are resident on the same SM and have the same shared memory.

    Warp: A group of 32 threads that execute the same instruction.

    Thread: A single thread within a warp.

    Shared memory (GPU) [8GZN]

    • May 12, 2026
    • Kellen Kanarios

    Shared memory on a GPU is a small (per-block) on-chip scratchpad with latency comparable to L1. Software-managed — the programmer explicitly stages data through it, the hardware does not cache for them. Used to:

    • Avoid non-coalesced global accesses by reorganizing data layouts on the fly.
    • Reuse loaded data across threads within a block.
    • Communicate between threads within a block.

    Bank conflicts serialize shared-memory accesses to the same bank [OGCRU]

    • May 25, 2026
    • Kellen Kanarios

    Shared memory is divided into banks (typically 32). Each bank can service one access per cycle. If two threads in a warp access different addresses in the same bank, the accesses serialize — a bank conflict.

    Three behaviors when warp threads access shared memory:

    • Different banks: full parallelism. No conflict.
    • Same bank, same word: hardware broadcasts the word to all threads. No conflict.
    • Same bank, different words: serialize.
    Question. How to avoid bank conflicts?
    Answer. Change the access pattern so threads hit different banks (often by transposing or by adding padding to break the stride that causes the conflict).
    Remark. The GPU memory hierarchy is software-managed at L1 (shared memory) and coherent only at L2 — by restricting L1 to local-to-block scope, the architecture sidesteps the cost of a global Cache coherence protocol on a thousand-way-parallel chip.

    Memory coalescing merges a warp's accesses into one transaction per cache line [U1FO]

    • May 12, 2026
    • Kellen Kanarios

    The hardware optimization on GPUs that combines simultaneous memory accesses by threads in a warp into a single transaction — when the accessed addresses fall in the same cache line.

    Global memory latency on a GPU is 400–800 cycles. Coalescing is the primary way to keep the SM fed:

    • If all 32 threads in a warp access addresses within one cache line: one memory transaction, full bandwidth.
    • If addresses span multiple cache lines: the hardware serializes into multiple transactions; bandwidth drops proportionally.
    Remark. Address permutations within a cache line don't hurt utilization — coalescing is about which lines are touched, not which exact bytes.

    Coalescing is why memory layout matters so much for GPU code. The standard idiom is structure-of-arrays (SoA) instead of array-of-structures (AoS): threads accessing the same field of consecutive records hit consecutive bytes, which coalesce; threads accessing the same struct don't.

    Question. Should you always cache on the GPU?
    Answer. No. If you're memory-bound and have no Spatial locality, cache is wasted on data you won't reuse. CUDA exposes the option to disable L1 caching for these cases, which lets misaligned loads transfer only the bytes needed instead of full cache lines.
    Remark. Use GPUs for parallel parts where throughput dominates. Use CPUs for sequential parts where latency dominates — the ILP vs TLP axis is the right frame for deciding which.

    Graphics Processing Cluster (GPC) [OGCS7]

    • June 1, 2026
    • Kellen Kanarios

    A Graphics Processing Cluster (GPC) is a top-level hardware partition inside an NVIDIA GPU. It's one level up from the SM in the chip's hierarchy.

    As of Hopper, SMs located in the same GPC are connected via an SM-to-SM network, enabling DSMEM.

    Cooperative Thread Array (CTA) [W2D5]

    • June 1, 2026

    NVIDIA's hardware-level term for what CUDA programmers call a thread block.

    A CTA is a group of threads that execute on the same Streaming Multiprocessor (SM), can synchronize via barriers (__syncthreads()), and share fast on-chip shared memory (Shared memory (GPU)). Threads within a CTA are scheduled and executed in groups of 32 called warps.

    Backlinks

      A two-loop scheduler makes warp readiness a constant-time scoreboard check [7SGF]

      • July 11, 2026
      • Kellen Kanarios

      Deciding which warp to issue each cycle by scanning every warp's operands against the scoreboard would be prohibitive — 64 warps against 4 operands each is 256 comparisons. A two-loop scheduler makes the readiness check constant-time instead. The first loop selects a warp with room in its instruction buffer, looks up its PC, and fetches; a small per-warp bit-vector of outstanding register dependencies rides alongside the fetched instruction and clears as writes retire. The second loop issues any buffered instruction whose dependency vector is clean.

      Because readiness is a single per-warp bit-vector test rather than a global scan, the scheduler stays cheap even with dozens of resident warps — the design tradeoff that lets a GPU keep far more work in flight than a CPU's CAM-based wakeup-select ever could. It is the scheduling-side counterpart to why the scoreboard replaces a CPU's dependency-tracking hardware.

      ILP vs TLP [RJEB]

      • May 13, 2026
      • Kellen Kanarios

      The two strategies a microarchitecture can use to keep arithmetic units busy in the face of memory latency and dependencies. Instruction-level parallelism mines parallelism from within a single instruction stream; Thread-level parallelism mines it across many. Every modern design is a position on this axis, and the position determines the rest of the microarchitecture.

      The choice is not really a choice — it is forced by the workload. ILP works only when the single thread has independent work to find; once dependency chains and branch mispredictions saturate the issue window, no amount of silicon spent on reorder buffers, wakeup logic, or speculation buys more throughput. TLP works only when many independent threads exist; throwing more warps at a single sequential dependency chain accomplishes nothing.

      The downstream consequences split everything else:

      • Control overhead. ILP machines spend most of their area on out-of-order execution machinery — Reorder buffer, Reservation stations, register renaming, CAM-based wakeup-select. TLP machines spend almost none; the schedule is "pick any ready thread."
      • Caches. ILP needs large caches because a single stalled thread blocks the whole pipeline. TLP can afford small caches because the latency is hidden by switching threads.
      • Register file. ILP keeps one (renamed) register set hot. TLP keeps every resident thread's register set hot simultaneously — GPU register files are an order of magnitude larger than CPU register files for this reason.
      • Readiness tracking. ILP pays O(N²) for CAM-based wakeup-select across N in-flight instructions; TLP pays O(1) per instruction via a per-register scoreboard.

      The slogan: CPUs spend a lot of silicon to find parallelism in one thread; GPUs spend almost none because they have many threads. Everything downstream — pipeline depth, cache hierarchy, scheduling policy, energy per operation — falls out of which side of this axis the chip sits on.

      Definition. Issue queue wakeup-select [UJPD]

      • May 13, 2026
      • Kellen Kanarios

      The mechanism by which a modern out-of-order CPU notices that a previously-stalled instruction now has all its operands ready. The issue queue holds the N in-flight instructions whose operands are not yet available; each entry watches a small set of physical register tags (the operands it is waiting for). When a producer finishes, it broadcasts "tag T is ready" to every entry, and each entry compares that tag against its own waiting tags through a content-addressable memory (CAM). Entries whose last operand arrives become eligible to issue this cycle; a separate select stage picks which of the now-ready entries actually fires.

      The cost is the reason CPU issue queues are small. Each entry runs a CAM lookup against each broadcast tag, so an N-entry queue with N producers per cycle scales as \(O(N^2)\) in comparators, wires, and power. The select stage compounds this — priority encoders across N entries are themselves a quadratic-area structure on the critical timing path. The cumulative wakeup-select latency is what bounds clock frequency once N grows past a few tens.

      This is the structural reason CPU out-of-order windows stay in the tens-to-low-hundreds of entries even as transistor budgets grow. Wider doesn't help: timing closure on the CAM degrades faster than the additional ILP repays. GPUs sidestep the whole problem by leaning on Thread-level parallelism — a cheap per-register scoreboard plus a warp scheduler covers stalls without any broadcast/compare structure.

      Remark. Wakeup-select sits at the boundary between Reservation station (which holds the instruction) and register renaming (which named the tags being broadcast). All three structures together are what people mean by "the out-of-order back-end."

      Remark. SIMT is sometimes described as "SIMD with a per-lane PC," which captures the implementation but understates the programming-model impact: vectorization becomes a runtime, not a compile-time, decision.

      Definition. Thread-level parallelism [G4ML]

      • May 13, 2026
      • Kellen Kanarios

      Parallelism extracted by running multiple independent threads concurrently and switching between them whenever any one of them stalls. Where Instruction-level parallelism mines a single instruction stream for hidden concurrency, TLP gives up on that ambition and instead assumes that enough independent work exists in the program (or workload) that some thread is always ready to make progress.

      TLP makes the per-thread microarchitecture cheap. There is no need for a deep issue window, aggressive speculation, or CAM-based wakeup-select: if the current thread can't proceed, the scheduler picks another. The cost shifts from per-instruction machinery to context-holding — registers, program counters, and stack pointers for every resident thread must live somewhere fast.

      Two dominant implementation styles:

      • Coarse-grained TLP. Conventional multicore CPUs and SMT: a handful of threads share a core, switched on long-latency events or interleaved cycle-by-cycle.
      • Massive TLP. GPUs keep dozens of warps resident per GPU architecture SM and switch on any stall. The GPU register scoreboard is the readiness check that makes this cheap.

      TLP fails when the workload genuinely has only one thread of useful work, or when the threads contend on shared state (locks, Cache coherence traffic). It wins overwhelmingly on data-parallel kernels where the threads are independent by construction.

        GPU architecture [82U4]

        • May 13, 2026
        • Kellen Kanarios

        A GPU sacrifices single-thread latency for aggregate throughput. The whole machine is built around the Thread-level parallelism design point — keep many threads resident, switch on every stall, spend almost no silicon on per-instruction control. The opposite end of the axis from a CPU:

        CPU GPU
        Caches Large Small (boost memory throughput)
        Control Sophisticated (branch prediction, OoO) Simple (no branch prediction, no forwarding)
        ALUs Powerful Energy-efficient, heavily pipelined
        Optimizes for Single-thread latency Aggregate throughput

        The GPU equivalent of a core is a streaming multiprocessor (SM). Each SM executes the same instruction across many threads in lockstep — the SIMT programming model — and uses a per-register GPU register scoreboard rather than CPU-style wakeup-select to decide which warp can issue next.

        Definition. SIMT [7F19]

        • May 13, 2026
        • Kellen Kanarios

        Single instruction, multiple threads. A GPU streaming multiprocessor issues one instruction per cycle to a group of threads — a warp — but each thread within the group keeps its own program counter and register state. The hardware enforces lockstep execution within the warp while preserving the illusion of independent threads to the programmer.

        SIMT is the GPU's answer to a question the Vector ISA / SIMD world struggled with: how do you get the throughput of wide vector execution without forcing the programmer to manually vectorize? Two mechanisms make this work.

        Warp divergence serializes both sides of a branch [OGCSM]

        • July 6, 2026
        • Kellen Kanarios

        When threads in a warp take different branches of a conditional, the warp does not split into smaller warps. Instead, the active mask selects which lanes execute each branch; lanes on the inactive side run dummy instructions until the warp reconverges at the post-conditional join point. The programmer writes ordinary scalar code with if/else, and the hardware silently masks the inactive lanes.

        This is the chief ergonomic win over SIMD: the compiler does not have to prove that all lanes will take the same path. The cost is that divergent code pays for both sides — a 32-wide warp where 16 threads take the if-branch and 16 take the else-branch executes both branches sequentially with half the lanes idle each time.

        The hardware sequences the two paths with a reconvergence stack — or, on recent GPUs, a barrier-based equivalent that avoids its deadlocks. The standard software mitigation is JKNL: group threads with the same control-flow trajectory into one warp so the branch never diverges in the first place.

        Grouping flexibility

        • May 13, 2026
        • Kellen Kanarios

        Threads are assigned to warps based on warp-ID, but the hardware and compiler are free to regroup. JKNL re-bundles threads with similar control-flow trajectories into the same warp so that future branches stay non-divergent. This is a class of optimization that pure SIMD cannot express — a SIMD ISA fixes the lane assignment at compile time.

        The hardware cost of SIMT relative to SIMD is the per-thread PC and register state. The benefit is that a single binary runs efficiently across a wide range of dynamic control flow, which is why the model has dominated GPU programming since CUDA. The scoreboard mechanism described in GPU register scoreboard operates per-warp rather than per-thread precisely because SIMT treats the warp as the unit of issue.

        Remark. SIMT is sometimes described as "SIMD with a per-lane PC," which captures the implementation but understates the programming-model impact: vectorization becomes a runtime, not a compile-time, decision.

        Warp scheduling hides memory latency by switching among resident warps [1PWL]

        • July 11, 2026
        • Kellen Kanarios

        A GPU keeps many warps resident on each Streaming Multiprocessor (SM) and switches between them on every stall. When a warp issues a memory op that misses cache it is dead for hundreds of cycles — but with dozens of resident warps the scheduler always has a runnable one, so aggregate throughput stays high even though single-warp latency is awful. This is the Thread-level parallelism design point cashed out on the GPU: spend silicon on resident warp state rather than on latency-reduction machinery. The switch is free because each SM's register file is large enough to hold every resident warp's registers at once, so a context switch is just changing which warp the scheduler picks — not saving state to memory the way a CPU context switch must.

        Two things bound how well this works. Occupancy — how many warps are resident — sets the ceiling on latency hiding: with too few warps the scheduler runs out of runnable candidates. And the scheduler itself must stay cheap enough to choose a warp every cycle, which it does with a two-loop scoreboard check, marshalling each issued instruction's register reads through an operand collector.

        Definition. GPU register scoreboard [XPAA]

        • May 13, 2026
        • Kellen Kanarios

        The readiness-tracking structure that lets a GPU issue instructions out of program order without any of the CAM machinery a CPU needs. A scoreboard is a small table — typically one busy bit per architectural register — that records whether some in-flight instruction is still computing a value that something else might want to read.

        The protocol is trivial. When an instruction issues, it sets the busy bit on its destination register. When it writes back, it clears the bit. Before any instruction can read its sources, the hardware checks the corresponding bits: if any source is busy, the instruction (or its entire warp) waits. The check is a single table lookup per source, so the per-instruction cost is \(O(1)\) regardless of how many other instructions are in flight.

        This is enough because GPUs lean on Thread-level parallelism rather than Instruction-level parallelism. When a warp's next instruction reads a busy register, the GPU architecture scheduler does not stall — it picks a different ready warp from the dozens of resident warps and runs that one instead. The scoreboard's job is only to answer "is this warp ready? yes/no," which is far cheaper than the broadcast/compare a CPU Issue queue wakeup-select does to find dependency satisfaction in a single thread's instruction window.

        The size difference falls out of this. CPU issue queues hold tens of entries; GPU instruction windows can hold many warps' worth of in-flight work, because every additional warp costs a row in the scoreboard and a register-file allocation, not a quadratic CAM expansion. See ILP vs TLP for why the design fork sits where it does.

        Remark. The scoreboard's simplicity is what lets GPUs spend their transistor budget on ALUs and the Register file instead of control logic. It is one of the most visible places where the Thread-level parallelism design choice cashes out as silicon savings.

        Warp [X9VZ]

        • May 12, 2026
        • Kellen Kanarios

        A warp is the unit of scheduling on an NVIDIA GPU: a group of ~32 threads that issue one instruction per cycle in lockstep. Each functional unit in the SM has as many lanes as a warp has threads, so a single fetched instruction fans out across all 32 lanes at once. The warp — not the thread — is what the hardware fetches, schedules, and tracks; the per-thread SIMT programming model is an illusion built on top of it.

        Because the warp is the atom of execution, the two facts that dominate GPU performance are both properties of the warp: it hides memory latency by switching among resident warps instead of stalling, and it serializes both sides of a branch when its threads diverge.

        Shared memory (GPU) [8GZN]

        • May 12, 2026
        • Kellen Kanarios

        Shared memory on a GPU is a small (per-block) on-chip scratchpad with latency comparable to L1. Software-managed — the programmer explicitly stages data through it, the hardware does not cache for them. Used to:

        • Avoid non-coalesced global accesses by reorganizing data layouts on the fly.
        • Reuse loaded data across threads within a block.
        • Communicate between threads within a block.

        Bank conflicts serialize shared-memory accesses to the same bank [OGCRU]

        • May 25, 2026
        • Kellen Kanarios

        Shared memory is divided into banks (typically 32). Each bank can service one access per cycle. If two threads in a warp access different addresses in the same bank, the accesses serialize — a bank conflict.

        Three behaviors when warp threads access shared memory:

        • Different banks: full parallelism. No conflict.
        • Same bank, same word: hardware broadcasts the word to all threads. No conflict.
        • Same bank, different words: serialize.
        Question. How to avoid bank conflicts?
        Answer. Change the access pattern so threads hit different banks (often by transposing or by adding padding to break the stride that causes the conflict).
        Remark. The GPU memory hierarchy is software-managed at L1 (shared memory) and coherent only at L2 — by restricting L1 to local-to-block scope, the architecture sidesteps the cost of a global Cache coherence protocol on a thousand-way-parallel chip.

        Memory coalescing merges a warp's accesses into one transaction per cache line [U1FO]

        • May 12, 2026
        • Kellen Kanarios

        The hardware optimization on GPUs that combines simultaneous memory accesses by threads in a warp into a single transaction — when the accessed addresses fall in the same cache line.

        Global memory latency on a GPU is 400–800 cycles. Coalescing is the primary way to keep the SM fed:

        • If all 32 threads in a warp access addresses within one cache line: one memory transaction, full bandwidth.
        • If addresses span multiple cache lines: the hardware serializes into multiple transactions; bandwidth drops proportionally.
        Remark. Address permutations within a cache line don't hurt utilization — coalescing is about which lines are touched, not which exact bytes.

        Coalescing is why memory layout matters so much for GPU code. The standard idiom is structure-of-arrays (SoA) instead of array-of-structures (AoS): threads accessing the same field of consecutive records hit consecutive bytes, which coalesce; threads accessing the same struct don't.

        Question. Should you always cache on the GPU?
        Answer. No. If you're memory-bound and have no Spatial locality, cache is wasted on data you won't reuse. CUDA exposes the option to disable L1 caching for these cases, which lets misaligned loads transfer only the bytes needed instead of full cache lines.
        Remark. Use GPUs for parallel parts where throughput dominates. Use CPUs for sequential parts where latency dominates — the ILP vs TLP axis is the right frame for deciding which.

        ILP vs TLP [RJEB]

        • May 13, 2026
        • Kellen Kanarios

        The two strategies a microarchitecture can use to keep arithmetic units busy in the face of memory latency and dependencies. Instruction-level parallelism mines parallelism from within a single instruction stream; Thread-level parallelism mines it across many. Every modern design is a position on this axis, and the position determines the rest of the microarchitecture.

        The choice is not really a choice — it is forced by the workload. ILP works only when the single thread has independent work to find; once dependency chains and branch mispredictions saturate the issue window, no amount of silicon spent on reorder buffers, wakeup logic, or speculation buys more throughput. TLP works only when many independent threads exist; throwing more warps at a single sequential dependency chain accomplishes nothing.

        The downstream consequences split everything else:

        • Control overhead. ILP machines spend most of their area on out-of-order execution machinery — Reorder buffer, Reservation stations, register renaming, CAM-based wakeup-select. TLP machines spend almost none; the schedule is "pick any ready thread."
        • Caches. ILP needs large caches because a single stalled thread blocks the whole pipeline. TLP can afford small caches because the latency is hidden by switching threads.
        • Register file. ILP keeps one (renamed) register set hot. TLP keeps every resident thread's register set hot simultaneously — GPU register files are an order of magnitude larger than CPU register files for this reason.
        • Readiness tracking. ILP pays O(N²) for CAM-based wakeup-select across N in-flight instructions; TLP pays O(1) per instruction via a per-register scoreboard.

        The slogan: CPUs spend a lot of silicon to find parallelism in one thread; GPUs spend almost none because they have many threads. Everything downstream — pipeline depth, cache hierarchy, scheduling policy, energy per operation — falls out of which side of this axis the chip sits on.

        Definition. Instruction-level parallelism [JAZI]

        • May 13, 2026
        • Kellen Kanarios

        Parallelism extracted from within a single thread by executing independent instructions concurrently. The instruction stream is sequential as written, but a sufficiently aggressive back-end can issue, execute, and complete instructions in a different order than they appear, as long as data dependencies are respected and the visible architectural state still looks in-order.

        The machinery a CPU spends on ILP is the bulk of its area and complexity:

        • out-of-order execution — issue instructions whenever their operands are ready, not in program order.
        • register renaming — break false dependencies by mapping the same architectural register name to different physical registers.
        • Reservation stations + Reorder buffer — hold dispatched instructions while waiting on operands, then commit them in original order.
        • Speculative execution and speculation — keep the pipeline fed across control-flow boundaries by predicting which way a branch will go.

        ILP runs into hard limits: dependency chains, branch mispredictions, and the quadratic scaling of issue-queue wakeup-select all cap how wide a single-threaded window can grow. Beyond that ceiling, more silicon stops buying more performance, which is the structural reason the industry pivoted to Thread-level parallelism for throughput-bound workloads.

        Remark. ILP is what classic uniprocessor texts mean by "out-of-order superscalar execution"; see Notebook: Inside the Machine for the unwound history.

        Definition. Issue queue wakeup-select [UJPD]

        • May 13, 2026
        • Kellen Kanarios

        The mechanism by which a modern out-of-order CPU notices that a previously-stalled instruction now has all its operands ready. The issue queue holds the N in-flight instructions whose operands are not yet available; each entry watches a small set of physical register tags (the operands it is waiting for). When a producer finishes, it broadcasts "tag T is ready" to every entry, and each entry compares that tag against its own waiting tags through a content-addressable memory (CAM). Entries whose last operand arrives become eligible to issue this cycle; a separate select stage picks which of the now-ready entries actually fires.

        The cost is the reason CPU issue queues are small. Each entry runs a CAM lookup against each broadcast tag, so an N-entry queue with N producers per cycle scales as \(O(N^2)\) in comparators, wires, and power. The select stage compounds this — priority encoders across N entries are themselves a quadratic-area structure on the critical timing path. The cumulative wakeup-select latency is what bounds clock frequency once N grows past a few tens.

        This is the structural reason CPU out-of-order windows stay in the tens-to-low-hundreds of entries even as transistor budgets grow. Wider doesn't help: timing closure on the CAM degrades faster than the additional ILP repays. GPUs sidestep the whole problem by leaning on Thread-level parallelism — a cheap per-register scoreboard plus a warp scheduler covers stalls without any broadcast/compare structure.

        Remark. Wakeup-select sits at the boundary between Reservation station (which holds the instruction) and register renaming (which named the tags being broadcast). All three structures together are what people mean by "the out-of-order back-end."

        Register file [ETWN]

        • May 12, 2026
        • Kellen Kanarios

        The CPU's nearest tier of storage — the first NUMA level. Registers sit physically next to the ALUs and are addressed by name in the instruction encoding rather than by memory address. Their count is limited by the chip's surface area, which is why main memory exists at all.

        The register file is the structure that holds the architecturally-visible registers. To support a superscalar machine, the file needs enough read and write ports to feed every ALU simultaneously: for an \(n\)-ALU chip with two operands per op, that's \(2n\) read ports and \(n\) write ports.

        Problem. Die area for a register file scales quadratically in the number of ports.
        Solution. Split it. Different execution units (integer, floating-point, vector) get their own smaller register files. Fewer ports per file, faster access, less area.

        Beyond the architecturally-named registers, modern chips keep many extra rename registers used by register renaming to break false data dependencies.

        Remark. Instructions distinguish "immediate types" from "register types" so the decoder knows whether a field encodes a register name or a literal value.
