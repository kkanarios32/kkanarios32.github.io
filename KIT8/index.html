Directory-based coherence scales by replacing broadcasts with point-to-point messages [KIT8]

  • May 12, 2026
  • Kellen Kanarios

A scalable Cache coherence organization for systems too large for a shared bus. Each memory address has a home node; the home node maintains a directory entry listing which caches currently hold copies and in what state. Coherence transactions are point-to-point messages routed via the directory rather than broadcasts. The directory is itself distributed — each node owns the entries for the addresses in its local DRAM, and a physical address uniquely identifies its home — so there is no central bottleneck.

Compared to snoop-based coherence, directory protocols trade a level of indirection (3 hops instead of 2 for some transactions) for vastly reduced bandwidth — making them viable at hundreds of cores.

Three refinements make directory coherence practical. Its metadata is kept from scaling with memory by sparse directories; its multi-hop transactions force the protocol to reason about transient states and races; and workload-specific patterns like migratory sharing let it collapse pairs of transactions into one. See Implementing Directory-based MESI in Murphi for a full worked implementation in Murphi.

    Directory protocols expose transient states because transactions span multiple hops [Z32G]

    • July 11, 2026
    • Kellen Kanarios

    Because a directory transaction takes several hops (requestor → home → owner → …), a line spends observable time in transient states between the stable ones — e.g. \(\mathrm {MI}^{A}\) ("modified, going to invalid, waiting on an acknowledgment"). These are not an implementation detail to wave away: concurrent transactions to the same line race, and the protocol must define a correct outcome for every interleaving. That combinatorial state space is what makes directory protocols hard to get right.

    Example. L holds a dirty copy and starts a writeback to home H. Concurrently R sends a read to H. R sits in \(\mathrm {S}^{D}\) ("shared, waiting for data") until the race resolves.
    Remark. Never resolve these races with NACKs — they cause starvation. The Implementing Directory-based MESI in Murphi Murphi implementation works through the channel-separation arguments that keep the transient-state space deadlock-free.

    Sparse directories keep coherence metadata from scaling with memory [QSWW]

    • July 11, 2026
    • Kellen Kanarios

    A directory must remember which caches hold each line, and the naive encoding — a sharer bitmap per cache line — does not scale: at 256 nodes that is 32 bytes of metadata per line, so roughly half of memory goes to coherence bookkeeping. Two ideas bring it back down. Limited-sharer encoding exploits the fact that most lines have ≤4 sharers: track those exactly and fall back to broadcast for the rare widely-shared line. Sparse directories go further and cache the directory entries themselves — an entry missing from the directory cache means nobody holds the line at all, and evicting an entry invalidates the corresponding lines. The directory then scales with the number of actively cached lines rather than with total memory.

    Snoop-based coherence keeps caches consistent by watching a shared bus [8C2H]

    • May 12, 2026
    • Kellen Kanarios

    A Cache coherence family where every cache watches (snoops) a shared bus. When one cache issues a coherence transaction (e.g. requests permission to read or write a line), every other cache sees it and updates its local state accordingly. No central directory needed; the bus itself is the coordination mechanism.

    The simplest snoop-based protocol is valid-invalid:

    • Invalid + load → BusRd to gain read permission; transition to valid.
    • Invalid + store → no transition (write-through assumed; the bus carries the write).
    • Valid + store → BusWr to notify other caches; they invalidate.

    Write-update coherence, the variant where stores update other caches' copies instead of invalidating them, works for very small bus-based systems but does not scale: every write reaches everyone, and copies of data the snooper isn't using pollute its Cache replacement policy.

    Snoop-based coherence relies on a shared physical bus, which limits it to roughly 8 cores. Beyond that, switch to directory-based coherence.

    Remark. Snoop-based SMPs use the Inclusion property so that snooping at L2 suffices — every L1 line is also tracked at L2.

    Cache coherence [GA7N]

    • May 12, 2026
    • Kellen Kanarios

    In a Multicore system where every core has its own Cache hierarchy, the same memory address can live in multiple caches simultaneously. A cache coherence protocol is the mechanism that keeps those copies consistent — propagating writes, invalidating stale copies, and preserving two invariants:

    1. Single-writer invariant: at any moment, at most one core has permission to write a given line. No other core can read or write it concurrently.
    2. Data-value invariant: all cores see the same order of updates to a given line.

    Two organizational families:

    • snoop-based coherence — every cache watches a shared bus and reacts to other cores' traffic. Scales to ~8 cores.
    • directory-based coherence — a structure tracks who has a copy of each line; coherence messages are point-to-point. Scales to hundreds of cores at the cost of an extra hop.

    The concrete state machines are protocols like MSI protocol, MESI protocol, MOESI protocol, and DEC Firefly, each adding states to handle more cases or to optimize specific sharing patterns. A worked implementation of directory-based MESI in Murphi lives in Implementing Directory-based MESI in Murphi.

      Migratory-sharing optimization collapses read-then-write into one coherence transaction [20SW]

      • July 11, 2026
      • Kellen Kanarios

      A common access pattern is migratory: a core reads a line and then writes it — think a lock or a hand-off variable that passes from core to core. Handled naively this costs two cache-coherence transactions: a read miss that fetches the line shared, then a write miss that upgrades it to modified. Upgrade-miss detection spots the pattern and, on the read miss, transitions the line straight to a migratory-exclusive state, so the following write is silent — one transaction instead of two.

      Remark. The same optimization hurts producer–consumer sharing, where one core writes and several read. There the right move on a read miss is to downgrade the previous writer rather than invalidate it, so later readers source the line from the last writer instead of racing to own it. The lesson generalizes: coherence optimizations are workload-shaped — a win for one sharing pattern is a loss for another.

      Starting with MSI

      • March 24, 2026
      • Kellen Kanarios

      The first thing we need to do is understand all the things that can go awry.

      First in the valid-invalid protocol example we were given they only have two virtual channels. However, we need 3 channels to handle, request messages, forwarded messages, and response messages.

      Let us briefly understand why this is the case. Detecting a deadlock is equivalent to detecting a cycle in a dependency graph, so it is sufficient to show a cyclic dependency.

      • If we issue a GetS this goes to the directory, giving us an edge N1 -> D.
      • Then the directory issues a Fwd-GetS to the ownder, giving us the chain N1 -> D -> N2
      • The owner then needs to send data to the requester, giving us the chain N1 -> D -> N2 -> N1.
      • Namely, the data must go through the same channel as all of the requests depending on it.

      To avoid this, we partition into 3 channels:

      • Request: GetS, GetM, PutM
      • Forwarded: Fwd-GetS, Fwd-GetM, PutAck, Inv
      • Response: InvAck, Data

      We directly enumerate these in our MessageType as 

        MessageType: enum {
    GetS,           -- request for shared copy
    GetM,           -- request for modified (exclusive) copy
    PutS,           -- evict shared copy
    PutM,           -- evict modified copy (w/ data)

    FwdGetS,        -- forwarded GetS to owner
    FwdGetM,        -- forwarded GetM to owner
    Inv,            -- invalidation request

    Data,           -- data response
    InvAck,         -- invalidation acknowledgment
    PutAck          -- put acknowledgment
  };

      Notably, we are missing some of the messages in the figures below. We can handle these based on the AckCount

    Record
      state: enum { 
        P_Invalid, P_Shared, P_Modified,       --stable states
        PT_IS_D,      --sent GetS, waiting for Data
        PT_IS_DI,     --sent GetS, got Inv, waiting for Data then go to I
        PT_IM_AD,     --sent GetM (from I), waiting for Data and InvAcks
        PT_SM_AD,     --sent GetM (from S), waiting for Data and InvAcks
        PT_IM_A,      --got Data for GetM, waiting for remaining InvAcks
        PT_MI_A,      --sent PutM, waiting for PutAck
        PT_SI_A,      --sent PutS, waiting for PutAck
        PT_II_A       --waiting for PutAck, already invalidated
      };
      val: Value;
      ack: AckCount;  -- tracks InvAck count
    End;
      Remark. The load, store, and replacement all only alter the states of the coherence protocol for the permanent states. Therefore, to simulate the system it is sufficient to only implement the rules that initiate these state transitions i.e. evict in modified state or load from invalid state.
      Directory controller. Now for the directory controller:
      Remark. We actually end up adding a second transient state to the directory controller \(M^D\). This is to handle the PutM, GetM race described below.
      Weird Races.
      Problem. When implementing the PutM rule to downgrade from modified to invalid. We must NOT get rid of the data until we have been acknowledged, so that we can still service forward requests. This is a transition to \(MI^A\). However, we can still receive the PutAck causing us to transition to \(I\) before servicing the Fwd-GetM request. This is lost data.
      Solution. We must add a \(M^D\) state to the directory controller. In this state, if we get a PutM request from a node that is not our current owner we stall and wait for data to arrive. For \(M^D\) to transition to \(M\), we must send the data back to the directory even though we are doing a three-hop, so that the directory knows the data has also been forwarded to the new owner. It is unclear to me the best way to do this without this state.
      Problem.

      The above solution causes a deadlock, where two processors both store miss and one load misses. The "winning processor" of the store will be the second one. The first one will get the data from the directory transitioning from \(IM^{AD} \to IM^{A}\), but the second one will then trigger a Fwd-GetM overwriting the first.

      The load miss transitions to \(IS^D\). However, when it receives an Inv it stalls according to the above figure. This blocks the Fwd-GetM from reaching the other winning processor because they are on the same VC. Thus, we have a deadlock where the owner never gets the data.

      Solution. We need to add another transient state not included in the figure. Namely, we add \(II^{A}\) that says we were in \(IS^{A}\), but we got invalidated and now we are waiting for our PutAck before we transition to \(I\).
      Remark. When the home directory is in \(M\) and we get a GetS we cannot clear owner as directed by figure. Otherwise in \(S^D\), we will not be able to tell if the PutM is from the owner and stall, so we only remove the owner once we have received the data.

      Moving to MESI

      • March 24, 2026
      • Kellen Kanarios
       Aside from the modifications discussed above, the rest was straightforward and we now are ready to move on to MESI.

      Cache controller. We will start with MESI cache controller, given by the following table:

      Design choice 1. Since we allow immediate upgrade to \(M\), without forwarding, if E upgrades to M, then the value in the directory sent to the next GetS / GetM would not have updated value. It seems we must either

      1. not allow E to silently upgrade to M and then we do not need forwarding, or
      2. allow E to silently upgrade to M and treat E as M basically and forward everything.
      Notably, neither of these have to write back on PutE, but we will go with 1 because that is how it is indicated in the figure.

      Design Choice 2. In order to indicate that the directory received the first GetS we introduce a ExclusiveData message. Upon receiving, ExclusiveData in \(IS^D\) we transition to \(E\).

      Remark. Once we have sent the PutE, or the PutM \(EI^A\) and \(MI^A\) are functionally the same. I include both for completeness but verified that transitioning directly to \(MI^A\) from \(E\) works the same.

      Directory controller. We start with the analagous table.

      Similar to the cache controller, we also add a \(E^D\) state for completion that can be absorbed into \(M^D\) introduced in the MSI protocol.

      Weird Races.

      Problem. First processor does GetS and is in \(IS^D\) but the Home directory sends it exclusive data and is now in \(E\). Then second processor does GetM and Home issues Fwd-GetM to processor one, which arrives before data.
      Solution. Need to stall on Fwd-GetM and Fwd-GetS in \(IS^D\) until data arrives
      Problem. An even crazier race happens when we get invalidated before the Fwd-GetM or Fwd-GetS arrive. Due to our previous design decision, we transition to \(II^A\), where if we receive a PutAck we transition to invalid. This then loses the needed forward data.
      Example. The timeline:
      1. Proc_1 performs a load miss → \(IS^D\)
      2. Home receives first GetS from Proc_1 → \(E\) and sends ExclusiveData to Proc_1
      3. Proc_2 issues GetM → \(IM^AD\)
      4. Home sends Inv to Proc_1
      5. Home forwards Fwd-GetM to Proc_1
      6. Proc_1 receives Inv → \(II^D\)
      7. Proc_1 receives ExclusiveData → \(I\)
      Data is lost
      Solution. We once again have to use the ExclusiveData message. We can then see that we got ExclusiveData in \(II^D\) and transition to \(E\) instead of \(I\). \(E\) will then correctly handle the Fwd-GetM or Fwd-GetS.
