  2. The backlog of things to do next

2. What to do when actually working on a task

  • June 25, 2026
  • Kellen Kanarios

This is the harder part (and requires the most discipline). When working on a task, especially a more open-ended one like research, it is very common that a million other things come up that seem immediately important. All of the sudden, you have a great idea for the research project you haven't worked on in months.

It is crucial to collect and forget about these things until they are scheduled. This is the one thing I did take from Getting Things Done. The system should flow like water. Making this achievable is where the todoist API becomes crucial.

  • I have a script bound by dwm that pops up vim for me to write down the task and, on save, it sends it to my inbox in todoist.
  • I have more involved ones that allow me to actually allocate it to the specific work stream / priority if it is immediately obvious.

Another thing that I have found useful to actually achieving this watery flow is the pomodoro technique. This basically lets me gaslight myself into thinking I can go to the other things as soon as the timer ends (which is intentionally always just close enough). By the end, I often no longer have this urge and can continue on the scheduled tasks.

      Multi-level feedback queue [4WG5]

      • May 12, 2026
      • Kellen Kanarios

      A practical scheduler that approximates STCF scheduling's turnaround properties and Round-robin scheduling's response properties without requiring foreknowledge of job lengths. Maintain multiple priority queues; within a queue, schedule round-robin. Never service a lower-priority queue while a higher-priority queue has a runnable job.

      The priority of a job is not assigned by the user — it is learned from behavior:

      • New jobs enter at the top priority.
      • Each priority level has an allotment — the total CPU time a job may consume before being demoted.
      • Once a job exhausts its allotment at a level, it drops to the next level.

      Short jobs that finish quickly never leave the top queue (good turnaround). Interactive jobs that yield often stay at the top (good response time). CPU-bound jobs sink to the bottom and get scheduled when nothing better is runnable.

      Problem. Gaming the allotment: a job can issue I/O just before its allotment expires to avoid demotion, since I/O time doesn't count.
      Solution. Charge the job's allotment for total time at the priority level, including time spent blocked.
      Problem. Starvation of low-priority queues if higher queues never empty.
      Solution. Periodically reset all jobs to the top priority. Cheap and effective.

      CPU basics

      • March 15, 2026
      • Kellen Kanarios

      Register file [ETWN]

      • May 12, 2026
      • Kellen Kanarios

      The CPU's nearest tier of storage — the first NUMA level. Registers sit physically next to the ALUs and are addressed by name in the instruction encoding rather than by memory address. Their count is limited by the chip's surface area, which is why main memory exists at all.

      The register file is the structure that holds the architecturally-visible registers. To support a superscalar machine, the file needs enough read and write ports to feed every ALU simultaneously: for an \(n\)-ALU chip with two operands per op, that's \(2n\) read ports and \(n\) write ports.

      Problem. Die area for a register file scales quadratically in the number of ports.
      Solution. Split it. Different execution units (integer, floating-point, vector) get their own smaller register files. Fewer ports per file, faster access, less area.

      Beyond the architecturally-named registers, modern chips keep many extra rename registers used by register renaming to break false data dependencies.

      Remark. Instructions distinguish "immediate types" from "register types" so the decoder knows whether a field encodes a register name or a literal value.

      Fetch-execute loop [VZDH]

      • May 12, 2026
      • Kellen Kanarios

      The classical model of CPU operation: each instruction goes through three phases.

      1. Fetch: read the instruction from memory into the instruction register; increment the program counter.
      2. Decode: extract the opcode, the operand register names, and the destination from the instruction register.
      3. Execute: route to the appropriate functional unit — ALU for arithmetic, load/store unit for memory ops, branch unit for branches.
      Remark. Instructions don't take arbitrary time. The motherboard's clock cycle bounds when latched values become visible to the next stage.

      pipeline decomposes this loop into independently-running pipeline stages so multiple instructions are in flight at once. Superscalar machines run multiple instructions per stage. Both are extensions of this same three-phase model.

      Branch instruction [IJ7Q]

      • May 12, 2026
      • Kellen Kanarios

      An instruction that potentially changes the program counter to something other than "the next sequential instruction." Two flavors:

      1. Unconditional: the PC is set to the destination address, period.
      2. Conditional: the PC is set to the destination only if some condition holds.

      Conditional branches require the hardware to remember the outcome of recent arithmetic — typically in a processor status word (PSW) register, with one bit per condition (zero, sign, carry, overflow). Each conditional-branch variant tests a particular PSW bit.

      In a pipelined CPU, branches stall the front-end: until the branch resolves, the fetcher doesn't know what to fetch next. This is the control hazard, and modern machines mitigate it with Speculative execution driven by Branch prediction.

      Remark. If the program uses relative addressing, the OS supplies a code-segment base to add to the branch destination.

      Boot sequence [XSFO]

      • May 12, 2026
      • Kellen Kanarios

      The fixed series of steps a machine performs from power-on until a usable operating system is running. Each stage runs from a known location and hands off control to the next.

      1. Power signal resets the CPU to a known state.
      2. The CPU fetches from a hard-wired address — the BIOS (or UEFI firmware). The BIOS runs basic hardware checks (POST).
      3. The BIOS loads and jumps to the bootloader (e.g. GRUB) from a known location on disk.
      4. The bootloader places the kernel image at the correct address and jumps to its entry point.
      5. The kernel performs early setup: installs the Trap table (a special privileged instruction), starts the timer interrupt, sets up memory management, and finally enters user space.
      Remark. The hardware is hard-wired only to find the BIOS. Everything past that is software the user can replace, which is why dual-booting and custom bootloaders work.

      Pipelining and superscalar execution

      • March 15, 2026
      • Kellen Kanarios

      Pipelining overlaps instruction stages to retire one per cycle [COV5]

      • May 12, 2026
      • Kellen Kanarios

      Decomposing the Fetch-execute loop into independently-running stages so that multiple instructions are in flight simultaneously. A classical four-stage pipeline:

      1. Fetch from memory.
      2. Decode.
      3. Execute (the original execute phase, minus the write).
      4. Write the result back to register or memory.

      Each stage runs concurrently on different instructions. The clock cycle is bounded by the slowest stage, not the whole loop — so dividing the work into roughly-equal-length stages is essential. With \(n\) stages running in steady state, the machine retires one instruction per cycle even though each instruction takes \(n\) cycles to traverse.

      Metrics

      • May 12, 2026
      • Kellen Kanarios
      1. Completion rate: instructions completed per nanosecond. Program time is \(\frac {\text {instructions per program}}{\text {avg completion rate}}\).
      2. Instruction throughput: instructions retired per clock cycle.
      3. Instruction latency: cycles to complete one instruction (\(\geq \) the number of stages, often more in practice).
      Remark. Pipelining doesn't make any one instruction faster — it raises throughput. Pushing this further (deeper pipelines, more stages) shrinks the clock cycle but increases per-instruction latency and the cost of pipeline stalls or flushes.

      What stops the steady-state throughput of one instruction per cycle is hazards: data hazard, structural hazard, and control hazard. The next layer of complication is going superscalar — multiple instructions per stage — which is covered in superscalar.

      Superscalar CPUs issue multiple instructions per cycle [6JU6]

      • April 3, 2026
      • Kellen Kanarios

      Multiple Execution unitss on the same chip running in parallel during the same pipeline stage. Decode becomes decode/dispatch: the front-end must decide which instructions can issue in parallel and route them to the right units. Going superscalar amplifies the cost of every pipeline data hazard and control hazard, which is why the rest of this chapter's machinery — Forwarding, register renaming, Branch prediction — exists.

      Execution units [DA6Q]

      • May 12, 2026
      • Kellen Kanarios

      The functional circuitry that actually performs an instruction. A superscalar CPU contains several, dispatched to in parallel during the execute stage of the pipeline. The common families:

      • ALU — arithmetic logic unit, performs arithmetic (add, sub, mul, div) and logical (AND, OR, NOT) operations on integers and on special registers like the PSW. Typically split into integer logic units (ILU) and floating-point units (FPU).
      • Load-store unit — performs memory loads and stores. Carries a small address-calculation adder for address-mode resolution.
      • Branch execution unit — evaluates branch conditions (often by reading the PSW) and signals whether the branch is taken.
      • Vector / SIMD units — operate on wide registers (SSE, AVX, AltiVec) for data-parallel work.
      Remark. Per the convention in Inside the Machine, "ALU" often refers to any execution unit for arithmetic or logical operations — including integer logic units (ILU) and floating-point units (FPU).

      Each unit's reads and writes flow through a Register file. Splitting the register file per-unit reduces port count and access time.

      Microarchitecture [87GH]

      • May 12, 2026
      • Kellen Kanarios

      The internal organization of a CPU that implements a given instruction set architecture (ISA). Two chips can present identical ISAs to software while having radically different microarchitectures — e.g. Intel and AMD both implement x86, but their pipelines, branch predictors, and execution units differ.

      The bridge between the two is the microcode engine: a mini-CPU inside the CPU. It holds small programs in a microcode ROM and runs them when a complex instruction needs to be translated into the simpler primitives the hardware actually executes. Conceptually, it's an emulator.

      Remark. CISC vs RISC-style ISAs need large microcode engines to implement their complex instructions. RISC designs minimize the microcode engine, pushing translation to the compiler instead.

      The translated primitives are Micro-operations — a finer-grained, more uniform instruction stream that the actual hardware schedules. The translation step is the main reason an x86 chip's pipeline starts with such complex decode logic.

      Hazards in a superscalar machine

      • April 3, 2026
      • Kellen Kanarios

      Data hazards stall when an instruction needs a not-yet-written value [ITFA]

      • May 12, 2026
      • Kellen Kanarios

      When one instruction's result is the next instruction's input, a naive pipeline would have to stall: the second instruction can't execute until the first has written back. Data hazards are the most pervasive source of stalls in a CPU.

      Two standard mitigations:

      • Forwarding — route the producer's output directly into the consumer's input, before the writeback stage commits it. The consumer sees the result one cycle earlier.
      • register renaming — false dependencies, where two instructions write to the same architectural register but operate on independent values, can be broken by mapping the architectural register to two different physical registers.

      Forwarding [USSP]

      • May 12, 2026
      • Kellen Kanarios

      A data hazard mitigation: the producing instruction's result is routed from the output of the execute stage directly to the input of the next instruction's execute stage, bypassing the register-file write and read. The dependent consumer can execute one cycle later instead of waiting for writeback.

      Conceptually, the pipeline grows wiring rather than stages. The result is "forwarded" from where it appears to where it's needed.

      Register renaming breaks false dependencies [5CT9]

      • May 12, 2026
      • Kellen Kanarios

      Register renaming is the data hazard mitigation that makes out-of-order execution viable: the hardware dynamically remaps each architectural-register write to a fresh physical register from a larger pool, so two instructions that reuse the same architectural name for independent values never collide. The false data dependency — same architectural register, unrelated values — simply disappears, because each write lands in a different physical register.

      Example. Consider: 
      add r1, r2, r3      // r2 := r1 + r3
add r4, r5, r2      // r2 := r4 + r5   (reuses r2 for an independent value)
      The second add only depends on r2 as a destination, not as a source. Without renaming, it would have to wait for the first instruction's writeback to avoid clobbering. With renaming, the second write goes to a fresh physical register, and the two instructions can proceed in parallel.

      The programming model exposes only the architecturally-named registers; the Register file hides a larger pool of rename registers underneath. This is one of the central tricks that makes out-of-order execution viable.

      Structural hazards stall when instructions contend for one resource [7WV1]

      • May 12, 2026
      • Kellen Kanarios

      Two simultaneous instructions need the same hardware resource, and there's only one of it. The classic example is a single-port Register file facing two writes in one cycle, or two memory-accessing instructions facing a single load-store unit.

      Mitigations cost area:

      • Duplicate the resource (extra ALU, more register-file write ports).
      • Pipeline the resource (multi-cycle dividers that accept a new input each cycle).
      • Stall — accept the cost in throughput.
      Problem. Register file area scales quadratically in port count, so duplicating ports for many ALUs gets expensive quickly.
      Solution. Per-unit register files (one for integer, one for FP, one for vector) with fewer ports each. Discussed under Register file.

      Control hazards stall fetch until the branch resolves [I2B5]

      • May 12, 2026
      • Kellen Kanarios

      When the CPU reaches a Branch instruction in the pipeline, it doesn't yet know what to fetch next. Until the branch's condition resolves and its target is computed — several stages later — fetch is stuck.

      Mitigations:

      • Branch prediction — guess where control flow is going.
      • Speculative execution — actually run instructions down the predicted path, ready to throw the work away if the guess was wrong.
      • Instruction caching — keep recently-fetched instructions handy so the cost of restarting fetch on a mispredict is at least bounded by cache latency, not DRAM latency.
      Remark. The cost of a mispredicted branch is roughly the pipeline depth — every speculatively-issued instruction has to be squashed and the pipeline refilled. Deeper pipelines amplify this cost, which is one reason Pentium 4 vs G4e: design philosophies (deep pipeline, narrow & deep) and superscalar-style designs (wider, shallower) make different tradeoffs.

      Branch prediction and speculation

      • March 15, 2026
      • Kellen Kanarios

      Branch prediction [1LCS]

      • May 12, 2026
      • Kellen Kanarios

      The mechanism that lets the front-end of a pipelined CPU keep fetching past a Branch instruction before the branch has actually resolved. The fetcher consults a predictor; if the prediction is right, the pipeline runs at full throughput; if wrong, the speculatively-issued instructions are squashed.

      Static prediction

      • May 12, 2026
      • Kellen Kanarios

      Program-agnostic rules: e.g. "always predict taken", "always predict not-taken", or "predict backward branches taken, forward branches not-taken". Cheap, but accuracy is poor outside of loop-heavy code.

      Dynamic prediction

      • May 12, 2026
      • Kellen Kanarios

      Driven by per-branch history kept in two hardware tables:

      • Branch history table (BHT) — indexed by branch address, holds a small counter (typically 2-bit saturating) tracking how often the branch was taken.
      • Branch target buffer (BTB) — holds the target address each branch was observed to jump to. A BTB miss is expensive: even if you correctly predict "taken", you still don't know where to fetch from.

      Per-branch prediction allows a counter to learn pattern-specific behavior — e.g. a loop counter takes the back-edge \(N - 1\) times then falls through.

      The Pentium famously combined the two: static prediction on a BHT miss, dynamic prediction otherwise. Later machines added refinements — PowerPC 750 added a Trace cache-like branch target instruction cache (BTIC) that stores the first few instructions at the target, eliminating the re-fetch latency. The Pentium M added a loop detector for loops that always iterate the same number of times.

      Remark. See Speculative execution for what happens once a prediction has been made.

      Speculative execution [IIF9]

      • May 12, 2026
      • Kellen Kanarios

      Run instructions before knowing whether they should actually be run, on the assumption that you'll usually be right. Used after Branch prediction to keep the pipeline busy past a Branch instruction: instructions on the predicted path enter the pipeline, execute, and stash their results in non-architectural state. Only when the branch resolves are speculated results "committed" to architectural state — or, if the prediction was wrong, discarded.

      Speculative state lives in the Reorder buffer (and the register renaming rename registers). Entries are tagged as speculative; on misprediction the speculative entries are flushed.

      Remark. Modern speculative execution drives a lot of CPU complexity — and a lot of CPU security work. Spectre-class side channels exploit the fact that speculative execution can leave traces in caches even after the speculative results are discarded.

      Out-of-order execution

      • March 15, 2026
      • Kellen Kanarios

      Out-of-order execution runs ready instructions past a stalled one, committing in program order [U0NY]

      • May 12, 2026
      • Kellen Kanarios

      Execute instructions in whatever order their operands become available, then commit them back in program order. The programmer's model still sees sequential execution; the hardware reorders aggressively underneath to keep Execution units busy across data hazards and memory latency.

      Two structures make this work:

      • Reservation stations — hold instructions waiting on operands, so a stalled instruction doesn't block independent ones behind it.
      • Reorder buffer — preserves program order so commits look sequential and mispredicted Speculative execution paths can be unwound.

      Combined with register renaming to break false dependencies, the back-end of a modern CPU is essentially a dynamic scheduler that uses the program order only as a hint about what to retire next.

      Example. An out-of-order pipeline can issue a load early (because its address operands are ready), execute several unrelated ALU ops while waiting for the memory, then have all four instructions ready to commit in program order. The illusion of sequential execution is preserved; the parallelism is real.

      Reorder buffer [BI4H]

      • May 13, 2026
      • Kellen Kanarios

      The structure that lets an out-of-order execution CPU look in-order to the programmer. The ROB holds every in-flight instruction in original program order, regardless of the order they actually executed. When the oldest entry has its result ready and all earlier entries have already committed, the ROB commits it — writing its result to architectural state and freeing the slot.

      For stores, commit is only half the story. The ROB entry is freed and the instruction is architecturally real, but the value has not yet reached the cache — it is handed off to the Store buffer for asynchronous drain. This is what keeps the ROB from stalling behind every cache miss: once a store is known to be non-speculative, the front-end can move on.

      Each ROB entry typically tracks:

      • Instruction identity and program order position.
      • Operand register names (and renamed sources).
      • Status — pending, executing, complete, speculated.
      • The instruction's result, held in a data field that doubles as a register renaming register.

      Speculatively-issued instructions also enter the ROB but are tagged. On a branch misprediction, all speculative entries past the mispredicted branch are squashed.

      The Pentium Pro introduced this design with a 40-entry ROB. The ROB sits at both ends of the pipeline — dispatch enters it, retirement leaves it — which is what makes the rest of the back-end free to issue and execute out of order.

      Remark. ROB size, Reservation station count, and pipeline depth set each other's bounds. A shallow pipeline can afford a small ROB; a deep one (Pentium 4 had 126) needs a large one to keep the back-end fed.

      Reservation station [FQ9N]

      • May 12, 2026
      • Kellen Kanarios

      A buffer in front of an Execution units (or group of them) that holds dispatched instructions waiting on their operands. Once all operands arrive — either by writeback or by Forwarding — the instruction is issued to its execution unit. Reservation stations exist so that an instruction waiting on a slow operand doesn't block dispatch of unrelated instructions behind it.

      Two organizational choices:

      • One reservation station per execution unit (Pentium Pro). Smaller, simpler.
      • One shared station feeding multiple units (also seen on Pentium Pro, where it issues up to five micro-ops per cycle).

      Within a single station, instructions issue in FIFO order. Across stations, there is no guarantee — that's what out-of-order execution actually means.

      Dispatching an instruction into a reservation station has four prerequisites:

      1. In-order rule: instructions dispatch in program order (they execute out of order, not dispatch).
      2. Execution unit availability: the matching reservation station has room.
      3. Completion buffer availability: the Reorder buffer has room to track the new in-flight instruction.
      4. Register-rename availability: enough rename registers exist to hold the new instruction's destination.

      Definition. Issue queue wakeup-select [UJPD]

      • May 13, 2026
      • Kellen Kanarios

      The mechanism by which a modern out-of-order CPU notices that a previously-stalled instruction now has all its operands ready. The issue queue holds the N in-flight instructions whose operands are not yet available; each entry watches a small set of physical register tags (the operands it is waiting for). When a producer finishes, it broadcasts "tag T is ready" to every entry, and each entry compares that tag against its own waiting tags through a content-addressable memory (CAM). Entries whose last operand arrives become eligible to issue this cycle; a separate select stage picks which of the now-ready entries actually fires.

      The cost is the reason CPU issue queues are small. Each entry runs a CAM lookup against each broadcast tag, so an N-entry queue with N producers per cycle scales as \(O(N^2)\) in comparators, wires, and power. The select stage compounds this — priority encoders across N entries are themselves a quadratic-area structure on the critical timing path. The cumulative wakeup-select latency is what bounds clock frequency once N grows past a few tens.

      This is the structural reason CPU out-of-order windows stay in the tens-to-low-hundreds of entries even as transistor budgets grow. Wider doesn't help: timing closure on the CAM degrades faster than the additional ILP repays. GPUs sidestep the whole problem by leaning on Thread-level parallelism — a cheap per-register scoreboard plus a warp scheduler covers stalls without any broadcast/compare structure.

      Remark. Wakeup-select sits at the boundary between Reservation station (which holds the instruction) and register renaming (which named the tags being broadcast). All three structures together are what people mean by "the out-of-order back-end."

      Instruction set architecture

      • March 15, 2026
      • Kellen Kanarios

      CISC vs RISC [L896]

      • May 12, 2026
      • Kellen Kanarios

      Two design philosophies for an instruction set architecture.

      • Complex instruction set computing (CISC) — many specialized instructions, including ones that combine memory access with arithmetic or operate on complex types (strings, decimal numbers). One instruction often does the work of several. x86 is the canonical example.
      • Reduced instruction set computing (RISC) — a small set of simple, uniform instructions; explicit load and store; arithmetic only on registers. Each instruction does little but maps cleanly to hardware. PowerPC, MIPS, ARM are RISC.
      Problem. CISC needs a fat Microarchitecture engine to translate its complex instructions into hardware-executable primitives. That's silicon area, power, and design complexity that doesn't directly add throughput.
      Solution. Once RISC clearly won as a hardware model, x86 vendors adopted instruction translation: a CISC front-end decodes x86 into RISC-like Micro-operations internally. The chip is RISC on the inside, CISC on the outside.
      Remark. The cost of legacy is real. The original Pentium spent roughly 30% of its transistors on legacy support (mostly the microcode ROM). The Pentium Pro spent 40% on backward compatibility. Modern x86 chips have absorbed this overhead because they can — transistor budgets grew faster than the legacy tax.

      Micro-operations [WEW2]

      • May 12, 2026
      • Kellen Kanarios

      The hardware-internal instruction stream that an x86 chip actually schedules. Each user-visible x86 instruction is decoded into one or more micro-ops, which are simpler, more uniform, RISC-like primitives that the back-end can execute, reorder, and rename freely.

      Many common x86 instructions translate to just one or two micro-ops, so decoders are split into simple and complex variants — the simple decoders are specific to single-micro-op instructions and run in parallel with the complex one.

      Optimizations that operate on the micro-op stream:

      • Micro-op fusion (Pentium M) — two related micro-ops that the back-end tracks together while sharing a slot in the issue queue. A store decomposes into a store-address and store-data pair, which are fused; a read-modify decomposes into a load + ALU op that fuse and execute serially.
      • Macro fusion (Core 2 Duo) — fuse a pair of x86 instructions (e.g. compare + branch) before decode, so a single micro-op represents both. Increases peak decode bandwidth without growing ROB or RS pressure.
      • Trace cache (Pentium 4) — cache decoded micro-op sequences for hot code so decoding doesn't have to happen on every iteration.
      Remark. Micro-ops exist because CISC vs RISC-style ISAs need to be translated to something the hardware can schedule cleanly. Pure RISC chips don't need them.

      Trace cache [3DIC]

      • May 12, 2026
      • Kellen Kanarios

      An L1-level cache of decoded Micro-operations sequences. Instead of caching raw x86 instructions that have to be decoded on every fetch, a trace cache stores micro-op traces ready for the back-end. Introduced in the Pentium 4 to keep its very deep pipeline fed when normal x86 decoding couldn't run fast enough.

      The trace cache operates in two modes:

      • Execution mode: on a trace hit, feed the cached micro-ops directly to the back-end. Decode is bypassed entirely.
      • Build mode: on a trace miss, fetch the instructions from L2, decode them, and assemble a new trace while feeding it forward.

      The trace cache is indexed during Branch prediction, so speculative paths beyond a branch can be stored in the same trace and fed forward without restarting fetch. This lets a single cache line hold six micro-ops that may include a speculated branch target.

      Problem. Long x86 instructions that decode to hundreds of micro-ops would blow out the trace cache.
      Solution. Tag those instructions. When the trace cache encounters a tag, it stores a pointer to the microcode ROM rather than the micro-ops themselves, and the ROM streams them on demand.

      Caches [M91V]

      • April 3, 2026
      • Kellen Kanarios

      The CPU clock is decreasing much faster than main-memory latency, so without a cache hierarchy a fast CPU spends most of its time stalled on memory.

      Example. If you are a really fast factory, the size of the truck doesn't matter — you must still wait the full duration of the first delivery.

      Cache hierarchy [KBYI]

      • May 12, 2026
      • Kellen Kanarios

      Multiple tiers of progressively-larger, progressively-slower storage between the CPU's registers and main memory. Each tier holds the most-recently-used contents of the next slower tier, exploiting Spatial locality and Temporal locality in the workload.

      Approximate tiers on a modern x86 chip:

      • Registers — ns, dozens of bytes per core.
      • L1 cache — 2–8 ns, 8–128 KB per core. Often split into instruction and data L1s.
        • Built from static RAM (4–6 transistors per cell), which is fast but area-expensive vs. DRAM's single-transistor cell.
      • L2 cache — 5–12 ns, 0.5–8 MB. Closes the latency gap between L1 and DRAM.
      • L3 cache — slower, shared across cores.
      • Main memory — 10–60 ns, GBs.
      Example. A byte's journey. L1 checked first; on miss, L2; on miss, main memory; on miss in main memory, Page fault to disk.
      Remark. L1 is split into instruction and data caches because the locality profiles of code and data are very different — re-running the same code over different data has high temporal locality for code but low for data.

      Cache miss types [8CGB]

      • May 12, 2026
      • Kellen Kanarios

      A cache miss happens when the requested data isn't in cache and has to be fetched from a slower tier. Misses fall into four categories — the "four Cs" of cache misses:

      1. Compulsory miss — the data was never in cache. Cold-start misses are unavoidable for a given cache size and working set.
      2. Conflict miss — the data was evicted because of an associativity collision. A different access mapped to the same set kicked it out. Reduced by higher Cache associativity.
      3. Coherence miss — the data was invalidated because another core wrote to it. Specific to shared-memory multicore systems.
      4. Capacity miss — the cache filled up and the data was evicted by Cache replacement policy. The working set doesn't fit.

      Distinguishing them matters because they have different mitigations: conflict misses respond to associativity changes, capacity misses respond to size, coherence misses respond to cache-coherence protocol design, and compulsory misses respond to prefetching.

      Locality, the reason caches work

      • April 3, 2026
      • Kellen Kanarios

      Spatial locality [PMP9]

      • May 12, 2026
      • Kellen Kanarios

      The empirical observation that if a program accesses memory address \(x\), it is likely to access addresses near \(x\) soon. Sequential array traversal, struct field access, and instruction fetch all exhibit it.

      Caches exploit spatial locality by loading whole cache lines (typically 64 bytes) on every miss, not just the single requested byte. The byte that triggered the load is the critical word; the surrounding bytes come along for free under the assumption you'll want them.

      Remark. RAM and caches are organized in cache-line granularity throughout. Bandwidth is sized for whole-line transfers; tags are per-line, not per-byte.

      The complement is Temporal locality — repeated access to the same data over time. Temporal locality implies spatial locality (you re-touch nearby bytes when you re-touch the original), but not vice versa.

      Temporal locality [9RWN]

      • May 12, 2026
      • Kellen Kanarios

      If a program accesses memory address \(x\), it is likely to access \(x\) again soon. Loop variables, frequently-called functions, and re-used data structures all exhibit it.

      Caches exploit temporal locality by retaining recently-accessed lines until something more important takes their slot. The Cache replacement policy is what decides "more important" — typically LRU and the clock algorithm-ish.

      Temporal locality implies spatial locality (re-touching a byte tends to re-touch its neighbors) but not vice versa: a sequential scan has spatial locality with no temporal locality.

      Cache structure

      • April 3, 2026
      • Kellen Kanarios

      Cache associativity [C74U]

      • May 12, 2026
      • Kellen Kanarios

      The rule that decides which slots in the cache an address is allowed to occupy. Three points on the spectrum:

      • Direct mapped — every address maps to exactly one slot. Fast (only one tag to check) but suffers conflict misses: two hot addresses that map to the same slot evict each other.
      • Fully associative — any address can live in any slot. No conflict misses but tag check is expensive (compare against every entry in parallel).
      • \(n\)-way set associative — middle ground. The cache is divided into sets of \(n\) slots; an address maps to a specific set but can occupy any of its \(n\) slots. The hot setting on real caches is somewhere between 4-way and 16-way.

      Higher associativity reduces conflict misses but increases tag-check hardware and power. The cache tag of an address is the bits that aren't used to pick the set — they're stored alongside the data and compared on every access.

      Remark. The tag array is stored separately from the data array (tag RAM) so the tag comparison can happen in parallel with the data lookup. Tags are small, allowing many in-flight comparisons.

      Cache replacement policy [UG51]

      • May 12, 2026
      • Kellen Kanarios

      Within a Cache associativity set, when a new line arrives and the set is full, which existing line gets evicted? Same question as LRU and the clock algorithm for pages, different timescales.

      The hot policy is least recently used — but pure LRU requires tracking a per-line ordering and updating it on every hit, which is too expensive at L1 latencies. Real caches use pseudo-LRU — a tree of single bits per set that approximates LRU with a fraction of the bookkeeping. The clock algorithm is one well-known pseudo-LRU scheme.

      Remark. Block (cache line) size is part of the same tradeoff. Larger blocks improve Spatial locality but cause cache pollution (you fetch bytes you don't use) and eat memory bandwidth. The sweet spot is around 64 bytes on modern x86.

      Write-through vs write-back [54BZ]

      • May 12, 2026
      • Kellen Kanarios

      When a cached line is modified, when do you propagate the change to the next tier?

      • Write-through — every store updates both the cache and main memory. Simple, but generates memory traffic on every write.
      • Write-back — stores update only the cache. The line is marked dirty via a per-line bit; the next-tier copy is updated only when the line is evicted. Less memory traffic but the cache is now the sole authoritative source until eviction, which complicates correctness.
      Remark. Real CPUs run write-back at L1 and L2. Write-through is occasionally used for specific lines (e.g. memory-mapped device registers where every write must hit the device immediately).

      Write-back is one of the constraints behind cache-coherence protocols on multicore chips: every core must agree on which one currently owns a dirty line.

      Power and multicore

      • March 15, 2026
      • Kellen Kanarios

      Power wall [XQMQ]

      • May 12, 2026
      • Kellen Kanarios

      The hard limit on per-core clock speed that emerged in the early 2000s, ending the era of frequency scaling. Power dissipation has two components on a chip:

      • Dynamic power density — watts per unit area dissipated by transistor switching. Scales with clock frequency: faster switching, more energy per second. You can also drive it up by cramming more transistors into the same area.
      • Static power density — watts per unit area dissipated by leakage current through "off" transistors. Smaller transistors leak more, so denser chips leak more even when idle.

      By the mid-2000s, single-core frequency scaling was thermally unsustainable. The industry pivoted to low-power design and Multicore — get more throughput from many slower cores instead of one faster one. Pentium M and the Core series were the first generations explicitly designed under this constraint.

      Multicore [JRP0]

      • May 12, 2026
      • Kellen Kanarios

      Multiple independent CPU cores on the same chip. A core, for this purpose, is the bundle of front-end, back-end, Execution unitss, and L1 Cache hierarchy that runs one architectural thread of execution.

      Traditional multiprocessor systems put each core on its own physical chip; chip multiprocessing (CMP) puts multiple cores on the same die. The key advantage is shared lower-level caches — typically an L2 or L3 shared across all cores on the chip — which speeds up inter-core communication and amortizes silicon.

      Remark. The shift to multicore was forced by the Power wall more than chosen for its own merits. The software cost is real: shared-memory parallelism requires correctness work that single-threaded code didn't.

      Multicore is what makes Concurrency a hardware problem, not just a scheduling problem: real simultaneous execution on different physical cores means software has to handle Race conditions with real synchronization primitives like Locks.

      Hardware interrupt path [K325]

      • May 15, 2026
      • Kellen Kanarios

      What the CPU does, entirely in hardware, the moment an interrupt arrives — whether a device IRQ, a fault like a Page fault, or a software int N. The point of the hardware path is to land in the kernel at a clean instruction boundary with enough saved state that user execution can resume later as if nothing happened.

      Four things happen, in order, with no software involvement:

      1. Reach a clean instruction boundary. The CPU cannot pause halfway through, say, a memory write — architectural state would be inconsistent. So the current instruction either finishes normally (typical for asynchronous interrupts like a timer) or is rolled back as if it never started (faults, where the instruction couldn't complete anyway and needs to be retried). A page-fault handler maps in the missing page and UHNL returns to re-execute the faulting instruction; a timer handler returns to the next instruction.
      2. Save the minimum resume state to the kernel stack. The CPU pushes RIP (where to resume), RSP (the user stack pointer it's about to abandon), the privilege level (CS — were we in ring 3 or ring 0?), and RFLAGS. The save target is the kernel stack, whose address the CPU reads from the Task state segment that the OS set up at boot. This is a security boundary: the kernel never trusts the user stack, since user code could have corrupted it or pointed RSP at something malicious. Note what is not saved: general-purpose registers. The handler itself pushes RAX, RBX, ... if it uses them, which keeps the hardware path fast.
      3. Jump to the handler via the Interrupt descriptor table. The CPU takes the interrupt number \(N\), indexes entry \(N\) of the IDT (found via the IDTR register), loads the handler address into RIP, and loads the new code segment, which switches the CPU from ring 3 to ring 0. After this step, the CPU is running kernel code on the kernel stack at the right privilege level — all set up automatically.
      4. Flush the Reorder buffer of work past the interrupt point. In an out-of-order execution CPU the ROB may already contain instructions from after the interrupt — they've been issued speculatively because the front-end ran ahead. Those entries are discarded so the handler sees clean architectural state, as if execution had stopped exactly at the interrupt point. When UHNL returns, those instructions re-execute normally. This is the same squash machinery used for branch mispredictions in Speculative execution — and, famously, the reason Spectre-class side channels exist: speculative work leaves observable traces in caches even after the ROB is flushed.

      Once the hardware path finishes, the OS's interrupt handler is running. Its first job is typically to save the remaining registers before doing real work — the rest is software.

      Remark. Steps 2 and 3 are why an OS needs to set up the IDT and TSS before enabling interrupts. Until both structures point somewhere valid, the first interrupt would either jump to garbage or push to a user-controlled stack — both fatal.

      Definition. Interrupt descriptor table [YK46]

      • May 15, 2026
      • Kellen Kanarios

      The CPU's lookup table from interrupt number to handler. The OS allocates the IDT in memory, fills entry \(N\) with the address of the handler for interrupt \(N\), and points the IDTR register at the table's base. From then on the Hardware interrupt path uses the IDT to dispatch every interrupt without any further software help.

      Each entry — a gate descriptor on x86 — carries enough information for the CPU to switch into the handler safely: the handler's linear address, the code segment to load (which sets the new privilege level, typically ring 0), and a type byte distinguishing interrupt gates (which mask further interrupts on entry) from trap gates (which don't). Entries 0–31 are reserved for CPU-defined exceptions (divide-by-zero, Page fault, double fault, ...); 32–255 are available for device IRQs and software interrupts.

      Remark. The IDT is one of three OS-installed tables that the CPU consults transparently — the others are the GDT (segment descriptors) and the page tables. Each is a hardware contract: the OS lays bytes out in a format the silicon understands, then hands over a pointer.

      Definition. Task state segment [ZAAT]

      • May 15, 2026
      • Kellen Kanarios

      A small OS-allocated structure that the CPU reads to find the kernel stack pointer when it raises privilege. Its historical purpose — full task switching in hardware — is dead on modern x86-64; what survives is one essential field: RSP0, the address of the kernel stack to switch to on entry from ring 3.

      During the Hardware interrupt path, the moment the CPU decides to push RIP/RSP/CS/RFLAGS, it reads RSP0 from the current TSS (located via the GDT entry pointed at by the TR register) and switches RSP to that value before pushing. If the TSS were missing or pointed at user memory, the very first interrupt would either fault while trying to push, or push the user's registers onto attacker-controlled memory. So the TSS is set up at boot, before interrupts are enabled.

      Remark. This is why a kernel needs a per-CPU TSS, not a per-process one: RSP0 is consulted on every transition into the kernel, and each CPU needs its own kernel stack to land on.

      Pentium and Pentium Pro [7XOX]

      • April 3, 2026
      • Kellen Kanarios

      Machine-instance notes — concepts these chips introduced live in their own atomic notes; this tree records what was novel about each implementation.

      The original Pentium

      • April 3, 2026
      • Kellen Kanarios

      Concepts in play:

      • Cache hierarchy — Pentium adopted L1/L2 caching and split the L1 into separate instruction and data caches.
      • Branch prediction — branch history table (BHT) and branch target buffer (BTB) combined with static prediction as a fallback on BHT miss.
      • Speculative execution — execute past a predicted branch, write back only after the prediction is confirmed.

      Pipeline.

      1. Fetch / Prefetch: read from instruction cache.
      2. Decode 1: decode to internal format; branch prediction.
      3. Decode 2: convert complex instructions via Microarchitecture to Micro-operations.
      4. Execute.
      5. Write.

      Back-end: two integer ALUs (also doing address calculations) and one floating-point ALU. The FP register file was arranged as a stack, with most FP ops operating on the top — zero-cycle stack swaps existed but still occupied a decode slot.

      Remark. Estimated 30% of Pentium transistors went to legacy support, mostly the microcode ROM.

      The Pentium Pro

      • April 3, 2026
      • Kellen Kanarios

      Concepts introduced:

      • out-of-order execution — instructions execute in dataflow order, commit in program order.
      • Reorder buffer — 40-entry ROB tracks all in-flight instructions and assists register renaming.
      • Reservation station — three-instruction dispatch, up to five-issue per cycle.
      • CISC vs RISC decoupling — x86 in front, Micro-operations internally.
      Remark. Simple and complex decoders are split because many CISC instructions translate to one or two micro-ops — the simple decoders handle those in parallel with the slow complex decoder.
      Remark. 40% of Pentium Pro transistors went to backward-compatibility support.

      PowerPC chip family [3LIK]

      • April 3, 2026
      • Kellen Kanarios

      Machine-instance notes for the PowerPC line — a RISC implementation of the same concepts covered in superscalar and Branch prediction, with a few innovations specific to this family.

      PowerPC 601

      • April 3, 2026
      • Kellen Kanarios

      Front-end. standard 4-stage pipeline. Instruction queue dispatches 3 instructions per cycle.

      • Identifies unconditional branches (and resolved conditional ones) in the queue; if taken, deletes the branch instruction and starts fetching from the target — branch folding.
      • Untaken branches "fall out" — fall through.
      • No Reorder buffer-equivalent. Static scheduling.

      Back-end. one integer ALU (also handles addresses), one floating-point ALU with single-cycle fused multiply-add. For FP-heavy code the integer ALU just does addresses.

      Remark. Competed with two-integer-ALU designs because simpler logic left room for 4× the L1 cache (32KB vs 8KB).

      PowerPC 603 and 603e

      • April 3, 2026
      • Kellen Kanarios

      Front-end.

      • Branch dispatch on a separate bus, so branches don't compete with non-branch dispatch.
      • Up to 2 non-branch instructions dispatched per cycle.
      • Adds a commit unit and completion queue — the PowerPC names for Reorder buffer. Dynamic scheduling with an instruction window.

      Back-end.

      • Two added execution units: a system unit for misc integer ops and a load/store unit. Lighter load on the integer ALU.
      • FPU took hits: 3 instructions every 4 cycles (mandatory bubble); double precision (including fmadd) takes 2 cycles in execute.

      PowerPC 604

      • April 3, 2026
      • Kellen Kanarios

      Front-end.

      • Splits decode/dispatch and writeback into two stages — enables shorter clock cycles.
      • Up to 6 instructions dispatched per cycle.
      • Dynamic Branch prediction with a 512-entry BHT (2 bits per entry) and a 64-entry branch address cache (BTB).
      • Adds a condition-register unit sharing dispatch bus with the branch unit.

      Back-end. two simple integer units + one complex; fully pipelined FPU.

      Dispatch obeys Reservation station rules: in-order dispatch, execution-unit availability, ROB availability, and rename-register availability.

      Remark. Shallow pipeline → small ROB and few reservation stations are acceptable. Tradeoff: lower clock speed.
      Remark. The 604e doubled L1 to 64KB and added a condition-register unit, freeing the integer and branch units from condition computations.

      PowerPC 750

      • April 3, 2026
      • Kellen Kanarios

      Same 4-stage pipeline as the 603e. Two integer units instead of one; 3-cycle single-precision FP; 6-entry instruction queue, up to 4 fetched per cycle.

      The 750's signature trick: a branch target instruction cache (BTIC). A traditional BTB stores the address of the branch target — you still have to re-fetch that instruction. The BTIC caches the actual first instructions at the target, eliminating the re-fetch latency. This makes Branch prediction dramatically more useful.

      Remark. Did not support a SIMD instruction set.

      PowerPC 7400

      • April 3, 2026
      • Kellen Kanarios

      Added a vector permute unit (VPU) and vector ALU (VALU) for SIMD; full double-precision FPU; 8-entry instruction queue.

      Pentium 4 vs G4e: design philosophies [037I]

      • April 3, 2026
      • Kellen Kanarios

      Two contrasting approaches to using a transistor budget, well-paired as instance studies because they target the same constraints from opposite directions.

      The G4e: wide and shallow

      • April 3, 2026
      • Kellen Kanarios

      Optimizes for instruction-level parallelism. Seven-stage pipeline.

      1. Additional second fetching stage.
      2. Issue stage. To avoid in-order issue stalls, the G4e uses issue queues (general / vector / float) that issue 3, 2, and 1 instruction per cycle respectively. Issue queues differ from Reservation stations in that each queue feeds a group of execution units, not a single one.
      3. Complete stage.

      The Pentium 4: narrow and deep

      • April 3, 2026
      • Kellen Kanarios

      Optimizes for clock speed. 20-stage pipeline — extreme depth for fast clock, with the obvious downside that mispredicted branches are very expensive.

      • Needs deep buffers (126-entry Reorder buffer) to keep the fast back-end fed.

      Trace cache.

      • x86 decoding was too slow to keep a 20-stage pipeline fed.
      • The trace cache moves to L1, storing already-decoded Micro-operations traces; the raw instruction cache moves to L2.
      • Trace cache is indexed during Branch prediction, so speculative paths past a branch can be stored in the same trace.
      • Operates in two modes — execution mode (trace hit, feed micro-ops directly) and build mode (trace miss, decode and assemble while feeding forward).

      Pentium M, Core Duo, Core 2 Duo [DKZ3]

      • April 3, 2026
      • Kellen Kanarios

      The post-Pentium-4 generations, designed under the Power wall. Each generation contributed an innovation to the concept stack.

      Remark. A processor often has three names: a brand name, a microarchitecture brand name, and a microarchitecture code name.

      The Pentium M

      • April 3, 2026
      • Kellen Kanarios

      First chip explicitly designed for low power. Innovations:

      • Widened fetch buffer to 64 bytes — decodes up to 4 16-byte x86 instructions at a time.
      • Micro-op fusion — pairs of Micro-operations from the same x86 instruction (e.g. store-address + store-data, or load + ALU op for a read-modify) are tracked together as one queue entry until issue. See Micro-operations.
      • Loop detector in Branch prediction — counters watch loop trip counts; loops that iterate the same number of times each invocation can be predicted with 100% accuracy.
      • Indirect predictor — like a BHT but stores each branch's favorite target address.
      • Stack execution unit — dedicated unit with its own adder to handle push/pop ESP updates without going through the integer ALU. Stack micro-ops never reach the back-end.

      The Core Duo

      • April 3, 2026
      • Kellen Kanarios

      The first true Multicore x86: two cores on the same die, sharing L2. Innovations relative to Pentium M:

      • Every decoder is capable of micro-op fusion (up to 3 fused entries into the micro-op queue per cycle).
      • Fuses vector SSE instructions in addition to scalars.
      • Improved loop detector for short loops.
      Remark. Until this point, the "core" was an abstraction — front-end + back-end + L1. The Core Duo is when "core" became a thing chips have more than one of.

      The Core 2 Duo

      • April 3, 2026
      • Kellen Kanarios

      Theme: "more is always better" — wider decode/dispatch/issue, bigger Reorder buffer, more Reservation stations, more decode logic. Two innovations made the extra resources pay off:

      • Macro fusion (front-end) — fuse pairs of x86 instructions (e.g. compare + branch) before decode. Max decode bandwidth grows to 5 x86 instructions per cycle without burning ROB or RS entries.
      • Memory disambiguation (back-end) — apply Speculative execution to memory aliasing. Loads can speculatively execute ahead of stores with unknown target addresses; on actual alias detected at commit time, the load and dependent ops are squashed and re-executed.

      The point of memory disambiguation: stores must commit in program order relative to other stores, and no load can commit ahead of an aliased store. But the rule "no load past a store-data with unknown target" forced a lot of unnecessary stalls — most unknowns aren't actually aliases. Speculate and verify.

      Back-end also gained single-cycle vector ops: a 128-bit data bus to the vector ALU replaced the previous 80-bit bus, so 128-bit SSE instructions decode as a single micro-op.

      Remark. A load is one micro-op because address computation and memory access are inherently serial. A store is two micro-ops (store-address + store-data) because address computation and reading the register file can happen in parallel.

      64-bit computing

      • March 15, 2026
      • Kellen Kanarios

      \(n\)-bit computing refers to how many bits a register can hold. 64-bit means registers hold 64 bits, 64-bit instructions means their operands are 64 bits wide.

      Remark. Instruction encoding size hasn't actually changed — the PC widened because the virtual address space did.
      Remark. Only integer and addressing units widened. Floating point and vector units already operated on wider registers.

      Benefits.

      • Increased dynamic range — values can range over \(2^{64}\) instead of \(2^{32}\). Overflow/underflow happens at the extremes of the representable range.
      • 64-bit address space. x86-64 currently implements 40-bit virtual addresses but the architecture allows far more.
        • Drawback: pointers now take up twice the cache space.
      Remark. It is not always beneficial to use 64-bit integers. If you don't need the range, the wider type wastes cache space. The default integer size on x86-64 is still 32 bits; widening requires a one-byte prefix, adding less than 10% to code size.

      The process abstraction, the syscalls for working with processes on Unix, the hardware-software contract that makes time-sharing safe, and the scheduling policies that decide who runs when.

      The abstraction

      • August 16, 2025
      • Kellen Kanarios

      Process [FCHM]

      • May 12, 2026
      • Kellen Kanarios

      A running program, plus all the state the OS needs to pause it and resume it later. A process consists of:

      1. Memory: an Address space holding the program's code, stack, and heap.
      2. Registers: general-purpose registers carrying current computation, plus special-purpose registers:
        • Program counter (PC): the next instruction to run.
        • Stack pointer and frame pointer: where the process sits on its stack.

      A process moves between three states:

      • running: currently on a CPU.
      • ready: runnable, waiting for the Scheduling to give it a CPU.
      • blocked: waiting on I/O or another event, not runnable.
      Remark. Threads (see Concurrency Intro) share an address space within a process but each have their own PC, register set, and stack.

      Process API [JJ45]

      • May 12, 2026
      • Kellen Kanarios

      The minimum operations any OS must expose for working with Processes:

      1. Create: load a program from disk, set up its address space (stack, heap, code), and jump to main. The OS also initializes argc/argv.
      2. Destroy: kill a misbehaving or unwanted process.
      3. Wait: block until another process completes — useful for ordering dependent work.
      4. Miscellaneous control: e.g. suspend / resume.
      5. Status: report on running processes.
      Remark. The stack is allocated up front. The heap starts small and grows via malloc, which calls into the OS (brk/sbrk) when it runs out of headroom.

      On Unix, the concrete realization of this API uses fork(), wait(), and exec() — covered separately.

      Process control block [JE53]

      • May 12, 2026
      • Kellen Kanarios

      The kernel-side data structure holding all per-process state. There is one PCB per Process, threaded together into the process list. The PCB is what the OS reads and writes during a Context switch: the saved register set lives here, as does the process state, the open files, the parent pointer, and so on.

      An example from xv6:

      // the registers xv6 will save and restore
// to stop and subsequently restart a process
struct context {
  int eip;
  int esp;
  int ebx;
  int ecx;
  int edx;
  int esi;
  int edi;
  int ebp;
};
// the different states a process can be in
enum proc_state { UNUSED, EMBRYO, SLEEPING, RUNNABLE, RUNNING, ZOMBIE };
// the information xv6 tracks about each process
// including its register context and state
struct proc {
  char *mem;                 // Start of process memory
  uint sz;                   // Size of process memory
  char *kstack;              // Bottom of kernel stack for this process
  enum proc_state state;     // Process state
  int pid;                   // Process ID
  struct proc *parent;       // Parent process
  void *chan;                // If non-zero, sleeping on chan
  int killed;                // If non-zero, have been killed
  struct file *ofile[NOFILE]; // Open files
  struct inode *cwd;         // Current directory
  struct context context;    // Switch here to run process
  struct trapframe *tf;      // Trap frame for the current interrupt
};
      Remark. Threads need an analogous thread control block; see Concurrency Intro.

      Unix process syscalls

      • August 16, 2025
      • Kellen Kanarios

      fork() [O1S9]

      • May 12, 2026
      • Kellen Kanarios

      The Unix system call that duplicates the calling Process. After fork() returns, two processes exist where there was one. The child gets its own copy of:

      1. Address space.
      2. General-purpose registers.
      3. Program counter — the child returns from fork() at the same instruction the parent does.

      The two processes are distinguished by fork()'s return value:

      int main()
  int rc = fork();
  if (rc < 0)
    printf("error with fork");
  else if (rc == 0)
    printf("in child");
  else
    printf("in parent");        // rc holds child PID
      Remark. Which process is scheduled first after fork() is non-deterministic. Use wait() if you need ordering.
      Remark. The copy is logical, not physical: real implementations use Copy-on-write so the duplicated address space only allocates fresh pages when one side writes to them.

      wait() [J6VZ]

      • May 12, 2026
      • Kellen Kanarios

      Blocks the calling Process until one of its children terminates. The OS marks the child's PCB ZOMBIE on exit, keeping enough state (exit status, resource usage) for the parent to retrieve via wait(); only when the parent collects it is the child fully reaped.

      wait() returns the PID of the child that died. Variant waitpid(pid, ...) blocks on a specific child — useful when a parent has multiple children and only one is interesting.

      Used to enforce ordering between parent and child code paths after fork(), or to know when an exec()'d program finishes.

      Remark. Calling wait() with no children to wait on returns -1.

      exec() [YHJR]

      • May 12, 2026
      • Kellen Kanarios

      Replaces the calling Process's program with a new executable. exec() does not return on success — the original code, data, stack, and heap are discarded and the new program runs in their place. The PID is unchanged.

      What exec() does internally:

      1. Loads the new executable's code and static data, overwriting the current.
      2. Re-initializes stack and heap.
      3. Jumps to the new program's entry point with arguments passed as argv.

      The canonical Unix pattern is fork() then exec() — the shell uses this to launch a command without becoming that command:

      int main()
  while (true) {
    command = getcommand();
    int rc = fork();
    if (rc < 0)
      printf("error with fork");
    else if (rc == 0)
      exec(command);            // child becomes the command
    else
      int rc = wait(NULL);      // parent waits for it
  }

      Redirection is implemented by opening a file in the child between fork() and exec(), attached to STDOUT_FILENO:

      if (rc == 0)
  close(STDOUT_FILENO);
  open("output.txt");
  exec(command);
      Remark. The many variants — execl, execle, execlp, execv, execvp, execvP — differ in how they take arguments (list vs. vector), whether they use the PATH environment variable, and whether they take an explicit environment.

      Unix signals [9V3H]

      • May 12, 2026
      • Kellen Kanarios

      An asynchronous notification delivered to a Process. A process sends a signal with kill(pid, signum) (the name is misleading — most signals don't kill); the recipient can install a handler with signal(signum, handler) to "catch" it, otherwise the default action runs (often termination).

      Signals are the OS's mechanism for events the recipient didn't ask to be told about: SIGINT from Ctrl-C, SIGSEGV from a Page fault on an invalid page, SIGCHLD when a child exits, SIGTERM from an administrative kill.

      Who can signal whom depends on the user owning the process — a user can signal their own processes; the superuser can signal any process.

      Limited Direct Execution [VE8C]

      • July 19, 2025
      • Kellen Kanarios

      The OS lets user code run directly on the CPU for speed, but with hardware-enforced limits that prevent a buggy or malicious program from compromising the rest of the system. Two questions motivate the mechanism:

      Question.
      1. How do we make sure a program doesn't do what we don't want it to do?
      2. How do we enable time-sharing?

      The first question is answered by hardware User mode vs kernel mode enforced via the Trap table, with a controlled Syscall mechanism as the only path from one to the other. The second is answered by periodic timer interrupts that trap to the OS, which performs a Context switch and runs the scheduler.

      The whole stack is bootstrapped during the Boot sequence, when the kernel installs the trap table and starts the timer before handing control to user space.

      User mode vs kernel mode [QUQC]

      • May 12, 2026
      • Kellen Kanarios

      Two privilege levels supported by the CPU. User mode restricts the running process to a limited set of instructions and a limited view of memory — usually just its own address space. Kernel mode runs the OS with full access to all instructions and all of physical memory.

      Switching between modes is mediated by hardware via a Trap table populated at boot. The OS sets up the table; the CPU enforces it. A process cannot promote itself to kernel mode arbitrarily — it must invoke a Syscall mechanism, which traps to a fixed entry point in the OS.

      Remark. Kernel mode is dangerous because the address space is effectively the entire machine. The mode bit + the trap table together are what makes it safe to time-share a CPU between mutually distrusting programs.

      Trap table [0NVY]

      • May 12, 2026
      • Kellen Kanarios

      A hardware-readable table installed by the OS at boot that maps each kind of exceptional control-flow event (syscall, interrupt, page fault, illegal instruction, ...) to the address of an OS handler. When the CPU takes a trap, it consults this table to decide where to jump, switches into kernel mode, and runs the handler.

      The table is what makes it safe to expose privileged operations to user mode: user code never gets to pick the destination of a mode-switch, only the trap number. The OS controls every entry point, the CPU enforces the indirection.

      Remark. Installing the trap table is itself a privileged operation. It happens once, early in boot, before any user process exists.

      Syscall mechanism [A5S6]

      • May 12, 2026
      • Kellen Kanarios

      A syscall is a user-mode-initiated request for a privileged service from the kernel. Because user code cannot jump into kernel code directly, the mechanism is built around a single trap instruction plus a pre-agreed register convention identifying which service to perform.

      The end-to-end sequence:

      1. User code places the syscall number in a pre-defined register and invokes the trap instruction.
      2. Hardware saves enough state to resume the process and switches to kernel mode.
      3. Hardware jumps to the OS-installed handler via the Trap table.
      4. The handler reads the syscall number from the agreed register and dispatches to the corresponding OS routine.
      5. The OS validates the request, performs the work, and arranges return values per the calling convention.
      6. The OS restores user state and returns the process to user mode.
      Remark. Because the register convention is fixed by ABI, the user-mode side of a syscall is typically hand-written in assembly within libc.
      Remark. Interrupts work via a similar trap-handler mechanism, but are initiated by hardware (timer, device) rather than user code.
      Remark. The kernel must validate any pointers or arguments the user supplies — user code cannot be trusted not to ask the kernel to read kernel memory on its behalf.

      Context switch [OKCI]

      • May 12, 2026
      • Kellen Kanarios

      The operation of saving the full execution state of one process (or thread) and restoring another's so the CPU can resume it as if it had never been interrupted. On a trap into the kernel, the hardware stashes the user-visible state (PC, condition codes, stack pointer) into a known location, and the OS saves the remaining general-purpose registers as part of the switching code.

      Two flavors of state are involved:

      1. User registers: the current program's context (general-purpose registers, stack vars).
      2. Kernel registers: meta-information used to resume the trap correctly (PC at trap, syscall number, etc.).

      Context switches are not free — they incur the cost of cache and TLB pollution in addition to the direct cost of saving/restoring state. This is the cost amortized by larger scheduling quanta and a recurring concern for any Concurrency system.

      Boot sequence [XSFO]

      • May 12, 2026
      • Kellen Kanarios

      The fixed series of steps a machine performs from power-on until a usable operating system is running. Each stage runs from a known location and hands off control to the next.

      1. Power signal resets the CPU to a known state.
      2. The CPU fetches from a hard-wired address — the BIOS (or UEFI firmware). The BIOS runs basic hardware checks (POST).
      3. The BIOS loads and jumps to the bootloader (e.g. GRUB) from a known location on disk.
      4. The bootloader places the kernel image at the correct address and jumps to its entry point.
      5. The kernel performs early setup: installs the Trap table (a special privileged instruction), starts the timer interrupt, sets up memory management, and finally enters user space.
      Remark. The hardware is hard-wired only to find the BIOS. Everything past that is software the user can replace, which is why dual-booting and custom bootloaders work.

      Scheduling [B8GC]

      • July 26, 2025
      • Kellen Kanarios

      The scheduler decides which runnable Process to put on the CPU at each moment. There is no single right answer because the metrics conflict:

      1. Turnaround time: time from arrival to completion, \(T_{\text {completion}} - T_{\text {arrival}}\). Minimized by running short jobs first.
      2. Response time: time from arrival to first scheduling. Minimized by frequent rotation.
      3. Fairness: every job makes progress; nobody starves.

      FIFO scheduling [JA6Z]

      • May 12, 2026
      • Kellen Kanarios

      First-in, first-out. Jobs are run to completion in the order they arrive. Trivially fair under any reasonable notion of fairness, but suffers from the convoy effect: a single long-running job at the head delays every shorter job behind it, inflating average turnaround time.

      Problem. If the first arriving job is much longer than the rest, all subsequent jobs wait at least its full duration before getting any CPU.

      SJF scheduling [VJ4F]

      • May 12, 2026
      • Kellen Kanarios

      Shortest job first. Run the job with the smallest total work first, run-to-completion. If all jobs arrive at the same time, SJF provably minimizes average turnaround time.

      Problem. Real workloads don't all arrive at \(t = 0\). A long job that arrives slightly before a short one still gets to monopolize the CPU until it finishes — SJF degenerates to FIFO scheduling under sequential arrivals.

      Removing the run-to-completion constraint and allowing preemption gives STCF scheduling.

      STCF scheduling [7R07]

      • May 12, 2026
      • Kellen Kanarios

      Shortest time-to-completion first. A preemptive refinement of SJF scheduling: whenever the scheduler is invoked, it picks the job with the least remaining work. A short job arriving mid-execution can preempt a long one.

      STCF is provably optimal for minimizing average turnaround time over all preemptive schedulers.

      Problem. Long jobs starve in response time. The lowest-priority work might not get a CPU slice for a very long time, which is unacceptable for interactive workloads.

      Round-robin scheduling [NEL8]

      • May 12, 2026
      • Kellen Kanarios

      Cycle through all runnable jobs, giving each a fixed time slice — the stride, time slice, or quantum length — before preempting and moving to the next. The best you can do at minimizing response time: every job sees the CPU within at most \(N\) quanta where \(N\) is the runnable-job count.

      The quantum length is a tradeoff:

      • Short quantum: better response time, more frequent context switches, worse cache locality.
      • Long quantum: amortizes context-switch cost, worse response time.

      Worst-case response time for \(N\) jobs at quantum \(S\) is \(S(N-1)\).

      Remark. Round-robin minimizes response time but is poor for turnaround time relative to STCF scheduling. A scheduler that wants both has to combine ideas — see Multi-level feedback queue.

      Multi-level feedback queue [4WG5]

      • May 12, 2026
      • Kellen Kanarios

      A practical scheduler that approximates STCF scheduling's turnaround properties and Round-robin scheduling's response properties without requiring foreknowledge of job lengths. Maintain multiple priority queues; within a queue, schedule round-robin. Never service a lower-priority queue while a higher-priority queue has a runnable job.

      The priority of a job is not assigned by the user — it is learned from behavior:

      • New jobs enter at the top priority.
      • Each priority level has an allotment — the total CPU time a job may consume before being demoted.
      • Once a job exhausts its allotment at a level, it drops to the next level.

      Short jobs that finish quickly never leave the top queue (good turnaround). Interactive jobs that yield often stay at the top (good response time). CPU-bound jobs sink to the bottom and get scheduled when nothing better is runnable.

      Problem. Gaming the allotment: a job can issue I/O just before its allotment expires to avoid demotion, since I/O time doesn't count.
      Solution. Charge the job's allotment for total time at the priority level, including time spent blocked.
      Problem. Starvation of low-priority queues if higher queues never empty.
      Solution. Periodically reset all jobs to the top priority. Cheap and effective.

      Lottery scheduling [AM8U]

      • May 12, 2026
      • Kellen Kanarios

      A proportional-share scheduler. Each runnable job holds some number of tickets, drawn from a fixed total. On each scheduling decision, the scheduler picks a random ticket and runs the job that owns it. Over time, a job receives CPU time proportional to its share of the tickets.

      The implementation is trivial: one random number per decision, plus a sum table. No global state about historical CPU usage is needed.

      Mechanisms layered on top:

      • Ticket currency — give each user a fixed number of tickets to split among their processes however they like.
      • Ticket transfer — a client process can hand tickets to a server to boost its priority while servicing the request.
      • Ticket inflation — a process can grow its share temporarily, with bounds set by trust.
      Remark. The unsolved problem is how to assign tickets in the first place. The mechanism is simple; the policy is not.

      A deterministic version of the same idea is Stride scheduling, which avoids the variance of random sampling at the cost of needing global state.

      Stride scheduling [HCJ7]

      • May 12, 2026
      • Kellen Kanarios

      A deterministic proportional-share scheduler. Each job has a stride inversely proportional to its ticket count (the more tickets, the smaller the stride) and a running pass counter that accumulates its stride every time it's scheduled.

      The decision is:

      1. Pick the job with the lowest pass.
      2. Run it for one quantum.
      3. Increment its pass by its stride.

      A job with twice as many tickets has half the stride, so its pass grows half as fast, so it gets scheduled twice as often. Same long-run behavior as Lottery scheduling, but with zero variance.

      Problem. New jobs need an initial pass value. Setting it to 0 lets them dominate; setting it to the global minimum lets them merge in cleanly but requires tracking that minimum.
      Remark. Each context switch is not free. Choosing a longer time slice amortizes that cost at the price of response time.
      Remark. I/O-bound jobs are commonly handled by splitting a job into its non-I/O segments and scheduling them individually — turning one long-but-mostly-blocked job into many short, runnable ones.

      Memory [TQSZ]

      • August 16, 2025
      • Kellen Kanarios

      The address-space abstraction, the translation schemes that implement it, the algorithms that manage free space within a process, and what happens when physical memory is not enough.

      We are now moving on from scheduling into the wonderful world of all things memory. The starting point is the address space abstraction — each process is handed its own private virtual memory and the OS, with hardware help, manages the mapping from virtual to physical addresses.

      The progression of translation schemes covered in TPE:

      1. Base and bounds — one base+bound pair per process. Simple but wasteful.
      2. Segmentation — per-region base+bound. Removes internal fragmentation, introduces external fragmentation.
      3. Paging — fixed-size translation units. Eliminates external fragmentation at the cost of a translation table.

      Address space [7MH4]

      • May 12, 2026
      • Kellen Kanarios

      The chunk of memory the OS gives a process to use at its own discretion. Each process has its own address space, so that one process cannot read or overwrite another's program state. An address space typically contains three regions:

      1. Code: the program text.
      2. Stack: local variables, function arguments, return values.
      3. Heap: dynamically allocated memory.

      Every address a process sees is virtual — not the location in physical memory. The hardware translates virtual addresses to physical ones at access time. Approaches to that translation include Base and bounds, Segmentation, and Paging.

      Remark. The API the user sees is tiny: a way to get memory and a way to release it, exposed as malloc and free. The common bugs are forgetting to initialize, forgetting to free, freeing too early (dangling pointer), freeing twice (double free), and freeing memory not from malloc (invalid free).
      Remark. malloc and free are library calls, not syscalls. The heap is pre-allocated; malloc extends it by invoking brk/sbrk when it runs out.
      Remark. When a process dies, the OS reclaims its memory, so a bounded leak is not catastrophic — just bad practice.
      Remark. You can also acquire an anonymous heap-like region directly via mmap.

      Base and bounds [9MPY]

      • May 12, 2026
      • Kellen Kanarios

      The simplest hardware mechanism for Address space translation. The CPU has two dedicated registers per process — a base pointing to the start of the process's region in physical memory and a bound marking the end. Translation is just phys = base + virt, with a check that the result stays below bound. The unit performing this translation in hardware is the memory management unit (MMU).

      The hardware must expose privileged instructions for the OS to set base and bound on a context switch, and trap on out-of-bound accesses.

      Problem. To support a large address space, base-and-bounds suffers significant internal fragmentation — the unused gap between the stack and heap is allocated but wasted. With 16KB total memory and 4KB-but-mostly-empty processes, you can run 4 processes when you'd really have room for 8.

      Segmentation is the natural next step — give each region (code, stack, heap) its own base and bound so the gaps don't have to be allocated.

      Segmentation [REKW]

      • May 12, 2026
      • Kellen Kanarios

      An Address space-translation scheme that gives each logical region of a process (code, heap, stack, ...) its own base and bound. Instead of allocating a single contiguous chunk with empty space between regions, the OS allocates each segment separately and on demand. A virtual address is split into a segment number and an offset within that segment:

      \[\underbrace {0 1}_{\text {segment}}\underbrace {0 1 \cdots 11}_{\text {offset}}\]

      The hardware uses the segment number to index a segment table, retrieves that segment's base and bound, and adds the offset. Per-segment protection bits make read-only code and no-execute data straightforward.

      Segmentation also enables memory sharing — two processes can map the same physical segment (e.g. shared code) into both their address spaces.

      Remark. The stack grows downward, so the bounds check has to be direction-aware based on the segment number.
      Remark. Segmentation does not solve fragmentation; it pushes it from internal (per-process) to external (per-system). Variable-sized segments leave variably-sized holes in physical memory. The fix that eventually won is Paging.

      Paging [SB61]

      • July 31, 2025
      • Kellen Kanarios

      Instead of allocating memory in variable-sized chunks (which fragments badly), paging divides virtual and physical memory into fixed-size pages and maps virtual pages to physical pages independently. This is the address-translation scheme that won.

      The mechanism rests on three concepts:

      1. The Page table — per-process map from virtual page number to physical page number.
      2. The Translation lookaside buffer — small hardware cache that absorbs the cost of consulting the page table on every access.
      3. The Multi-level page table — the page table is itself paged, so processes only pay for the parts of the address space they use.

      When a process touches a virtual address whose page isn't in physical memory, the page table's present bit is unset; the hardware traps to the OS as a page fault, and the OS retrieves the page from swap. The eviction policies that decide what to swap out are covered by Page replacement.

      The two-level decomposition did not arrive fully formed. The Segmented page tables design tried to give each named segment of the address space its own variable-sized page table; the fragmentation that produced is what motivated the move to fixed-size paged page tables.

      Page table [WALE]

      • May 12, 2026
      • Kellen Kanarios

      The per-process data structure that maps each virtual page number (VPN) to a physical page number (PPN). A virtual address is split into a VPN and an offset within the page; the page table translates the VPN, the offset is appended unchanged.

      \[\underbrace {01}_{\text {VPN}}\underbrace {110\cdots 01}_{\text {offset}}\]

      Each page-table entry (PTE) carries more than just a PPN:

      1. Valid bit: whether this VPN has memory allocated. Gaps between stack and heap need not be allocated.
      2. Protection bits: read/write/execute permissions.
      3. Present bit: whether the page currently resides in physical memory (vs. swap).
      4. Dirty bit: has the page been modified since being paged in.
      5. Reference bit: has the page been accessed (used by eviction policy).
      Problem. The table gets very large. For a 32-bit address space with 4KB pages and 4-byte PTEs, a linear page table costs \(2^{32}/2^{12} \cdot 4 = 2^{22}\) bytes — 4MB per process.
      Problem. Translation requires accessing the table in memory, which is slow on every memory reference. Mitigated by the Translation lookaside buffer.

      The size problem is mitigated by Multi-level page table, which only allocates the parts of the table that are actually used.

      Translation lookaside buffer [FGOK]

      • May 12, 2026
      • Kellen Kanarios

      A small hardware cache of recent virtual-to-physical translations. Without it, every memory reference would incur the cost of walking the Page table in memory.

      An access proceeds as:

      1. Extract VPN from the virtual address.
      2. Check whether the VPN is in the TLB.
      3. TLB hit: read PPN from the entry, append offset, access memory.
      4. TLB miss: walk the page table (in hardware on CISC, by trapping to OS on RISC), insert the translation into the TLB, retry.

      A TLB entry holds the VPN, the PPN, a valid bit (separate from the PTE valid bit — invalid on init, eviction, context switch), and protection bits.

      Problem. Each process has different translations for the same VPN, so a naive TLB must be flushed on every context switch.
      Solution. Add an address space identifier (ASID) to each TLB entry and an ASID register set on context switch. Now entries from different processes coexist.

      Definition. Culler's Law [P9UM]

      • July 31, 2025
      • Kellen Kanarios

      "Ram is not always ram". Referring to the fact that a physical memory access takes a substantial different amount of time depending on whether it hit in the TLB.

      Remark. In real software-managed TLBs, an entry has global bits for shared pages, an 8-bit ASID, coherence bits, and a wired register reserving slots for code that cannot afford a TLB miss (e.g. the TLB-miss handler itself).
      Remark. For very fast caches, even a TLB access is a bottleneck. Virtually indexed caches sidestep the TLB on the index path.

      Definition. Segmented page tables [HBFS]

      • May 13, 2026
      • Kellen Kanarios

      An early translation scheme that splits the virtual address space into a small number of named segments (code, data, stack, heap, ...) and gives each segment its own Page table sized only to the range of addresses that segment actually uses. The virtual address is interpreted as a triple: segment number, page number within the segment, and offset within the page.

      Translation is a two-step walk. The segment number indexes a segment table, which yields the base and length of that segment's page table. The page number then indexes that page table to find the physical frame, and the offset is appended. Empty or unused segments need no page table at all, so a process pays for memory it actually uses — the same motivation that drives every non-flat translation scheme.

      The mechanism has appealing side benefits. Each segment carries its own protection bits, so the architecture can enforce no-execute on the stack or read-only on the code segment at the segment-table level without per-page complexity. Translation overhead is bounded by the segment count, which is typically very small.

      The problems are what killed it. Page tables are now variable-sized objects that still have to live in contiguous physical memory. The stack segment's page table might be small; the heap's might be large and grow over time. Allocating, growing, and freeing variably-sized tables in kernel memory produces external fragmentation — exactly the headache pure segmentation has at the user level, now pushed down a layer. If the heap outgrows its current page table, the kernel has to allocate a larger contiguous chunk and copy.

      Secondary issues compound. Translation costs two memory references (segment table, then page table) before reaching data, which hurts without a Translation lookaside buffer. The segment count is fixed by the architecture, which is awkward for programs whose layout does not fit "code/data/stack/heap" — a sprawling memory-mapped region or a JIT wanting many small protection domains has nowhere to go. And segmentation tends to leak into the programming model: x86 carried segment registers around for decades after they had stopped being useful.

      The fix that won is the Multi-level page table — page the page table itself, in fixed-size chunks, and recover the "only pay for what you use" property without variable-sized contiguous allocations. Every modern architecture descends from that decision.

      Multi-level page table [VX1M]

      • May 12, 2026
      • Kellen Kanarios

      A page table that is itself paged: instead of one linear array of PTEs, the table is split into page-sized chunks indexed by a top-level page directory. Subtrees of the address space that are entirely unused get a single null entry at the top level — no second-level page is allocated for them. This is the design that displaced Segmented page tables — it preserves the "don't pay for empty regions" property while keeping every piece of the structure page-sized, eliminating the variable-allocation fragmentation that doomed the segmented version.

      Advantages over a linear Page table:

      1. Fits easily in memory — only allocate a second-level page when an entry in it is used.
      2. Memory cost is proportional to the amount of address space actually in use.
      Problem. On a Translation lookaside buffer miss, the walk now costs two memory loads — page directory entry first, then the PTE.
      Remark. Real architectures push this further (x86-64 uses 4 or 5 levels) and lean on the TLB to absorb the walk cost.

      Page replacement [HK73]

      • August 8, 2025
      • Kellen Kanarios

      Once swap space is in play, the OS must decide which pages to evict from physical memory. The mechanism is the Page fault machinery; the question of policy is what this hub covers.

      Page fault [CXFM]

      • May 12, 2026
      • Kellen Kanarios

      Physical memory is finite, so the OS reserves a region of disk — swap space — to hold pages that don't fit. Pages currently on disk have their present bit cleared in the Page table entry; the PTE instead holds a disk address.

      When a process touches a virtual address whose page is not present, the hardware traps to the OS. This is a page fault. The handler:

      1. Looks at the PTE: if invalid, the access is illegal — terminate the process.
      2. If valid but not present: read the page in from swap, update the PTE, retry the access.
      Remark. The hardware never handles page faults itself — that would require it to understand disk I/O and the OS's swap layout. Page-fault handling is always in software.

      The three cases on every memory access:

      1. Present + valid: hardware reads the PPN from the PTE and proceeds.
      2. Not present + valid: hardware invokes the page-fault handler to retrieve from swap.
      3. Not valid: hardware raises an exception; the OS typically kills the process.
      Example. What happens when you dereference a nullptr? The TLB misses on VPN 0, the page-table walk finds the PTE marked invalid, and the hardware traps to the OS — which delivers SIGSEGV.

      LRU and the clock algorithm [7WG2]

      • May 12, 2026
      • Kellen Kanarios

      Least recently used is the standard heuristic for picking a page to evict: throw out the one that has been untouched the longest. It works because of the principle of locality — recently-used pages are likely to be used again soon (temporal locality).

      Pure LRU requires updating a data structure on every memory access, which is too expensive. The clock algorithm approximates it cheaply:

      1. Maintain a reference bit per page, set by hardware on access.
      2. Loop a pointer around the pages. If the reference bit is 1, clear it and advance.
      3. If the reference bit is 0, evict that page.
      Remark. LRU is not optimal. The pathological case is the looping workload: if the working set is one page larger than the cache, LRU misses 100% of the time.

      The quality of any replacement policy is measured by its average memory access time:

      \[AMAT = (1 - P_{miss}) T_P + P_{miss} T_D\]

      where \(T_P\) is physical-memory access time and \(T_D\) is disk access time. The disk gap is so large that even tiny miss rates dominate AMAT.

      Belady's anomaly [LQV3]

      • May 12, 2026
      • Kellen Kanarios

      A desirable property of a page-replacement policy is the stack property: increasing the cache size never makes the miss rate worse. Surprisingly, not every reasonable-looking policy has it.

      FIFO eviction violates the stack property — on certain access sequences, growing the cache strictly increases the number of misses. This is Belady's anomaly. It is what motivates LRU and the clock algorithm-style policies, which do have the stack property because they only depend on relative access recency.

      The optimal eviction policy (offline) is to evict the page whose next use is farthest in the future. Unimplementable in practice (you don't know the future), but useful as a theoretical upper bound to compare against.

      Two refinements make eviction practical on real workloads. A pure global policy lets one greedy process hog memory, which Segmented FIFO fixes with a per-process resident-set cap; and writing victims back one page at a time wastes disk bandwidth, which Page clustering amortizes by batching the dirty list.

      Segmented FIFO [HF6H]

      • July 11, 2026
      • Kellen Kanarios

      A process-aware refinement of LRU and the clock algorithm. Pure FIFO is a global policy — it does not distinguish memory by owner, so one greedy process can evict everyone else's pages. Segmented FIFO instead maintains a per-process FIFO queue capped at that process's resident set size, bounding how much physical memory any single process can hold.

      Evicted pages are not discarded outright; they move to one of two secondary lists:

      1. Clean page free list — evicted pages that were never modified.
      2. Dirty page list — evicted pages that must be written back before reuse.

      The secondary lists act as a victim cache: if a process faults on a page still sitting on the dirty list, it can be reclaimed without going to disk. The resident-set cap gives fairness; the secondary lists soften the cost of capping it too aggressively.

      Copy-on-write [DEJB]

      • May 12, 2026
      • Kellen Kanarios

      An optimization where two address spaces share the same physical page until one writes to it. Both processes have PTEs pointing at the same PPN, but with the write bit cleared. The first write traps to the OS, which allocates a fresh page, copies the contents, points the writing process's PTE at the new page (with writes re-enabled), and resumes.

      Used pervasively:

      • fork() — the child shares the parent's pages until either modifies one. Avoids the cost of copying an entire address space the child often immediately overwrites with exec().
      • Shared libraries — many processes map the same library text pages.
      • Kernel mapped into every process — same physical kernel code seen from every address space.

      On-demand zeroing [PW7Q]

      • May 12, 2026
      • Kellen Kanarios

      When a process asks for new heap memory, the OS must guarantee the returned pages contain zeros (to avoid leaking other processes' data). The naive implementation zeroes the page at allocation time. On-demand zeroing defers that work to first touch: until the process actually accesses a freshly-allocated page, no zero-fill happens.

      The OS marks the page as not-present in the Page table. The first access traps as a Page fault, the handler zeroes a physical page, installs it in the PTE, and resumes. Heap memory that is allocated but never used costs nothing.

      Page clustering [YMHF]

      • July 11, 2026
      • Kellen Kanarios

      An amortization trick for the write side of page replacement. Writing one dirty page to swap at a time pays a full disk I/O per eviction, and small random writes are the worst case for a disk. Page clustering batches them: the OS collects a group of dirty pages from the global dirty list and writes them out together in a single larger, more sequential I/O. The per-page cost of paging out falls because the fixed overhead of a disk write is spread across the whole batch.

      Remark. A common convention is to map the kernel into every process's address space (via Copy-on-write) so kernel code and structures appear at the same virtual addresses everywhere.

      Free-space management [67X5]

      • July 29, 2025
      • Kellen Kanarios

      Once we have an Address space, we still have to actually allocate physical bytes within it. Free-space management is the discipline of tracking which bytes are free, picking which free chunk to satisfy a request, and merging adjacent freed chunks back together. Two fragmentation hazards to watch:

      1. External fragmentation: many small unallocated non-contiguous chunks.
      2. Internal fragmentation: allocated chunks larger than what the requester uses.

      Simplifying assumptions for the algorithms below: allocations are immovable once placed, total memory is fixed, and the user does not specify a size in free. The last assumption is handled by storing the size in a header preceding the returned pointer.

      Malloc placement policies [1ABF]

      • May 12, 2026
      • Kellen Kanarios

      Given a free list of unallocated memory chunks and a request of size \(n\), which chunk do you carve the allocation out of? Different answers trade off fragmentation against search time.

      1. Best fit: find the smallest chunk \(\geq n\). Minimizes wasted space per allocation but tends to leave many small unusable fragments behind, and requires scanning the whole list.
      2. Worst fit: find the largest chunk. Aims to leave usefully-sized leftovers but also requires a full scan.
      3. First fit: take the first chunk that fits. Fast (early exit) but biases small fragments toward the start of the list.
      4. Next fit: like first fit, but resume the search from where the previous allocation ended. Spreads small fragments more evenly.

      On free, the allocator searches the list for adjacent free chunks and coalesces them into a larger chunk. The size of each allocation is stored in a header immediately preceding the returned pointer so free doesn't need a size argument.

      Segregated lists [FB66]

      • May 12, 2026
      • Kellen Kanarios

      A data-structure-level optimization for allocators that maintains separate free lists per common object size. Instead of one global free list scanned by Malloc placement policies, you keep a list per size class. Allocations of that size pop straight off the matching list — no search.

      • Grow a per-size list by requesting larger segments from the main allocator and chopping them up.
      • Drain back to the main list when no references remain.
      • Optionally leave objects pre-initialized on the list to skip constructor/destructor calls.
      Remark. This is the structure behind kernel slab allocators and many object-pool patterns. It works when the workload allocates many objects of a few common sizes.

      Buddy allocation [8M72]

      • May 12, 2026
      • Kellen Kanarios

      An allocator for memory of total size \(2^n\) that maintains a binary-tree of free blocks at every power-of-two size. To service a request, walk down to the smallest block \(\geq \) requested size; if no block of that size exists, split a larger one (and its "buddy" becomes free). On free, check whether the freed block's buddy is also free, and if so, coalesce upward — recursively.

      The key trick is that buddies differ by exactly one bit in their address at each level. To check whether a block's buddy is free, flip that bit and consult a bitmap. To coalesce upward, mask off the bit and recurse.

      Remark. Internal fragmentation is bounded by 2× the request size (you may have to round up to the next power of two), but external fragmentation is essentially eliminated within the buddy region.

      Threads and shared state

      • August 16, 2025
      • Kellen Kanarios

      Concurrency Intro [16TG]

      • August 16, 2025
      • Kellen Kanarios

      Concurrency comes from sharing: multiple Threads within a single process share the address space, which is what makes them cheap to spawn and to communicate between, and is also what makes them dangerous. When threads interleave on a shared variable, you get a Race condition. The fragile region is the Critical section, protected by synchronization primitives like Locks and Barriers.

      Thread [63U3]

      • May 12, 2026
      • Kellen Kanarios

      An independent unit of execution within a Process. Each thread has its own program counter, register set, and stack — but shares the surrounding address space, file descriptors, and other process-level resources with its siblings.

      To support a context switch between threads, the OS keeps a thread control block (TCB) per thread — the per-thread analog of Process control block. Each thread also gets its own stack within the process's address space.

      Why threads instead of processes:

      1. Intra-process parallelism: parallelizable work within one program, without paying the context switch cost of separate processes.
      2. Avoiding blocking I/O: one thread blocks waiting on I/O while others keep running.
      Remark. Sharing an address space is exactly what makes threads cheap to communicate between and exactly what makes them dangerous — see Race condition and Critical section.

      Race condition [VOQN]

      • May 12, 2026
      • Kellen Kanarios

      When the correctness of a program depends on the order in which concurrent Threads schedule, you have a race condition. The classic example: two threads each execute

      mov addr reg     // load
add 1 reg        // increment
mov reg addr     // store

      intending to increment a shared variable twice. If the OS interleaves the threads at the wrong instruction boundary:

      1. Thread 1 stores its incremented value.
      2. Thread 2 stores its incremented value.
      3. Thread 1 increments (in its register).
      4. Thread 2 increments (in its register).

      The result is a single increment, not two — both threads loaded the same starting value and the second store clobbered the first.

      The fix is to prevent interleaving inside the code region that touches shared state — the Critical section — using a synchronization primitive like Locks.

      Critical section [KG2K]

      • May 12, 2026
      • Kellen Kanarios

      A region of code that accesses shared state in a way that is incorrect if more than one Thread executes it concurrently. Critical sections are exactly the regions that need protection by a synchronization primitive: a Locks excludes other threads while one is inside; a Barriers forces all threads to reach the same point before any may continue.

      The art of writing correct concurrent code is to identify critical sections and protect them — without making them so large that you lose all parallelism.

      Locks [Q2DT]

      • March 9, 2026
      • Kellen Kanarios

      A lock is a synchronization primitive that grants mutually exclusive access to a critical section. The implementations below trade off coherence traffic, fairness, and memory in different ways. The naive locks (Test-and-set lock and Test-and-test-and-set lock) are simple but generate \(O(n)\) coherence traffic per release; queue-based locks (Ticket lock, Array-based lock, MCS lock) scale better at the cost of extra state per waiter.

      Test-and-set lock [MJ5L]

      • May 12, 2026
      • Kellen Kanarios

      The simplest spin lock. The acquire loop spins on an atomic test_and_set: it writes LOCKED and reads the previous value in one step. If the previous value was UNLOCKED, the caller has the lock; otherwise it tries again.

      acquire(lock_ptr):
  while (true):
    // Perform "test-and-set"
    // UNLOCKED = 0, LOCKED = 1
    test_and_set(old, lock_ptr)
    if (old == UNLOCKED):
      break // lock acquired!
    // keep spinning, back to top of while loop

release(lock_ptr):
  store[lock_ptr] <- UNLOCKED
      Problem. Major coherence traffic. Every spinning core must acquire write permissions and invalidate everyone else on each test_and_set, so contention scales badly with the number of waiters.

      Test-and-test-and-set lock [PXY9]

      • May 12, 2026
      • Kellen Kanarios

      A refinement of the Test-and-set lock that reduces coherence traffic during the spin. Threads first perform a plain load to check whether the lock looks free, and only attempt the atomic test_and_set when it does. Spinning is then read-only, so cores can share the cache line.

      acquire(lock_ptr):
  while (true):
    // Perform "test"
    load [lock_ptr] -> original_value
    if (original_value == UNLOCKED):
      // Perform "test-and-set"
      test_and_set(old, lock_ptr)
      if (old == UNLOCKED):
        break // lock acquired!
    // keep spinning, back to top of while loop

release(lock_ptr):
  store[lock_ptr] <- UNLOCKED
      Remark. Spinning is a read-only operation now.
      Problem. When the lock is released, the line must be invalidated on every waiter, and they all simultaneously read and try to write. The contention is shifted from steady state to release time but is still \(O(n)\) in the number of waiters.

      Ticket lock [V7KP]

      • May 12, 2026
      • Kellen Kanarios

      A fair spin lock based on two counters. Each arriving thread takes a "ticket" by atomically incrementing next_ticket and then spins until now_serving equals its ticket. Release simply increments now_serving. Threads acquire the lock in FIFO order, which prevents starvation but does not by itself reduce contention.

      acquire(lock_ptr):
  my_ticket = fetch_and_increment(lock_ptr->next_ticket)
  while (lock_ptr->now_serving != my_ticket); // spin

release(lock_ptr):
  lock_ptr->now_serving = lock_ptr->now_serving + 1
      Problem. Still \(O(n)\) coherence traffic on release: incrementing now_serving invalidates the line on every waiter, and they all reload it to recheck their ticket.

      Array-based lock [BJ7N]

      • May 12, 2026
      • Kellen Kanarios

      Each waiter is assigned its own slot in an array and spins only on that slot. Release just flips the next slot to "go-ahead", so the cache line touched by release is the one read by exactly one other thread. Coherence traffic per release is constant rather than \(O(n)\), at the cost of one slot per waiter.

      • Give each waiter its own "slot" to wait on.
      • Only need to set the next slot to "go-ahead" on release.
      • Constant coherence traffic.
      Problem. \(O(n)\) memory for the lock array, and padding each slot to \(64\)b (one cache line) to avoid false sharing makes that cost real.

      MCS lock [IN64]

      • May 12, 2026
      • Kellen Kanarios

      A scalable, list-based queue lock. The lock variable is a pointer to the tail of a linked list of per-thread nodes. Each thread enqueues its own node and spins on a flag in its own cache line. The releaser wakes only its direct successor, so coherence traffic per handoff is constant and memory cost is proportional only to threads currently waiting, not to the lock's fanout.

      acquire(lock):
  I->next = null
  predecessor = fetch_and_store(lock, I)
  if predecessor != nil:                   // some node holds the lock
    I->must_wait = true
    predecessor->next = I                  // predecessor must wake us
    repeat while I->must_wait              // spin till lock is free

release(lock):
  if (I->next == null):                    // no known successor
    if compare_and_swap(lock, I, nil):     // make sure...
      return                               // CAS succeeded; lock freed
    repeat while I->next == nil            // spin to learn successor
  I->next->must_wait = false               // wake successor

      Barriers [9QUX]

      • May 12, 2026
      • Kellen Kanarios

      A barrier holds threads at a synchronization point until all \(P\) participants have arrived, then releases them simultaneously. The naive implementation uses a shared counter that each arriving thread decrements; the last arrival resets the counter and the rest spin until it returns to \(P\).

      Naive central barrier

      • May 12, 2026
      • Kellen Kanarios
      global (shared) count : integer := P

procedure central_barrier
  if fetch_and_decrement(&count) == 1
    count := P
  else
    repeat until count == P
      Problem. When count == P, a fast thread can reach the next barrier and decrement count again before the slow threads observe the reset. The slow threads then spin forever waiting for a value that has already passed.

      Sense-reversing barrier

      • May 12, 2026
      • Kellen Kanarios

      The fix is to give each thread a private local_sense that flips on each barrier. The global sense flag is flipped by the last arrival, and waiters spin on sense == local_sense, so a fresh barrier cannot be confused with a stale one.

      global (shared) count : integer := P
global (shared) sense : bool := true
global (private) local_sense : bool := false

procedure central_barrier
  local_sense = !local_sense
  if fetch_and_decrement(&count) == 1
    count := P
    sense = local_sense
  else
    repeat until sense == local_sense
