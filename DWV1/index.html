Routing deadlock [DWV1]

  • May 12, 2026
  • Kellen Kanarios

Routing deadlock occurs when a packet cannot make progress because the buffer at its next link is full, and the packet holding that buffer is itself blocked waiting on a buffer this packet holds — a cycle in the channel dependency graph.

Two structural responses:

  1. Avoid deadlock by construction. Dimension-order routing enforces an ordering on dimension transitions so the dependency graph is acyclic by design.
  2. Detect and recover from deadlock. Allow all turns. Detect cycles via timeouts. Drain stuck packets onto a backup network (a "slow channel" — physical, or more commonly an escape virtual channel) that uses dimension-order routing.

The relationship to other related concepts:

  • Backpressure (Network backpressure) is the signaling mechanism that lets a router downstream tell upstream it is congested, before the upstream actually fills the link.
  • Adaptive routing avoids deadlock locally; backpressure lets adaptive routing react to remote congestion.

Context

    On-chip interconnection networks [667T]

    • April 8, 2026
    • Kellen Kanarios

    The network connecting cores, L2/L3 banks, and memory controllers on a Multicore chip. The same abstractions that govern off-chip networks (topology, routing, flow control) apply, but the constraints are different: wires are abundant but die area is scarce, latency is dominated by hops rather than propagation, and the network is part of the Cache coherence critical path.

    On-chip network metrics [OMZU]

    • May 12, 2026
    • Kellen Kanarios

    The measurements used to compare network topologies and traffic patterns.

    Traffic-independent

    • May 12, 2026
    • Kellen Kanarios
    • Degree: number of links at each node. A proxy for router complexity — each port costs a buffer, crossbar input, state machine.
    • Bisection bandwidth: bandwidth across a cut that splits the network into two equal-sized halves. The minimum over all such cuts. A proxy for peak achievable throughput, especially for uniformly random traffic.
    • Diameter: maximum hop distance between any two nodes. The worst-case hop count.
    Remark. Bisection bandwidth is less useful for on-chip networks than off-chip because global on-chip wiring is plentiful.
    Remark. Ring bisection is 2 (two cuts to bisect); torus bisection is 4.

    Traffic-dependent

    • May 12, 2026
    • Kellen Kanarios
    • Hop count: hops between two specific nodes. The average over a traffic distribution is a useful proxy for latency.
    • Maximum channel load: the most-loaded link's load relative to per-node injection bandwidth, for a given traffic pattern. The proxy for sustained throughput.
    • Path diversity: number of distinct shortest paths between two nodes. Higher path diversity allows adaptive routing to spread load.

    Performance regimes

    • May 12, 2026
    • Kellen Kanarios
    • Zero-load latency: average hop count with no contention.
    • Saturation throughput: the injection rate at which latency curves upward exponentially.
    Remark. Hop count is not the same as latency. Per-hop pipeline depth and link propagation matter — more short hops can beat fewer slow ones. Router pipeline depth isn't always known in advance.

    Direct vs indirect networks [95AC]

    • May 12, 2026
    • Kellen Kanarios

    Two organizational families for on-chip networks.

    • Direct — every terminal node (core, memory controller) is associated with a router. Most on-chip designs are direct because area is precious and there's no point spending a router with no compute attached. The "\(k\)-ary \(n\)-cube" family (Ring topology, Mesh topology, Torus topology) is direct.
    • Indirect — routers may exist that connect only to other routers, with no terminal attached. Butterfly network and Clos network are indirect; a router in their middle stages just shuffles bits between input and output stages.
    Remark. Indirect networks can shorten the diameter for the same node count, but they pay for extra silicon area for routers that don't serve traffic locally. Often the right call off-chip; usually the wrong call on-chip.

    Topologies

    • April 8, 2026
    • Kellen Kanarios

    Definition. Crossbar [YV5J]

    • May 12, 2026
    • Kellen Kanarios

    The simplest indirect network: an \(n \times m\) grid of switch points connecting \(n\) inputs to \(m\) outputs. Any input can be connected to any free output simultaneously — non-blocking.

    • One hop per transit (minimum possible).
    • Serializes if multiple senders target the same receiver.
    • Switch area grows as \(O(n \cdot m)\) — the dominant cost as systems grow.
    Remark. Crossbar router area scales as the square of the port count, which is why high-radix routers are dramatically more expensive than low-radix ones.

    Crossbars are common as the inner switch of larger structured networks (Clos network, Fat tree) — large flat crossbars are impractical at scale, but small crossbars are cheap.

    Definition. Ring topology [UEA0]

    • May 12, 2026
    • Kellen Kanarios

    A direct network where each node connects to two neighbors, forming a closed loop. Bidirectional rings allow either direction of travel; unidirectional rings allow only one. The simplest topology to implement and reason about.

    In the \(k\)-ary \(n\)-cube family, a ring is a \(k\)-ary 1-cube.

    • Degree 2 (every node).
    • Diameter \(\lfloor k/2 \rfloor \) for a bidirectional \(k\)-node ring.
    • Bisection bandwidth 2 — cut the ring at exactly two links.
    • Hardware overhead very low: routers have two ports.
    Problem. Worst-case hop count is \(O(k)\). Doesn't scale beyond a small number of nodes.
    Remark. Despite the poor scaling, rings are common in commercial CPUs (Intel's L3 ring bus on Sandy/Ivy Bridge) because they're cheap and 8–16 cores is enough.

    Definition. Mesh topology [9SA2]

    • May 12, 2026
    • Kellen Kanarios

    A 2D grid of nodes with neighbors connected on the four compass directions, no wrap-around. The dominant on-chip topology in commercial designs (Xeon Phi, recent server x86 chips) because it lays out naturally on a 2D die.

    • Interior nodes degree 4; edge nodes degree 3; corners degree 2.
    • Average hop count for an even-\(k\) mesh: \[H_{avg} = \frac {nk}{3}\]
    • Maximum channel load for uniform random traffic: \(k/4\) — twice that of a Torus topology.
    Problem. Non-uniform node degree. Edge nodes carry less traffic than they could, while interior nodes bottleneck on traffic flowing through them.
    Remark. Mesh dominates Torus topology in practice despite the topology metrics being worse, because torus wrap-around wires are physically long and force expensive folding to equalize link lengths. Manufacturing wins over theoretical performance.

    Definition. Torus topology [DDKT]

    • May 12, 2026
    • Kellen Kanarios

    A Mesh topology with wrap-around links connecting the ends of each dimension. In the \(k\)-ary \(n\)-cube classification, a \(k\)-ary \(n\)-cube. Uniform node degree (every node has degree \(2n\)), uniform load distribution, half the hop count of a mesh.

    • Every node has degree \(2n\).
    • Average hop count for even \(k\): \[H_{avg} = \frac {nk}{4}\]
    • Maximum channel load for uniform random traffic: \(k/8\) — half that of a Mesh topology.
    • Good path diversity: many shortest paths between any two nodes.
    Problem. Wrap-around links are physically long. To equalize wire delays, the torus must be folded, doubling effective wire lengths and link area. The wrap-around also doubles the link count versus a mesh of the same dimensions.
    Remark. Despite the theoretical advantage, mesh wins on real chips. Tori show up in HPC interconnects (Cray) where the wires can run on dedicated cables.

    Variants:

    • \(k\)-ary \(n\)-cube: \(k\) nodes along each of \(n\) dimensions, all ends wrapped.
    • Asymmetric: \((i, j, k)\)-ary — different node count per dimension.
    • Mesh: same as torus but without wrap-around.
    • Ring: \(k\)-ary 1-cube.

    Definition. Butterfly network [5WX9]

    • May 12, 2026
    • Kellen Kanarios

    A multistage indirect network. A \(k\)-ary \(n\)-fly has \(k^n\) terminal nodes connected through \(n\) stages of \(k \times k\) crossbar switches. At each stage, the routing decision selects which of \(k\) sub-groups to enter; after \(n\) stages, every terminal can reach every other.

    • Hop count \(n - 1\).
    • Under uniform random traffic, maximum channel load is 1 (perfectly balanced).
    • Channel load drops further for non-uniform traffic that respects the natural partitioning.
    Problem. Zero path diversity. There is exactly one path between any source and destination. A single hot link wedges traffic.
    Solution. Add shuffle stages that permute paths after each routing stage, so the same source-destination pair can be reached by multiple routes.

    Definition. Clos network [YKAN]

    • May 12, 2026
    • Kellen Kanarios

    A three-stage indirect network parameterized by \((m, n, r)\): \(r\) input switches and \(r\) output switches each of size \(n \times m\), with \(m\) middle-stage switches each \(r \times r\). Total terminal nodes \(N = nr\). Non-blocking when \(m \geq 2n - 1\).

    • Number of total switching elements scales as \(N^{1.5}\) versus \(N^2\) for a flat Crossbar while preserving non-blocking.
    • Path diversity is \(m\) — multiple distinct paths between any source and destination.
    Proof. There are \(rmn\) input-switch elements, \(mr^2\) middle-switch elements, and \(rmn\) output-switch elements: \[S = 2rmn + mr^2.\] With \(r = N/n\): \[S = 2Nm + m(N/n)^2.\] Choosing \(n = \sqrt {N}\) and \(m \geq 2n - 1 = O(\sqrt {N})\): \[S = 4mN = O(N^{1.5}).\]
    Remark. A folded Clos network — input and output stages combined, making \(n\) input ports and \(m\) output ports bidirectional — is structurally the same as a Fat tree.

    Definition. Fat tree [DFC7]

    • May 12, 2026
    • Kellen Kanarios

    A tree topology with bandwidth that doubles at every level toward the root. Each interior switch has logical degree 4, but as you climb the tree, link width doubles to absorb the combined traffic of its subtree — preserving bisection bandwidth.

    Structurally, a fat tree is a folded Clos network: the input and output stages of a Clos are merged into a single bidirectional stage at each level of the tree, and the middle stage becomes the higher levels of the tree.

    • Each switch has logical degree 4 (2 down, 2 up — with link doubling).
    • Bisection bandwidth scales linearly with the number of leaves.
    • Path diversity grows toward the root.
    Remark. Fat trees are the dominant topology for off-chip HPC interconnects (InfiniBand fat-tree clusters). Less common on-chip because the doubling links are hard to lay out on a 2D die.

    Network routing strategies [VTS4]

    • May 12, 2026
    • Kellen Kanarios

    Given a topology and a packet's source and destination, which path through the network does it take? Three classes of answer:

    • Greedy (shortest-path) — always pick the shortest available route.
      • (+) Minimizes latency under light load.
      • (-) Overloads central links; worst-case throughput poor.
    • Uniform / oblivious random — pick a direction at random.
      • (+) Spreads load; avoids systematic hotspots.
      • (-) Wastes bandwidth on non-minimal paths; latency unpredictable.
    • Adaptive — at each hop, pick the outgoing link with the lowest local congestion.
      • (+) Dynamically balances load; best worst-case throughput.
      • (-) Routers need congestion state; risk of livelock without care.
    Example. Tornado traffic on an 8-node bidirectional ring: node \(i\) sends to \((i+3) \bmod 8\).
    • Greedy: all packets travel counter-clockwise. Bidirectional bandwidth wasted; \(\frac {1}{3}\) utilization.
    • Uniform: randomly clockwise/counter-clockwise. Counter-clockwise gets \(\frac {2}{3}\) utilization, clockwise \(\frac {2}{5}\); worst-case \(\frac {2}{5}\).
    • Adaptive: sends \(\frac {5}{8}\) counter-clockwise to match traffic exactly. Best.
    Remark. Adaptive routing tends to need Network backpressure signaling to know where congestion is, and care to avoid livelock when many adaptive decisions chase each other.
    Remark. Oblivious strategies like Dimension-order routing and Valiant's routing algorithm give deterministic guarantees (deadlock-free, bounded path length) at the cost of giving up the load-balancing of true adaptive routing.

    Definition. Dimension-order routing [83P2]

    • May 12, 2026
    • Kellen Kanarios

    An oblivious Network routing strategies for \(k\)-ary \(n\)-cube networks (Mesh topology, Torus topology): always exhaust the offset in one dimension completely before changing to the next. E.g. in 2D mesh routing, route X-then-Y — never turn back to X after starting Y.

    The dimension order prevents routing deadlock: cycles in the channel dependency graph are impossible because no packet ever requests a channel in a "lower" dimension once it has started using a higher one. The dependency graph is therefore acyclic, and the standard cycle-equals-deadlock argument applies.

    • (+) Provably deadlock-free without virtual channels.
    • (-) Zero path diversity. Hot spots can't be routed around.
    • (-) Unaware of current network state — oblivious.

    The standard remedy for the path-diversity problem is Valiant's routing algorithm, which keeps the deadlock-freedom of dimension-order routing while spreading load with a randomized intermediate hop.

    Definition. Valiant's routing algorithm [XLDA]

    • May 12, 2026
    • Kellen Kanarios

    An oblivious Network routing strategies that achieves provably optimal worst-case throughput by introducing randomness in path selection:

    1. For a packet going from \(s\) to \(d\), randomly pick an intermediate node \(w\).
    2. Route \(s \to w\) using Dimension-order routing.
    3. Eject the packet at \(w\), then re-inject it.
    4. Route \(w \to d\) using Dimension-order routing.

    To preserve deadlock-freedom, the intermediate node must remove the packet from the network before re-injecting — otherwise the channel dependencies of the two halves could interlock.

    Remark. Restricting \(w\) to lie in the sub-quadrant defined by \(s\) and \(d\) keeps paths near-minimal while still spreading load — the practical compromise between guaranteed throughput and added latency.
    Remark. Valiant's algorithm trades average-case latency (paths are longer) for worst-case throughput (any traffic pattern looks roughly uniform once randomized). The right call for adversarial workloads; overkill for uniform traffic.

    Network backpressure [EZDD]

    • May 12, 2026
    • Kellen Kanarios

    A signaling mechanism that lets a downstream router tell upstream "I am congested — slow down" before its buffers actually overflow. With backpressure information, adaptive routing can route around remote congestion, not just local.

    The mechanism propagates: a congested link signals back to its sender, which then knows its next link will fill, which signals further back. Useful information flows in the opposite direction of the data.

    Problem. Acting only on local state can cause livelock — packets bounce between hotspots, never converging.

    Backpressure breaks the livelock by giving local routers global-ish information without an actual global view.

    Remark. Backpressure is mechanism; adaptive routing is policy. The two together are what real high-radix on-chip networks use.

    Interface with the cache hierarchy

    • April 8, 2026
    • Kellen Kanarios

    The on-chip network connects each core (with its L1) and the L2 cache banks (which may be shared or private). Each L2 bank has a cache controller (manages Cache coherence) and, at memory boundaries, a memory controller (talks to DRAM).

    Two cache-organization choices interact with the network design:

    • Private L2 caches: each tile owns its L2 slice; cross-tile traffic happens only on private-L2 miss. (+) Lower hit latency; less interconnect traffic. (-) Replicated data; more off-chip traffic on misses.
    • Shared L2 cache: a single logical L2 split into banks, with each address mapped to a home bank. (+) Better utilization, no replication. (-) Every L1 miss goes on the network.

    The interface to the network sits at the cache controller:

    • Miss status handling register (MSHR): on a cache miss, records what request the cache is waiting on. Matched against the reply when it arrives.
    • Transaction status handling register (TSHR): tracks outstanding memory transactions to off-chip memory from the directory cache.
    Remark. Memory controllers are commonly placed at the chip perimeter, near the I/O pads. A memory controller can share a router with a core/cache tile (saves area; needs arbitration) or have its own (more area; no contention).
    Remark. Hardware-managed shared memory is one design point; another is explicit message passing, where software (not hardware) is responsible for moving data between processes.

    Implementation considerations

    • April 8, 2026
    • Kellen Kanarios

    Links. Thicker wires with larger spacing → lower resistance, higher speed, better energy efficiency. The wire budget determines how many ports you can afford per router.

    Routers. Each port costs a buffer, a crossbar input, state logic, and a link to its neighbor. Crossbar area scales as the square of port count, so high-radix routers are dramatically expensive. Rings have very simple routers but poor performance.

    Floorplan. Tori need to be folded to equalize wire lengths, doubling effective wire length; meshes don't. Tori also have twice as many links as meshes, which may force narrower links and lower per-link bandwidth.

    Folding can also equalize path lengths inside a network at the cost of increasing average latency.

    Concentration allocates more cores per router. (-) Decreases throughput per node. (+) Programs with high locality do little remote communication, so concentration is a good fit there.

    Custom and hierarchical topologies

    • April 8, 2026
    • Kellen Kanarios

    General topologies aren't always optimal. Two construction strategies:

    • Splitting: start with a large crossbar; split into smaller switches until constraints are met.
    • Merging: start with many small switches and combine.

    Hierarchical topologies use one topology within clusters and another across clusters — common when traffic has locality.

      Starting with MSI

      • March 24, 2026
      • Kellen Kanarios

      The first thing we need to do is understand all the things that can go awry.

      First in the valid-invalid protocol example we were given they only have two virtual channels. However, we need 3 channels to handle, request messages, forwarded messages, and response messages.

      Let us briefly understand why this is the case. Detecting a deadlock is equivalent to detecting a cycle in a dependency graph, so it is sufficient to show a cyclic dependency.

      • If we issue a GetS this goes to the directory, giving us an edge N1 -> D.
      • Then the directory issues a Fwd-GetS to the ownder, giving us the chain N1 -> D -> N2
      • The owner then needs to send data to the requester, giving us the chain N1 -> D -> N2 -> N1.
      • Namely, the data must go through the same channel as all of the requests depending on it.

      To avoid this, we partition into 3 channels:

      • Request: GetS, GetM, PutM
      • Forwarded: Fwd-GetS, Fwd-GetM, PutAck, Inv
      • Response: InvAck, Data

      We directly enumerate these in our MessageType as 

        MessageType: enum {
    GetS,           -- request for shared copy
    GetM,           -- request for modified (exclusive) copy
    PutS,           -- evict shared copy
    PutM,           -- evict modified copy (w/ data)

    FwdGetS,        -- forwarded GetS to owner
    FwdGetM,        -- forwarded GetM to owner
    Inv,            -- invalidation request

    Data,           -- data response
    InvAck,         -- invalidation acknowledgment
    PutAck          -- put acknowledgment
  };

      Directory controller. Now for the directory controller:
      Remark. We actually end up adding a second transient state to the directory controller \(M^D\). This is to handle the PutM, GetM race described below.
      Weird Races.
      Problem. When implementing the PutM rule to downgrade from modified to invalid. We must NOT get rid of the data until we have been acknowledged, so that we can still service forward requests. This is a transition to \(MI^A\). However, we can still receive the PutAck causing us to transition to \(I\) before servicing the Fwd-GetM request. This is lost data.
      Solution. We must add a \(M^D\) state to the directory controller. In this state, if we get a PutM request from a node that is not our current owner we stall and wait for data to arrive. For \(M^D\) to transition to \(M\), we must send the data back to the directory even though we are doing a three-hop, so that the directory knows the data has also been forwarded to the new owner. It is unclear to me the best way to do this without this state.
      Problem.

      The above solution causes a deadlock, where two processors both store miss and one load misses. The "winning processor" of the store will be the second one. The first one will get the data from the directory transitioning from \(IM^{AD} \to IM^{A}\), but the second one will then trigger a Fwd-GetM overwriting the first.

      The load miss transitions to \(IS^D\). However, when it receives an Inv it stalls according to the above figure. This blocks the Fwd-GetM from reaching the other winning processor because they are on the same VC. Thus, we have a deadlock where the owner never gets the data.

      Solution. We need to add another transient state not included in the figure. Namely, we add \(II^{A}\) that says we were in \(IS^{A}\), but we got invalidated and now we are waiting for our PutAck before we transition to \(I\).
      Remark. When the home directory is in \(M\) and we get a GetS we cannot clear owner as directed by figure. Otherwise in \(S^D\), we will not be able to tell if the PutM is from the owner and stall, so we only remove the owner once we have received the data.

      Moving to MESI

       Aside from the modifications discussed above, the rest was straightforward and we now are ready to move on to MESI.

      Cache controller. We will start with MESI cache controller, given by the following table:

      Design choice 1. Since we allow immediate upgrade to \(M\), without forwarding, if E upgrades to M, then the value in the directory sent to the next GetS / GetM would not have updated value. It seems we must either

      1. not allow E to silently upgrade to M and then we do not need forwarding, or
      2. allow E to silently upgrade to M and treat E as M basically and forward everything.
      Notably, neither of these have to write back on PutE, but we will go with 1 because that is how it is indicated in the figure.

      Design Choice 2. In order to indicate that the directory received the first GetS we introduce a ExclusiveData message. Upon receiving, ExclusiveData in \(IS^D\) we transition to \(E\).

      Remark. Once we have sent the PutE, or the PutM \(EI^A\) and \(MI^A\) are functionally the same. I include both for completeness but verified that transitioning directly to \(MI^A\) from \(E\) works the same.

      Directory controller. We start with the analagous table.

      Similar to the cache controller, we also add a \(E^D\) state for completion that can be absorbed into \(M^D\) introduced in the MSI protocol.

      Weird Races.

      Problem. First processor does GetS and is in \(IS^D\) but the Home directory sends it exclusive data and is now in \(E\). Then second processor does GetM and Home issues Fwd-GetM to processor one, which arrives before data.
      Solution. Need to stall on Fwd-GetM and Fwd-GetS in \(IS^D\) until data arrives
      Problem. An even crazier race happens when we get invalidated before the Fwd-GetM or Fwd-GetS arrive. Due to our previous design decision, we transition to \(II^A\), where if we receive a PutAck we transition to invalid. This then loses the needed forward data.
      Example. The timeline:
      1. Proc_1 performs a load miss → \(IS^D\)
      2. Home receives first GetS from Proc_1 → \(E\) and sends ExclusiveData to Proc_1
      3. Proc_2 issues GetM → \(IM^AD\)
      4. Home sends Inv to Proc_1
      5. Home forwards Fwd-GetM to Proc_1
      6. Proc_1 receives Inv → \(II^D\)
      7. Proc_1 receives ExclusiveData → \(I\)
      Data is lost
      Solution. We once again have to use the ExclusiveData message. We can then see that we got ExclusiveData in \(II^D\) and transition to \(E\) instead of \(I\). \(E\) will then correctly handle the Fwd-GetM or Fwd-GetS.

        Virtual Channels [HUDE]

        • May 15, 2026
        • Kellen Kanarios

        A virtual channel (VC) is a separate logical buffer that shares the same physical link with other VCs. Multiple VCs on one link let the network distinguish classes of traffic — if one VC fills, others on the same link still flow.

        Two motivations for VCs:

        • Routing-deadlock avoidance: provide an escape channel routed via DOR.
        • Protocol-deadlock avoidance: separate message classes (request, response, forwarded) onto distinct VCs so a stuck response cannot block a request that would have unstuck it. See Implementing Directory-based MESI in Murphi for a worked Murphi example where 3 VCs are required for the directory protocol.

        The dimension order prevents routing deadlock: cycles in the channel dependency graph are impossible because no packet ever requests a channel in a "lower" dimension once it has started using a higher one. The dependency graph is therefore acyclic, and the standard cycle-equals-deadlock argument applies.

        • (+) Provably deadlock-free without virtual channels.
        • (-) Zero path diversity. Hot spots can't be routed around.
        • (-) Unaware of current network state — oblivious.

        The standard remedy for the path-diversity problem is Valiant's routing algorithm, which keeps the deadlock-freedom of dimension-order routing while spreading load with a randomized intermediate hop.
