Interpreting the role of a single neuron

  • September 10, 2025
  • Kellen Kanarios

This is done in Network dissection

Idea 1: find neuron with maximum activation for subsets of image. Need subset dataset.

Problem: How do you do this for all of the neurons in your network? This is not scalable!

Approach 1: use segmentation network to segment images and use these as subsets. This requires the concepts to be detectabl by a segmentation model.

This is done in MILAN

Approach 2: train a "captioner" model i.e. an LSTM to generate captions for parts of images. Can also get labels using clips as done in CLIP-dissect

Problem with these approaches is they are one-shot i.e. we train the model to produce labels and use those in one-shot. We instead want a "loop" that allows us to refine labeler. Namely

  1. Train model to label something like "human faces"
  2. Find sensitive neuron to images labeled "human faces" by the labeler.
  3. Remove the neuron and see if the model becomes unable to detect human faces.
  4. If this isn't the case then we need to refine the labeler.

How can we automate this process?

Example. 

  • September 10, 2025
  • Kellen Kanarios

Given a picture of kids with some amount of muffins, answer the question

How many muffins can each kid have for it to be fair?

The LLM writes code to identify children / muffins in the image, counting them, and dividing number of muffins by number of kids.

MAIA API:

  • takes image, language description along with model and specific neuron i.e. "neuron 0 layer 4" and it will get the activation for the subset of the image focusing on the language description.
  • Can also take a dataset, or generate new images to test hypothesis.
  • Can go image to text

End to end user can ask "what function does neuron 122 perform" and MAIA will be able to answer.

How do we know the description is correct? One approach is "synthetic neurons". Provide the value of the neuron as the confidence score from SAM for a given language concept description.

Another approach: take the description given by MAIA, then generate images from the prompts along with "negatives" that have nothing to do with the label of MAIA. Then look at the activation value for each of the examples.

Whole-Body Conditioned Egocentric Video Prediction [83S7]

  • September 8, 2025
  • Kellen Kanarios

Presentation on PEVA. Related work COSMOS.

Observation: you can only completely predict changes caused by your own actions. You cannot necessarily predict changes caused by the world. \[\text {Prediction} = \text {change via your actions} + \text {change via the world}\]

Problem formulation: given history frames, actions predict what is going to happen in pixel space.

Challenges:

  1. There is a strong contextual dependence for both action and vision. Depending on context, you may hold / see things in a different way.
  2. Human actions are harder than global condition. Language vs. high DoF control.

Auto-Regressive diffusion model. Spatial-attention on image space, causal attention on temporal space.

Need architecture, so that actions only interact with next image.

Adaptive layer-norm. \[(1 + \Delta \gamma (c)) \cdot \mathrm {LN}(x) + \Delta \beta (c)\] Adaptive per-frame layer-norm. \[(1 + \Delta \gamma (c_t)) \cdot \mathrm {LN}(x_t) + \Delta \beta (c_t)\]

Using PEVA for control. Other approaches

  1. Diffusion forcing
  2. NWM (CDiT)

Visual geometrially grounded transformer (VGGT).

    Implementing Directory-based MESI in Murphi [HHZO]

    • March 24, 2026

    Murphi

    • March 24, 2026
    • Kellen Kanarios

    Syntactical things I need to remember from Murphi docs. Most things are straightforward, but there are a few things to note:

    Declarations. A declaration is a variable or a collection of variables that are used in the rules.

    Rules. A rule is a transition from one state to another

    • Each rule can optionally have an initial condition required to be fired
    • We can easily define rules for all the processors or collection of \(n\) things with ruleset.
    • For our use case, rules are used to define
      1. Controller actions
      2. Receiving a request from the network
      3. Retrying stalled requests

    Startstates. A startstate is the intial configuration for all our declarations.

    • Must assign a value to every global variable

    Invariants. An invariant is something that Murphi checks to ensure is true at all possible configurations of the system.

    Overview straight from the docs. Repeat forever:

    1. Find all rules whose conditions are true in the current state. (i.e. conditional expressions are true, given the current values of the global variables).
    2. Choose one arbitrarily and execute the action, yielding a new state.

    Starting with MSI

    • March 24, 2026
    • Kellen Kanarios

    The first thing we need to do is understand all the things that can go awry.

    First in the valid-invalid protocol example we were given they only have two virtual channels. However, we need 3 channels to handle, request messages, forwarded messages, and response messages.

    Let us briefly understand why this is the case. Detecting a deadlock is equivalent to detecting a cycle in a dependency graph, so it is sufficient to show a cyclic dependency.

    • If we issue a GetS this goes to the directory, giving us an edge N1 -> D.
    • Then the directory issues a Fwd-GetS to the ownder, giving us the chain N1 -> D -> N2
    • The owner then needs to send data to the requester, giving us the chain N1 -> D -> N2 -> N1.
    • Namely, the data must go through the same channel as all of the requests depending on it.

    To avoid this, we partition into 3 channels:

    • Request: GetS, GetM, PutM
    • Forwarded: Fwd-GetS, Fwd-GetM, PutAck, Inv
    • Response: InvAck, Data

    We directly enumerate these in our MessageType as 

      MessageType: enum {
    GetS,           -- request for shared copy
    GetM,           -- request for modified (exclusive) copy
    PutS,           -- evict shared copy
    PutM,           -- evict modified copy (w/ data)

    FwdGetS,        -- forwarded GetS to owner
    FwdGetM,        -- forwarded GetM to owner
    Inv,            -- invalidation request

    Data,           -- data response
    InvAck,         -- invalidation acknowledgment
    PutAck          -- put acknowledgment
  };
    Cache controller. We will start with MSI cache controller, given by the following table:

    Notably, we are missing some of the messages in the figures below. We can handle these based on the AckCount

      ProcState:
    Record
      state: enum { 
        P_Invalid, P_Shared, P_Modified,       --stable states
        PT_IS_D,      --sent GetS, waiting for Data
        PT_IS_DI,     --sent GetS, got Inv, waiting for Data then go to I
        PT_IM_AD,     --sent GetM (from I), waiting for Data and InvAcks
        PT_SM_AD,     --sent GetM (from S), waiting for Data and InvAcks
        PT_IM_A,      --got Data for GetM, waiting for remaining InvAcks
        PT_MI_A,      --sent PutM, waiting for PutAck
        PT_SI_A,      --sent PutS, waiting for PutAck
        PT_II_A       --waiting for PutAck, already invalidated
      };
      val: Value;
      ack: AckCount;  -- tracks InvAck count
    End;

    Weird Race Conditions

    Problem. 

    • March 24, 2026
    • Kellen Kanarios
     When implementing the PutM rule to downgrade from modified to invalid. We must NOT get rid of the data until we have been acknowledged, so that we can still service forward requests. This is a transition to \(MI^A\). However, we can still receive the PutAck causing us to transition to \(I\) before servicing the Fwd-GetM request. This is lost data.

    Solution. 

    • March 24, 2026
    • Kellen Kanarios
     We must add a \(M^D\) state to the directory controller. In this state, if we get a PutM request from a node that is not our current owner we stall and wait for data to arrive. For \(M^D\) to transition to \(M\), we must send the data back to the directory even though we are doing a three-hop, so that the directory knows the data has also been forwarded to the new owner. It is unclear to me the best way to do this without this state.

    Problem. 

    • March 24, 2026
    • Kellen Kanarios

    The above solution causes a deadlock, where two processors both store miss and one load misses. The "winning processor" of the store will be the second one. The first one will get the data from the directory transitioning from \(IM^{AD} \to IM^{A}\), but the second one will then trigger a Fwd-GetM overwriting the first.

    The load miss transitions to \(IS^D\). However, when it receives an Inv it stalls according to the above figure. This blocks the Fwd-GetM from reaching the other winning processor because they are on the same VC. Thus, we have a deadlock where the owner never gets the data.

    Solution. 

    • March 24, 2026
    • Kellen Kanarios
    We need to add another transient state not included in the figure. Namely, we add \(II^{A}\) that says we were in \(IS^{A}\), but we got invalidated and now we are waiting for our PutAck before we transition to \(I\).

    Remark. 

    • March 24, 2026
    • Kellen Kanarios
     When the home directory is in \(M\) and we get a GetS we cannot clear owner as directed by figure. Otherwise in \(S^D\), we will not be able to tell if the PutM is from the owner and stall, so we only remove the owner once we have received the data.

    Moving to MESI

    • March 24, 2026
    • Kellen Kanarios
     Aside from the modifications discussed above, the rest was straightforward and we now are ready to move on to MESI.

    Cache controller. We will start with MESI cache controller, given by the following table:

    Design choice 1. Since we allow immediate upgrade to \(M\), without forwarding, if E upgrades to M, then the value in the directory sent to the next GetS / GetM would not have updated value. It seems we must either

    1. not allow E to silently upgrade to M and then we do not need forwarding, or
    2. allow E to silently upgrade to M and treat E as M basically and forward everything.
    Notably, neither of these have to write back on PutE, but we will go with 1 because that is how it is indicated in the figure.

    Design Choice 2. In order to indicate that the directory received the first GetS we introduce a ExclusiveData message. Upon receiving, ExclusiveData in \(IS^D\) we transition to \(E\).

    • March 24, 2026
    • Kellen Kanarios
     Once we have sent the PutE, or the PutM \(EI^A\) and \(MI^A\) are functionally the same. I include both for completeness but verified that transitioning directly to \(MI^A\) from \(E\) works the same.

    Weird Races.

    Problem. 

    • March 24, 2026
    • Kellen Kanarios
    First processor does GetS and is in \(IS^D\) but the Home directory sends it exclusive data and is now in \(E\). Then second processor does GetM and Home issues Fwd-GetM to processor one, which arrives before data.

    Solution. 

    • March 24, 2026
    • Kellen Kanarios
    Need to stall on Fwd-GetM and Fwd-GetS in \(IS^D\) until data arrives

    Problem. 

    • March 24, 2026
    • Kellen Kanarios
    An even crazier race happens when we get invalidated before the Fwd-GetM or Fwd-GetS arrive. Due to our previous design decision, we transition to \(II^A\), where if we receive a PutAck we transition to invalid. This then loses the needed forward data.

    Example. 

    • March 24, 2026
    • Kellen Kanarios
     The timeline:
    1. Proc_1 performs a load miss → \(IS^D\)
    2. Home receives first GetS from Proc_1 → \(E\) and sends ExclusiveData to Proc_1
    3. Proc_2 issues GetM → \(IM^AD\)
    4. Home sends Inv to Proc_1
    5. Home forwards Fwd-GetM to Proc_1
    6. Proc_1 receives Inv → \(II^D\)
    7. Proc_1 receives ExclusiveData → \(I\)
    Data is lost

    Solution. 

    • March 24, 2026
    • Kellen Kanarios
     We once again have to use the ExclusiveData message. We can then see that we got ExclusiveData in \(II^D\) and transition to \(E\) instead of \(I\). \(E\) will then correctly handle the Fwd-GetM or Fwd-GetS.

    3. Remove the neuron and see if the model becomes unable to detect human faces.
    4. If this isn't the case then we need to refine the labeler.

    Observation: you can only completely predict changes caused by your own actions. You cannot necessarily predict changes caused by the world. \[\text {Prediction} = \text {change via your actions} + \text {change via the world}\]

    Problem formulation: given history frames, actions predict what is going to happen in pixel space.

    Challenges:

    1. There is a strong contextual dependence for both action and vision. Depending on context, you may hold / see things in a different way.

    Auto-Regressive diffusion model. Spatial-attention on image space, causal attention on temporal space.

    Need architecture, so that actions only interact with next image.

    Adaptive layer-norm. \[(1 + \Delta \gamma (c)) \cdot \mathrm {LN}(x) + \Delta \beta (c)\] Adaptive per-frame layer-norm. \[(1 + \Delta \gamma (c_t)) \cdot \mathrm {LN}(x_t) + \Delta \beta (c_t)\]
