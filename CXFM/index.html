Page fault [CXFM]

  • May 12, 2026
  • Kellen Kanarios

Physical memory is finite, so the OS reserves a region of disk — swap space — to hold pages that don't fit. Pages currently on disk have their present bit cleared in the Page table entry; the PTE instead holds a disk address.

When a process touches a virtual address whose page is not present, the hardware traps to the OS. This is a page fault. The handler:

  1. Looks at the PTE: if invalid, the access is illegal — terminate the process.
  2. If valid but not present: read the page in from swap, update the PTE, retry the access.
Remark. The hardware never handles page faults itself — that would require it to understand disk I/O and the OS's swap layout. Page-fault handling is always in software.

The three cases on every memory access:

  1. Present + valid: hardware reads the PPN from the PTE and proceeds.
  2. Not present + valid: hardware invokes the page-fault handler to retrieve from swap.
  3. Not valid: hardware raises an exception; the OS typically kills the process.
Example. What happens when you dereference a nullptr? The TLB misses on VPN 0, the page-table walk finds the PTE marked invalid, and the hardware traps to the OS — which delivers SIGSEGV.

    My Current Workflow (constantly changing) [OGCSJ]

    • June 25, 2026
    • Kellen Kanarios

    I have been using todoist and habitify. I decided on these tools primarily because of their API availability. I wanted to be able to add / complete / see outstanding tasks or habits for the day without the allure of everything else my browser has to offer. Also visualization for my Weeknotes.

    • I had Claude write a little script to have my next tasks listed when clicked in my status bar, so I do not have to open the browser to check what I need to do today.

    It seems how to properly use these (todoist in particular) is the subject of much debate. Here, I will outline what I converged to / works for ME right now. I have broken down my system into a hierarchy:

    1. Higher level: how to actually choose what to do.
    2. Lower level: how to actually do the chosen task.

    Figuring out what to do

    • June 25, 2026
    • Kellen Kanarios

    This is effectively a task management system.

    Work streams. I took this name from my investment banker friends (sounds cool / important). Basically, these are groupings I have made and allocated time for each day. They roughly consist of the following:

    • Research / work (4hrs first thing in the morning):
      • Active work towards new artifact or paper i.e. coding up ideas.
      • In the absence of ideas, studying existing relevant papers.
    • Deliberate practice (2hrs after run / lunch):
      • Name taken from Cal Newport.
      • As of writing this, a current example is spending ~2hrs a day writing / optimizing known kernels in CuTe / Pallas / NKI.
      • Basically, something I need to understand or master for research that is not novel.
    • Extraneous Study (30min - 1hr):
      • Things not directly related to research, but I am curious about.
      • I try to maintain study of related computer science fields. Primarily, studying a textbook in something like OS, computer architecture, etc.
    • Habits / daily upkeep (1hr normally end of day):
      • Mainly consists of reviewing Anki cards daily that come from extraneous study / deliberate practice.

    To decide on the next task, I use a priority-based system:

    1. Things I will be working on today / until they are done. At most ONE thing per work stream.
    2. The backlog of things to do next
      • An index to find things to move to priority 1 for the next day
      • Keep relatively small to avoid the crippling nature of choice - roughly on the order of things I want done this week.
    3. A larger backlog of things - should be completed in the next month or so.
    4. Things I am not working on right now, or have any need to be completed any time soon.

    This idea is mainly based on the MLFQ, which has been proven effective for balancing response / turnaround time. To maintain this system, I have reviews each day and week (tracking with habitify) to make sure I stay consistent.

    Algorithm. Weekly review.
    Algorithm. Daily review.
    • For each work stream,
      • If I finished the allocated task for the day, get new task from priority 2.
      • Else, do nothing and continue with the current task.
    • Review inbox and move new tasks to corresponding work stream for review on Friday.
      • If immediately important, can move directly to priority 1 / 2.
    • Look at calendar for next day and timeblock work streams for the next day.
      • Since PhD work is so hard to predict, I have found timeblocking any further than one day in advance to be a futile effort.
    • Typically done on Fridays, so that I can dynamically allocate time on weekends if needed.
    • For each work stream,
      1. Look in remaining priority 2 and priority 3 tasks.
      2. Define what I think a successful week would look like (e.g. if I get all these things done next week I will be happy).
        • Look at calendar for week to guide this decision.
      3. Get rid of at least two of those tasks (initial plan always too ambitious).
      4. Move / keep remaining tasks in priority 2.
    • Write Weeknotes for the week.

    What to do when actually working on a task

    • June 25, 2026
    • Kellen Kanarios

    This is the harder part (and requires the most discipline). When working on a task, especially a more open-ended one like research, it is very common that a million other things come up that seem immediately important. All of the sudden, you have a great idea for the research project you haven't worked on in months.

    It is crucial to collect and forget about these things until they are scheduled. This is the one thing I did take from Getting Things Done. The system should flow like water. Making this achievable is where the todoist API becomes crucial.

    • I have a script bound by dwm that pops up vim for me to write down the task and, on save, it sends it to my inbox in todoist.
    • I have more involved ones that allow me to actually allocate it to the specific work stream / priority if it is immediately obvious.

    Another thing that I have found useful to actually achieving this watery flow is the pomodoro technique. This basically lets me gaslight myself into thinking I can go to the other things as soon as the timer ends (which is intentionally always just close enough). By the end, I often no longer have this urge and can continue on the scheduled tasks.

    Stack vs. Heap [0025]

    • December 28, 2024
    • Kellen Kanarios

    I often hear this question come up amongst performance-driven developers, and for the longest time I could never get a straight answer as to why the stack was so much better? After a long foray into the inner workings of how memory allocation is actually done, I think I finally have an answer. I do not want to forget this, so I am doing a quick write up here.

    Why stack is faster.

    • December 28, 2024
    • Kellen Kanarios
     This all comes down to what happens during each memory allocation. Prior to actually running the program, the operating system allocates some amount of memory for both the stack and the heap (I think around 2-4MB?). When you allocate memory on the stack i.e. int x = 1;
    • Stack just increments stack pointer.
    • Writes value to new address pointed to by stack pointer.
    When you allocate memory on the heap i.e. int *x = malloc(sizeof(int)), we must instead choose how best to fulfill the request, which is a whole research direction in itself.
    • One such implementation is the buddy algorithm
    However, malloc itself is not a system call and therefore is not terribly slow. A real issue occurs when we run out of pre-allocated heap memory. This requires invoking a system call like sbrk to increment the size of the allocated memory for the heap.
    • If no available pages, OS must evict and write to disk...
    • Formally, this is known as "heap growth".
    Generally, if you know how much memory you need it is good practice to just allocated all up front to minimize any potential system calls. With this, the performance difference is not too drastic :)

    Rebuilding My (Neo)Vim Config From Scratch [005F]

    • March 4, 2025
    • Kellen Kanarios

    I used LazyVim for a long time, but I kept hitting a wall: understanding how LazyVim did something was consistently harder than if I had just written the thing myself. So I gave myself one day to rebuild from scratch, with the modest goal of first-class support for \(\TeX \), Python, Forester, and C/C++. That one-day sprint is long past, and the config has kept evolving since — this note describes where it landed, not the frantic first pass. The through-line is that almost everything now leans on Neovim's own primitives instead of a framework, which is exactly the transparency I was missing.

    Remark. The whole config lives in my dotfiles. It is a thin init.lua that loads four modules — config.lazy, config.options, config.keymaps, and core.lsp — with the actual plugin specs under lua/plugins/ and per-server LSP tables under lsp/.

    Sane defaults

    • March 4, 2025
    • Kellen Kanarios

    To my surprise, a lot of the behavior I had taken for granted turned out to be options LazyVim set internally. I was greeted on first launch by 8-space indents and a clipboard that would not survive a trip between terminal instances. So the first job was archaeology: I dug through LazyVim for every option I actually liked and copied them into config/options.lua. System clipboard, sensible folding via Treesitter (foldexpr = v:lua.vim.treesitter.foldexpr()), a global statusline, conceallevel for markup — the unglamorous foundation everything else sits on.

    Plugin management

    • March 4, 2025
    • Kellen Kanarios

    The plugin manager is lazy.nvim, the de facto standard, and it genuinely just works. The bootstrap snippet from their docs clones it on first run, and after that every file under lua/plugins/ that returns a spec table is imported automatically. No central manifest to keep in sync — a new file is a new plugin.

    Native LSP, no framework

    • March 4, 2025
    • Kellen Kanarios

    This is where the config diverged most sharply from where it started. The first pass used mason.nvim, mason-lspconfig, and nvim-lspconfig — the standard stack. All three are now gone. Neovim 0.11 ships a native LSP configuration API: vim.lsp.config defines a server and vim.lsp.enable turns it on, and any table returned from lsp/<name>.lua on the runtime path is picked up as that server's config. So core.lsp is just a list —

     vim.lsp.enable({ "forester", "texlab", "clangd", "basedpyright", "lua_ls", "ruff" })

    — and each server gets a plain Lua file in lsp/. No package manager, no bridge plugin translating between two config schemas. I install the servers themselves through my system package manager and let Neovim wire them up. It is dramatically less machinery for the same result, and when something breaks there is exactly one place to look.

    The Python setup is the clearest upgrade. I dropped pyright for basedpyright and split responsibilities: basedpyright does type checking and inlay hints, while ruff runs as its own language server handling import organization and diagnostics. In the LazyVim days I fought constantly with import resolution, hand-writing a function to locate each project's virtualenv and pin pythonPath to its interpreter. basedpyright resolves environments well enough on its own that the hack is gone. clangd, texlab, and lua_ls round out the set, each configured in its own file — texlab in particular carries a fair amount of custom command wiring for building, forward search into Zathura, and environment manipulation.

    A shared LspAttach autocommand in core.lsp binds the buffer-local LSP keymaps once, for every server: go-to-definition and friends route through fzf-lua pickers, and document highlighting, inlay-hint toggling, and native completion get enabled per-buffer based on what each client actually supports.

    Forester finally has a real language server

    • March 4, 2025
    • Kellen Kanarios

    The original post described an ugly hack to get Forester completion. The forester.nvim plugin exposed a completion source, but only for nvim-cmp; blink.cmp (which I use — see below) needs a blink.compat shim, and that shim expected a plugin to be a source rather than to bury one inside a larger feature set. Getting reference completion working meant reading blink.compat internals and registering the source by hand.

    None of that survives, because Forester now ships an actual language server. My lsp/forester.lua just launches forester lsp and hands it the forest root. The one wrinkle worth recording: forester locates the forest by its process working directory and ignores the LSP rootUri, and Neovim does not spawn the server in root_dir by default. So the config resolves the nearest forest.toml and starts the RPC process with an explicit cwd — otherwise the server exits with status 1 whenever Neovim's cwd has no forest.toml (e.g. when I open a tree from $HOME). Reference completion — indispensable for writing this very blog — now flows through the standard LSP path with no shim in sight.

    Evidence Lower Bound [0082]

    • June 30, 2025
    • Kellen Kanarios

    Setup. We are given a joint distribution \(p(X, Z; \theta )\), where \(X\) and \(Z\) are random variables with their joint parametrized by \(\theta \). We are also given a dataset \(\mathcal {D}\) containing realizations of only \(X\) i.e. \(Z\) is a latent random variable.

    Goal. We want to learn \(p(x; \theta )\).

    However, \[p(x; \theta ) = \int p(x, z; \theta ) \mathrm {d}z = \int p(x | z; \theta )p(z) \mathrm {d}z,\] which is computationally intractable for most continuous \(Z\).

    Question. Can we update \(\theta \) for a given realization of \(x\) and \(z\)?
    Answer. Yes!!!

    Motivation. Suppose we had some \(q(z)\) that we could sample realizations of \(Z\) from. Then we want to find some reasonable \(\ell _{\theta }\), such that \[\theta ^* \approx \arg \max _{\theta } \mathbb {E}_{x, \sim p(x; \theta ^*), z \sim q(z)}[\ell _{\theta }(X, Z)]\]

    Remark. \(q(z)\) can potentially (and will) depend on the sample \(x_i\). This is OK because we are given \(x_i\)'s, which we can then just pass to \(q\) to get the corresponding \(z_i\).
    Why is this the case? Because then (for well-behaved \(\ell _{\theta }\)) \[\theta _{t + 1} = \theta _{t} - \alpha \nabla _{\theta _{t}}\ell _{\theta _{t}}(x_i, z_i) \to \theta ^*\] i.e. SGD is a reasonable thing to do, where the gradient is for a single sample of \(x_i\) and \(z_i\) which we have readily available from our dataset \(\mathcal {D}\) and our distribution \(q\) that we can sample from. This idea comes up a lot in optimization, such as the \(\log \) trick in the original policy gradient.

    Similar to the \(\log \) trick, the Evidence Lower Bound (ELBO) comes from a simple multiply and divide trick. Namely, \[\begin {align*} \log p(x; \theta ) &= \log \int p(x, z; \theta ) \mathrm {d}z \\ &= \log \int p(x, z; \theta ) \frac {q(z)}{q(z)} \mathrm {d}z \\ &= \log \mathbb {E}_{z \sim q(z)} \frac {p(x, Z; \theta )}{q(z)} \\ &\geq \mathbb {E}_{z \sim q(z)} \log \frac {p(x, Z; \theta )}{q(z)} .\end {align*}\] The last inequality follows directly from Jensen's Inequality. However, note that this is only a lower bound and therefore not exactly what we want. For example, if we were to pick \(q \sim N(0, 1)\) and then optimize the ELBO for \(\theta \) we would not necessarily find a good \(\theta \) for the likelihood \(\log p(x; \theta )\). Luckily, we can actually get the difference \(\log p(x; \theta ) - \)ELBO in closed form! Namely, \[\begin {align*} \log p(x; \theta ) - \mathrm {ELBO} &= \log p(x; \theta ) - \mathbb {E}_{z \sim q(z)} \log \frac {p(x, Z; \theta )}{q(z)} \\ &= \int \left (\log p(x; \theta ) - \log \frac {p(x, z; \theta )}{q(z)}\right )q(z) \\ &= \int \left (\log \frac {p(x; \theta )q(z)}{p(x, z; \theta )}\right )q(z) \\ &= \int \left (\log \frac {p(x; \theta )q(z)}{p(z | x; \theta )p(x; \theta )}\right )q(z) \\ &= \int \left (\log \frac {q(z)}{p(z | x; \theta )}\right )q(z) \\ &= \mathrm {KL}\big (q(z) \mid \mid p(z | x; \theta )\big ) .\end {align*}\] Thus, to tighten the lower bound we can optimize \(q_{\phi }(z)\) to try to match \(p(z | x; \theta )\). As a sanity check, lets make sure these match when \(q(z) = p(z \mid x; \theta )\). \[\begin {align*} \mathbb {E}_{z \sim q(z)} \log \frac {p(x, Z; \theta )}{q(z)} &= \int \log \frac {p(x, z; \theta )}{q(z)}\mathrm {d}z \\ &= \int \log \frac {p(x; \theta )p(z \mid x; \theta )}{p(z \mid x; \theta )}\mathrm {d}z \\ &= \int \log p(x; \theta )\mathrm {d}z \\ &= \log (x; \theta ) .\end {align*}\] You may be wondering how we use this algorithmically to find \(\theta ^*\)? This will be the main idea of expectation maximization. Feel free to check it out!

    Expectation Maximization [0088]

    • July 1, 2025
    • Kellen Kanarios

    This is the follow up to the previous note on the ELBO. Mainly, here we aim to show how we can use the ELBO to find \(\theta ^*\) i.e. \[ \begin {align*} \theta ^* &= \arg \max \mathbb {E}_{x \sim \mathcal {D}} \log p(x; \theta ) \\ &= \arg \max \mathbb {E}_{x \sim \mathcal {D}} \log \int p(x, z; \theta ) \mathrm {d}z .\end {align*} \] Notably, ELBO gives us the lower bound \[\log p(x; \theta ) \geq \mathbb {E}_{z \sim q}\left [\log \frac {p(x, Z; \theta )}{q(z)}\right ] \eqqcolon \mathsf {ELBO}(x, q, \theta )\] for a fixed theta and \(q\). Therefore, for each \(\theta \), we want to find some \(q\), such that \[\mathsf {ELBO}(x, q, \theta ) = \log p(x, \theta )\] With this, we can easily perform gradient ascent \[\theta _{t + 1} \gets \theta _t + \nabla _{\theta } \mathsf {ELBO}(x, q, \theta _t),\] to maximize the log-likelihood \(\log p(x; \theta )\).

    Remark. Note that the gradient is with respect to \(\theta \), where \(\theta \) is just the third argument to \(\mathsf {ELBO}\) as defined above. Notably, \(\theta _t\) will show up elsewhere later but will NOT be backpropagated through.

    Expectation

    • July 1, 2025
    • Kellen Kanarios

    The above motivates the expectation step of the EM algorithm. Namely how can we compute the lower bound expectation for the best possible \(q\).

    Recall (or review) from the discussion on ELBO that \(q(z) = p(z | x; \theta )\) maximizes the ELBO for a given \(x\) and \(\theta \). Therefore, we actually want to learn \(q_{\phi }(z, x, \theta ) \approx p(z | x; \theta )\). To do so, we can simply maximize the ELBO (duh). Formally, \[\phi _{t+1} = \argmax _{\phi } \mathbb {E}_{x \sim \mathcal {D}, z \sim q}\left [\log \frac {p(x, Z; \theta _t)}{q_{\phi }(Z, x, \theta _t)}\right ]\]

    Question. Now this is all cool theoretically but how TF do we actually maximize this thing?
    Answer. It is highly dependent on the parametrization of \(p\) and \(q\).
    Example.

    We define the problem as

    1. \(p(z) \sim N(0, 1)\)
    2. \(p(x | z; \theta ) \sim N(\mu _{\theta }(z), \sigma _{\theta }(z))\)
    3. \(q_{\phi }(z | x, \theta ) \sim N(\mu _{\phi }(x), \sigma _\phi (x))\)
    where \(\mu _{\{\phi , \theta \}}, \sigma _{\{\phi , \theta \}}\) are parametrized neural networks. Then we can rewrite the ELBO as (ignoring constants) \[ \begin {align*} \mathsf {ELBO}(x, q, \theta ) &= \mathbb {E}_{x \sim \mathcal {D}, z \sim q(\cdot \mid x, \theta )} \left [\log p(Z) - \log \frac {p(x \mid Z; \theta )}{q_{\phi }(Z, x, \theta )}\right ] \\ &= \mathbb {E}_{x \sim \mathcal {D}, z \sim q(\cdot \mid x, \theta )} \left [-\frac {Z^2}{2} - \frac {\sigma _{\phi }(x)^2}{\sigma _{\theta }(z)^2}\big ((Z - \mu _{\theta }(Z))^2 - (Z - \mu _{\phi }(x))^2\big )\right ] .\end {align*} \]

    Problem. We need to sample realization \(z \sim q_{\phi }(\cdot |x, \theta )\). So isn't our realization \(z\) a function of \(\phi \). How would we optimize this?
    Answer. The reparametrization trick!!

    Now for a given sample \(x_i \sim \mathcal {D}\), we can get a sample \(z_i(\phi ) \sim q(\cdot \mid x_i, \theta )\) that is differentiable with respect to \(\phi \). Using this, we can get the stochastic gradient estimate \[\nabla _q \widetilde {\mathsf {ELBO}}(x, q, \theta ) = \nabla _{\phi } \left [\frac {z_i(\phi )^2}{2} - \frac {\sigma _{\phi }(x)^2}{\sigma _{\theta }(z)^2}\big ((z_i(\phi ) - \mu _{\theta }(z_i(\phi )))^2 - (z_i(\phi ) - \mu _{\phi }(x))^2\big )\right ]\]

    Algorithm. input \(\theta _t\)
    1. Sample \(x_i\) from dataset \(\mathcal {D}\)
    2. Sample \(z_i(\phi ) \sim q_{\phi }(\cdot \mid x_i, \theta _t)\) via trick.
    3. Compute \(\nabla _q \widetilde {\mathsf {ELBO}}\)
    4. Update via SGD

    Maximization

    • July 1, 2025
    • Kellen Kanarios

    Finally, we have reached the maximization step. This step is fairly straightforward after all the progress we have made. Once we have a tight lower bound i.e. \(q(z) \approx p(z \mid x; \theta )\) then we simply fix \(q\) and maximize the ELBO for \(\theta \) i.e. in the above example and algorithm we instead compute \[\nabla _\theta \widetilde {\mathsf {ELBO}}(x, q, \theta ) = \nabla _{\theta } \left [\frac {z_i(\phi )^2}{2} - \frac {\sigma _{\phi }(x)^2}{\sigma _{\theta }(z)^2}\big ((z_i(\phi ) - \mu _{\theta }(z_i(\phi )))^2 - (z_i(\phi ) - \mu _{\phi }(x))^2\big )\right ]\] Alternating between these steps yields the expectation-maximization algorithm the ML oldheads mythologize.

    Definition. Reparametrization Trick [0089]

    • July 1, 2025

    For \(q_{\phi }(\cdot | x) \sim N(\mu _{\phi }(x), \sigma _{\phi }(x))\). The reparametrization trick allows us to differentate through a random sample by sampling from \(q_{\phi }(\cdot | x)\) via \[z(\phi ) = \mu _{\phi }(x) + \sigma _{\phi }(x) \cdot \epsilon ,\] where \(\epsilon \sim N(0, 1)\). Then, for a given sample \(\epsilon \), we can easily compute \[\nabla _{\phi } z(\phi ) = \nabla _{\phi } \mu _{\phi }(x) + \nabla _{\phi }\sigma _{\phi }(x) \cdot \epsilon \]

    Multi Query Attention [W4YP]

    • July 14, 2025
    • Kellen Kanarios

    Notes on the paper Fast Transformer Decoding: One Write-Head is All You Need.

    For,

    • \(b\) batch dimension,
    • \(n\) sequence length,
    • \(d\) hidden dimension,
    • \(h\) number of heads.
    The total FLOPS for attention are roughly \(bnd^2\). This comes from
    1. \(3nd^2\) FLOPS to compute \(Q = XW_Q, K = XW_K, V = XW_V\).
    2. Need to do this \(b\) times for each input in batch.
    and the total memory accesses are roughly \(\underbrace {bnd}_{X} + \underbrace {bhn^2}_{\text {softmax}} + \underbrace {d^2}_{\text {projection}}\). This gives high arithmetic intensity \[O\left (\left (\frac {h}{d} + \frac {1}{bn}\right )^{-1}\right )\]

    However, if we do incremental inference then we must multiply our total number of memory accesses by \(n\) i.e. \(bn^2d + nd^2\). This gives arithmetic intensity \[O\left (\left (\frac {n}{d} + \frac {1}{b}\right )^{-1}\right ),\] which requires large batch and short sequence length.

    Remark. I believe we just ignore the softmax memory contribution in the inference case because it does not scale with \(n\) anymore as we are computing the logits just for the next token. Therefore, it becomes a \(\frac {d}{h}\) term, which we can safely ignore?

    The key is that the \(\frac {n}{d}\) term comes from the \(bn^2d\) term, where we are loading \(h\) \((b \times n \times d / h)\) \(K\) and \(V\) matrices \(n\) times. Thus, we can improve the \(\frac {n}{d}\) term by a factor of \(h\) by simply not using a different \(K\) and \(V\) matrix for each head. This is the entire idea behind MQA.

    Remark. Similar to the KV-cache, we still can use \(h\) \(Q\) matrices because we do not need to load \(n\) of them into memory because only the last one matters for inference.

    Eligibility Traces [DJPO]

    • July 24, 2025
    • Kellen Kanarios

    As I grow more interested in abandoning batched updates and the replay buffer, the importance of elibility traces grows with me. Due to this, I thought it finally time to re-prove / understand what makes these traces so eligible.

    The reference followed is Sutton and Barto (Ch 12.).

    The eligibility trace vector is given by \[ \begin {align*} \mathbf {z}_{-1} &= 0 \\ \mathbf {z}_t &= \gamma \lambda \mathbf {z}_{t - 1} + \nabla \hat {v}(S_t; \mathbf {w}_t) .\end {align*} \] Recall the temporal difference error is given by \[\begin {align*} \mathbf {\delta }_t = r_{t + 1} + \gamma \hat {v}(s_{t + 1}; \mathbf {w}_t) - \hat {v}(s_t; \mathbf {w}_t) .\end {align*}\] For TD(\(\lambda \)), the incremental update is given by \[\begin {align*} \mathbf {w}_{t + 1} = \mathbf {w}_t + \alpha \mathbf {\delta }_{t} \mathbf {z}_t .\end {align*}\]

    We can rewrite the \(\lambda \)-return recursively in terms of reward and itself as follows: \[\begin {align*} G_t^{\lambda } &= (1 - \lambda ) \sum _{n=1}^{\infty } \lambda ^{n-1}G_{t:t+n} \\ G_t^{\lambda } &= (1 - \lambda ) \sum _{n=1}^{\infty } \lambda ^{n-1}(R_{t + 1} + \gamma G_{t+1:t+n}) \\ G_t^{\lambda } &= R_{t + 1} + \gamma (1 - \lambda ) \left [\hat {v}(S_{t + 1}; \mathbf {w}_t) + \sum _{n=2}^{\infty } \lambda ^{n-1} G_{t+1:t+n}\right ] \\ G_t^{\lambda } &= R_{t + 1} + (1 - \lambda )\gamma \hat {v}(S_{t + 1}; \mathbf {w}_t) + \gamma \lambda G_{t + 1}^{\lambda } .\end {align*}\]

    Now we will show that the prediction error of TD(\(\lambda \)) can be re-written as the sum of TD-errors. The prediction error is written as \[PE = G_t^{\lambda } - \hat {v}(S_t, \mathbf {w}_t)\] Rewriting the \(\lambda \)-return as above \[ \begin {align*} PE &= R_{t + 1} + \gamma \hat {v}(S_{t + 1} ; \mathbf {w}_t) + \gamma \lambda G_{t + 1}^{\lambda } - \gamma \lambda \hat {v}(S_{t + 1}, \mathbf {w}_t) - \hat {v}(S_t; \mathbf {w}_t) &= (R_{t + 1} + \gamma \hat {v}(S_{t + 1} ; \mathbf {w}_t) - \hat {v}(S_t, \mathbf {w}_t)) + \gamma \lambda G_{t + 1}^{\lambda } - \gamma \lambda \hat {v}(S_{t + 1}; \mathbf {w}_t) &= \mathbf {\delta }_t + \gamma \lambda G_{t + 1}^{\lambda } - \gamma \lambda \hat {v}(S_{t + 1}; \mathbf {w}_t) .\end {align*} \]

    We now are left with \(\gamma \lambda (G_{t + 1}^{\lambda } - \hat {v}(S_{t + 1}, \mathbf {w}_t))\), where we can proceed in the same manner recursively, such that \[PE = \sum _{t=1}^{\infty } (\gamma \lambda )^{t - 1} \delta _t\] It is easy to see the same thing goes through for the finite time slice \(\lambda \)-return i.e. \[G_{t:t + k}^{\lambda } - \hat {v}(S_t, \mathbf {w}_{t - 1}) = \sum _{i = t}^{t + k} (\gamma \lambda )^{i - t} \delta _i\]

    Going back to the eligibility trace update, if we instead do not apply each elibgibility trace update but apply the sum of all the updates for the same fixed \(\mathbf {w}_t\), then we get that the total update is \[ \begin {align*} \mathbf {w}_{t+1} &= \sum _{t=1}^{T} \alpha \delta _t \mathbf {z}_t \\ &= \sum _{t=1}^{T} \alpha \delta _t \sum _{i=0}^{t} (\lambda \gamma )^{i} \nabla \hat {v}(S_i, \mathbf {w}_t) \\ &= \sum _{t=1}^{T} \alpha (r_t + \gamma \hat {v}(S_{t+1}, \mathbf {w}_t) - \hat {v}(S_t, \mathbf {w}_t)) \underbrace {\sum _{i=0}^{t} (\lambda \gamma )^{i} \nabla \hat {v}(S_i, \mathbf {w}_t)}_{X_t} .\end {align*} \] Then \[\begin {align*} &= \sum _{t=1}^{T} \alpha r_t X_t + \gamma \hat {v}(S_{t+1}, \mathbf {w}_t) X_t - \hat {v}(S_t, \mathbf {w}_t) X_t .\end {align*}\]

    This shows that the backward-view and forward-view are "equivalent". The forward-view in practice makes an update at each step, so it is typically better.

    Rapidly Exploring Random Trees [QQC1]

    • August 14, 2025
    • Kellen Kanarios

    Rapidly exploring random trees (RRT) is a very simple, elegant and widely used algorithm for motion planning. The algorithm proceeds as follows:

    Algorithm. For \(k = 1\) to \(K\) do
    1. \(x_{rand} \gets \mathsf {RANDOM\_STATE}()\)
    2. \(x_{near} \gets \mathsf {NEAREST\_NEIGHBOR}(x_{rand}, \mathcal {T})\)
    3. \(u \gets \mathsf {SELECT\_INPUT}(x_{near})\)
    4. \(x_{new} \gets \mathsf {NEW\_STATE}(x_{near}, u, \Delta t)\)
    5. \(\mathcal {T}.add\_vertex(x_{new})\)
    6. \(\mathcal {T}.add\_edge((x_{near}, u, x_{new}))\)
    return \(\mathcal {T}\)
    Remark. Given some dynamics model \(\dot {x} = f(x, u)\), \(\mathsf {NEW\_STATE}\) is some approximate integration, such as Euler integration i.e. \[x_{new} = x_{near} + f(x_{near}, u) \Delta t\] Preferable to use higher-order method like Runge-Kutta.
    Remark. \(\mathsf {SELECT\_INPUT}\) is determined by whether the system is holonomic or non-holomonic. If the agent is holonomic, then \(u\) will just be the control corresponding to the difference in the two states.

    Page replacement [HK73]

    • August 8, 2025
    • Kellen Kanarios

    Once swap space is in play, the OS must decide which pages to evict from physical memory. The mechanism is the Page fault machinery; the question of policy is what this hub covers.

    Page fault [CXFM]

    • May 12, 2026
    • Kellen Kanarios

    Physical memory is finite, so the OS reserves a region of disk — swap space — to hold pages that don't fit. Pages currently on disk have their present bit cleared in the Page table entry; the PTE instead holds a disk address.

    When a process touches a virtual address whose page is not present, the hardware traps to the OS. This is a page fault. The handler:

    1. Looks at the PTE: if invalid, the access is illegal — terminate the process.
    2. If valid but not present: read the page in from swap, update the PTE, retry the access.
    Remark. The hardware never handles page faults itself — that would require it to understand disk I/O and the OS's swap layout. Page-fault handling is always in software.

    The three cases on every memory access:

    1. Present + valid: hardware reads the PPN from the PTE and proceeds.
    2. Not present + valid: hardware invokes the page-fault handler to retrieve from swap.
    3. Not valid: hardware raises an exception; the OS typically kills the process.
    Example. What happens when you dereference a nullptr? The TLB misses on VPN 0, the page-table walk finds the PTE marked invalid, and the hardware traps to the OS — which delivers SIGSEGV.

    LRU and the clock algorithm [7WG2]

    • May 12, 2026
    • Kellen Kanarios

    Least recently used is the standard heuristic for picking a page to evict: throw out the one that has been untouched the longest. It works because of the principle of locality — recently-used pages are likely to be used again soon (temporal locality).

    Pure LRU requires updating a data structure on every memory access, which is too expensive. The clock algorithm approximates it cheaply:

    1. Maintain a reference bit per page, set by hardware on access.
    2. Loop a pointer around the pages. If the reference bit is 1, clear it and advance.
    3. If the reference bit is 0, evict that page.
    Remark. LRU is not optimal. The pathological case is the looping workload: if the working set is one page larger than the cache, LRU misses 100% of the time.

    The quality of any replacement policy is measured by its average memory access time:

    \[AMAT = (1 - P_{miss}) T_P + P_{miss} T_D\]

    where \(T_P\) is physical-memory access time and \(T_D\) is disk access time. The disk gap is so large that even tiny miss rates dominate AMAT.

    Belady's anomaly [LQV3]

    • May 12, 2026
    • Kellen Kanarios

    A desirable property of a page-replacement policy is the stack property: increasing the cache size never makes the miss rate worse. Surprisingly, not every reasonable-looking policy has it.

    FIFO eviction violates the stack property — on certain access sequences, growing the cache strictly increases the number of misses. This is Belady's anomaly. It is what motivates LRU and the clock algorithm-style policies, which do have the stack property because they only depend on relative access recency.

    The optimal eviction policy (offline) is to evict the page whose next use is farthest in the future. Unimplementable in practice (you don't know the future), but useful as a theoretical upper bound to compare against.

    Two refinements make eviction practical on real workloads. A pure global policy lets one greedy process hog memory, which Segmented FIFO fixes with a per-process resident-set cap; and writing victims back one page at a time wastes disk bandwidth, which Page clustering amortizes by batching the dirty list.

    Segmented FIFO [HF6H]

    • July 11, 2026
    • Kellen Kanarios

    A process-aware refinement of LRU and the clock algorithm. Pure FIFO is a global policy — it does not distinguish memory by owner, so one greedy process can evict everyone else's pages. Segmented FIFO instead maintains a per-process FIFO queue capped at that process's resident set size, bounding how much physical memory any single process can hold.

    Evicted pages are not discarded outright; they move to one of two secondary lists:

    1. Clean page free list — evicted pages that were never modified.
    2. Dirty page list — evicted pages that must be written back before reuse.

    The secondary lists act as a victim cache: if a process faults on a page still sitting on the dirty list, it can be reclaimed without going to disk. The resident-set cap gives fairness; the secondary lists soften the cost of capping it too aggressively.

    Copy-on-write [DEJB]

    • May 12, 2026
    • Kellen Kanarios

    An optimization where two address spaces share the same physical page until one writes to it. Both processes have PTEs pointing at the same PPN, but with the write bit cleared. The first write traps to the OS, which allocates a fresh page, copies the contents, points the writing process's PTE at the new page (with writes re-enabled), and resumes.

    Used pervasively:

    • fork() — the child shares the parent's pages until either modifies one. Avoids the cost of copying an entire address space the child often immediately overwrites with exec().
    • Shared libraries — many processes map the same library text pages.
    • Kernel mapped into every process — same physical kernel code seen from every address space.

    On-demand zeroing [PW7Q]

    • May 12, 2026
    • Kellen Kanarios

    When a process asks for new heap memory, the OS must guarantee the returned pages contain zeros (to avoid leaking other processes' data). The naive implementation zeroes the page at allocation time. On-demand zeroing defers that work to first touch: until the process actually accesses a freshly-allocated page, no zero-fill happens.

    The OS marks the page as not-present in the Page table. The first access traps as a Page fault, the handler zeroes a physical page, installs it in the PTE, and resumes. Heap memory that is allocated but never used costs nothing.

    Page clustering [YMHF]

    • July 11, 2026
    • Kellen Kanarios

    An amortization trick for the write side of page replacement. Writing one dirty page to swap at a time pays a full disk I/O per eviction, and small random writes are the worst case for a disk. Page clustering batches them: the OS collects a group of dirty pages from the global dirty list and writes them out together in a single larger, more sequential I/O. The per-page cost of paging out falls because the fixed overhead of a disk write is spread across the whole batch.

    Remark. A common convention is to map the kernel into every process's address space (via Copy-on-write) so kernel code and structures appear at the same virtual addresses everywhere.

      Hardware interrupt path [K325]

      • May 15, 2026
      • Kellen Kanarios

      What the CPU does, entirely in hardware, the moment an interrupt arrives — whether a device IRQ, a fault like a Page fault, or a software int N. The point of the hardware path is to land in the kernel at a clean instruction boundary with enough saved state that user execution can resume later as if nothing happened.

      Four things happen, in order, with no software involvement:

      1. Reach a clean instruction boundary. The CPU cannot pause halfway through, say, a memory write — architectural state would be inconsistent. So the current instruction either finishes normally (typical for asynchronous interrupts like a timer) or is rolled back as if it never started (faults, where the instruction couldn't complete anyway and needs to be retried). A page-fault handler maps in the missing page and IRET returns to re-execute the faulting instruction; a timer handler returns to the next instruction.
      2. Save the minimum resume state to the kernel stack. The CPU pushes RIP (where to resume), RSP (the user stack pointer it's about to abandon), the privilege level (CS — were we in ring 3 or ring 0?), and RFLAGS. The save target is the kernel stack, whose address the CPU reads from the Task state segment that the OS set up at boot. This is a security boundary: the kernel never trusts the user stack, since user code could have corrupted it or pointed RSP at something malicious. Note what is not saved: general-purpose registers. The handler itself pushes RAX, RBX, ... if it uses them, which keeps the hardware path fast.
      3. Jump to the handler via the Interrupt descriptor table. The CPU takes the interrupt number \(N\), indexes entry \(N\) of the IDT (found via the IDTR register), loads the handler address into RIP, and loads the new code segment, which switches the CPU from ring 3 to ring 0. After this step, the CPU is running kernel code on the kernel stack at the right privilege level — all set up automatically.
      4. Flush the Reorder buffer of work past the interrupt point. In an out-of-order execution CPU the ROB may already contain instructions from after the interrupt — they've been issued speculatively because the front-end ran ahead. Those entries are discarded so the handler sees clean architectural state, as if execution had stopped exactly at the interrupt point. When IRET returns, those instructions re-execute normally. This is the same squash machinery used for branch mispredictions in Speculative execution — and, famously, the reason Spectre-class side channels exist: speculative work leaves observable traces in caches even after the ROB is flushed.

      Once the hardware path finishes, the OS's interrupt handler is running. Its first job is typically to save the remaining registers before doing real work — the rest is software.

      Remark. Steps 2 and 3 are why an OS needs to set up the IDT and TSS before enabling interrupts. Until both structures point somewhere valid, the first interrupt would either jump to garbage or push to a user-controlled stack — both fatal.

      Definition. Interrupt descriptor table [YK46]

      • May 15, 2026
      • Kellen Kanarios

      The CPU's lookup table from interrupt number to handler. The OS allocates the IDT in memory, fills entry \(N\) with the address of the handler for interrupt \(N\), and points the IDTR register at the table's base. From then on the Hardware interrupt path uses the IDT to dispatch every interrupt without any further software help.

      Each entry — a gate descriptor on x86 — carries enough information for the CPU to switch into the handler safely: the handler's linear address, the code segment to load (which sets the new privilege level, typically ring 0), and a type byte distinguishing interrupt gates (which mask further interrupts on entry) from trap gates (which don't). Entries 0–31 are reserved for CPU-defined exceptions (divide-by-zero, Page fault, double fault, ...); 32–255 are available for device IRQs and software interrupts.

      Remark. The IDT is one of three OS-installed tables that the CPU consults transparently — the others are the GDT (segment descriptors) and the page tables. Each is a hardware contract: the OS lays bytes out in a format the silicon understands, then hands over a pointer.

      Cache hierarchy [KBYI]

      • May 12, 2026
      • Kellen Kanarios

      Multiple tiers of progressively-larger, progressively-slower storage between the CPU's registers and main memory. Each tier holds the most-recently-used contents of the next slower tier, exploiting Spatial locality and Temporal locality in the workload.

      Approximate tiers on a modern x86 chip:

      • Registers — ns, dozens of bytes per core.
      • L1 cache — 2–8 ns, 8–128 KB per core. Often split into instruction and data L1s.
        • Built from static RAM (4–6 transistors per cell), which is fast but area-expensive vs. DRAM's single-transistor cell.
      • L2 cache — 5–12 ns, 0.5–8 MB. Closes the latency gap between L1 and DRAM.
      • L3 cache — slower, shared across cores.
      • Main memory — 10–60 ns, GBs.
      Example. A byte's journey. L1 checked first; on miss, L2; on miss, main memory; on miss in main memory, Page fault to disk.
      Remark. L1 is split into instruction and data caches because the locality profiles of code and data are very different — re-running the same code over different data has high temporal locality for code but low for data.

      Unix signals [9V3H]

      • May 12, 2026
      • Kellen Kanarios

      An asynchronous notification delivered to a Process. A process sends a signal with kill(pid, signum) (the name is misleading — most signals don't kill); the recipient can install a handler with signal(signum, handler) to "catch" it, otherwise the default action runs (often termination).

      Signals are the OS's mechanism for events the recipient didn't ask to be told about: SIGINT from Ctrl-C, SIGSEGV from a Page fault on an invalid page, SIGCHLD when a child exits, SIGTERM from an administrative kill.

      Who can signal whom depends on the user owning the process — a user can signal their own processes; the superuser can signal any process.

      On-demand zeroing [PW7Q]

      • May 12, 2026
      • Kellen Kanarios

      When a process asks for new heap memory, the OS must guarantee the returned pages contain zeros (to avoid leaking other processes' data). The naive implementation zeroes the page at allocation time. On-demand zeroing defers that work to first touch: until the process actually accesses a freshly-allocated page, no zero-fill happens.

      The OS marks the page as not-present in the Page table. The first access traps as a Page fault, the handler zeroes a physical page, installs it in the PTE, and resumes. Heap memory that is allocated but never used costs nothing.

      Trap table [0NVY]

      • May 12, 2026
      • Kellen Kanarios

      A hardware-readable table installed by the OS at boot that maps each kind of exceptional control-flow event (syscall, interrupt, page fault, illegal instruction, ...) to the address of an OS handler. When the CPU takes a trap, it consults this table to decide where to jump, switches into kernel mode, and runs the handler.

      The table is what makes it safe to expose privileged operations to user mode: user code never gets to pick the destination of a mode-switch, only the trap number. The OS controls every entry point, the CPU enforces the indirection.

      Remark. Installing the trap table is itself a privileged operation. It happens once, early in boot, before any user process exists.

      Page replacement [HK73]

      • August 8, 2025
      • Kellen Kanarios

      Once swap space is in play, the OS must decide which pages to evict from physical memory. The mechanism is the Page fault machinery; the question of policy is what this hub covers.

      Page fault [CXFM]

      • May 12, 2026
      • Kellen Kanarios

      Physical memory is finite, so the OS reserves a region of disk — swap space — to hold pages that don't fit. Pages currently on disk have their present bit cleared in the Page table entry; the PTE instead holds a disk address.

      When a process touches a virtual address whose page is not present, the hardware traps to the OS. This is a page fault. The handler:

      1. Looks at the PTE: if invalid, the access is illegal — terminate the process.
      2. If valid but not present: read the page in from swap, update the PTE, retry the access.
      Remark. The hardware never handles page faults itself — that would require it to understand disk I/O and the OS's swap layout. Page-fault handling is always in software.

      The three cases on every memory access:

      1. Present + valid: hardware reads the PPN from the PTE and proceeds.
      2. Not present + valid: hardware invokes the page-fault handler to retrieve from swap.
      3. Not valid: hardware raises an exception; the OS typically kills the process.
      Example. What happens when you dereference a nullptr? The TLB misses on VPN 0, the page-table walk finds the PTE marked invalid, and the hardware traps to the OS — which delivers SIGSEGV.

      LRU and the clock algorithm [7WG2]

      • May 12, 2026
      • Kellen Kanarios

      Least recently used is the standard heuristic for picking a page to evict: throw out the one that has been untouched the longest. It works because of the principle of locality — recently-used pages are likely to be used again soon (temporal locality).

      Pure LRU requires updating a data structure on every memory access, which is too expensive. The clock algorithm approximates it cheaply:

      1. Maintain a reference bit per page, set by hardware on access.
      2. Loop a pointer around the pages. If the reference bit is 1, clear it and advance.
      3. If the reference bit is 0, evict that page.
      Remark. LRU is not optimal. The pathological case is the looping workload: if the working set is one page larger than the cache, LRU misses 100% of the time.

      The quality of any replacement policy is measured by its average memory access time:

      \[AMAT = (1 - P_{miss}) T_P + P_{miss} T_D\]

      where \(T_P\) is physical-memory access time and \(T_D\) is disk access time. The disk gap is so large that even tiny miss rates dominate AMAT.

      Belady's anomaly [LQV3]

      • May 12, 2026
      • Kellen Kanarios

      A desirable property of a page-replacement policy is the stack property: increasing the cache size never makes the miss rate worse. Surprisingly, not every reasonable-looking policy has it.

      FIFO eviction violates the stack property — on certain access sequences, growing the cache strictly increases the number of misses. This is Belady's anomaly. It is what motivates LRU and the clock algorithm-style policies, which do have the stack property because they only depend on relative access recency.

      The optimal eviction policy (offline) is to evict the page whose next use is farthest in the future. Unimplementable in practice (you don't know the future), but useful as a theoretical upper bound to compare against.

      Two refinements make eviction practical on real workloads. A pure global policy lets one greedy process hog memory, which Segmented FIFO fixes with a per-process resident-set cap; and writing victims back one page at a time wastes disk bandwidth, which Page clustering amortizes by batching the dirty list.

      Segmented FIFO [HF6H]

      • July 11, 2026
      • Kellen Kanarios

      A process-aware refinement of LRU and the clock algorithm. Pure FIFO is a global policy — it does not distinguish memory by owner, so one greedy process can evict everyone else's pages. Segmented FIFO instead maintains a per-process FIFO queue capped at that process's resident set size, bounding how much physical memory any single process can hold.

      Evicted pages are not discarded outright; they move to one of two secondary lists:

      1. Clean page free list — evicted pages that were never modified.
      2. Dirty page list — evicted pages that must be written back before reuse.

      The secondary lists act as a victim cache: if a process faults on a page still sitting on the dirty list, it can be reclaimed without going to disk. The resident-set cap gives fairness; the secondary lists soften the cost of capping it too aggressively.

      Copy-on-write [DEJB]

      • May 12, 2026
      • Kellen Kanarios

      An optimization where two address spaces share the same physical page until one writes to it. Both processes have PTEs pointing at the same PPN, but with the write bit cleared. The first write traps to the OS, which allocates a fresh page, copies the contents, points the writing process's PTE at the new page (with writes re-enabled), and resumes.

      Used pervasively:

      • fork() — the child shares the parent's pages until either modifies one. Avoids the cost of copying an entire address space the child often immediately overwrites with exec().
      • Shared libraries — many processes map the same library text pages.
      • Kernel mapped into every process — same physical kernel code seen from every address space.

      Page clustering [YMHF]

      • July 11, 2026
      • Kellen Kanarios

      An amortization trick for the write side of page replacement. Writing one dirty page to swap at a time pays a full disk I/O per eviction, and small random writes are the worst case for a disk. Page clustering batches them: the OS collects a group of dirty pages from the global dirty list and writes them out together in a single larger, more sequential I/O. The per-page cost of paging out falls because the fixed overhead of a disk write is spread across the whole batch.

      Remark. A common convention is to map the kernel into every process's address space (via Copy-on-write) so kernel code and structures appear at the same virtual addresses everywhere.

        Page table [WALE]

        • May 12, 2026
        • Kellen Kanarios

        The per-process data structure that maps each virtual page number (VPN) to a physical page number (PPN). A virtual address is split into a VPN and an offset within the page; the page table translates the VPN, the offset is appended unchanged.

        \[\underbrace {01}_{\text {VPN}}\underbrace {110\cdots 01}_{\text {offset}}\]

        Each page-table entry (PTE) carries more than just a PPN:

        1. Valid bit: whether this VPN has memory allocated. Gaps between stack and heap need not be allocated.
        2. Protection bits: read/write/execute permissions.
        3. Present bit: whether the page currently resides in physical memory (vs. swap).
        4. Dirty bit: has the page been modified since being paged in.
        5. Reference bit: has the page been accessed (used by eviction policy).
        Problem. The table gets very large. For a 32-bit address space with 4KB pages and 4-byte PTEs, a linear page table costs \(2^{32}/2^{12} \cdot 4 = 2^{22}\) bytes — 4MB per process.
        Problem. Translation requires accessing the table in memory, which is slow on every memory reference. Mitigated by the Translation lookaside buffer.

        The size problem is mitigated by Multi-level page table, which only allocates the parts of the table that are actually used.
