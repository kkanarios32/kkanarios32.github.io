  1. For what types of workloads does SJF scheduling deliver the same turnaround as FIFO scheduling?
    2. Answer. For workloads that arrive in increasing order of total job time.
  2. For what types of workloads and quantum lengths does SJF deliver the same response times as Round-robin scheduling?
    3. Answer. For shorter jobs and longer quantum lengths, RR and SJF should deliver similar response times.
  3. What happens to response time with SJF as job lengths increase?
    4. Answer. Response time degrades because the longest jobs have to wait for each of the shorter jobs to finish. If these shorter jobs become longer than the response time grows linearly.
  4. What happens to response time with RR as quantum lengths increase? Can you write an equation that gives the worst-case response time, given \(N\) jobs?
    5. Answer. The response time is the average over stride length number of preceding jobs. The formula can be written as \[\mathrm {RT} = \frac {1}{N} \sum _{i = 1}^{N} (i - 1)S = \frac {S(N - 1) N}{N} = S(N - 1)\]

Deliberate-Practice (6/66).

  1. Write a program that calls fork(). Before calling fork(), have the main process access a variable (e.g., x) and set its value to something (e.g., 100). What value is the variable in the child process? What happens to the variable when both the child and parent change the value of x
    2. Answer. Both the parent and the child get a copy of the variable values i.e. \(x\). When they modify \(x\) it only affects the copy of \(x\) in their local address space.
  2. Write a program that opens a file (with the open() system call) and then calls fork() to create a new process. Can both the child and parent access the file descriptor returned by open()? What happens when they are writing to the file concurrently, i.e., at the same time?
    3. Answer. Both do have access to the same file descriptor. I believe that they can interleave writes if writing at the same time but for me whichever process got ran first always finished writing prior the other starting. This may be a scheduling thing, where one process is able to complete before the other is scheduled?
  3. Write another program using fork(). The child process should print "hello"; the parent process should print "goodbye". You should try to ensure that the child process always prints first; can you do this without calling wait() in the parent?
    4. Answer. Using the previous problem, in the child, we can write something to a file. Then, in the parent, we can enforce an ordering by making the parent wait in a while loop until it reads the value written by the child. However, this is terribly inconvenient (and probably why wait() exists).
  4. Now write a program that uses wait() to wait for the child process to finish in the parent. What does wait() return? What happens if you use wait() in the child?
    5. Answer. wait() returns the pid of the child process that died. Using wait() in the child just immediately returns \(-1\).
  5. Write a program that calls fork() and then calls some form of exec() to run the program /bin/ls. See if you can try all of the variants of exec(), including execl(), execle(), execlp(), execv(), execvp(), and execvP(). Why do you think there are so many variants of the same basic call?
    6. Answer. It seems that some of them support different types of arguments i.e. execv family allows variable size argument list. Additionally, there is a distinction in the environment. This seems to be a list of environment variable values that exec{l,v}e() will search over. The \(p\) family use the same search as the shell i.e. the PATH.
  6. Write a slight modification of the previous program, this time using waitpid() instead of wait(). When would waitpid() be useful?
    7. Answer. If a parent process has multiple children, then wait() will only wait until the first one dies. waitpid() lets you specify waiting on a specific child process.
  7. Write a program that creates a child process, and then in the child closes standard output (STDOUT_FILENO). What happens if the child calls printf() to print some output after closing the descriptor?
    8. Answer. Nothing prints in the child process. However, stdout is still open in the parent process and able to print.
  8. Write a program that creates two children, and connects the standard output of one to the standard input of the other, using the pipe() system call
    9. Answer. This required fflush() for me. I am not sure if this is true in general?

      Problem. Review the methods discussed in the first part of the book. What are their strengths and weaknesses? When should they be used?
      Solution.
      Strength Weakness Use case
      Bandits Simplest case. Can isolate exploration problem. No credit assignment. Action only effect next timestep. Clinical trials. Things without temporally extended outcomes.
      Dynamic Programming Exact solution. Complexity blows up with state and action space. Model-based. Tabular MDPs.
      Monte Carlo Unbiased estimator. Model-free. Extremely high variance. Must wait till end of episode. When interacting with environment is inexpensive, episode length is short.
      TD methods Model-free. Online. Biased estimator. Environment interaction is cheap but also episode length is long or continuing.
      N step Balance bias-variance tradeoff. Must choose \(n\). Lower variance continuing environments?
      Dyna Sample efficiency. Model-based. When interacting with environment is costly i.e. driving.

      Show that tabular methods are a special case of linear function approximation. What would the feature vectors be?

      Solution.

      As a linear function approximation method, we have two quantities \(\boldsymbol {w}\) and \(\boldsymbol {x}(s)\). We make the update \[w_{t + 1} = w_{t} + \alpha (r_t + \gamma w_{t}^{\top } x_{t+1} - w_t^{\top } x_t) x_t\] In the tabular setting, we just assume that we have the capacity to represent every possible state. This means that \(x(s) \in \mathbb {R}^{|S|}\). Therefore, if we define the features as \(x : s_i \mapsto \boldsymbol {e}_i\) then we can recover policy evaluation by taking, \[r_i = \mathbb {E}_{a \sim \pi , s \sim p}[r(s_i)], \quad w_i = v_{\pi }(s_i)\]

      Substituting, we get \[ w_{t + 1} = w_{t} + \alpha (\mathbb {E}_{a \sim \pi , s \sim p}[r(s_i)] + v(s_{t + 1}) - v(s_{t})) \cdot \boldsymbol {e}_i \] Since \(\boldsymbol {w}\) is our vector of values, updating the \(i\)th entry is exactly performing exactly one update to \(v_{\pi }(s_i)\) in the tabular setting.

      1. A discrete probability space cannot contain an infinite sequence of independent events \(A_1, A_2, \ldots \) each with probability \(1/2\). To see this, denote the event space as \(\mathcal {X}\). Note that \[P(A_1 \cap A_2) + P(A_1 \cap A_2^c) + P(A_1^c \cap A_2) + P(A_1^c \cap A_2^c) = 1\] Therefore, every \(x \in \mathcal {X}\) must be in one of those four events and have probability at most \(1/4\). Similarly, we can construct \(n\) disjoint sets in the same manner and bound \(\mathbb {P}(\{x\})\) by \(1/n\). Taking \(n \to \infty \), we have that \(\mathbb {P}(\{x\}) = 0\). Since \(\mathcal {X}\) is a discrete probability space, we must have that \[\sum _{x \in \mathcal {X}} \mathbb {P}(\{x\}) = 1\] However, we have just shown that for each \(x \in \mathcal {X}\), \(\mathbb {P}(\{x\}) = 0\). Thus, we have reached a contradiction.

       Let \(P\) be a probability measure on a field \(\mathscr {F}_{0}\) and for every subset \(A\) of \(\Omega \) define \(P^{\star }(A)\) by (3.1). Denote also by \(P\) the extension of \(P\) to \(\mathscr {F}=\sigma (\mathscr {F}_{0})\).
      1. Show that \(A\) is \(P^*\)-measurable if and only if \(P^*(A) = P_*(A)\).
      Solution.

      First suppose that \(P^*(A) = P_*(A)\). Let \(E \subset \Omega \). We must show that \[P^*(E \cap A) + P^*(E \cap A^c) = P^*(E)\] From part (1) and our hypothesis, \(\exists \overline {A}, \underline {A} \in \mathscr {F}\) such that \(\underline {A} \subset A \subset \overline {A}\) and \[P(\overline {A}) = P^*(A) = P_*(A) = P(\underline {A})\]. Additionally, \(\exists B \in \mathscr {F}\) such that \(E \subset B\) and \(P^*(E) = P(B)\). Then we have that \[P^*(E \cap A) \leq P^*(B \cap \overline {A}) = P(B \cap \overline {A})\] and \[P^*(B \cap A) \leq P^*(B \cap \underline {A}^c) = P(B \cap \underline {A}^c)\] Thus, \[P^*(E \cap A) + P^*(E \cap A^c) \leq P(B \cap \underline {A}^c) + P(B \cap \overline {A}) = P^*(E)\] as desired.

      Remark. I really struggled for a minute on even understanding this problem. The import thing to understand for this problem is what space everything is defined on.
      1. \(P\) is defined on the initial field \(\mathscr {F}_0\), which is the closure over finite unions and complements of \(\Omega \).
      2. \(P^*\) is defined on the powerset \(\mathcal {P}(\Omega )\). \(P^*\) is NOT necessarily a measure on all of these subsets though.
        • \(E\) is NOT necessarily in \(\mathscr {F}\) hence the need for \(B\).
      3. \(P\) (abuse of notation) is \(P^*\) restricted to the \(P^*\)-measurable subsets of \(\Omega \).
      Common question is why we cannot just apply finite additivity and have everything \(P^*\)-measurable? The answer is that \(P^*\) is not countably subadditive on \(\mathcal {P}(\Omega )\) but only the \(P^*\)-measurable sets.

        The process abstraction, what the OS has to support to maintain it, and the data structure that holds it together.

        Process [FCHM]

        • May 12, 2026
        • Kellen Kanarios

        A running program, plus all the state the OS needs to pause it and resume it later. A process consists of:

        1. Memory: an Address space holding the program's code, stack, and heap.
        2. Registers: general-purpose registers carrying current computation, plus special-purpose registers:
          • Program counter (PC): the next instruction to run.
          • Stack pointer and frame pointer: where the process sits on its stack.

        A process moves between three states:

        • running: currently on a CPU.
        • ready: runnable, waiting for the Scheduling to give it a CPU.
        • blocked: waiting on I/O or another event, not runnable.
        Remark. Threads (see Concurrency Intro) share an address space within a process but each have their own PC, register set, and stack.

        Process API [JJ45]

        • May 12, 2026
        • Kellen Kanarios

        The minimum operations any OS must expose for working with Processes:

        1. Create: load a program from disk, set up its address space (stack, heap, code), and jump to main. The OS also initializes argc/argv.
        2. Destroy: kill a misbehaving or unwanted process.
        3. Wait: block until another process completes — useful for ordering dependent work.
        4. Miscellaneous control: e.g. suspend / resume.
        5. Status: report on running processes.
        Remark. The stack is allocated up front. The heap starts small and grows via malloc, which calls into the OS (brk/sbrk) when it runs out of headroom.

        On Unix, the concrete realization of this API uses fork(), wait(), and exec() — covered separately.

        Process control block [JE53]

        • May 12, 2026
        • Kellen Kanarios

        The kernel-side data structure holding all per-process state. There is one PCB per Process, threaded together into the process list. The PCB is what the OS reads and writes during a Context switch: the saved register set lives here, as does the process state, the open files, the parent pointer, and so on.

        An example from xv6:

  int eip;
  int esp;
  int ebx;
  int ecx;
  int edx;
  int esi;
  int edi;
  int ebp;
};
// the different states a process can be in
enum proc_state { UNUSED, EMBRYO, SLEEPING, RUNNABLE, RUNNING, ZOMBIE };
// the information xv6 tracks about each process
// including its register context and state
struct proc {
  char *mem;                 // Start of process memory
  uint sz;                   // Size of process memory
  char *kstack;              // Bottom of kernel stack for this process
  enum proc_state state;     // Process state
  int pid;                   // Process ID
  struct proc *parent;       // Parent process
  void *chan;                // If non-zero, sleeping on chan
  int killed;                // If non-zero, have been killed

          Round-robin scheduling [NEL8]

          • May 12, 2026
          • Kellen Kanarios

          Cycle through all runnable jobs, giving each a fixed time slice — the stride, time slice, or quantum length — before preempting and moving to the next. The best you can do at minimizing response time: every job sees the CPU within at most \(N\) quanta where \(N\) is the runnable-job count.

          The quantum length is a tradeoff:

          • Short quantum: better response time, more frequent context switches, worse cache locality.
          • Long quantum: amortizes context-switch cost, worse response time.

          Worst-case response time for \(N\) jobs at quantum \(S\) is \(S(N-1)\).

          Remark. Round-robin minimizes response time but is poor for turnaround time relative to STCF scheduling. A scheduler that wants both has to combine ideas — see Multi-level feedback queue.

          SJF scheduling [VJ4F]

          • May 12, 2026
          • Kellen Kanarios

          Shortest job first. Run the job with the smallest total work first, run-to-completion. If all jobs arrive at the same time, SJF provably minimizes average turnaround time.

          Problem. Real workloads don't all arrive at \(t = 0\). A long job that arrives slightly before a short one still gets to monopolize the CPU until it finishes — SJF degenerates to FIFO scheduling under sequential arrivals.

          Removing the run-to-completion constraint and allowing preemption gives STCF scheduling.

          FIFO scheduling [JA6Z]

          • May 12, 2026
          • Kellen Kanarios

          First-in, first-out. Jobs are run to completion in the order they arrive. Trivially fair under any reasonable notion of fairness, but suffers from the convoy effect: a single long-running job at the head delays every shorter job behind it, inflating average turnaround time.

          Problem. If the first arriving job is much longer than the rest, all subsequent jobs wait at least its full duration before getting any CPU.

          Unix signals [9V3H]

          • May 12, 2026
          • Kellen Kanarios

          An asynchronous notification delivered to a Process. A process sends a signal with kill(pid, signum) (the name is misleading — most signals don't kill); the recipient can install a handler with signal(signum, handler) to "catch" it, otherwise the default action runs (often termination).

          Signals are the OS's mechanism for events the recipient didn't ask to be told about: SIGINT from Ctrl-C, SIGSEGV from a Page fault on an invalid page, SIGCHLD when a child exits, SIGTERM from an administrative kill.

          Who can signal whom depends on the user owning the process — a user can signal their own processes; the superuser can signal any process.

          exec() [YHJR]

          • May 12, 2026
          • Kellen Kanarios

          Replaces the calling Process's program with a new executable. exec() does not return on success — the original code, data, stack, and heap are discarded and the new program runs in their place. The PID is unchanged.

          What exec() does internally:

          1. Loads the new executable's code and static data, overwriting the current.
          2. Re-initializes stack and heap.
          3. Jumps to the new program's entry point with arguments passed as argv.

          The canonical Unix pattern is fork() then exec() — the shell uses this to launch a command without becoming that command:

          int main()
  while (true) {
    command = getcommand();
    int rc = fork();
    if (rc < 0)
      printf("error with fork");
    else if (rc == 0)
      exec(command);            // child becomes the command
    else
      int rc = wait(NULL);      // parent waits for it
  }

          Redirection is implemented by opening a file in the child between fork() and exec(), attached to STDOUT_FILENO:

          if (rc == 0)
  close(STDOUT_FILENO);
  open("output.txt");
  exec(command);
          Remark. The many variants — execl, execle, execlp, execv, execvp, execvP — differ in how they take arguments (list vs. vector), whether they use the PATH environment variable, and whether they take an explicit environment.

          wait() [J6VZ]

          • May 12, 2026
          • Kellen Kanarios

          Blocks the calling Process until one of its children terminates. The OS marks the child's PCB ZOMBIE on exit, keeping enough state (exit status, resource usage) for the parent to retrieve via wait(); only when the parent collects it is the child fully reaped.

          wait() returns the PID of the child that died. Variant waitpid(pid, ...) blocks on a specific child — useful when a parent has multiple children and only one is interesting.

          Used to enforce ordering between parent and child code paths after fork(), or to know when an exec()'d program finishes.

          Remark. Calling wait() with no children to wait on returns -1.

          fork() [O1S9]

          • May 12, 2026
          • Kellen Kanarios

          The Unix system call that duplicates the calling Process. After fork() returns, two processes exist where there was one. The child gets its own copy of:

          1. Address space.
          2. General-purpose registers.
          3. Program counter — the child returns from fork() at the same instruction the parent does.

          The two processes are distinguished by fork()'s return value:

          int main()
  int rc = fork();
  if (rc < 0)
    printf("error with fork");
  else if (rc == 0)
    printf("in child");
  else
    printf("in parent");        // rc holds child PID
          Remark. Which process is scheduled first after fork() is non-deterministic. Use wait() if you need ordering.
          Remark. The copy is logical, not physical: real implementations use Copy-on-write so the duplicated address space only allocates fresh pages when one side writes to them.
