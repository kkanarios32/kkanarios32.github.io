Snoop-based coherence keeps caches consistent by watching a shared bus [8C2H]

  • May 12, 2026

A Cache coherence family where every cache watches (snoops) a shared bus. When one cache issues a coherence transaction (e.g. requests permission to read or write a line), every other cache sees it and updates its local state accordingly. No central directory needed; the bus itself is the coordination mechanism.

The simplest snoop-based protocol is valid-invalid:

  • Invalid + load → BusRd to gain read permission; transition to valid.
  • Invalid + store → no transition (write-through assumed; the bus carries the write).
  • Valid + store → BusWr to notify other caches; they invalidate.

Write-update coherence, the variant where stores update other caches' copies instead of invalidating them, works for very small bus-based systems but does not scale: every write reaches everyone, and copies of data the snooper isn't using pollute its Cache replacement policy.

Snoop-based coherence relies on a shared physical bus, which limits it to roughly 8 cores. Beyond that, switch to directory-based coherence.

Remark. Snoop-based SMPs use the Inclusion property so that snooping at L2 suffices — every L1 line is also tracked at L2.

    Directory-based coherence scales by replacing broadcasts with point-to-point messages [KIT8]

    • May 12, 2026

    A scalable Cache coherence organization for systems too large for a shared bus. Each memory address has a home node; the home node maintains a directory entry listing which caches currently hold copies and in what state. Coherence transactions are point-to-point messages routed via the directory rather than broadcasts. The directory is itself distributed — each node owns the entries for the addresses in its local DRAM, and a physical address uniquely identifies its home — so there is no central bottleneck.

    Compared to snoop-based coherence, directory protocols trade a level of indirection (3 hops instead of 2 for some transactions) for vastly reduced bandwidth — making them viable at hundreds of cores.

    Three refinements make directory coherence practical. Its metadata is kept from scaling with memory by sparse directories; its multi-hop transactions force the protocol to reason about transient states and races; and workload-specific patterns like migratory sharing let it collapse pairs of transactions into one. See Implementing Directory-based MESI in Murphi for a full worked implementation in Murphi.

    Inclusion property [4W5G]

    • May 12, 2026

    A cache hierarchy property: every line held in a higher-level cache (closer to the core, e.g. L1) must also be present in the next-level cache (L2). The hierarchy contains its own contents at every level — L2 ⊇ L1.

    Why it matters for snoop-based coherence: with inclusion, a snoop only needs to consult L2. If a snoop hits in L2, then either the line is also in L1 (and L2 forwards the snoop) or it's not (and L1 doesn't need to look). This means L1 doesn't need a snoop port.

    Problem. L1 hits don't update LRU metadata at L2 — so L2 might evict a line that L1 is actively using, violating inclusion.
    Solution. Either (a) refuse to evict an L2 line that's still in L1, or (b) when evicting from L2, also evict from L1.
    Remark. Inclusive caches waste capacity (every line is duplicated). Non-inclusive and exclusive variants trade some snoop simplicity for better effective capacity.

    Cache coherence [GA7N]

    • May 12, 2026

    In a Multicore system where every core has its own Cache hierarchy, the same memory address can live in multiple caches simultaneously. A cache coherence protocol is the mechanism that keeps those copies consistent — propagating writes, invalidating stale copies, and preserving two invariants:

    1. Single-writer invariant: at any moment, at most one core has permission to write a given line. No other core can read or write it concurrently.
    2. Data-value invariant: all cores see the same order of updates to a given line.

    Two organizational families:

    • snoop-based coherence — every cache watches a shared bus and reacts to other cores' traffic. Scales to ~8 cores.
    • directory-based coherence — a structure tracks who has a copy of each line; coherence messages are point-to-point. Scales to hundreds of cores at the cost of an extra hop.

    The concrete state machines are protocols like MSI protocol, MESI protocol, MOESI protocol, and DEC Firefly, each adding states to handle more cases or to optimize specific sharing patterns. A worked implementation of directory-based MESI in Murphi lives in Implementing Directory-based MESI in Murphi.

      Cache replacement policy [UG51]

      • May 12, 2026

      Within a Cache associativity set, when a new line arrives and the set is full, which existing line gets evicted? Same question as LRU and the clock algorithm for pages, different timescales.

      The hot policy is least recently used — but pure LRU requires tracking a per-line ordering and updating it on every hit, which is too expensive at L1 latencies. Real caches use pseudo-LRU — a tree of single bits per set that approximates LRU with a fraction of the bookkeeping. The clock algorithm is one well-known pseudo-LRU scheme.

      Remark. Block (cache line) size is part of the same tradeoff. Larger blocks improve Spatial locality but cause cache pollution (you fetch bytes you don't use) and eat memory bandwidth. The sweet spot is around 64 bytes on modern x86.
