Software pipelining vs. loop unrolling [MOPC]

  • July 17, 2026

See [hennessy2019computer, H.3] for more details.

Loop unrolling at its core performs no instruction re-ordering. It literally just expands the loop. When combined with a compiler, this increase the size of your basic blocks allowing for more optimization opportunities. This also eliminates overhead of branch prediction

Software Pipelining does not rely on the unroll window.

Example. Loop unrolling cannot make use of the large distance between dependencies for large \(k\) below. 
for (i = k; i < n; i++)
    a[i] += a[i - k];
For \(k = 3\), Software pipelining can perform something like 
for (i = 3; i < n; i++)
    a[i - 2] += a[i - 2 - k];
    a[i - 1] += a[i - 1 - k];
    a[i] += a[i - k];
You can imagine how this becomes more useful as \(k\) grows large.
Example. There is an actual dependency between instructions here that cannot be circumvented by OoO hardware 
for (i = 0; i < n; i++)
    sum += a[i];
However, this can be re-written at the software level to get some parallelism within loop bodies as: 
for (i = 0; i < n; i += 4) {
    s0 += a[i];   s1 += a[i+1];
    s2 += a[i+2]; s3 += a[i+3];
}
sum = (s0 + s1) + (s2 + s3);}

        Definition. Software Pipelining [AOWY]

        • July 17, 2026
        • Kellen Kanarios

        A compiler transformation that restructures a loop so the steady-state body contains instructions from multiple logical iterations, each at a different stage — statically constructing the overlap an OoO core would find dynamically. Canonical algorithm: modulo scheduling.

        Out-of-order execution runs ready instructions past a stalled one, committing in program order [U0NY]

        • May 12, 2026
        • Kellen Kanarios

        Execute instructions in whatever order their operands become available, then commit them back in program order. The programmer's model still sees sequential execution; the hardware reorders aggressively underneath to keep Execution units busy across data hazards and memory latency.

        Two structures make this work:

        • Reservation stations — hold instructions waiting on operands, so a stalled instruction doesn't block independent ones behind it.
        • Reorder buffer — preserves program order so commits look sequential and mispredicted Speculative execution paths can be unwound.

        Combined with register renaming to break false dependencies, the back-end of a modern CPU is essentially a dynamic scheduler that uses the program order only as a hint about what to retire next.

        Example. An out-of-order pipeline can issue a load early (because its address operands are ready), execute several unrelated ALU ops while waiting for the memory, then have all four instructions ready to commit in program order. The illusion of sequential execution is preserved; the parallelism is real.
