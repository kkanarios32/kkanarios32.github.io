Locks [Q2DT]

  • March 9, 2026

A lock is a synchronization primitive that grants mutually exclusive access to a critical section. The implementations below trade off coherence traffic, fairness, and memory in different ways. The naive locks (Test-and-set lock and Test-and-test-and-set lock) are simple but generate \(O(n)\) coherence traffic per release; queue-based locks (Ticket lock, Array-based lock, MCS lock) scale better at the cost of extra state per waiter.

1. Test-and-set lock [MJ5L]

  • May 12, 2026

The simplest spin lock. The acquire loop spins on an atomic test_and_set: it writes LOCKED and reads the previous value in one step. If the previous value was UNLOCKED, the caller has the lock; otherwise it tries again.

acquire(lock_ptr):
  while (true):
    // Perform "test-and-set"
    // UNLOCKED = 0, LOCKED = 1
    test_and_set(old, lock_ptr)
    if (old == UNLOCKED):
      break // lock acquired!
    // keep spinning, back to top of while loop

release(lock_ptr):
  store[lock_ptr] <- UNLOCKED
Problem. Major coherence traffic. Every spinning core must acquire write permissions and invalidate everyone else on each test_and_set, so contention scales badly with the number of waiters.

2. Test-and-test-and-set lock [PXY9]

  • May 12, 2026

A refinement of the Test-and-set lock that reduces coherence traffic during the spin. Threads first perform a plain load to check whether the lock looks free, and only attempt the atomic test_and_set when it does. Spinning is then read-only, so cores can share the cache line.

acquire(lock_ptr):
  while (true):
    // Perform "test"
    load [lock_ptr] -> original_value
    if (original_value == UNLOCKED):
      // Perform "test-and-set"
      test_and_set(old, lock_ptr)
      if (old == UNLOCKED):
        break // lock acquired!
    // keep spinning, back to top of while loop

release(lock_ptr):
  store[lock_ptr] <- UNLOCKED
Remark. Spinning is a read-only operation now.
Problem. When the lock is released, the line must be invalidated on every waiter, and they all simultaneously read and try to write. The contention is shifted from steady state to release time but is still \(O(n)\) in the number of waiters.

3. Ticket lock [V7KP]

  • May 12, 2026

A fair spin lock based on two counters. Each arriving thread takes a "ticket" by atomically incrementing next_ticket and then spins until now_serving equals its ticket. Release simply increments now_serving. Threads acquire the lock in FIFO order, which prevents starvation but does not by itself reduce contention.

acquire(lock_ptr):
  my_ticket = fetch_and_increment(lock_ptr->next_ticket)
  while (lock_ptr->now_serving != my_ticket); // spin

release(lock_ptr):
  lock_ptr->now_serving = lock_ptr->now_serving + 1
Problem. Still \(O(n)\) coherence traffic on release: incrementing now_serving invalidates the line on every waiter, and they all reload it to recheck their ticket.

4. Array-based lock [BJ7N]

  • May 12, 2026

Each waiter is assigned its own slot in an array and spins only on that slot. Release just flips the next slot to "go-ahead", so the cache line touched by release is the one read by exactly one other thread. Coherence traffic per release is constant rather than \(O(n)\), at the cost of one slot per waiter.

  • Give each waiter its own "slot" to wait on.
  • Only need to set the next slot to "go-ahead" on release.
  • Constant coherence traffic.
Problem. \(O(n)\) memory for the lock array, and padding each slot to \(64\)b (one cache line) to avoid false sharing makes that cost real.

5. MCS lock [IN64]

  • May 12, 2026

A scalable, list-based queue lock. The lock variable is a pointer to the tail of a linked list of per-thread nodes. Each thread enqueues its own node and spins on a flag in its own cache line. The releaser wakes only its direct successor, so coherence traffic per handoff is constant and memory cost is proportional only to threads currently waiting, not to the lock's fanout.

acquire(lock):
  I->next = null
  predecessor = fetch_and_store(lock, I)
  if predecessor != nil:                   // some node holds the lock
    I->must_wait = true
    predecessor->next = I                  // predecessor must wake us
    repeat while I->must_wait              // spin till lock is free

release(lock):
  if (I->next == null):                    // no known successor
    if compare_and_swap(lock, I, nil):     // make sure...
      return                               // CAS succeeded; lock freed
    repeat while I->next == nil            // spin to learn successor
  I->next->must_wait = false               // wake successor

    Dennard scaling [S0TN]

    • May 12, 2026

    The observation that as transistor dimensions shrink by a factor \(\alpha \), you can also drop voltage by \(\alpha \), keeping power per transistor constant:

    \[P = C \left (\frac {V}{\alpha }\right )^2 f\]

    In the Dennard era, every process shrink got more transistors at the same power budget — the foundation of decades of frequency scaling.

    Post-Dennard. Below ~65nm, voltage can no longer be scaled because the supply-to-threshold ratio \(V_{dd}/V_{th}\) must stay above ~3 for transistors to switch reliably. The consequences:

    • Power per transistor scales only as \(1/\alpha \), not \(1/\alpha ^2\).
    • Power density therefore scales as \(\alpha \) — denser chips dissipate more per unit area.

    This is the same constraint that drove the Power wall in commercial CPUs: single-core frequency scaling stopped working, and the industry pivoted to multicore and energy-efficient architectures.

    Transactional memory [Y8JE]

    • May 12, 2026

    An alternative to fine-grained Locks: mark code regions as transactions that should appear atomic, execute them speculatively in parallel, and detect conflicts after the fact. Aims at the performance of fine-grained locking with the programming ergonomics of coarse-grained locking.

    Conflict detection

    • May 12, 2026

    Two transactions conflict if they access the same memory location and at least one of them is a write. The implementation tracks per-transaction read and write sets.

    • Lazy detection. Build the entire read/write set as the transaction runs; check for conflicts only at commit time. If conflict, roll back.
    • Eager detection. Check on every access against other transactions' sets; abort immediately on conflict.

    Eager detection can piggyback on the cache coherence protocol: add read/write bits to each cache line, set them during a transaction, and let a coherence invalidation signal a conflict.

    Version management (rollback)

    • May 12, 2026

    Two operations: commit and abort. Implementations:

    1. Lazy. stores go to a "write table" rather than memory; loads check the table first, falling back to memory. Commit writes the table back (slow); abort just clears the table (fast).
    2. Eager (coherence-backed). stores write through but track read/write sets per cache line. Commit clears the sets (fast); abort clears the sets (also fast — work already on overwrite if not aborted before commit).
    Problem. If a transaction touches more cache lines than the cache holds, the tracking overflows.
    Solution. Fall back to software transactional memory (STM) on overflow — a hybrid scheme that keeps hardware fast-path for common cases.

    Cache coherence [GA7N]

    • May 12, 2026

    In a Multicore system where every core has its own Cache hierarchy, the same memory address can live in multiple caches simultaneously. A cache coherence protocol is the mechanism that keeps those copies consistent — propagating writes, invalidating stale copies, and preserving two invariants:

    1. Single-writer invariant: at any moment, at most one core has permission to write a given line. No other core can read or write it concurrently.
    2. Data-value invariant: all cores see the same order of updates to a given line.

    Two organizational families:

    • snoop-based coherence — every cache watches a shared bus and reacts to other cores' traffic. Scales to ~8 cores.
    • directory-based coherence — a structure tracks who has a copy of each line; coherence messages are point-to-point. Scales to hundreds of cores at the cost of an extra hop.

    The concrete state machines are protocols like MSI protocol, MESI protocol, MOESI protocol, and DEC Firefly, each adding states to handle more cases or to optimize specific sharing patterns. A worked implementation of directory-based MESI in Murphi lives in Implementing Directory-based MESI in Murphi.

    Worked instance: Implementing Directory-based MESI in Murphi implements directory-based MESI in Murphi.

    Memory consistency

    • February 8, 2026
    • Kellen Kanarios

    Sequential consistency [OW4D]

    • May 12, 2026

    The strongest practical memory model. The result of any execution is the same as if the operations of all threads were executed in some single sequential order, and the operations of each thread appear in that order in the same sequence as the program specifies.

    Equivalent operational definition: imagine a single switch that, at each step, selects one thread to run its next operation. Any valid execution is the result of some sequence of such selections. There is no reordering across threads, no reordering within a thread, and no buffering that becomes visible to other threads.

    Sequential consistency is what most programmers reason about by default — and what real hardware does not provide, because it forbids almost every optimization that makes a CPU fast. Relaxed consistency is what we get in exchange.

    Remark. SC does not provide guarantees between threads beyond ordering. Two threads racing on the same variable are still racing — SC just promises a consistent global timeline.
    Remark. Enforcing SC purely in hardware requires loads to wait for the Store buffer to drain and stores to retire strictly in order. Both kill performance. SC-preserving compilers try to push the constraint to compile time by refusing to reorder shared accesses.

    Relaxed consistency [BCFG]

    • May 12, 2026
    • with contributions from Kellen Kanarios

    A memory model weaker than Sequential consistency that allows the hardware (and compiler) to reorder memory operations for performance, with explicit Memory fence instructions where the programmer wants to forbid reordering.

    A fence is implemented with a counter — incremented on issue of each memory op, decremented on completion. The fence stalls dispatch until the counter reaches zero, which forces the Store buffer to drain before later accesses may proceed.

    Real machines pick a relaxation along a spectrum: Total store order permits only store-load reordering and is what x86 ships; Relaxed memory order permits anything and pushes the ordering burden onto the programmer. Either way, the next move is to make the fences themselves cheaper, which is what Release consistency does.

    Definition. Total store order [VH8Y]

    • May 24, 2026
    • Kellen Kanarios

    A Relaxed consistency relaxation in which loads may be hoisted above prior stores, but no other reorderings are permitted. The motivation: a store can sit in the Store buffer waiting for cache, while subsequent loads — which do not touch the buffered address — proceed without stalling.

    Same-core reads of a still-buffered store get the right value through Store-to-load forwarding; cross-core reads see the old value until the buffer drains. This asymmetry is the entire point of TSO's relaxation: the store buffer hides latency from the issuing thread without forcing other threads to wait. x86 implements TSO.

    Remark. Under TSO, Dekker's algorithm is still broken without Memory fence — the classic example where TSO is too weak for naive software synchronization.
    Remark. The IBM 370 variant is TSO without Store-to-load forwarding — it preserves global store atomicity at the cost of same-core stall-on-self-read.

    Definition. Relaxed memory order [WT44]

    • May 24, 2026
    • Kellen Kanarios

    A Relaxed consistency relaxation in which any pair of memory operations may be reordered. RMO is the most permissive end of the consistency spectrum; the programmer recovers ordering only through explicit Memory fence instructions.

    Each stronger reordering still has to be justified per case:

    • Load → store — no performance gain to reorder; leave alone.
    • Store → store — keep ordering with a slightly larger Store buffer.
    • Load → load — speculate the second load while waiting for the first; the reorder is invisible if the speculation is verified.

    Because every fence is expensive under RMO, the natural next move is to weaken the fences themselves into half-fences, which is what Release consistency does.

    Definition. Release consistency [ZWHV]

    • May 24, 2026
    • Kellen Kanarios

    A Relaxed consistency discipline that recovers data-race freedom (see Data race) under Relaxed memory order not by issuing fewer fences, but by issuing weaker ones. The full-fence cost of Sequential consistency is replaced by a pair of half-fences placed exactly at the boundaries of critical sections — acquire on entry, release on exit.

    The intuition is that correctness around a critical section only requires that work cannot leak out of it; work is free to sink into it. Half-fences encode exactly this asymmetry, which is why they are the right primitive for lock-based and lock-free synchronization alike.

    Definition. Full fence [S80U]

    • May 24, 2026
    • Kellen Kanarios

    A Memory fence across which no memory operation may move in either direction. Prior accesses must become globally visible before any subsequent access may issue. x86's mfence is the canonical example — the hammer that drains the Store buffer and serializes all prior memory operations against all subsequent ones.

    Full fences are the default ordering primitive under Sequential consistency — every memory operation behaves as if a full fence sat on either side. They are also the only fence one needs to reason about correctness, which is why they are the building block for the half-fences of Release consistency.

    Definition. Half fence [S5D0]

    • May 24, 2026
    • Kellen Kanarios

    A Memory fence that constrains motion in only one direction, splitting the symmetric guarantee of a Full fence into two cheaper primitives:

    • Acquire — operations after the fence may not move above it. Used at the entry to a critical section: subsequent reads and writes cannot be hoisted before the lock is held.
    • Release — operations before the fence may not move below it. Used at the exit of a critical section: prior writes cannot be deferred past the unlock.

    The asymmetry is exactly the freedom one wants around a critical section: expanding the section is fine — work may sink into it (past acquire) or rise into it (before release) — but work may not leak out of it. This is the basis of Release consistency and the reason C++ std::atomic exposes memory_order_acquire / memory_order_release as its core primitives.

    C++ std::atomic compiles to half-fences in the right places, which is why it is the standard primitive for portable lock-free code.

    Definition. Store buffer [UNB6]

    • May 13, 2026
    • Kellen Kanarios

    A FIFO of retired-but-not-yet-globally-visible stores that decouples instruction commit from cache write. When a store reaches the head of the Reorder buffer and commits, its value has not yet reached L1 — commit only hands the store off to the store buffer. The actual cache write, called drain, happens asynchronously and may be delayed by cache misses, coherence traffic, or memory-ordering rules.

    Without this split, the ROB would have to stall on a cache miss for every store. With it, once a store is known to be non-speculative, the front-end moves on while the store buffer absorbs whatever latency the memory hierarchy imposes. Commit and drain are therefore two distinct moments in the lifetime of a store:

    • Commit — the store is architecturally real. Younger speculation cannot squash it, but other cores do not see it yet.
    • Drain — the value is written to L1 and becomes visible to the coherence protocol, and therefore to other cores.

    Between these two moments the value lives only in the store buffer. Younger same-core loads can still observe it through Store-to-load forwarding, but other cores cannot — which is the structural reason Total store order permits store-load reordering across cores. A Memory fence like mfence forces the buffer to drain so that subsequent operations see a consistent global view.

    Remark. Retirement bandwidth (ROB commits per cycle) and store bandwidth (drain rate to L1) are separate throughput limits. A workload that retires stores faster than the cache can absorb them eventually backs up the ROB anyway, but the store buffer absorbs short bursts at no cost.

    Definition. Store-to-load forwarding [GXCN]

    • May 13, 2026
    • Kellen Kanarios

    The mechanism by which a younger load on the same core observes the value of an older store that is still sitting in the Store buffer and has not yet drained to L1. The load-store unit checks the store buffer for any in-flight store to the same address; if one exists, the load's data comes directly from the buffer entry rather than from the cache, satisfying the same-core sequential semantics that the program expects.

    Forwarding is the reason the commit/drain split does not break the single-core illusion. Within one thread, a store followed by a load to the same address must read the just-stored value — and would, on a machine without a store buffer, because the cache is the single source of truth. With a store buffer, the cache is briefly stale relative to architectural state; forwarding patches that hole.

    The structural consequence shows up in Total store order: store-load reordering across cores is permitted precisely because forwarding makes same-core reads see the new value, while cross-core reads continue to see the old one until drain. This asymmetry is the entire point of TSO's relaxation — the store buffer can hide latency from the issuing thread without forcing other threads to wait.

    Remark. Address matching in the store buffer is partial: most implementations check enough low-order bits to detect a hit but fall back to a stall (or a re-execution) on aliasing or partial-overlap cases. Reliable wide-then-narrow forwarding is one of the harder corners of memory disambiguation.

    Definition. Memory fence [R03A]

    • May 13, 2026
    • Kellen Kanarios

    An instruction that forces an ordering constraint between memory operations on either side of it. Fences exist because Relaxed consistency hardware reorders accesses aggressively for performance, and the programmer occasionally needs a point in the instruction stream past which prior accesses must be globally visible before subsequent ones may issue. A fence is the explicit knob for "wait until the world catches up here."

    The x86 family carries three:

    • sfence — store fence. All prior stores must drain from the Store buffer before any later store may issue. Used to publish a buffer whose writes must be visible to consumers in order.
    • lfence — load fence. All prior loads must complete before any later load issues. Less commonly needed under TSO, but useful as a speculation barrier (Spectre mitigation cases) and on architectures with load-load reordering.
    • mfence — full fence. Drains the entire store buffer and serializes all prior memory operations against all subsequent ones. The hammer.

    What a fence actually does in hardware is force the store buffer to drain (or wait for it to drain) rather than just waiting for Reorder buffer commit. This is the distinction worth keeping straight: ROB commit makes a store architecturally real but invisible to other cores; the fence is what makes it visible. The cost is whatever latency the cache hierarchy adds — fences are expensive precisely because they undo the latency-hiding the Store buffer exists to provide.

    Under Sequential consistency every memory operation behaves as if a full fence sat on either side of it, which is why SC is fast to reason about and slow to implement. Real-world consistency models pick relaxations and require fences only at the points the programmer marks as ordering-sensitive — acquire and release Half fences (see Release consistency) are the common compromise.

    Remark. A fence does not flush caches or invalidate anything — coherence traffic does that. A fence only constrains the issue order observed by the rest of the system.

    Data race [P0NQ]

    • May 12, 2026

    Two accesses to the same memory location, from different threads, where at least one is a write and there is no happens-before relationship between them. The happens-before relation is what synchronization primitives like Locks and atomics induce — when one access happens before another, they are not racing.

    Remark. Competing for the same lock is not a data race. Whichever thread acquires first happens before the next; the access inside each critical section is sequentially ordered.

    Data-race freedom (DRF) is the implicit contract behind every relaxed memory model: if your program has no data races, Relaxed consistency hardware behaves as if it were Sequential consistency. Conversely, in the presence of a data race, the result is undefined by the language and the model.

    Problem. Writing data-race-free programs is hard. Treating every shared access as a synchronization point destroys performance.

    SC-preserving compilers approach: forbid the compiler from reordering shared accesses, leaving the hardware free to do TSO-style reordering. The optimization most disrupted is common-subexpression elimination across shared accesses — solved by inserting a "monitor" check on the coherence bit between expressions.

    Hardware approach. Treat all pages as local/safe by default and extend page protection bits to the thread level: a non-owning thread accessing the page triggers a fault, exposing the data race at runtime.

    Definition. Instruction-level parallelism [JAZI]

    • May 13, 2026
    • Kellen Kanarios

    Parallelism extracted from within a single thread by executing independent instructions concurrently. The instruction stream is sequential as written, but a sufficiently aggressive back-end can issue, execute, and complete instructions in a different order than they appear, as long as data dependencies are respected and the visible architectural state still looks in-order.

    The machinery a CPU spends on ILP is the bulk of its area and complexity:

    • out-of-order execution — issue instructions whenever their operands are ready, not in program order.
    • register renaming — break false dependencies by mapping the same architectural register name to different physical registers.
    • Reservation stations + Reorder buffer — hold dispatched instructions while waiting on operands, then commit them in original order.
    • Speculative execution and speculation — keep the pipeline fed across control-flow boundaries by predicting which way a branch will go.

    ILP runs into hard limits: dependency chains, branch mispredictions, and the quadratic scaling of issue-queue wakeup-select all cap how wide a single-threaded window can grow. Beyond that ceiling, more silicon stops buying more performance, which is the structural reason the industry pivoted to Thread-level parallelism for throughput-bound workloads.

    Remark. ILP is what classic uniprocessor texts mean by "out-of-order superscalar execution"; see Notebook: Inside the Machine for the unwound history.

    Definition. Thread-level parallelism [G4ML]

    • May 13, 2026
    • Kellen Kanarios

    Parallelism extracted by running multiple independent threads concurrently and switching between them whenever any one of them stalls. Where Instruction-level parallelism mines a single instruction stream for hidden concurrency, TLP gives up on that ambition and instead assumes that enough independent work exists in the program (or workload) that some thread is always ready to make progress.

    TLP makes the per-thread microarchitecture cheap. There is no need for a deep issue window, aggressive speculation, or CAM-based wakeup-select: if the current thread can't proceed, the scheduler picks another. The cost shifts from per-instruction machinery to context-holding — registers, program counters, and stack pointers for every resident thread must live somewhere fast.

    Two dominant implementation styles:

    • Coarse-grained TLP. Conventional multicore CPUs and SMT: a handful of threads share a core, switched on long-latency events or interleaved cycle-by-cycle.
    • Massive TLP. GPUs keep dozens of warps resident per GPU architecture SM and switch on any stall. The GPU register scoreboard is the readiness check that makes this cheap.

    TLP fails when the workload genuinely has only one thread of useful work, or when the threads contend on shared state (locks, Cache coherence traffic). It wins overwhelmingly on data-parallel kernels where the threads are independent by construction.

    ILP vs TLP [RJEB]

    • May 13, 2026
    • Kellen Kanarios

    The two strategies a microarchitecture can use to keep arithmetic units busy in the face of memory latency and dependencies. Instruction-level parallelism mines parallelism from within a single instruction stream; Thread-level parallelism mines it across many. Every modern design is a position on this axis, and the position determines the rest of the microarchitecture.

    The choice is not really a choice — it is forced by the workload. ILP works only when the single thread has independent work to find; once dependency chains and branch mispredictions saturate the issue window, no amount of silicon spent on reorder buffers, wakeup logic, or speculation buys more throughput. TLP works only when many independent threads exist; throwing more warps at a single sequential dependency chain accomplishes nothing.

    The downstream consequences split everything else:

    • Control overhead. ILP machines spend most of their area on out-of-order execution machinery — Reorder buffer, Reservation stations, register renaming, CAM-based wakeup-select. TLP machines spend almost none; the schedule is "pick any ready thread."
    • Caches. ILP needs large caches because a single stalled thread blocks the whole pipeline. TLP can afford small caches because the latency is hidden by switching threads.
    • Register file. ILP keeps one (renamed) register set hot. TLP keeps every resident thread's register set hot simultaneously — GPU register files are an order of magnitude larger than CPU register files for this reason.
    • Readiness tracking. ILP pays O(N²) for CAM-based wakeup-select across N in-flight instructions; TLP pays O(1) per instruction via a per-register scoreboard.

    The slogan: CPUs spend a lot of silicon to find parallelism in one thread; GPUs spend almost none because they have many threads. Everything downstream — pipeline depth, cache hierarchy, scheduling policy, energy per operation — falls out of which side of this axis the chip sits on.

    Vector ISA / SIMD [Q7ZD]

    • May 12, 2026

    Single instruction, multiple data. Extend the Fetch-execute loop so that one fetch+decode applies the same operation to a vector of operands. The fetch-decode overhead amortizes over the vector width; loop bookkeeping (increment, compare, branch) is replaced by a single vector op.

    Concretely, a vector ISA adds:

    • A vector register file — wider than scalar registers; specify the data width per element and total bits per register.
    • Vector load/store instructions for moving contiguous chunks of memory to/from vector registers efficiently.
    • Vector predication (masking) — apply the op only where the mask is true, allowing data-dependent control flow without scalar branches.

    Trade-offs:

    • Need the array length to be divisible by the vector width for perfect speedup; tail handling is annoying.
    • Vector pipelines cannot do Forwarding cheaply — would need \(O(n^2)\) connections to every ALU.
    • Wider hardware vectors can be supported on narrower hardware by microcoded decomposition; backward compatibility falls out for free.

    In practice, vector code is generated by:

    1. Intrinsic functions (architecture-specific C builtins).
    2. Vector data types in a language extension.
    3. External libraries (BLAS, etc.).
    4. Compiler auto-vectorization.
    Problem. Aliasing: modifying a vector you're also reading from causes a race within the operation.
    Remark. SIMD is hardware-driven data parallelism. The GPU descendant is SIMT, which softens many of SIMD's rigidities by giving each lane its own program counter and active mask.

    GPU architecture [82U4]

    • May 13, 2026
    • with contributions from Kellen Kanarios

    A GPU sacrifices single-thread latency for aggregate throughput. The whole machine is built around the Thread-level parallelism design point — keep many threads resident, switch on every stall, spend almost no silicon on per-instruction control. The opposite end of the axis from a CPU:

    CPU GPU
    Caches Large Small (boost memory throughput)
    Control Sophisticated (branch prediction, OoO) Simple (no branch prediction, no forwarding)
    ALUs Powerful Energy-efficient, heavily pipelined
    Optimizes for Single-thread latency Aggregate throughput

    The GPU equivalent of a core is a streaming multiprocessor (SM). Each SM executes the same instruction across many threads in lockstep — the SIMT programming model — and uses a per-register GPU register scoreboard rather than CPU-style wakeup-select to decide which warp can issue next.

    Definition. SIMT [7F19]

    • May 13, 2026
    • Kellen Kanarios

    Single instruction, multiple threads. A GPU streaming multiprocessor issues one instruction per cycle to a group of threads — a warp — but each thread within the group keeps its own program counter and register state. The hardware enforces lockstep execution within the warp while preserving the illusion of independent threads to the programmer.

    SIMT is the GPU's answer to a question the Vector ISA / SIMD world struggled with: how do you get the throughput of wide vector execution without forcing the programmer to manually vectorize? Two mechanisms make this work.

    Warp divergence serializes both sides of a branch [OGCSM]

    • July 6, 2026
    • Kellen Kanarios

    When threads in a warp take different branches of a conditional, the warp does not split into smaller warps. Instead, the active mask selects which lanes execute each branch; lanes on the inactive side run dummy instructions until the warp reconverges at the post-conditional join point. The programmer writes ordinary scalar code with if/else, and the hardware silently masks the inactive lanes.

    This is the chief ergonomic win over SIMD: the compiler does not have to prove that all lanes will take the same path. The cost is that divergent code pays for both sides — a 32-wide warp where 16 threads take the if-branch and 16 take the else-branch executes both branches sequentially with half the lanes idle each time.

    The hardware sequences the two paths with a reconvergence stack — or, on recent GPUs, a barrier-based equivalent that avoids its deadlocks. The standard software mitigation is JKNL: group threads with the same control-flow trajectory into one warp so the branch never diverges in the first place.

    Grouping flexibility

    • May 13, 2026
    • Kellen Kanarios

    Threads are assigned to warps based on warp-ID, but the hardware and compiler are free to regroup. JKNL re-bundles threads with similar control-flow trajectories into the same warp so that future branches stay non-divergent. This is a class of optimization that pure SIMD cannot express — a SIMD ISA fixes the lane assignment at compile time.

    The hardware cost of SIMT relative to SIMD is the per-thread PC and register state. The benefit is that a single binary runs efficiently across a wide range of dynamic control flow, which is why the model has dominated GPU programming since CUDA. The scoreboard mechanism described in GPU register scoreboard operates per-warp rather than per-thread precisely because SIMT treats the warp as the unit of issue.

    Remark. SIMT is sometimes described as "SIMD with a per-lane PC," which captures the implementation but understates the programming-model impact: vectorization becomes a runtime, not a compile-time, decision.

    Warp scheduling hides memory latency by switching among resident warps [1PWL]

    • July 11, 2026
    • Kellen Kanarios

    A GPU keeps many warps resident on each Streaming Multiprocessor (SM) and switches between them on every stall. When a warp issues a memory op that misses cache it is dead for hundreds of cycles — but with dozens of resident warps the scheduler always has a runnable one, so aggregate throughput stays high even though single-warp latency is awful. This is the Thread-level parallelism design point cashed out on the GPU: spend silicon on resident warp state rather than on latency-reduction machinery. The switch is free because each SM's register file is large enough to hold every resident warp's registers at once, so a context switch is just changing which warp the scheduler picks — not saving state to memory the way a CPU context switch must.

    Two things bound how well this works. Occupancy — how many warps are resident — sets the ceiling on latency hiding: with too few warps the scheduler runs out of runnable candidates. And the scheduler itself must stay cheap enough to choose a warp every cycle, which it does with a two-loop scoreboard check, marshalling each issued instruction's register reads through an operand collector.

    Definition. GPU register scoreboard [XPAA]

    • May 13, 2026
    • Kellen Kanarios

    The readiness-tracking structure that lets a GPU issue instructions out of program order without any of the CAM machinery a CPU needs. A scoreboard is a small table — typically one busy bit per architectural register — that records whether some in-flight instruction is still computing a value that something else might want to read.

    The protocol is trivial. When an instruction issues, it sets the busy bit on its destination register. When it writes back, it clears the bit. Before any instruction can read its sources, the hardware checks the corresponding bits: if any source is busy, the instruction (or its entire warp) waits. The check is a single table lookup per source, so the per-instruction cost is \(O(1)\) regardless of how many other instructions are in flight.

    This is enough because GPUs lean on Thread-level parallelism rather than Instruction-level parallelism. When a warp's next instruction reads a busy register, the GPU architecture scheduler does not stall — it picks a different ready warp from the dozens of resident warps and runs that one instead. The scoreboard's job is only to answer "is this warp ready? yes/no," which is far cheaper than the broadcast/compare a CPU Issue queue wakeup-select does to find dependency satisfaction in a single thread's instruction window.

    The size difference falls out of this. CPU issue queues hold tens of entries; GPU instruction windows can hold many warps' worth of in-flight work, because every additional warp costs a row in the scoreboard and a register-file allocation, not a quadratic CAM expansion. See ILP vs TLP for why the design fork sits where it does.

    Remark. The scoreboard's simplicity is what lets GPUs spend their transistor budget on ALUs and the Register file instead of control logic. It is one of the most visible places where the Thread-level parallelism design choice cashes out as silicon savings.

    Warp [X9VZ]

    • May 12, 2026

    A warp is the unit of scheduling on an NVIDIA GPU: a group of ~32 threads that issue one instruction per cycle in lockstep. Each functional unit in the SM has as many lanes as a warp has threads, so a single fetched instruction fans out across all 32 lanes at once. The warp — not the thread — is what the hardware fetches, schedules, and tracks; the per-thread SIMT programming model is an illusion built on top of it.

    Because the warp is the atom of execution, the two facts that dominate GPU performance are both properties of the warp: it hides memory latency by switching among resident warps instead of stalling, and it serializes both sides of a branch when its threads diverge.

    Shared memory (GPU) [8GZN]

    • May 12, 2026
    • with contributions from Kellen Kanarios

    Shared memory on a GPU is a small (per-block) on-chip scratchpad with latency comparable to L1. Software-managed — the programmer explicitly stages data through it, the hardware does not cache for them. Used to:

    • Avoid non-coalesced global accesses by reorganizing data layouts on the fly.
    • Reuse loaded data across threads within a block.
    • Communicate between threads within a block.

    Bank conflicts serialize shared-memory accesses to the same bank [OGCRU]

    • May 25, 2026
    • Kellen Kanarios

    Shared memory is divided into banks (typically 32). Each bank can service one access per cycle. If two threads in a warp access different addresses in the same bank, the accesses serialize — a bank conflict.

    Three behaviors when warp threads access shared memory:

    • Different banks: full parallelism. No conflict.
    • Same bank, same word: hardware broadcasts the word to all threads. No conflict.
    • Same bank, different words: serialize.
    Question. How to avoid bank conflicts?
    Answer. Change the access pattern so threads hit different banks (often by transposing or by adding padding to break the stride that causes the conflict).
    Remark. The GPU memory hierarchy is software-managed at L1 (shared memory) and coherent only at L2 — by restricting L1 to local-to-block scope, the architecture sidesteps the cost of a global Cache coherence protocol on a thousand-way-parallel chip.

    Memory coalescing merges a warp's accesses into one transaction per cache line [U1FO]

    • May 12, 2026

    The hardware optimization on GPUs that combines simultaneous memory accesses by threads in a warp into a single transaction — when the accessed addresses fall in the same cache line.

    Global memory latency on a GPU is 400–800 cycles. Coalescing is the primary way to keep the SM fed:

    • If all 32 threads in a warp access addresses within one cache line: one memory transaction, full bandwidth.
    • If addresses span multiple cache lines: the hardware serializes into multiple transactions; bandwidth drops proportionally.
    Remark. Address permutations within a cache line don't hurt utilization — coalescing is about which lines are touched, not which exact bytes.

    Coalescing is why memory layout matters so much for GPU code. The standard idiom is structure-of-arrays (SoA) instead of array-of-structures (AoS): threads accessing the same field of consecutive records hit consecutive bytes, which coalesce; threads accessing the same struct don't.

    Question. Should you always cache on the GPU?
    Answer. No. If you're memory-bound and have no Spatial locality, cache is wasted on data you won't reuse. CUDA exposes the option to disable L1 caching for these cases, which lets misaligned loads transfer only the bytes needed instead of full cache lines.
    Remark. Use GPUs for parallel parts where throughput dominates. Use CPUs for sequential parts where latency dominates — the ILP vs TLP axis is the right frame for deciding which.

    On-chip interconnection networks [667T]

    • April 8, 2026

    The network connecting cores, L2/L3 banks, and memory controllers on a Multicore chip. The same abstractions that govern off-chip networks (topology, routing, flow control) apply, but the constraints are different: wires are abundant but die area is scarce, latency is dominated by hops rather than propagation, and the network is part of the Cache coherence critical path.

    On-chip network metrics [OMZU]

    • May 12, 2026

    The measurements used to compare network topologies and traffic patterns.

    Traffic-independent

    • May 12, 2026
    • Degree: number of links at each node. A proxy for router complexity — each port costs a buffer, crossbar input, state machine.
    • Bisection bandwidth: bandwidth across a cut that splits the network into two equal-sized halves. The minimum over all such cuts. A proxy for peak achievable throughput, especially for uniformly random traffic.
    • Diameter: maximum hop distance between any two nodes. The worst-case hop count.
    Remark. Bisection bandwidth is less useful for on-chip networks than off-chip because global on-chip wiring is plentiful.
    Remark. Ring bisection is 2 (two cuts to bisect); torus bisection is 4.

    Traffic-dependent

    • May 12, 2026
    • Hop count: hops between two specific nodes. The average over a traffic distribution is a useful proxy for latency.
    • Maximum channel load: the most-loaded link's load relative to per-node injection bandwidth, for a given traffic pattern. The proxy for sustained throughput.
    • Path diversity: number of distinct shortest paths between two nodes. Higher path diversity allows adaptive routing to spread load.

    Performance regimes

    • May 12, 2026
    • Zero-load latency: average hop count with no contention.
    • Saturation throughput: the injection rate at which latency curves upward exponentially.
    Remark. Hop count is not the same as latency. Per-hop pipeline depth and link propagation matter — more short hops can beat fewer slow ones. Router pipeline depth isn't always known in advance.

    Direct vs indirect networks [95AC]

    • May 12, 2026

    Two organizational families for on-chip networks.

    • Direct — every terminal node (core, memory controller) is associated with a router. Most on-chip designs are direct because area is precious and there's no point spending a router with no compute attached. The "\(k\)-ary \(n\)-cube" family (Ring topology, Mesh topology, Torus topology) is direct.
    • Indirect — routers may exist that connect only to other routers, with no terminal attached. Butterfly network and Clos network are indirect; a router in their middle stages just shuffles bits between input and output stages.
    Remark. Indirect networks can shorten the diameter for the same node count, but they pay for extra silicon area for routers that don't serve traffic locally. Often the right call off-chip; usually the wrong call on-chip.

    Topologies

    • April 8, 2026

    Crossbar [YV5J]

    • May 12, 2026

    The simplest indirect network: an \(n \times m\) grid of switch points connecting \(n\) inputs to \(m\) outputs. Any input can be connected to any free output simultaneously — non-blocking.

    • One hop per transit (minimum possible).
    • Serializes if multiple senders target the same receiver.
    • Switch area grows as \(O(n \cdot m)\) — the dominant cost as systems grow.
    Remark. Crossbar router area scales as the square of the port count, which is why high-radix routers are dramatically more expensive than low-radix ones.

    Crossbars are common as the inner switch of larger structured networks (Clos network, Fat tree) — large flat crossbars are impractical at scale, but small crossbars are cheap.

    Ring topology [UEA0]

    • May 12, 2026

    A direct network where each node connects to two neighbors, forming a closed loop. Bidirectional rings allow either direction of travel; unidirectional rings allow only one. The simplest topology to implement and reason about.

    In the \(k\)-ary \(n\)-cube family, a ring is a \(k\)-ary 1-cube.

    • Degree 2 (every node).
    • Diameter \(\lfloor k/2 \rfloor \) for a bidirectional \(k\)-node ring.
    • Bisection bandwidth 2 — cut the ring at exactly two links.
    • Hardware overhead very low: routers have two ports.
    Problem. Worst-case hop count is \(O(k)\). Doesn't scale beyond a small number of nodes.
    Remark. Despite the poor scaling, rings are common in commercial CPUs (Intel's L3 ring bus on Sandy/Ivy Bridge) because they're cheap and 8–16 cores is enough.

    Mesh topology [9SA2]

    • May 12, 2026

    A 2D grid of nodes with neighbors connected on the four compass directions, no wrap-around. The dominant on-chip topology in commercial designs (Xeon Phi, recent server x86 chips) because it lays out naturally on a 2D die.

    • Interior nodes degree 4; edge nodes degree 3; corners degree 2.
    • Average hop count for an even-\(k\) mesh: \[H_{avg} = \frac {nk}{3}\]
    • Maximum channel load for uniform random traffic: \(k/4\) — twice that of a Torus topology.
    Problem. Non-uniform node degree. Edge nodes carry less traffic than they could, while interior nodes bottleneck on traffic flowing through them.
    Remark. Mesh dominates Torus topology in practice despite the topology metrics being worse, because torus wrap-around wires are physically long and force expensive folding to equalize link lengths. Manufacturing wins over theoretical performance.

    Torus topology [DDKT]

    • May 12, 2026

    A Mesh topology with wrap-around links connecting the ends of each dimension. In the \(k\)-ary \(n\)-cube classification, a \(k\)-ary \(n\)-cube. Uniform node degree (every node has degree \(2n\)), uniform load distribution, half the hop count of a mesh.

    • Every node has degree \(2n\).
    • Average hop count for even \(k\): \[H_{avg} = \frac {nk}{4}\]
    • Maximum channel load for uniform random traffic: \(k/8\) — half that of a Mesh topology.
    • Good path diversity: many shortest paths between any two nodes.
    Problem. Wrap-around links are physically long. To equalize wire delays, the torus must be folded, doubling effective wire lengths and link area. The wrap-around also doubles the link count versus a mesh of the same dimensions.
    Remark. Despite the theoretical advantage, mesh wins on real chips. Tori show up in HPC interconnects (Cray) where the wires can run on dedicated cables.

    Variants:

    • \(k\)-ary \(n\)-cube: \(k\) nodes along each of \(n\) dimensions, all ends wrapped.
    • Asymmetric: \((i, j, k)\)-ary — different node count per dimension.
    • Mesh: same as torus but without wrap-around.
    • Ring: \(k\)-ary 1-cube.

    Butterfly network [5WX9]

    • May 12, 2026

    A multistage indirect network. A \(k\)-ary \(n\)-fly has \(k^n\) terminal nodes connected through \(n\) stages of \(k \times k\) crossbar switches. At each stage, the routing decision selects which of \(k\) sub-groups to enter; after \(n\) stages, every terminal can reach every other.

    • Hop count \(n - 1\).
    • Under uniform random traffic, maximum channel load is 1 (perfectly balanced).
    • Channel load drops further for non-uniform traffic that respects the natural partitioning.
    Problem. Zero path diversity. There is exactly one path between any source and destination. A single hot link wedges traffic.
    Solution. Add shuffle stages that permute paths after each routing stage, so the same source-destination pair can be reached by multiple routes.

    Clos network [YKAN]

    • May 12, 2026

    A three-stage indirect network parameterized by \((m, n, r)\): \(r\) input switches and \(r\) output switches each of size \(n \times m\), with \(m\) middle-stage switches each \(r \times r\). Total terminal nodes \(N = nr\). Non-blocking when \(m \geq 2n - 1\).

    • Number of total switching elements scales as \(N^{1.5}\) versus \(N^2\) for a flat Crossbar while preserving non-blocking.
    • Path diversity is \(m\) — multiple distinct paths between any source and destination.
    Proof. There are \(rmn\) input-switch elements, \(mr^2\) middle-switch elements, and \(rmn\) output-switch elements: \[S = 2rmn + mr^2.\] With \(r = N/n\): \[S = 2Nm + m(N/n)^2.\] Choosing \(n = \sqrt {N}\) and \(m \geq 2n - 1 = O(\sqrt {N})\): \[S = 2mN + 2m\sqrt {N} = O(N^{1.5}).\]
    Remark. A folded Clos network — input and output stages combined, making \(n\) input ports and \(m\) output ports bidirectional — is structurally the same as a Fat tree.

    Fat tree [DFC7]

    • May 12, 2026

    A tree topology with bandwidth that doubles at every level toward the root. Each interior switch has logical degree 4, but as you climb the tree, link width doubles to absorb the combined traffic of its subtree — preserving bisection bandwidth.

    Structurally, a fat tree is a folded Clos network: the input and output stages of a Clos are merged into a single bidirectional stage at each level of the tree, and the middle stage becomes the higher levels of the tree.

    • Each switch has logical degree 4 (2 down, 2 up — with link doubling).
    • Bisection bandwidth scales linearly with the number of leaves.
    • Path diversity grows toward the root.
    Remark. Fat trees are the dominant topology for off-chip HPC interconnects (InfiniBand fat-tree clusters). Less common on-chip because the doubling links are hard to lay out on a 2D die.

    Routing

    • April 8, 2026

    Network routing strategies [VTS4]

    • May 12, 2026

    Given a topology and a packet's source and destination, which path through the network does it take? Three classes of answer:

    • Greedy (shortest-path) — always pick the shortest available route.
      • (+) Minimizes latency under light load.
      • (-) Overloads central links; worst-case throughput poor.
    • Uniform / oblivious random — pick a direction at random.
      • (+) Spreads load; avoids systematic hotspots.
      • (-) Wastes bandwidth on non-minimal paths; latency unpredictable.
    • Adaptive — at each hop, pick the outgoing link with the lowest local congestion.
      • (+) Dynamically balances load; best worst-case throughput.
      • (-) Routers need congestion state; risk of livelock without care.
    Example. Tornado traffic on an 8-node bidirectional ring: node \(i\) sends to \((i+3) \bmod 8\).
    • Greedy: all packets travel counter-clockwise. Bidirectional bandwidth wasted; \(\frac {1}{3}\) utilization.
    • Uniform: randomly clockwise/counter-clockwise. Counter-clockwise gets \(\frac {2}{3}\) utilization, clockwise \(\frac {2}{5}\); worst-case \(\frac {2}{5}\).
    • Adaptive: sends \(\frac {5}{8}\) counter-clockwise to match traffic exactly. Best.
    Remark. Adaptive routing tends to need Network backpressure signaling to know where congestion is, and care to avoid livelock when many adaptive decisions chase each other.
    Remark. Oblivious strategies like Dimension-order routing and Valiant's routing algorithm give deterministic guarantees (deadlock-free, bounded path length) at the cost of giving up the load-balancing of true adaptive routing.

    Dimension-order routing [83P2]

    • May 12, 2026

    An oblivious Network routing strategies for \(k\)-ary \(n\)-cube networks (Mesh topology, Torus topology): always exhaust the offset in one dimension completely before changing to the next. E.g. in 2D mesh routing, route X-then-Y — never turn back to X after starting Y.

    The dimension order prevents routing deadlock: cycles in the channel dependency graph are impossible because no packet ever requests a channel in a "lower" dimension once it has started using a higher one. The dependency graph is therefore acyclic, and the standard cycle-equals-deadlock argument applies.

    • (+) Provably deadlock-free without virtual channels.
    • (-) Zero path diversity. Hot spots can't be routed around.
    • (-) Unaware of current network state — oblivious.

    The standard remedy for the path-diversity problem is Valiant's routing algorithm, which keeps the deadlock-freedom of dimension-order routing while spreading load with a randomized intermediate hop.

    Valiant's routing algorithm [XLDA]

    • May 12, 2026

    An oblivious Network routing strategies that achieves provably optimal worst-case throughput by introducing randomness in path selection:

    1. For a packet going from \(s\) to \(d\), randomly pick an intermediate node \(w\).
    2. Route \(s \to w\) using Dimension-order routing.
    3. Eject the packet at \(w\), then re-inject it.
    4. Route \(w \to d\) using Dimension-order routing.

    To preserve deadlock-freedom, the intermediate node must remove the packet from the network before re-injecting — otherwise the channel dependencies of the two halves could interlock.

    Remark. Restricting \(w\) to lie in the sub-quadrant defined by \(s\) and \(d\) keeps paths near-minimal while still spreading load — the practical compromise between guaranteed throughput and added latency.
    Remark. Valiant's algorithm trades average-case latency (paths are longer) for worst-case throughput (any traffic pattern looks roughly uniform once randomized). The right call for adversarial workloads; overkill for uniform traffic.

    Routing deadlock [DWV1]

    • May 12, 2026

    Routing deadlock occurs when a packet cannot make progress because the buffer at its next link is full, and the packet holding that buffer is itself blocked waiting on a buffer this packet holds — a cycle in the channel dependency graph.

    Two structural responses:

    1. Avoid deadlock by construction. Dimension-order routing enforces an ordering on dimension transitions so the dependency graph is acyclic by design.
    2. Detect and recover from deadlock. Allow all turns. Detect cycles via timeouts. Drain stuck packets onto a backup network (a "slow channel" — physical, or more commonly an escape virtual channel) that uses dimension-order routing.

    The relationship to other related concepts:

    • Backpressure (Network backpressure) is the signaling mechanism that lets a router downstream tell upstream it is congested, before the upstream actually fills the link.
    • Adaptive routing avoids deadlock locally; backpressure lets adaptive routing react to remote congestion.

    Network backpressure [EZDD]

    • May 12, 2026

    A signaling mechanism that lets a downstream router tell upstream "I am congested — slow down" before its buffers actually overflow. With backpressure information, adaptive routing can route around remote congestion, not just local.

    The mechanism propagates: a congested link signals back to its sender, which then knows its next link will fill, which signals further back. Useful information flows in the opposite direction of the data.

    Problem. Acting only on local state can cause livelock — packets bounce between hotspots, never converging.

    Backpressure breaks the livelock by giving local routers global-ish information without an actual global view.

    Remark. Backpressure is mechanism; adaptive routing is policy. The two together are what real high-radix on-chip networks use.

    Interface with the cache hierarchy

    • April 8, 2026

    The on-chip network connects each core (with its L1) and the L2 cache banks (which may be shared or private). Each L2 bank has a cache controller (manages Cache coherence) and, at memory boundaries, a memory controller (talks to DRAM).

    Two cache-organization choices interact with the network design:

    • Private L2 caches: each tile owns its L2 slice; cross-tile traffic happens only on private-L2 miss. (+) Lower hit latency; less interconnect traffic. (-) Replicated data; more off-chip traffic on misses.
    • Shared L2 cache: a single logical L2 split into banks, with each address mapped to a home bank. (+) Better utilization, no replication. (-) Every L1 miss goes on the network.

    The interface to the network sits at the cache controller:

    • Miss status handling register (MSHR): on a cache miss, records what request the cache is waiting on. Matched against the reply when it arrives.
    • Transaction status handling register (TSHR): tracks outstanding memory transactions to off-chip memory from the directory cache.
    Remark. Memory controllers are commonly placed at the chip perimeter, near the I/O pads. A memory controller can share a router with a core/cache tile (saves area; needs arbitration) or have its own (more area; no contention).
    Remark. Hardware-managed shared memory is one design point; another is explicit message passing, where software (not hardware) is responsible for moving data between processes.

    Implementation considerations

    • April 8, 2026

    Links. Thicker wires with larger spacing → lower resistance, higher speed, better energy efficiency. The wire budget determines how many ports you can afford per router.

    Routers. Each port costs a buffer, a crossbar input, state logic, and a link to its neighbor. Crossbar area scales as the square of port count, so high-radix routers are dramatically expensive. Rings have very simple routers but poor performance.

    Floorplan. Tori need to be folded to equalize wire lengths, doubling effective wire length; meshes don't. Tori also have twice as many links as meshes, which may force narrower links and lower per-link bandwidth.

    Folding can also equalize path lengths inside a network at the cost of increasing average latency.

    Concentration allocates more cores per router. (-) Decreases throughput per node. (+) Programs with high locality do little remote communication, so concentration is a good fit there.

    Custom and hierarchical topologies

    • April 8, 2026

    General topologies aren't always optimal. Two construction strategies:

    • Splitting: start with a large crossbar; split into smaller switches until constraints are met.
    • Merging: start with many small switches and combine.

    Hierarchical topologies use one topology within clusters and another across clusters — common when traffic has locality.

    Concurrency [5P43]

    • August 16, 2025

    Threads, the hazards that come with sharing state between them, and the primitives that make sharing safe.

    Threads and shared state

    • August 16, 2025

    Concurrency Intro [16TG]

    • August 16, 2025

    Concurrency comes from sharing: multiple Threads within a single process share the address space, which is what makes them cheap to spawn and to communicate between, and is also what makes them dangerous. When threads interleave on a shared variable, you get a Race condition. The fragile region is the Critical section, protected by synchronization primitives like Locks and Barriers.

    Thread [63U3]

    • May 12, 2026

    An independent unit of execution within a Process. Each thread has its own program counter, register set, and stack — but shares the surrounding address space, file descriptors, and other process-level resources with its siblings.

    To support a context switch between threads, the OS keeps a thread control block (TCB) per thread — the per-thread analog of Process control block. Each thread also gets its own stack within the process's address space.

    Why threads instead of processes:

    1. Intra-process parallelism: parallelizable work within one program, without paying the context switch cost of separate processes.
    2. Avoiding blocking I/O: one thread blocks waiting on I/O while others keep running.
    Remark. Sharing an address space is exactly what makes threads cheap to communicate between and exactly what makes them dangerous — see Race condition and Critical section.

    Race condition [VOQN]

    • May 12, 2026

    When the correctness of a program depends on the order in which concurrent Threads schedule, you have a race condition. The classic example: two threads each execute

    mov addr reg     // load
add 1 reg        // increment
mov reg addr     // store

    intending to increment a shared variable twice. If the OS interleaves the threads at the wrong instruction boundary:

    1. Thread 1 stores its incremented value.
    2. Thread 2 stores its incremented value.
    3. Thread 1 increments (in its register).
    4. Thread 2 increments (in its register).

    The result is a single increment, not two — both threads loaded the same starting value and the second store clobbered the first.

    The fix is to prevent interleaving inside the code region that touches shared state — the Critical section — using a synchronization primitive like Locks.

    Critical section [KG2K]

    • May 12, 2026

    A region of code that accesses shared state in a way that is incorrect if more than one Thread executes it concurrently. Critical sections are exactly the regions that need protection by a synchronization primitive: a Locks excludes other threads while one is inside; a Barriers forces all threads to reach the same point before any may continue.

    The art of writing correct concurrent code is to identify critical sections and protect them — without making them so large that you lose all parallelism.

    Synchronization primitives

    • August 16, 2025

    Locks [Q2DT]

    • March 9, 2026

    A lock is a synchronization primitive that grants mutually exclusive access to a critical section. The implementations below trade off coherence traffic, fairness, and memory in different ways. The naive locks (Test-and-set lock and Test-and-test-and-set lock) are simple but generate \(O(n)\) coherence traffic per release; queue-based locks (Ticket lock, Array-based lock, MCS lock) scale better at the cost of extra state per waiter.

    Test-and-set lock [MJ5L]

    • May 12, 2026

    The simplest spin lock. The acquire loop spins on an atomic test_and_set: it writes LOCKED and reads the previous value in one step. If the previous value was UNLOCKED, the caller has the lock; otherwise it tries again.

    acquire(lock_ptr):
  while (true):
    // Perform "test-and-set"
    // UNLOCKED = 0, LOCKED = 1
    test_and_set(old, lock_ptr)
    if (old == UNLOCKED):
      break // lock acquired!
    // keep spinning, back to top of while loop

release(lock_ptr):
  store[lock_ptr] <- UNLOCKED
    Problem. Major coherence traffic. Every spinning core must acquire write permissions and invalidate everyone else on each test_and_set, so contention scales badly with the number of waiters.

    Test-and-test-and-set lock [PXY9]

    • May 12, 2026

    A refinement of the Test-and-set lock that reduces coherence traffic during the spin. Threads first perform a plain load to check whether the lock looks free, and only attempt the atomic test_and_set when it does. Spinning is then read-only, so cores can share the cache line.

    acquire(lock_ptr):
  while (true):
    // Perform "test"
    load [lock_ptr] -> original_value
    if (original_value == UNLOCKED):
      // Perform "test-and-set"
      test_and_set(old, lock_ptr)
      if (old == UNLOCKED):
        break // lock acquired!
    // keep spinning, back to top of while loop

release(lock_ptr):
  store[lock_ptr] <- UNLOCKED
    Remark. Spinning is a read-only operation now.
    Problem. When the lock is released, the line must be invalidated on every waiter, and they all simultaneously read and try to write. The contention is shifted from steady state to release time but is still \(O(n)\) in the number of waiters.

    Ticket lock [V7KP]

    • May 12, 2026

    A fair spin lock based on two counters. Each arriving thread takes a "ticket" by atomically incrementing next_ticket and then spins until now_serving equals its ticket. Release simply increments now_serving. Threads acquire the lock in FIFO order, which prevents starvation but does not by itself reduce contention.

    acquire(lock_ptr):
  my_ticket = fetch_and_increment(lock_ptr->next_ticket)
  while (lock_ptr->now_serving != my_ticket); // spin

release(lock_ptr):
  lock_ptr->now_serving = lock_ptr->now_serving + 1
    Problem. Still \(O(n)\) coherence traffic on release: incrementing now_serving invalidates the line on every waiter, and they all reload it to recheck their ticket.

    Array-based lock [BJ7N]

    • May 12, 2026

    Each waiter is assigned its own slot in an array and spins only on that slot. Release just flips the next slot to "go-ahead", so the cache line touched by release is the one read by exactly one other thread. Coherence traffic per release is constant rather than \(O(n)\), at the cost of one slot per waiter.

    • Give each waiter its own "slot" to wait on.
    • Only need to set the next slot to "go-ahead" on release.
    • Constant coherence traffic.
    Problem. \(O(n)\) memory for the lock array, and padding each slot to \(64\)b (one cache line) to avoid false sharing makes that cost real.

    MCS lock [IN64]

    • May 12, 2026

    A scalable, list-based queue lock. The lock variable is a pointer to the tail of a linked list of per-thread nodes. Each thread enqueues its own node and spins on a flag in its own cache line. The releaser wakes only its direct successor, so coherence traffic per handoff is constant and memory cost is proportional only to threads currently waiting, not to the lock's fanout.

    acquire(lock):
  I->next = null
  predecessor = fetch_and_store(lock, I)
  if predecessor != nil:                   // some node holds the lock
    I->must_wait = true
    predecessor->next = I                  // predecessor must wake us
    repeat while I->must_wait              // spin till lock is free

release(lock):
  if (I->next == null):                    // no known successor
    if compare_and_swap(lock, I, nil):     // make sure...
      return                               // CAS succeeded; lock freed
    repeat while I->next == nil            // spin to learn successor
  I->next->must_wait = false               // wake successor

    Barriers [9QUX]

    • May 12, 2026

    A barrier holds threads at a synchronization point until all \(P\) participants have arrived, then releases them simultaneously. The naive implementation uses a shared counter that each arriving thread decrements; the last arrival resets the counter and the rest spin until it returns to \(P\).

    Naive central barrier

    • May 12, 2026
    global (shared) count : integer := P

procedure central_barrier
  if fetch_and_decrement(&count) == 1
    count := P
  else
    repeat until count == P
    Problem. When count == P, a fast thread can reach the next barrier and decrement count again before the slow threads observe the reset. The slow threads then spin forever waiting for a value that has already passed.

    Sense-reversing barrier

    • May 12, 2026

    The fix is to give each thread a private local_sense that flips on each barrier. The global sense flag is flipped by the last arrival, and waiters spin on sense == local_sense, so a fresh barrier cannot be confused with a stale one.

    global (shared) count : integer := P
global (shared) sense : bool := true
global (private) local_sense : bool := false

procedure central_barrier
  local_sense = !local_sense
  if fetch_and_decrement(&count) == 1
    count := P
    sense = local_sense
  else
    repeat until sense == local_sense

      Data race [P0NQ]

      • May 12, 2026

      Two accesses to the same memory location, from different threads, where at least one is a write and there is no happens-before relationship between them. The happens-before relation is what synchronization primitives like Locks and atomics induce — when one access happens before another, they are not racing.

      Remark. Competing for the same lock is not a data race. Whichever thread acquires first happens before the next; the access inside each critical section is sequentially ordered.

      Data-race freedom (DRF) is the implicit contract behind every relaxed memory model: if your program has no data races, Relaxed consistency hardware behaves as if it were Sequential consistency. Conversely, in the presence of a data race, the result is undefined by the language and the model.

      Problem. Writing data-race-free programs is hard. Treating every shared access as a synchronization point destroys performance.

      SC-preserving compilers approach: forbid the compiler from reordering shared accesses, leaving the hardware free to do TSO-style reordering. The optimization most disrupted is common-subexpression elimination across shared accesses — solved by inserting a "monitor" check on the coherence bit between expressions.

      Hardware approach. Treat all pages as local/safe by default and extend page protection bits to the thread level: a non-owning thread accessing the page triggers a fault, exposing the data race at runtime.

      Transactional memory [Y8JE]

      • May 12, 2026

      An alternative to fine-grained Locks: mark code regions as transactions that should appear atomic, execute them speculatively in parallel, and detect conflicts after the fact. Aims at the performance of fine-grained locking with the programming ergonomics of coarse-grained locking.

      Conflict detection

      • May 12, 2026

      Two transactions conflict if they access the same memory location and at least one of them is a write. The implementation tracks per-transaction read and write sets.

      • Lazy detection. Build the entire read/write set as the transaction runs; check for conflicts only at commit time. If conflict, roll back.
      • Eager detection. Check on every access against other transactions' sets; abort immediately on conflict.

      Eager detection can piggyback on the cache coherence protocol: add read/write bits to each cache line, set them during a transaction, and let a coherence invalidation signal a conflict.

      Version management (rollback)

      • May 12, 2026

      Two operations: commit and abort. Implementations:

      1. Lazy. stores go to a "write table" rather than memory; loads check the table first, falling back to memory. Commit writes the table back (slow); abort just clears the table (fast).
      2. Eager (coherence-backed). stores write through but track read/write sets per cache line. Commit clears the sets (fast); abort clears the sets (also fast — work already on overwrite if not aborted before commit).
      Problem. If a transaction touches more cache lines than the cache holds, the tracking overflows.
      Solution. Fall back to software transactional memory (STM) on overflow — a hybrid scheme that keeps hardware fast-path for common cases.

      Multicore [JRP0]

      • May 12, 2026

      Multiple independent CPU cores on the same chip. A core, for this purpose, is the bundle of front-end, back-end, Execution unitss, and L1 Cache hierarchy that runs one architectural thread of execution.

      Traditional multiprocessor systems put each core on its own physical chip; chip multiprocessing (CMP) puts multiple cores on the same die. The key advantage is shared lower-level caches — typically an L2 or L3 shared across all cores on the chip — which speeds up inter-core communication and amortizes silicon.

      Remark. The shift to multicore was forced by the Power wall more than chosen for its own merits. The software cost is real: shared-memory parallelism requires correctness work that single-threaded code didn't.

      Multicore is what makes Concurrency a hardware problem, not just a scheduling problem: real simultaneous execution on different physical cores means software has to handle Race conditions with real synchronization primitives like Locks.
