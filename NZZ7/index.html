Definition 2. Vector Processing Unit [OGCRN]

  • May 22, 2026
  • Kellen Kanarios

A core component of the TPU architecture responsible for elementwise operations.

  • 2D SIMD execution, where each of 1024 sites execute same program in lockstep.
  • Each site contains 4 ALUs, enabling 4096 FLOPs.
  • Fed a 8x128 Vector Register in one cycle.

Definition 3. Matrix Multiply Unit (MXU) [OGCRP]

  • May 22, 2026
  • Kellen Kanarios

A core component of the TPU architecture responsible for performing matrix multiplication via a systolic array.

  • 256x256 totaling 65,536 multiply accumulate units i.e. each unit can multiply and add in one cycle.
Remark. Can only do 256x256 matrix multiplication, so the order is
  1. Tiling the matrix to load from HBM to VMEM
  2. Compiler tiles from VMEM to 256x256 tiles to load into MXU.

    ASIC accelerators [OGCRS]

    • May 24, 2026
    • Kellen Kanarios

    Application-Specific Integrated Circuits fuse multiple operations into hardware that a general-purpose pipeline would have to schedule sequentially. The motivating intuition: if you frequently multiply-accumulate, the multiply and the add each traverse the entire pipeline separately. A dedicated multiply-accumulate ALU does both in one cycle.

    The cost-of-operation gap is large — 32-bit vs 8-bit addition has roughly an order of magnitude difference in energy per op. ASICs win by matching width to need.

    Processing in Memory for ML [2BGU]

    • April 6, 2026
    • Kellen Kanarios

    Notes on guest lecture by Yufeng Gu titled Architecture and System Co-Design for Scalable Large Language Model Inference

    TFLOPs increasing much faster than DRAM bandwidth -> Hardware scaling is hitting the memory wall.

    Data center power consumption also increasing \(\approx \) 10% a year.

    GPUs are under low utilization for LLM inference

    • Prefix stage: parallel
    • Decoding stage: sequential -> bottleneck!

    As batch size increases, the KV-cache dominates memory usage.

    • KV cache size grows with context length.
    • Tradeoff between context length and batch size.
    Remark. Unlike fully-connected cannot batch self-attention because it depends on the individual prompts.

    Recall. operational intensity is \[\frac {\text {compute throughput}}{\text {memory bandwidth}}\]

    Inference much lower operational intensity than GEMM!

    • Inference only doing matrix \(\times \) row
    • If embedding dim is \(D\), then this is only \(D\) ops / byte.

    Processing in memory (PIM) enables faster memory bandwidth.

    Problem. Has lower memory capacity (fewer GB) due to added processing units (PU).
    Solution. Use Compute eXpress Link (CXL) to do memory expansion to get high memory bandwidth + expend memory capacity.
    Remark. Put high use operations i.e. vector multiplication, element-wise multiplication in PIM module (99% usage). Put more complicated operations on PNM accelerator. TODO: ISAs for each specified on slides (not available yet).
    Remark. CENT supports inter and intra-device communication via connecting communication controller to shared buffer and CXL port.

    Communcation primitives. multicast and gather

    • Multicast: send to all CXL devices.
    • Gather: receive data sequentially from all CXL devices.
    • Multicsat-gather -> tensor parallel.

    Peer-to-peer: can also support pipeline parallelism.

    Remark. For fast communication, use tensor parallelism i.e. if connected via NVLink. If slower communication use pipeline parallelism. Can mix.
    Remark. Longer context -> better performance for PIM (TODO: why?). HBM + PIM does not scale. Cannot arbitrarily add PIM.

      Operator fusion is like ASIC caching [PMIC]

      • July 13, 2026

      On a traditional CPU, the hardware manages caching to ensure that data that is reused does not roundtrip through DRAM. On an ASIC accelerators, we forego the complexity of caching to allow for more execution units. However, now if we have two operations i.e. elementwise add \(\to \) softmax this would

      1. Load the input tensors from memory (HBM on TPU).
      2. Perform the elementwise add.
      3. Store the output tensor to memory.
      4. Load the input tensors from memory.
      5. Perform the softmax.
      6. Store the output tensor to memory.

      On a CPU, the input tensors would remain in the cache. However, this must be done explicitly on ASIC (or by the compiler in the case of Google / XLA).

      Scaling Ironwood

      • May 22, 2026
      • Kellen Kanarios

      Torus Cube:

      • 64 ironwood chips
      • Each chip connects to 6 neighbors via high-speed Inter-Chip Interconnect
      • Form a 3d torus, providing three axes for parallel bandwidth.

      Optical Circuit Switching:

      • Multiple cubes are linked by an OCS network to build pods.
      • If a cube fails, the OCS optically bypasses and swaps in a spare.
      • OCS allows for slices without system shutdown (?)
